Message Queues

Kafka, RabbitMQ, SQS/SNS, event-driven architecture, and distributed messaging patterns

1. Why Message Queues

Core Benefits

1
Decoupling - Producers and consumers evolve independently. Adding new consumers requires no producer changes.
2
Asynchronous Processing - Offload time-consuming tasks (email, image processing, reports) from the request path.
3
Load Leveling - Buffer traffic spikes. Queue absorbs bursts, consumers process at steady rate.
4
Resilience - If a consumer crashes, messages persist in the queue. Processing resumes when consumer recovers.

2. Messaging Patterns

Point-to-Point (Queue) vs Pub/Sub (Topic)
  POINT-TO-POINT (Queue):
  Each message consumed by exactly ONE consumer

  Producer ──> [ msg3 | msg2 | msg1 ] ──> Consumer A  (gets msg1)
                                     ──> Consumer B  (gets msg2)
                                     ──> Consumer C  (gets msg3)
  (Competing consumers - work distribution)


  PUB/SUB (Topic):
  Each message delivered to ALL subscribers

                              ┌──> Subscriber A (gets ALL messages)
  Publisher ──> [ Topic ] ────┼──> Subscriber B (gets ALL messages)
                              └──> Subscriber C (gets ALL messages)
  (Broadcast - event notification)
Fan-Out Pattern
                                  ┌──> Queue A ──> Order Service
                                  │
  Order Event ──> SNS Topic ──────┼──> Queue B ──> Email Service
                                  │
                                  └──> Queue C ──> Analytics Service

  Each service gets its own copy of the message.
  Services process independently at their own pace.
  Adding a new service = subscribe new queue to topic.

Competing Consumers

  • Multiple consumers read from same queue
  • Each message processed by exactly one consumer
  • Horizontal scaling: add more consumers for throughput
  • Load balancing happens naturally
  • Message ordering NOT guaranteed across consumers

Dead Letter Queue (DLQ)

  • Messages that fail processing after max retries
  • Prevents poison messages from blocking the queue
  • DLQ should be monitored and alerted on
  • Manual inspection and replay of failed messages
  • Configure: maxReceiveCount (SQS), x-death (RabbitMQ)

Exactly-Once Semantics (EOS)

Delivery Guarantees

1
At-Most-Once - Fire and forget. Message may be lost. Producer doesn't retry. acks=0
2
At-Least-Once - Producer retries on failure. Message may be delivered multiple times. acks=all + retries. Consumer must be idempotent.
3
Exactly-Once - Each message processed exactly once. Requires idempotent producer (enable.idempotence=true) + transactional API + consumer read_committed isolation.
kafka-producer.py Python 
from confluent_kafka import Producer

conf = {
    'bootstrap.servers': 'kafka-1:9092,kafka-2:9092,kafka-3:9092',
    'acks': 'all',                    # Wait for all ISR to acknowledge
    'enable.idempotence': True,       # Exactly-once producer
    'max.in.flight.requests.per.connection': 5,
    'retries': 2147483647,            # Infinite retries
    'compression.type': 'lz4',        # Compress for throughput
    'linger.ms': 5,                   # Batch for 5ms
    'batch.size': 65536,              # 64KB batches
}

producer = Producer(conf)

def delivery_report(err, msg):
    if err:
        print(f'Delivery failed: {err}')
    else:
        print(f'Delivered to {msg.topic()}[{msg.partition()}] @ offset {msg.offset()}')

# Produce with key for partition assignment
producer.produce(
    topic='orders',
    key=str(order.user_id).encode(),   # Same user always same partition
    value=json.dumps(order).encode(),
    callback=delivery_report
)
producer.flush()

4. RabbitMQ

RabbitMQ Architecture
  Producer                   RabbitMQ Broker                    Consumer
  ┌──────┐    ┌──────────────────────────────────────┐    ┌──────────┐
  │      │    │                                      │    │          │
  │ App  │───>│  Exchange ─── Binding ─── Queue  ───>│───>│  Worker  │
  │      │    │     │          (routing key)    │    │    │          │
  └──────┘    │     │                           │    │    └──────────┘
              │     │    ┌─── Queue A ──────────>│────────> Worker A
              │     │    │    (order.created)    │    │
              │  Exchange├─── Queue B ──────────>│────────> Worker B
              │  (topic) │    (order.*)          │    │
              │     │    │                       │    │
              │     │    └─── Queue C ──────────>│────────> Worker C
              │     │         (*.created)        │    │
              └─────┼────────────────────────────┘    │
                    │                                  │
              ┌─────▼─────┐                           │
              │ Dead Letter│  (rejected/expired msgs) │
              │   Exchange │                          │
              └───────────┘                           │

  Message Flow:
  1. Producer publishes to Exchange with routing key
  2. Exchange routes to Queue(s) based on binding rules
  3. Consumer pulls/is pushed messages from Queue
  4. Consumer ACKs (or NACKs/rejects) message

Exchange Types

Type Routing Logic Use Case Example
Direct Exact routing key match Task distribution Key "pdf" goes to PDF queue only
Topic Pattern matching (* = one word, # = zero+) Selective routing "order.*.created" matches "order.us.created"
Fanout All bound queues (ignores routing key) Broadcast Send to all consumers regardless of key
Headers Message header attributes Complex routing Route by content-type, priority, etc.

Acknowledgments

  • Manual ACK: Consumer explicitly acknowledges after processing. Message removed from queue.
  • Auto ACK: Message removed on delivery. Risk of data loss if consumer crashes.
  • NACK: Negative acknowledgment. Message requeued or sent to DLX.
  • Reject: Single message rejection. Can requeue or discard.

Best practice: Always use manual ACK in production. ACK only after successful processing and persistence.

Prefetch (QoS)

  • prefetch_count = how many unacknowledged messages a consumer can hold
  • prefetch=1: Fair dispatch, slow throughput (round-robin one at a time)
  • prefetch=10-50: Good balance for most workloads
  • prefetch=0 (unlimited): Max throughput, risk of memory issues
  • Lower prefetch = fairer distribution among consumers
  • Higher prefetch = better throughput, uneven work
rabbitmq-consumer.py Python 
import pika

connection = pika.BlockingConnection(
    pika.ConnectionParameters(
        host='rabbitmq',
        heartbeat=600,
        blocked_connection_timeout=300,
        credentials=pika.PlainCredentials('user', 'pass')
    )
)
channel = connection.channel()

# Declare queue with DLX
channel.queue_declare(
    queue='orders',
    durable=True,
    arguments={
        'x-dead-letter-exchange': 'dlx',
        'x-dead-letter-routing-key': 'orders.failed',
        'x-message-ttl': 86400000,  # 24h TTL
        'x-max-length': 100000,     # Max queue depth
    }
)

# Set prefetch
channel.basic_qos(prefetch_count=10)

def callback(ch, method, properties, body):
    try:
        order = json.loads(body)
        process_order(order)
        ch.basic_ack(delivery_tag=method.delivery_tag)
    except Exception as e:
        # Reject and send to DLQ (requeue=False)
        ch.basic_nack(delivery_tag=method.delivery_tag, requeue=False)

channel.basic_consume(queue='orders', on_message_callback=callback)
channel.start_consuming()

  Benefits:
  - Each service processes independently
  - Each queue has its own DLQ
  - Services scale independently
  - Adding new subscriber = new queue + subscription

Key SQS Concepts

1
Visibility Timeout - After a consumer receives a message, it's "invisible" for this duration (default 30s). If not deleted (ACKed), it reappears for other consumers. Set to ~6x your processing time.
2
Long Polling - WaitTimeSeconds=20 reduces empty responses and API calls. Consumer waits up to 20s for messages to arrive.
3
Dead Letter Queue - Configure maxReceiveCount. After N failed receives, message moves to DLQ. Monitor DLQ depth for alerts.
4
Message Retention - 1 min to 14 days (default 4 days). After retention period, messages are permanently deleted.
sqs-consumer.py AWS SDK 
import boto3
import json

sqs = boto3.client('sqs', region_name='us-east-1')
queue_url = 'https://sqs.us-east-1.amazonaws.com/123456789/orders'

while True:
    response = sqs.receive_message(
        QueueUrl=queue_url,
        MaxNumberOfMessages=10,      # Batch up to 10
        WaitTimeSeconds=20,          # Long polling
        VisibilityTimeout=60,        # 60s to process
        MessageAttributeNames=['All']
    )

    messages = response.get('Messages', [])

    for msg in messages:
        try:
            body = json.loads(msg['Body'])
            process_order(body)

            # Delete (ACK) on success
            sqs.delete_message(
                QueueUrl=queue_url,
                ReceiptHandle=msg['ReceiptHandle']
            )
        except Exception as e:
            # Message will reappear after visibility timeout
            # After maxReceiveCount failures -> DLQ
            print(f"Error processing: {e}")

6. Kafka vs RabbitMQ vs SQS

Feature Kafka RabbitMQ AWS SQS
Throughput Millions msg/s Tens of thousands msg/s Unlimited (standard)
Ordering Per partition Per queue FIFO: per message group
Retention Configurable (days/weeks/forever) Until consumed + ACKed Max 14 days
Replay Yes (seek to any offset) No (consumed = gone) No
Protocol Custom TCP (binary) AMQP 0.9.1, MQTT, STOMP HTTPS (REST API)
Routing Topic + partition key Exchanges (complex routing) Queue-level only
Operations Complex (brokers, ZK/KRaft) Moderate Zero (serverless)
Best For Event streaming, log aggregation, high throughput Task queues, complex routing, RPC Simple queuing, serverless, AWS integration
Consumer Model Pull (consumers poll) Push (broker pushes to consumers) Pull (long polling)

When to Choose What

  • Kafka: Event sourcing, stream processing, audit logs, high-throughput data pipelines, replay capability needed
  • RabbitMQ: Complex routing requirements, priority queues, request-reply pattern, multiple protocols needed
  • SQS: Simple decoupling, serverless architectures, minimal ops overhead, AWS-native stack

7. Event-Driven Architecture

Events vs Commands

Aspect Event Command
Intent "Something happened" "Do something"
Naming Past tense: OrderCreated Imperative: CreateOrder
Coupling Publisher unaware of consumers Sender knows the receiver
Consumers Zero or more Exactly one handler
Failure Publisher doesn't care if no one listens Sender expects a result

Event Sourcing

  • Store events as the source of truth (not current state)
  • Current state = replay all events from beginning
  • Complete audit trail built-in
  • Can rebuild any past state (time travel)
  • Event store is append-only (immutable)
  • Snapshots optimize replay for long event streams

Example events:

  • AccountCreated { id, name, email }
  • MoneyDeposited { id, amount, balance }
  • MoneyWithdrawn { id, amount, balance }

9. Interview Q&A

Q1: When would you choose Kafka over RabbitMQ?
Answer: Choose Kafka when you need: (1) very high throughput (millions msg/s); (2) message replay capability (reprocess historical events); (3) event sourcing or stream processing; (4) long-term retention (days/weeks); (5) ordered processing within partitions. Choose RabbitMQ when: (1) complex routing is needed (topic exchanges, headers-based); (2) message priority queues; (3) request-reply (RPC) pattern; (4) lower latency per-message delivery; (5) simpler operational requirements. Key difference: Kafka is a distributed log (append-only, consumer tracks offset), while RabbitMQ is a traditional message broker (messages deleted after ACK).
Q2: How do you achieve exactly-once delivery in a distributed system?
Answer: True exactly-once delivery is impossible in distributed systems (Two Generals Problem). Instead, we achieve exactly-once processing through: (1) Idempotent producers (Kafka: enable.idempotence=true prevents duplicates on retries); (2) Transactional outbox pattern - write to DB and outbox table in same transaction, then publish from outbox; (3) Idempotent consumers - track processed message IDs, skip duplicates; (4) Deduplication - SQS FIFO dedup ID, Kafka exactly-once semantics with transactions. The practical approach: at-least-once delivery + idempotent consumers = effective exactly-once processing.
Q3: How do you decide on the number of Kafka partitions?
Answer: Partitions determine parallelism - max consumers in a group equals number of partitions. Factors: (1) Throughput: Target throughput / throughput per partition (typically 10-50 MB/s per partition); (2) Consumer parallelism: At least as many partitions as max expected consumers; (3) Ordering: Messages with same key go to same partition - choose a key with good cardinality; (4) Over-partitioning risks: More metadata overhead, longer leader elections, more file handles. Rule of thumb: start with 6-12 partitions per topic for most workloads, scale up if throughput demands. You can increase partitions later but CANNOT decrease. Rebalancing doesn't redistribute existing data.
Q4: Explain consumer groups in Kafka.
Answer: A consumer group is a set of consumers that cooperatively consume a topic. Each partition is assigned to exactly one consumer within a group. If consumers > partitions, some consumers sit idle. If consumers < partitions, some consumers handle multiple partitions. Rebalancing occurs when consumers join/leave the group - partitions are reassigned. Multiple consumer groups can independently consume the same topic (each gets all messages). Group coordinator (a broker) manages membership and offsets. Consumer commits offset to __consumer_offsets topic to track progress. If a consumer fails, its partitions are reassigned to remaining consumers.
Q5: What is the purpose of a Dead Letter Queue?
Answer: A DLQ captures messages that cannot be successfully processed after a configured number of retries. Purpose: (1) Prevent poison messages from blocking the queue (one bad message would block all processing); (2) Preserve failed messages for debugging and manual replay; (3) Alerting - DLQ depth is a key operational metric. Implementation: SQS maxReceiveCount=5, RabbitMQ dead-letter-exchange, Kafka error topic. Best practices: alert on DLQ growth, implement tooling to inspect and replay DLQ messages, include original error context in DLQ metadata, set appropriate DLQ retention (longer than main queue).
Q6: How does event sourcing differ from traditional CRUD?
Answer: CRUD: Store current state. UPDATE overwrites previous state. History is lost. Event Sourcing: Store sequence of events (immutable facts). Current state is derived by replaying events. Benefits: (1) Complete audit trail; (2) Time travel - reconstruct any past state; (3) Event replay for new projections; (4) Natural fit for event-driven microservices. Challenges: (1) Event schema evolution (versioning); (2) Eventual consistency (read model lags write); (3) Complexity of rebuilding state from long event streams (use snapshots); (4) Storage grows unbounded (compaction strategies). Example: Bank account - instead of storing balance=500, store events: Deposited(200), Deposited(400), Withdrawn(100).
Q7: Explain the saga pattern for distributed transactions.
Answer: A saga is a sequence of local transactions where each step publishes an event/command triggering the next. If a step fails, compensating transactions undo completed steps. Two styles: (1) Choreography: Each service listens for events and reacts. Decentralized, simple for few steps, but hard to track overall progress. (2) Orchestration: Central coordinator directs steps. Easier to understand and monitor, but adds a single point that must be resilient. Example: Order saga - CreateOrder -> ReserveInventory -> ProcessPayment -> ShipOrder. If payment fails: compensate by releasing inventory and canceling order. Key rules: each step must be idempotent, each step must have a compensating action, consider timeout/deadline for the overall saga.
Q8: How do you make consumers idempotent?
Answer: Idempotent consumers produce the same result regardless of how many times a message is processed. Techniques: (1) Deduplication table: Store processed message IDs in DB, check before processing (use unique constraint); (2) Natural idempotency: Design operations that are inherently idempotent (SET balance=500 vs ADD 100); (3) Optimistic locking: Include version/timestamp in message, reject if already applied; (4) Idempotency key: Client-generated unique ID per logical operation; (5) Upsert operations: INSERT ON CONFLICT UPDATE instead of blind INSERT. Always test by replaying the same message twice and verifying no side effects.

10. Troubleshooting Scenarios

! Scenario: Consumer Lag Growing

Symptom: Kafka consumer lag increasing steadily. Messages produced faster than consumed. Alerts on kafka_consumer_lag metric. Processing delay growing.

1. Measure: Check lag per partition: kafka-consumer-groups.sh --describe --group my-group --bootstrap-server kafka:9092. Identify which partitions are lagging.
2. Check consumer health: Are consumers crashing/restarting? Look for frequent rebalances (consumer logs). Check if processing time per message has increased.
3. Immediate mitigation: Scale consumers (add instances). If consumers = partitions already, you cannot add more consumers. Increase partition count (requires topic reconfiguration).
4. Optimize processing: Batch processing (consume multiple messages, process together). Async I/O in consumer. Check for slow downstream calls. Increase max.poll.records.
5. Long-term: Right-size partitions. Optimize consumer code. Consider parallel processing within each consumer (thread pool). Add back-pressure mechanism to producers if lag exceeds threshold.
! Scenario: Message Ordering Broken

Symptom: Events for the same entity arriving out of order. User sees "order shipped" before "order confirmed". State machine violations.

1. Identify scope: Is ordering broken globally or for specific entities? Check if producer is using a partition key. Messages with same key should go to same partition.
2. Common causes: (a) Producer not setting message key (round-robin partitioning); (b) Producer retries with max.in.flight.requests > 1 and enable.idempotence=false; (c) Multiple producers writing same entity without coordination; (d) Consumer processing out of order (parallel threads).
3. Fix producer: Set message key to entity ID (order_id). Enable idempotence. For SQS FIFO, use MessageGroupId.
4. Fix consumer: Process messages from each partition sequentially. If using threads, route same key to same thread. Implement sequence number validation.
5. Design fix: Include sequence number or timestamp in messages. Consumer reorders within a window. Accept eventual consistency for non-critical paths.
! Scenario: DLQ Filling Up

Symptom: Dead Letter Queue depth increasing. Alerts firing on DLQ message count. Some messages being permanently lost after DLQ retention expires.

1. Inspect messages: Sample messages from DLQ. Check error attributes/headers for failure reason. Look for patterns (same error type, same producer, specific time range).
2. Classify failures: (a) Poison messages: Malformed data, schema mismatch - fix producer or add validation; (b) Transient failures: Downstream timeout, DB unavailable - increase retries/backoff; (c) Bug in consumer: Exception on valid data - fix consumer code, replay messages.
3. Fix root cause: Deploy fix for consumer bug. Fix schema validation. Add retry with exponential backoff for transient errors. Increase visibility timeout if processing takes longer than expected.
4. Replay DLQ: After fix is deployed, replay DLQ messages back to main queue. SQS: use DLQ redrive. Kafka: consumer tool to read from error topic and re-produce. Verify successful processing.
5. Prevention: Add schema validation at producer. Implement circuit breaker for downstream calls. Alert on DLQ arrival rate (not just depth). Regular DLQ review process.
! Scenario: Kafka Broker Failure

Symptom: One of three Kafka brokers goes down. Under-replicated partitions detected. Some producers getting errors. Consumer rebalancing triggered.

1. Assess impact: Check under-replicated partitions: kafka-topics.sh --describe --under-replicated-partitions. If min.insync.replicas=2 and RF=3, one broker down is survivable.
2. Verify availability: Producers with acks=all may get errors if ISR drops below min.insync.replicas. Check ISR count for affected partitions. Leader election should auto-promote an ISR follower.
3. Consumers: Group rebalancing will reassign partitions from dead broker's consumers. Expect brief pause in consumption. Check consumer group state: kafka-consumer-groups.sh --describe.
4. Recovery: If broker restarts, it will rejoin cluster and start catching up (fetching missed data from leaders). Monitor replication catch-up. If disk failed, broker needs fresh data (full replication from scratch).
5. Post-mortem: Verify RF >= 3 for all critical topics. Ensure brokers are in different failure domains (AZs in K8s: pod anti-affinity). Test broker failure regularly. Monitor ISR shrink rate as early warning.
