Databases

SQL, NoSQL, replication, sharding, caching, migrations, and production database patterns

1. SQL vs NoSQL

Comparison Overview

Aspect SQL (Relational) NoSQL
Structure Tables with rows and columns Documents, key-value, columns, graphs
Schema Fixed schema, predefined Dynamic/flexible schema
Scaling Vertical (scale up), limited horizontal Horizontal (scale out) by design
ACID Full ACID compliance Eventual consistency (some offer ACID)
Joins Complex joins supported Denormalized; joins are expensive/unsupported
Query Language SQL (standardized) Varies (MongoDB Query, CQL, etc.)
Use Cases Financial, ERP, complex queries, OLTP Real-time analytics, IoT, content management
Examples PostgreSQL, MySQL, Oracle MongoDB, Redis, Cassandra, DynamoDB

CAP Theorem

CAP Theorem - Pick Two (in presence of partition)
                        Consistency (C)
                            /\
                           /  \
                          /    \
                         / CA   \
                        /________\
                       /   CP     \
                      / PostgreSQL  \
                     /   MongoDB     \
                    /   (primary)     \
                   /                   \
                  /                     \
 Availability   /_________AP____________\ Partition
    (A)              Cassandra            Tolerance (P)
                     DynamoDB
                     CouchDB

  +---------------------------------------------------------+
  | In distributed systems, network partitions WILL happen. |
  | So the real choice is: CP or AP                         |
  |                                                         |
  | CP = Consistent + Partition Tolerant                    |
  |      (blocks writes during partition)                   |
  |      Examples: PostgreSQL, MongoDB, HBase, Redis        |
  |                                                         |
  | AP = Available + Partition Tolerant                     |
  |      (allows writes, resolves conflicts later)          |
  |      Examples: Cassandra, DynamoDB, CouchDB             |
  +---------------------------------------------------------+

2. Relational Databases

ACID Properties

A
Atomicity - Transaction is all-or-nothing. If any part fails, the entire transaction rolls back.
C
Consistency - Database moves from one valid state to another. All constraints, triggers, and rules are satisfied.
I
Isolation - Concurrent transactions don't interfere. Levels: Read Uncommitted, Read Committed, Repeatable Read, Serializable.
D
Durability - Once committed, data survives crashes. Written to WAL (Write-Ahead Log) before confirming.

Indexes

B-Tree Index (Default)

  • Balanced tree structure, O(log n) lookups
  • Best for range queries and equality
  • Supports ORDER BY efficiently
  • Works with: <, >, =, BETWEEN, LIKE 'prefix%'
CREATE INDEX idx_users_email ON users(email);

Hash Index

  • O(1) for equality lookups only
  • Cannot do range queries
  • Smaller than B-tree for simple equality
  • Use case: exact-match lookups (sessions, cache keys)
CREATE INDEX idx_session ON sessions USING hash(session_id);

Other Index Types

Type Use Case Example
Composite Multi-column queries CREATE INDEX ON orders(user_id, created_at)
Partial Index subset of rows CREATE INDEX ON orders(total) WHERE status='active'
GIN Full-text search, JSONB, arrays CREATE INDEX ON docs USING gin(content)
GiST Geometric/spatial data CREATE INDEX ON geo USING gist(location)

Query Optimization

EXPLAIN ANALYZE example PostgreSQL 
-- Identify slow queries
EXPLAIN ANALYZE SELECT * FROM orders WHERE user_id = 123 AND status = 'pending';

-- Key things to look for in output:
-- Seq Scan      = Full table scan (bad for large tables)
-- Index Scan    = Using index (good)
-- Bitmap Scan   = Multiple index conditions (good)
-- Nested Loop   = OK for small result sets
-- Hash Join     = Better for larger joins
-- Sort          = May need index for ORDER BY

-- Common optimizations:
-- 1. Add missing indexes for WHERE/JOIN columns
-- 2. Use composite indexes matching query patterns
-- 3. Avoid SELECT * (fetch only needed columns)
-- 4. Use LIMIT for pagination (with keyset pagination)
-- 5. Rewrite NOT IN as NOT EXISTS (avoids null issues)

3. NoSQL Types

Type Examples Data Model Best For Limitations
Document MongoDB, CouchDB JSON/BSON documents Content management, catalogs, user profiles Complex transactions, deep joins
Key-Value Redis, DynamoDB, Memcached Simple key-value pairs Caching, sessions, leaderboards, queues No complex queries, limited filtering
Column-Family Cassandra, HBase, ScyllaDB Wide columns, row keys Time-series, IoT, write-heavy workloads Limited ad-hoc queries, no joins
Graph Neo4j, Amazon Neptune Nodes + Edges + Properties Social networks, fraud detection, recommendations Not ideal for bulk analytics

MongoDB (Document)

  • Flexible JSON-like documents (BSON)
  • Rich query language with aggregation pipelines
  • Replica sets for HA, sharding for horizontal scale
  • Multi-document ACID transactions (since 4.0)
  • Change streams for real-time event processing

Redis (Key-Value)

  • In-memory, sub-millisecond latency
  • Rich data structures: strings, hashes, lists, sets, sorted sets, streams
  • Pub/Sub, Lua scripting, transactions (MULTI/EXEC)
  • Redis Cluster for sharding, Sentinel for HA
  • Persistence: RDB snapshots + AOF (append-only file)

Cassandra (Column-Family)

  • Masterless ring architecture (no SPOF)
  • Tunable consistency (ONE, QUORUM, ALL)
  • Excellent write throughput at scale
  • Data modeling: design tables for query patterns
  • CQL (Cassandra Query Language) - SQL-like syntax

DynamoDB (Key-Value / Document)

  • Fully managed, serverless scaling
  • Single-digit millisecond at any scale
  • Partition key + optional sort key
  • GSI/LSI for alternative access patterns
  • DynamoDB Streams for change data capture
  • On-demand or provisioned capacity modes

5. Sharding

Horizontal Partitioning

Sharding splits data across multiple database instances (shards). Each shard holds a subset of the data, allowing horizontal scaling of both storage and throughput.

Sharding Architecture
  Application
       │
       ▼
  ┌──────────────┐
  │  Router /    │    Determines which shard based on shard key
  │  Proxy       │    (e.g., user_id % num_shards)
  └──────┬───────┘
         │
    ┌────┼────┬────────────┐
    ▼    ▼    ▼            ▼
  ┌────┐┌────┐┌────┐   ┌────┐
  │ S1 ││ S2 ││ S3 │...│ SN │
  │    ││    ││    │   │    │
  │A-F ││G-M ││N-S │   │T-Z │
  └────┘└────┘└────┘   └────┘

  Shard Key Selection Criteria:
  ┌─────────────────────────────────────────────┐
  │ 1. High cardinality (many unique values)    │
  │ 2. Even distribution (avoid hot spots)      │
  │ 3. Query isolation (queries hit 1 shard)    │
  │ 4. Growth pattern (monotonic = bad)         │
  └─────────────────────────────────────────────┘
Consistent Hashing
  Traditional Hashing: hash(key) % N
  Problem: Adding/removing nodes reshuffles ALL keys

  Consistent Hashing Ring:
                    Node A
                   /      \
                  /   A's   \
                 /   range   \
        Node D  |             |  Node B
                 \   C's     /
                  \  range  /
                   \      /
                    Node C

  Adding Node E:
  - Only keys between D and E move to E
  - Other nodes unaffected (~1/N keys move)

  Virtual Nodes:
  - Each physical node gets multiple positions on ring
  - Improves distribution evenness
  - Node A: A1, A2, A3 (3 virtual nodes)

Sharding Pros vs Cons

Pros Cons
Horizontal scalability (near-linear) Cross-shard queries are expensive
Isolation: one shard failure doesn't affect others Distributed transactions are complex
Each shard can be independently tuned Rebalancing shards is operationally difficult
Geo-locality (shard per region) Application complexity increases significantly
Smaller indexes per shard (faster queries) Schema changes must apply to all shards

6. Backup Strategies

Type Description Restore Time Storage Frequency
Full Complete copy of entire database Fastest restore Highest Weekly
Incremental Changes since last backup (any type) Slowest (chain) Lowest Hourly/Daily
Differential Changes since last full backup Moderate Moderate Daily

PostgreSQL Backup Tools

pg_dump -Fc mydb > mydb_$(date +%Y%m%d).dump
pg_dumpall > all_databases.sql
pg_restore -d mydb mydb_20240101.dump

Point-in-Time Recovery (PITR):

  • Enable WAL archiving: archive_mode = on
  • Take base backup with pg_basebackup
  • Replay WAL to desired timestamp
  • Recovery target: recovery_target_time

RDS Backup Features

  • Automated backups: Daily full snapshot + transaction logs every 5 min
  • Retention: 1-35 days (default 7)
  • PITR: Restore to any second within retention
  • Manual snapshots: Retained until deleted
  • Cross-region: Copy snapshots to other regions
  • Backup window: Preferred time for daily backup
aws rds create-db-snapshot --db-instance-identifier mydb --db-snapshot-identifier pre-migration
backup-cron.sh Script 
#!/bin/bash
# PostgreSQL backup with retention
DB_NAME="production"
BACKUP_DIR="/backups/postgres"
RETENTION_DAYS=30
TIMESTAMP=$(date +%Y%m%d_%H%M%S)

# Full backup with compression
pg_dump -Fc -Z9 "$DB_NAME" > "$BACKUP_DIR/${DB_NAME}_${TIMESTAMP}.dump"

# Upload to S3
aws s3 cp "$BACKUP_DIR/${DB_NAME}_${TIMESTAMP}.dump" \
    "s3://my-backups/postgres/${DB_NAME}_${TIMESTAMP}.dump" \
    --storage-class STANDARD_IA

# Clean old local backups
find "$BACKUP_DIR" -name "*.dump" -mtime +$RETENTION_DAYS -delete

# Verify backup integrity
pg_restore --list "$BACKUP_DIR/${DB_NAME}_${TIMESTAMP}.dump" > /dev/null 2>&1
if [ $? -ne 0 ]; then
    echo "ALERT: Backup verification failed!" | mail -s "Backup Failure" ops@company.com
fi

8. Migrations

Zero-Downtime Migrations (Expand-Contract Pattern)

Expand-Contract Pattern for Column Rename
  Phase 1: EXPAND (add new column)
  ┌────────────────────────────────────────────────┐
  │  ALTER TABLE users ADD COLUMN full_name TEXT;   │
  │  -- Both old (name) and new (full_name) exist  │
  │  -- App writes to BOTH columns                 │
  └────────────────────────────────────────────────┘
              │
              ▼
  Phase 2: MIGRATE (backfill data)
  ┌────────────────────────────────────────────────┐
  │  UPDATE users SET full_name = name             │
  │  WHERE full_name IS NULL                       │
  │  -- Do in batches to avoid long locks!         │
  │  -- App reads from new column                  │
  └────────────────────────────────────────────────┘
              │
              ▼
  Phase 3: CONTRACT (remove old column)
  ┌────────────────────────────────────────────────┐
  │  -- Deploy app version that only uses new col  │
  │  ALTER TABLE users DROP COLUMN name;           │
  │  -- Only after all app instances updated       │
  └────────────────────────────────────────────────┘

  Timeline:  Deploy v2 ────> Backfill ────> Deploy v3 ────> Drop
             (writes both)   (copy data)    (reads new)   (cleanup)

Migration Tools

Tool Language Format Key Features
Flyway Java/JVM SQL files (V1__name.sql) Simple, convention-based, repeatable migrations
Alembic Python Python scripts Auto-generate from SQLAlchemy models, branching
Liquibase Java/JVM XML/YAML/JSON/SQL Rollback support, diff, multi-DB
golang-migrate Go SQL files (up/down) CLI + library, multiple DB drivers
V003__add_email_index.sql Flyway 
-- Safe migration: CREATE INDEX CONCURRENTLY (no lock)
-- Note: Cannot run inside a transaction in PostgreSQL
CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_users_email
    ON users (email);

-- Unsafe alternative (holds AccessExclusiveLock):
-- CREATE INDEX idx_users_email ON users (email);
-- This blocks ALL reads and writes!

Rollback Strategies

  • Forward-only: Never roll back, fix forward with new migration. Safest for production.
  • Down migrations: Each migration has up/down. Risk: down may lose data.
  • Snapshots: Take DB snapshot before migration. Roll back by restoring snapshot.
  • Feature flags: Decouple migration from feature. Disable feature without rolling back schema.

9. Caching

Cache-Aside Pattern (Lazy Loading)
  ┌──────────┐         ┌───────────┐         ┌──────────┐
  │          │── 1. ──>│           │         │          │
  │   App    │  Check  │   Cache   │         │ Database │
  │          │<── 2. ──│  (Redis)  │         │          │
  │          │  Miss!  │           │         │          │
  │          │─── 3. Read ──────────────────>│          │
  │          │<── 4. Data ──────────────────-│          │
  │          │── 5. ──>│           │         │          │
  │          │  Store  │           │         │          │
  └──────────┘         └───────────┘         └──────────┘

  Cache Hit:   App -> Cache -> Return (fast, ~1ms)
  Cache Miss:  App -> Cache -> DB -> Store in Cache -> Return

  Pros: Only requested data is cached (no waste)
  Cons: Cache miss = 3 round trips; stale data possible
Write-Through vs Write-Behind
  WRITE-THROUGH:
  App ──> Cache ──> Database    (synchronous, consistent but slower writes)

  WRITE-BEHIND (Write-Back):
  App ──> Cache ──(async)──> Database   (fast writes, risk of data loss)

  WRITE-AROUND:
  App ──> Database              (cache not updated until read miss)
         ╳ Cache                (avoids cache churn for write-heavy data)

Redis Caching Example

cache.py Python 
import redis
import json

r = redis.Redis(host='redis', port=6379, db=0)

def get_user(user_id):
    # Check cache first
    cache_key = f"user:{user_id}"
    cached = r.get(cache_key)

    if cached:
        return json.loads(cached)

    # Cache miss - query database
    user = db.query(
        "SELECT * FROM users WHERE id = %s",
        user_id
    )

    # Store in cache with TTL
    r.setex(
        cache_key,
        timedelta(minutes=15),
        json.dumps(user)
    )
    return user

def update_user(user_id, data):
    db.execute("UPDATE users SET ...", data)
    # Invalidate cache
    r.delete(f"user:{user_id}")

Cache Invalidation Strategies

1
TTL (Time-To-Live) - Set expiry; data refreshes on next miss. Simple but allows staleness.
2
Event-Driven Invalidation - Publish event on write; subscribers invalidate. Low staleness.
3
Version Keys - Append version to cache key. Increment version on update. Old entries expire naturally.
4
Cache Stampede Prevention - Use locking (SETNX) or probabilistic early expiration to prevent thundering herd.

Redis vs Memcached

Feature Redis Memcached
Data Structures Strings, hashes, lists, sets, sorted sets, streams Strings only
Persistence RDB + AOF None (pure cache)
Replication Primary-Replica, Cluster None (client-side sharding)
Max Value Size 512MB 1MB
Multithreading Single-threaded (I/O threads in 6.0+) Multi-threaded
Use Case Caching + data store + pub/sub Simple, high-throughput caching

10. Interview Q&A

Q1: How do you choose between SQL and NoSQL for a new project?
Answer: Consider these factors: (1) Data relationships - complex relationships with joins favor SQL; (2) Schema stability - evolving schemas favor NoSQL; (3) Consistency requirements - financial data needing ACID favors SQL; (4) Scale pattern - massive write throughput or geographic distribution favors NoSQL; (5) Query patterns - ad-hoc complex queries favor SQL, known access patterns favor NoSQL (especially DynamoDB). Many systems use both: SQL for transactional data, NoSQL for caching/sessions/analytics.
Q2: Explain the CAP theorem with a real-world example.
Answer: CAP states a distributed system can guarantee at most 2 of: Consistency, Availability, Partition tolerance. Since network partitions are inevitable, the real choice is CP vs AP. CP example: A banking system (PostgreSQL with sync replication) - during a network partition, it refuses writes rather than risk inconsistency. AP example: A shopping cart (DynamoDB) - during a partition, both sides accept writes and resolve conflicts later (last-writer-wins or merge). The system stays available but may show stale data temporarily.
Q3: What are the differences between synchronous and asynchronous replication?
Answer: Synchronous: Primary waits for at least one replica to confirm the write before acknowledging to the client. Guarantees zero data loss but adds latency (network round-trip to replica). Used for: financial systems, RDS Multi-AZ. Asynchronous: Primary acknowledges immediately and streams changes in background. Lower write latency but risks data loss if primary fails before replica catches up (replication lag). Used for: read replicas, cross-region replication. Semi-synchronous: Middle ground - waits for one replica, others are async. PostgreSQL supports this with synchronous_commit settings.
Q4: What are the trade-offs of database sharding?
Answer: Benefits: Horizontal scalability, reduced index sizes, isolation of failures, ability to scale writes (not just reads). Costs: (1) Cross-shard queries require scatter-gather (slow); (2) Distributed transactions need 2PC or saga patterns; (3) Resharding is extremely disruptive; (4) Application must be shard-aware; (5) Operational complexity multiplies. When to shard: Only after exhausting vertical scaling, read replicas, and caching. Start with a shard key that distributes evenly (UUID hash) and allows query isolation (most queries hit one shard). Avoid monotonic keys (timestamps) as shard keys - they create hot spots.
Q5: How do you perform a zero-downtime database migration?
Answer: Use the expand-contract pattern: (1) Expand: Add new column/table without removing old. Deploy app version that writes to both old and new. (2) Migrate: Backfill existing data in batches (avoid long locks). Verify data consistency. (3) Contract: Deploy app that reads only from new. Remove old column. Key rules: never rename columns in-place, use CREATE INDEX CONCURRENTLY, batch large updates, test migrations against production-size data, always have a rollback plan (which may be "fix forward").
Q6: Why is connection pooling important and how does it work?
Answer: Each database connection consumes ~10MB RAM (PostgreSQL) and spawns a backend process. With microservices, hundreds of instances each opening connections can exhaust the DB. Connection pooling (PgBouncer, HikariCP) maintains a pool of reusable connections. 100 app instances sharing a pool of 25 actual DB connections. PgBouncer's transaction mode releases connections after each transaction, maximizing reuse. Key settings: max_client_conn (how many apps can connect to pooler), default_pool_size (actual DB connections per database), reserve_pool_size (burst capacity).
Q7: How do you design an indexing strategy for a production database?
Answer: (1) Start with query patterns - index columns in WHERE, JOIN, and ORDER BY clauses; (2) Use composite indexes matching your most common multi-column queries (leftmost prefix rule); (3) Consider partial indexes for queries that filter on a specific condition (e.g., WHERE status = 'active'); (4) Monitor with pg_stat_user_indexes to find unused indexes (they slow writes); (5) Use EXPLAIN ANALYZE to verify index usage; (6) Avoid over-indexing - each index adds write overhead and storage. Rule of thumb: index columns with high selectivity that appear in frequent queries.
Q8: What are common Redis use cases beyond simple caching?
Answer: (1) Session store: Fast reads, TTL for expiry; (2) Rate limiting: INCR + EXPIRE for sliding window; (3) Leaderboards: Sorted sets with ZADD/ZRANGE; (4) Pub/Sub: Real-time notifications, chat; (5) Distributed locks: SETNX + TTL (Redlock algorithm); (6) Queue: Lists with LPUSH/BRPOP for simple job queues; (7) Geospatial: GEOADD/GEORADIUS for location-based queries; (8) Streams: Event log with consumer groups (like mini-Kafka); (9) Bloom filters: Probabilistic membership testing (RedisBloom module).
Q9: How does Redis persistence work and what are the trade-offs?
Answer: Two mechanisms: RDB (snapshots): Point-in-time snapshots at intervals. Fast restart, compact files. Risk: lose data between snapshots. AOF (Append-Only File): Logs every write command. Options: appendfsync always (safest, slowest), everysec (good balance, lose ~1s), no (OS decides, fastest). Best practice: use both - AOF for durability, RDB for fast restarts and backups. AOF rewrite compacts the file periodically. In Redis 7+, Multi-Part AOF improves reliability.
Q10: How would you handle a database that has grown too large for a single instance?
Answer: Progressive approach: (1) Optimize first: Better indexes, query optimization, connection pooling; (2) Read replicas: Offload reads (reporting, analytics) to replicas; (3) Caching layer: Redis for hot data, reducing DB load; (4) Vertical scaling: Bigger instance (quick but has limits); (5) Archival: Move old data to cold storage (S3, data warehouse); (6) Functional partitioning: Separate databases per domain (orders, users, inventory); (7) Sharding: Last resort - horizontal partitioning with shard key. Each step adds complexity, so only escalate when the simpler approach is insufficient.

11. Troubleshooting Scenarios

! Scenario: Database Connection Exhaustion

Symptom: Application throwing "too many connections" errors. New requests failing. pg_stat_activity shows max_connections reached.

1. Immediate: Identify and terminate idle connections: SELECT pg_terminate_backend(pid) FROM pg_stat_activity WHERE state = 'idle' AND query_start < now() - interval '10 min'
2. Investigate: Check for connection leaks: SELECT client_addr, count(*) FROM pg_stat_activity GROUP BY client_addr ORDER BY count DESC
3. Root cause: Common causes: missing connection pool, pool misconfigured (too large), connection leaks in app code (connections not returned to pool), long-running transactions holding connections.
4. Fix: Deploy PgBouncer in transaction mode. Set default_pool_size to ~25 per database. Configure app connection timeout and idle timeout. Add connection leak detection to app framework.
5. Prevention: Alert on pg_stat_activity count approaching max_connections (80% threshold). Monitor connection pool utilization metrics.
! Scenario: Replication Lag Growing

Symptom: Read replicas returning stale data. Users see inconsistent reads. pg_stat_replication shows increasing replay_lag.

1. Measure: Check lag: SELECT client_addr, state, sent_lsn, write_lsn, flush_lsn, replay_lsn, replay_lag FROM pg_stat_replication
2. Identify cause: (a) Heavy write load on primary; (b) Replica under-resourced (CPU/IO); (c) Long-running queries on replica blocking replay; (d) Network bottleneck between primary and replica.
3. Mitigate: Kill long queries on replica (hot_standby_feedback = on to prevent conflicts). Scale up replica instance. Check network throughput.
4. Application fix: Route critical reads (read-your-writes) to primary. Use session affinity for recently-written data. Set max_standby_streaming_delay appropriately.
5. Long-term: If lag is persistent, add more replicas to distribute read load, or consider synchronous replication for critical replicas.
! Scenario: Split-Brain During Failover

Symptom: After network partition heals, two nodes both believe they are primary. Conflicting writes detected. Data divergence.

1. Detection: Monitor for multiple nodes reporting as primary. Patroni/etcd leader check. Fencing alerts.
2. Immediate: Fence (STONITH) the old primary - shut it down forcefully to prevent further writes. This is why fencing mechanisms exist.
3. Assess damage: Compare WAL positions of both nodes. Identify divergent transactions using pg_rewind or WAL analysis.
4. Recovery: Use pg_rewind to rewind old primary to the fork point, then replay as replica. Or restore old primary from backup and resync. Manually reconcile conflicting writes if needed.
5. Prevention: Use consensus-based failover (Patroni + etcd with 3+ nodes). Configure fencing (STONITH). Set synchronous_commit = on to prevent unacknowledged writes from being lost.
! Scenario: Slow Queries Degrading Performance

Symptom: Application response times increasing. CPU on database at 90%+. pg_stat_statements shows queries taking seconds instead of milliseconds.

1. Identify: Find top slow queries: SELECT query, mean_exec_time, calls FROM pg_stat_statements ORDER BY total_exec_time DESC LIMIT 10
2. Analyze: Run EXPLAIN (ANALYZE, BUFFERS) on slow queries. Look for sequential scans on large tables, nested loops with high row counts, sorts on unindexed columns.
3. Quick wins: Add missing indexes (use CREATE INDEX CONCURRENTLY). Rewrite NOT IN as NOT EXISTS. Add LIMIT to unbounded queries. Update table statistics: ANALYZE tablename.
4. Medium-term: Implement query result caching for hot paths. Denormalize frequently-joined data. Consider materialized views for complex aggregations. Move analytics queries to read replicas.
5. Monitor: Set log_min_duration_statement = 1000 to log slow queries. Alert on pg_stat_statements mean_exec_time regression. Track index hit ratio: should be >99%.
