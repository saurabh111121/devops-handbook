2. IAM Deep Dive

IAM Core Concepts

  • Users: Individual identities with long-term credentials (password, access keys)
  • Groups: Collection of users. Policies attached to groups apply to all members
  • Roles: Assumable identities with temporary credentials. Used by services, applications, cross-account access
  • Policies: JSON documents defining permissions (Effect, Action, Resource, Condition)

  +-------------------+                  |

  Flow:
  1. Pod mounts projected service account token (JWT)
  2. AWS SDK in pod calls STS AssumeRoleWithWebIdentity
  3. STS validates JWT against EKS OIDC provider
  4. STS returns temporary credentials for the IAM role
  5. Pod uses credentials to access AWS services

  Trust Policy on IAM Role:
  {
    "Principal": {
      "Federated": "arn:aws:iam::ACCOUNT:oidc-provider/oidc.eks.REGION.amazonaws.com/id/CLUSTER_ID"
    },
    "Condition": {
      "StringEquals": {
        "oidc.eks.REGION.amazonaws.com/id/CLUSTER_ID:sub":
          "system:serviceaccount:NAMESPACE:SA_NAME"
      }
    }
  }

IAM Best Practices

  • Always follow least privilege - grant only permissions needed
  • Use roles instead of users for applications and services
  • Enable MFA for all human users, especially for privileged access
  • Use conditions to restrict by IP, region, time, MFA status
  • Regularly audit with IAM Access Analyzer and remove unused permissions
  • Use permission boundaries to limit maximum permissions delegatable
  • Never embed credentials in code - use IAM roles, IRSA, or Secrets Manager

Security Groups vs NACLs

Feature Security Groups Network ACLs (NACLs)
Scope Instance/ENI level Subnet level
State Stateful (return traffic auto-allowed) Stateless (must explicitly allow return traffic)
Rule Type Allow rules only (implicit deny all) Allow AND Deny rules (processed in order)
Evaluation All rules evaluated (most permissive wins) Rules processed by number (first match wins)
Reference Can reference other SGs as source/destination CIDR-based rules only
Default Deny all inbound, allow all outbound Allow all inbound and outbound
Best For Primary firewall, micro-segmentation Subnet guardrails, blocking known bad CIDRs

VPC Peering vs Transit Gateway

VPC Peering

  • 1:1 connection between two VPCs
  • Non-transitive (A-B and B-C does NOT give A-C)
  • Works cross-region and cross-account
  • No single point of failure, no bandwidth bottleneck
  • Scales poorly: N VPCs need N*(N-1)/2 peerings
  • Best for: few VPCs, simple architectures

Transit Gateway

  • Hub-and-spoke model: all VPCs connect to TGW
  • Transitive routing (A-TGW-C works)
  • Route tables for traffic segmentation
  • Supports VPN, Direct Connect attachments
  • Scales well: add VPCs without new connections
  • Best for: many VPCs, complex network topologies

EKS Networking (VPC CNI)

AWS VPC CNI Plugin

  • How it works: Each pod gets a real VPC IP address (from the subnet CIDR)
  • ENI allocation: Each node has multiple ENIs, each ENI has multiple secondary IPs
  • Max pods per node: Limited by instance type ENI count and IPs per ENI
  • Benefit: Pods are first-class VPC citizens - SGs, NACLs, route tables all apply directly
  • Prefix delegation: Assign /28 prefixes (16 IPs) per ENI slot for more pod density
  • Custom networking: Pod IPs from different subnets than node IPs (for IP exhaustion)

S3 Key Features

  • Versioning: Keep all versions of objects. Protects against accidental deletes (delete marker instead of actual deletion)
  • Replication: CRR (Cross-Region) for DR, SRR (Same-Region) for compliance/aggregation. Requires versioning enabled
  • Encryption: SSE-S3 (AWS managed), SSE-KMS (customer KMS key), SSE-C (customer-provided key)
  • Consistency: Strong read-after-write consistency for all operations (since Dec 2020)
  • Event Notifications: Trigger Lambda, SQS, SNS on object creation/deletion

Health Checks

Route 53 Health Check Types

  • Endpoint: Monitor an IP/domain - HTTP, HTTPS, or TCP. Configurable interval (10s or 30s), failure threshold
  • Calculated: Combine multiple health checks (AND/OR logic). "Healthy if 2 of 3 children are healthy"
  • CloudWatch Alarm: Based on CloudWatch alarm state. Useful for monitoring internal resources

Health checkers run from multiple AWS regions. Resource considered unhealthy if >18% of checkers report unhealthy.

7. CloudWatch

CloudWatch Components

  • Metrics: Time-series data points (CPU, memory, custom). Namespace/dimension/metric-name. Standard (5min) or detailed (1min) monitoring
  • Logs: Log groups > log streams. Filter patterns, Logs Insights for querying. Retention configurable
  • Alarms: Trigger actions based on metric thresholds. States: OK, ALARM, INSUFFICIENT_DATA. Actions: SNS, Auto Scaling, EC2
  • Dashboards: Custom visualization of metrics and logs. Cross-account, cross-region supported

CloudWatch Agent

  • Collects system-level metrics (memory, disk, processes)
  • Collects custom application logs
  • Runs on EC2, on-premises, or ECS/EKS
  • Configured via JSON config file
  • StatsD and collectd protocol support

CloudWatch vs Prometheus

Feature CloudWatch Prometheus
Type Managed AWS service (push-based) Open-source (pull-based scraping)
Hosting Fully managed, no infra to maintain Self-hosted or AMP (AWS Managed Prometheus)
Query Language Metric Math, Logs Insights PromQL (more powerful)
Resolution 1 second (high-res) to 5 minutes Configurable (typically 15s-60s)
Retention 15 months (auto aggregation) Configurable (Thanos/Cortex for long-term)
Alerting CloudWatch Alarms + SNS Alertmanager (more flexible routing)
Cost Per metric, per API call, per GB logs Infrastructure cost (or AMP per-sample pricing)
Best For AWS-native monitoring, simple setup Kubernetes-native, high-cardinality metrics

Cold Starts

Understanding Cold Starts

  • What: First invocation (or scale-up) requires downloading code, starting runtime, initializing handler. Adds 100ms-10s latency
  • When: First request, after idle timeout (~5-15 min), during scale-out
  • Factors affecting duration: Runtime (Java/C# worst, Python/Node fastest), package size, VPC attachment, init code complexity

Lambda Best Practices

Performance

  • Use Provisioned Concurrency for latency-sensitive workloads
  • Minimize package size (remove unused deps, use layers)
  • Initialize SDK clients outside the handler (reuse across invocations)
  • Use ARM64 (Graviton2) for better price-performance
  • Increase memory to get proportionally more CPU

Architecture

  • Keep functions small and focused (single responsibility)
  • Use environment variables for configuration
  • Use dead letter queues (DLQ) for failed async invocations
  • Avoid VPC unless necessary (adds cold start overhead)
  • Use Step Functions for orchestrating multiple Lambdas

9. Cost Optimization

Pricing Models

Model Savings Commitment Flexibility Best For
On-Demand 0% (baseline) None Full Unpredictable workloads, short-term
Reserved Instances Up to 72% 1 or 3 years Low (instance family/region locked) Steady-state, predictable workloads
Savings Plans Up to 72% 1 or 3 years ($/hr) Medium (Compute SP is flexible) Flexible compute commitment
Spot Instances Up to 90% None (can be interrupted) High (but no guarantee) Fault-tolerant, batch, CI/CD, ML training

Cost Optimization Strategies

1
Right-sizing

Use AWS Compute Optimizer and CloudWatch metrics to identify over-provisioned instances. Downsize instances with consistently low CPU/memory utilization (<40%). Consider Graviton (ARM) instances for 20% better price-performance.

2
Spot Instances for EKS

Use spot instances for non-critical node groups. Implement pod disruption budgets, use multiple instance types/AZs, and run stateless workloads. Combine with Karpenter for intelligent spot instance selection and fallback to on-demand.

3
S3 Lifecycle Policies

Transition objects through storage classes based on access patterns. Standard -> Standard-IA (30 days) -> Glacier (90 days) -> Deep Archive (365 days). Use S3 Intelligent-Tiering for unknown patterns.

4
Tagging Strategy

Enforce cost allocation tags: Team, Environment, Project, CostCenter. Use AWS Organizations SCPs to enforce tagging. Enable Cost Explorer with tag-based grouping for chargeback/showback.

5
Eliminate Waste

Identify and remove: unattached EBS volumes, unused Elastic IPs, idle load balancers, oversized NAT Gateways, forgotten dev/test environments. Use AWS Trusted Advisor and Cost Explorer for anomaly detection.

10. Interview Q&A

Q1: When would you choose EKS vs ECS vs Lambda?
EKS: When you need Kubernetes-specific features (CRDs, operators, service mesh), multi-cloud portability, or have existing K8s expertise. More complex but more flexible. ECS: AWS-native container orchestration, simpler than EKS, deep AWS integration (native IAM task roles, CloudMap service discovery). Good when you don't need K8s ecosystem. Lambda: Event-driven, short-lived tasks (<15min), unpredictable traffic patterns, minimal ops overhead. Not suitable for long-running processes, websockets, or workloads needing persistent local state.
Q2: Explain IAM OIDC federation and IRSA. Why is it important for EKS?
IRSA (IAM Roles for Service Accounts) allows individual Kubernetes pods to assume specific IAM roles without sharing the node's role. Without IRSA, all pods on a node share the node's IAM role (overly permissive). With IRSA: (1) EKS exposes an OIDC provider, (2) K8s ServiceAccount is annotated with an IAM role ARN, (3) Pod gets a projected JWT token, (4) AWS SDK calls STS AssumeRoleWithWebIdentity, (5) STS validates the token against the OIDC provider and returns temporary credentials. This enables least privilege per pod.
Q3: Design a VPC for a production EKS cluster. What are the key considerations?
Key decisions: (1) CIDR sizing: /16 VPC with /19 subnets (8192 IPs each) for pod IP density with VPC CNI. (2) Multi-AZ: minimum 3 AZs. (3) Subnet tiers: public (ALB, NAT GW), private (EKS nodes), private-data (RDS). (4) IP planning: VPC CNI assigns pod IPs from subnet - plan for max pods. Consider prefix delegation or custom networking for scale. (5) NAT GW per AZ for resilience. (6) VPC endpoints for ECR, S3, STS, Logs (reduce NAT costs, improve security). (7) DNS: enable DNS hostnames and resolution. (8) No overlapping CIDRs with other VPCs (for peering/TGW).
Q4: What is S3 strong consistency and why did it matter when AWS introduced it?
Since December 2020, S3 provides strong read-after-write consistency for all operations (PUT, DELETE, LIST). Previously, S3 had eventual consistency for overwrite PUTs and DELETEs (you might read stale data). This meant applications needed workarounds (DynamoDB for consistency tracking, retry logic). Now: immediately after a successful PUT, any GET returns the latest version. LIST operations immediately reflect new/deleted objects. This simplified architectures that depend on S3 as a source of truth (data lakes, config storage, artifact repositories).
Q5: Compare Route 53 routing policies. When would you use Weighted vs Latency vs Failover?
Weighted: Traffic distribution by percentage. Use for canary deployments (send 5% to new version), A/B testing, or gradual migration between environments. Latency: Routes to the AWS region with lowest measured latency from the user. Use for multi-region active-active deployments to optimize user experience. Failover: Active-passive with health checks. Primary receives all traffic unless health check fails, then DNS points to secondary (DR region). Use when you want a hot standby but don't want active-active complexity. You can also combine them: Latency routing with Failover within each region.
Q6: How do you optimize costs for an EKS cluster running in production?
Compute: (1) Savings Plans for baseline capacity, (2) Spot instances for fault-tolerant workloads with Karpenter (diverse instance types, multiple AZs), (3) Fargate for bursty/infrequent workloads. Right-sizing: VPA recommendations, Goldilocks tool, Kubecost for per-pod cost allocation. Cluster: Cluster Autoscaler/Karpenter to scale down idle nodes. Schedule dev/staging clusters to shutdown off-hours. Networking: VPC endpoints to reduce NAT GW data processing costs. Storage: GP3 over GP2 (20% cheaper, better performance). Lifecycle policies for logs. Monitoring: Tag everything for cost attribution.
Q7: What is the difference between ALB and NLB? When do you use each?
ALB (Application Load Balancer): Layer 7, understands HTTP/HTTPS. Supports path-based routing, host-based routing, gRPC, WebSocket, Lambda targets, WAF integration. Use for: web applications, microservices with URL-based routing, HTTP/2. NLB (Network Load Balancer): Layer 4, handles TCP/UDP/TLS. Ultra-low latency (~100us), millions of requests/sec, static IP/Elastic IP per AZ, preserves source IP. Use for: non-HTTP protocols (MQTT, gaming), extreme performance needs, when you need static IPs, or TCP passthrough. In EKS: ALB via AWS Load Balancer Controller for Ingress, NLB for LoadBalancer-type Services.
Q8: How does CloudFront work and how would you configure it for a web application?
CloudFront is a CDN with 400+ edge locations. It caches content close to users. Architecture: (1) Client hits nearest edge location, (2) Cache hit = instant response, (3) Cache miss = fetch from origin (S3, ALB, custom). Configuration for web app: Origin 1: S3 bucket for static assets (path: /static/*). Origin 2: ALB for API (path: /api/*). Behaviors: cache static assets aggressively (TTL: 1 year with cache-busting), forward all headers for /api/* (no caching or short TTL). Add custom error pages, HTTPS redirect, Origin Access Control for S3 (block direct S3 access). Enable compression, HTTP/2, and Lambda@Edge for auth or A/B testing.
Q9: Explain VPC endpoints. What are the types and why are they important?
VPC Endpoints allow private connectivity to AWS services without internet (no IGW, NAT GW, VPN needed). Gateway Endpoints: Free, route-table based. Only S3 and DynamoDB. Traffic stays on AWS backbone. Interface Endpoints (PrivateLink): ENI in your subnet with private IP. Works for 100+ services (ECR, STS, CloudWatch, Secrets Manager, etc.). Costs: per-hour + per-GB. Why important: (1) Security: no internet exposure, (2) Cost: avoid NAT GW data processing fees (~$0.045/GB), (3) Reliability: no internet dependency, (4) Compliance: data never leaves AWS network. For EKS: essential for ECR pulls, STS (IRSA), CloudWatch Logs in private clusters.
Q10: How would you implement a disaster recovery strategy on AWS?
DR strategies (increasing cost and lower RTO/RPO): (1) Backup & Restore: S3 cross-region replication, RDS automated backups, AMI copies. RPO: hours, RTO: hours. (2) Pilot Light: Core components running in DR region (RDS replica, minimal infra). Scale up on failover. RPO: minutes, RTO: 10-30min. (3) Warm Standby: Scaled-down version running in DR. Route 53 failover routing. RPO: seconds, RTO: minutes. (4) Active-Active: Full deployment in both regions, Route 53 latency routing, DynamoDB global tables, Aurora global database. RPO: ~0, RTO: ~0. Key services: Route 53 health checks trigger failover, S3 CRR, RDS cross-region read replicas, Aurora global DB, DynamoDB global tables.

11. Scenarios

1. EKS Node Group Scaling Issues

Symptom: Pods stuck in Pending state. Cluster Autoscaler not adding new nodes.

1. Check pod events: kubectl describe pod <pod> - look for "Insufficient cpu/memory" or "0/N nodes are available"
2. Check Cluster Autoscaler logs: kubectl logs -n kube-system -l app=cluster-autoscaler
3. Common causes: ASG max size reached, instance type unavailable in AZ, node group IAM role missing permissions, subnet has no available IPs
4. Fix: Increase ASG max size, add instance type diversity, check subnet CIDR space, verify IAM permissions (autoscaling:SetDesiredCapacity, ec2:RunInstances)
5. Prevention: Use Karpenter instead (faster scaling, automatic instance type selection, consolidation). Set appropriate pod resource requests (not too high)
2. Cross-Account Access Setup

Requirement: EKS pods in Account A need to access S3 bucket in Account B.

1. In Account A: Create IAM role with IRSA trust policy (EKS OIDC provider as principal)
2. In Account A: Role policy allows s3:GetObject on Account B bucket ARN
3. In Account B: Add bucket policy allowing Account A's IAM role as principal
4. In Account A (K8s): Create ServiceAccount annotated with the IAM role ARN
5. Verify: Deploy test pod with SA, run aws s3 ls s3://account-b-bucket/ from inside pod
6. Common issues: Missing bucket policy in Account B, OIDC provider not associated, wrong namespace/SA in trust policy condition
3. S3 Permission Denied (Access Denied)

Symptom: Application returns "Access Denied" when accessing S3 objects.

1. Identify the caller: aws sts get-caller-identity - confirm which IAM principal is being used
2. Check IAM policy: Does the identity have s3:GetObject on the correct resource ARN? (bucket/* not just bucket)
3. Check bucket policy: Is there an explicit Deny? Does the bucket policy allow the principal?
4. Check S3 Block Public Access settings (account-level and bucket-level)
5. Check VPC endpoint policy (if using S3 Gateway endpoint - it may restrict which buckets are accessible)
6. Check KMS key policy if bucket uses SSE-KMS (caller needs kms:Decrypt permission on the key)
7. Use IAM Policy Simulator or CloudTrail logs to identify exactly which permission is being denied
4. Lambda Cold Start Optimization

Symptom: API responses are slow (~3-5s) intermittently. CloudWatch shows init duration spikes.

1. Measure: Check CloudWatch metrics for "Init Duration" - this is the cold start. Look at "Duration" for warm execution time
2. Reduce package size: Remove unused dependencies, use Lambda layers for shared code, use webpack/esbuild for JS bundling
3. Move initialization outside handler: SDK clients, DB connections, config loading should be at module level (reused across invocations)
4. If in VPC: Use VPC with Hyperplane ENIs (now default - dramatically reduced VPC cold starts). Consider if VPC is even necessary
5. Use Provisioned Concurrency for latency-critical paths (keeps N instances pre-warmed). Set up Application Auto Scaling on provisioned concurrency
6. Consider SnapStart (Java only): snapshots initialized function for near-zero cold starts
7. Runtime choice: Python/Node.js have fastest cold starts (~100-300ms). Java/C# are slowest (1-5s without SnapStart)
