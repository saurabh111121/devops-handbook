Linux & Networking Fundamentals

Core Linux internals, process management, TCP/IP networking, and cloud infrastructure patterns

1. Process Management

Process Fundamentals

  • PID: Process ID - unique identifier assigned by the kernel. PID 1 is the init process (systemd on modern Linux)
  • PPID: Parent Process ID - the PID of the process that spawned this one
  • init/systemd: First process started by the kernel. Adopts orphaned processes. Manages service lifecycle
  • Fork/Exec: How processes are created: fork() creates a copy, exec() replaces it with a new program

Zombie Processes

What is a Zombie Process?

A zombie is a process that has finished execution but still has an entry in the process table. It exists so the parent can read its exit status via wait().

  • Cause: Parent did not call wait() or waitpid()
  • Problem: Zombies consume PID entries (limited resource)
  • Detection: ps aux | grep Z shows state as Z (zombie)
  • Fix: Kill the parent process (orphaned zombies are adopted by init/systemd which reaps them)
  • In containers: PID 1 must handle SIGCHLD - this is the "PID 1 problem" (use tini or --init)

2. Containers & Linux Primitives

What Makes a Container?

A container is NOT a VM. It is a regular Linux process with namespaces for isolation and cgroups for resource limits. The container sees its own isolated view of the system.

CPU Cgroup Controls

  • cpu.shares: Relative weight (requests). 1024 = 1 CPU core equivalent
  • cpu.cfs_quota_us: Hard limit (limits). Max CPU time per period
  • cpu.cfs_period_us: Period length (default 100ms)
  • cpuset.cpus: Pin to specific CPU cores

K8s 100m = 100 millicores = 10% of one core

Memory Cgroup Controls

  • memory.limit_in_bytes: Hard limit. OOM kill if exceeded
  • memory.soft_limit_in_bytes: Soft limit (reclaim target)
  • memory.usage_in_bytes: Current usage
  • memory.oom_control: OOM killer behavior

K8s 128Mi = 128 mebibytes memory limit

5. HTTP/2 & gRPC

HTTP/2 Key Features

  • Multiplexing: Multiple requests/responses over a single TCP connection simultaneously (no head-of-line blocking at HTTP level)
  • Header Compression: HPACK reduces redundant headers (cookies, user-agent sent once)
  • Server Push: Server can send resources before client requests them
  • Binary Framing: More efficient parsing than HTTP/1.1 text-based
  • Stream Prioritization: Client can indicate priority of requests

gRPC

  • Built on HTTP/2: Gets multiplexing, streaming, header compression for free
  • Protocol Buffers: Binary serialization (smaller, faster than JSON)
  • Strongly typed: Service contracts defined in .proto files
  • Code generation: Client/server stubs auto-generated in any language
  • Streaming types:
  1. Unary (request-response)
  2. Server streaming (one request, stream of responses)
  3. Client streaming (stream of requests, one response)
  4. Bidirectional streaming (both stream simultaneously)

6. Load Balancing

Algorithm How It Works Best For Drawback
Round Robin Requests distributed sequentially to each server Homogeneous servers, stateless apps Ignores server load/capacity differences
Least Connections Routes to server with fewest active connections Long-lived connections (WebSocket, DB) Doesn't account for request weight
Weighted Round Robin Round robin with weights (bigger server gets more) Heterogeneous server fleet Static weights don't adapt to load
IP Hash Hash client IP to pick server (same client -> same server) Session affinity without cookies Uneven distribution if clients behind NAT
Consistent Hashing Hash ring. Minimal redistribution when servers added/removed Caches, distributed systems, session stores More complex implementation
Least Response Time Routes to server with lowest response time + fewest connections Latency-sensitive applications Requires health check overhead
Random Randomly select a server Simple, works well at scale (power of 2 random choices) Can create hotspots at small scale

7. Resilience Patterns

Circuit Breaker

Circuit Breaker State Machine
                    Success
              +-------------------+
              |                   |
              v                   |
  +--------+------+     +--------+--------+
  |    CLOSED     |     |   HALF-OPEN     |
  | (Normal flow) |     | (Test with      |
  |               |     |  limited reqs)  |
  +-------+-------+     +--------+--------+
          |                       ^
          | Failure threshold     | Timer expires
          | exceeded              |
          v                       |
  +-------+-------+              |
  |     OPEN      |--------------+
  | (Fail fast,   |
  |  reject calls)|  Timeout (e.g., 30s)
  +--------------+

  CLOSED:    All requests pass through. Count failures.
             If failures > threshold (e.g., 5 in 10s) --> OPEN

  OPEN:      All requests immediately fail (no network call).
             After timeout --> HALF-OPEN

  HALF-OPEN: Allow limited requests through.
             If success --> CLOSED
             If failure --> OPEN

Retry with Exponential Backoff

Formula

wait_time = base_delay * 2^attempt + random_jitter

  • Attempt 1: wait 1s + jitter
  • Attempt 2: wait 2s + jitter
  • Attempt 3: wait 4s + jitter
  • Attempt 4: wait 8s + jitter
  • Max retries: Cap at N attempts or max_delay

Jitter is critical: Without jitter, all clients retry at the same time ("thundering herd"). Random jitter spreads retries over time.

Full jitter: wait_time = random(0, base_delay * 2^attempt)

Rate Limiting - Token Bucket

Token Bucket Algorithm
  +-------------------------------------------+
  |              TOKEN BUCKET                  |
  |                                           |
  |  Tokens added at fixed rate               |
  |  (e.g., 10 tokens/second)                 |
  |           |                               |
  |           v                               |
  |  +------------------+                     |
  |  | Bucket           |  Max capacity: 100  |
  |  | [T][T][T][T][T]  |                     |
  |  | [T][T][T][ ][ ]  |  Current: 8 tokens  |
  |  +--------+---------+                     |
  |           |                               |
  |           v                               |
  |  Request arrives:                         |
  |    - Has token? --> Allow + remove token  |
  |    - No token?  --> Reject (429) or queue |
  +-------------------------------------------+

  Properties:
  - Allows bursts up to bucket capacity
  - Steady-state rate = refill rate
  - Smooth traffic over time

  Alternatives:
  - Leaky Bucket: Fixed output rate, queue excess
  - Sliding Window: Count requests in time window
  - Fixed Window: Count per time period (boundary issues)

TTL (Time To Live)

  • What: How long resolvers should cache this record (in seconds)
  • Low TTL (60-300s): Faster propagation of changes, more DNS queries (cost)
  • High TTL (3600-86400s): Better caching, slower to update
  • Strategy: Lower TTL before migrations, raise after stable

10. Interview Q&A

Q1: What is the difference between cgroups and namespaces?

Answer: Namespaces provide isolation (what a process can see) - PID namespace gives own PID tree, network namespace gives own interfaces/IPs. Cgroups provide resource limits (how much a process can use) - CPU shares/quotas, memory limits, I/O bandwidth. Together they make containers: namespaces isolate the view, cgroups limit the resources. A container is just a process with both applied.

Q2: What is the PID 1 problem in containers?

Answer: In a container, the main process becomes PID 1 (init). PID 1 has special responsibilities: (1) it must reap zombie processes (call wait() on children), (2) it must forward signals properly (SIGTERM for graceful shutdown). Most applications are NOT designed to be PID 1 and ignore SIGTERM or don't reap zombies. Solution: use a lightweight init like tini (Docker's --init flag) or dumb-init as the entrypoint, which handles signal forwarding and zombie reaping, then execs your application.

Q3: Explain the TCP three-way handshake and why it is needed.

Answer: The 3-way handshake (SYN, SYN-ACK, ACK) establishes a reliable TCP connection. (1) Client sends SYN with initial sequence number x. (2) Server responds with SYN-ACK: acknowledges x+1 and sends its own sequence number y. (3) Client sends ACK acknowledging y+1. Purpose: both sides agree on initial sequence numbers (needed for ordering), confirm both can send AND receive, and allocate resources (buffers). Without it, you cannot guarantee reliable, ordered communication.

Q4: What is the difference between a reverse proxy and a load balancer?

Answer: A reverse proxy sits in front of servers, terminating client connections and creating new ones to backends. It can cache, compress, terminate TLS, and hide backend topology. A load balancer distributes traffic across multiple servers using algorithms (round-robin, least-connections). In practice, they overlap heavily: Nginx/Envoy/HAProxy do both. Key distinction: a reverse proxy can work with one backend (security, caching), a load balancer implies multiple backends for availability and scaling.

Q5: Walk through what happens when you type a URL in a browser.

Answer: (1) DNS resolution: browser cache -> OS cache -> recursive resolver -> root -> TLD -> authoritative NS. (2) TCP connection: 3-way handshake with server IP. (3) TLS handshake: client hello, server hello + cert, key exchange, finished. (4) HTTP request: GET / with headers. (5) Server processes request, returns response (status code + body). (6) Browser parses HTML, discovers sub-resources (CSS, JS, images), fetches them (HTTP/2 multiplexing). (7) Render: DOM tree + CSSOM -> render tree -> layout -> paint.

Q6: How would you design a VPC for a production workload?

Answer: Multi-AZ for HA (minimum 2, preferably 3). Three subnet tiers: public (ALB, NAT GW), private (application), private-data (RDS, ElastiCache). Use /16 VPC CIDR with /24 subnets (allows growth). Security: SGs for instance-level (reference other SGs, not CIDRs), NACLs for subnet guardrails. NAT GW in each AZ for resilience. VPC endpoints for AWS services (avoid internet routing). Flow logs enabled for troubleshooting. Consider: CIDR ranges should not overlap if VPC peering is needed later.

Q7: What is consistent hashing and why is it used in load balancing?

Answer: Consistent hashing maps both servers and requests onto a ring (0 to 2^32). A request is routed to the nearest server clockwise on the ring. When a server is added/removed, only K/N keys need to remap (K=total keys, N=servers) rather than reshuffling everything. This is critical for caches (Memcached, Redis cluster) where remapping means cache misses. Virtual nodes (multiple points per server on the ring) improve distribution uniformity.

Q8: How does a circuit breaker prevent cascade failures?

Answer: Without a circuit breaker, when service B is down, service A keeps sending requests that timeout, consuming threads/connections, eventually making A unresponsive too (cascade failure). The circuit breaker monitors failure rate and OPENS (trips) after a threshold, immediately rejecting calls to B without waiting for timeouts. This frees A's resources to serve other requests. After a timeout, it enters HALF-OPEN to test if B recovered. This pattern gives failing services time to recover without being overwhelmed by retry storms.
