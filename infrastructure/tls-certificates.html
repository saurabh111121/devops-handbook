TLS & Certificates

TLS handshake, certificate management, cert-manager in Kubernetes, mTLS, and OpenSSL operations

1. TLS/SSL Basics

Symmetric Encryption

  • Same key for encryption and decryption
  • Fast: AES-256-GCM is hardware-accelerated
  • Problem: How to share the key securely?
  • Used for: Bulk data encryption after handshake
  • Algorithms: AES-128, AES-256, ChaCha20-Poly1305

Asymmetric Encryption

  • Key pair: Public key (encrypt) + Private key (decrypt)
  • Slow: 1000x slower than symmetric
  • Solves: Key exchange problem (send public key openly)
  • Used for: Key exchange, digital signatures, authentication
  • Algorithms: RSA, ECDSA, Ed25519, ECDH (key exchange)

How TLS Combines Both

TLS uses asymmetric encryption during the handshake to securely exchange a session key, then switches to symmetric encryption for the actual data transfer. This gives you the security of asymmetric key exchange with the speed of symmetric data encryption.

2. TLS Handshake

TLS 1.2 Handshake (2 Round Trips)
  Client                                              Server
    |                                                    |
    |  1. ClientHello                                   |
    |    - TLS version (1.2)                            |
    |    - Random bytes (client_random)                 |
    |    - Supported cipher suites                      |
    |    - SNI (server name indication)                 |
    |--------------------------------------------------->|
    |                                                    |
    |  2. ServerHello                                   |
    |    - Chosen cipher suite                          |
    |    - Random bytes (server_random)                 |
    |    - Session ID                                   |
    |<---------------------------------------------------|
    |                                                    |
    |  3. Certificate                                   |
    |    - Server's certificate chain                   |
    |<---------------------------------------------------|
    |                                                    |
    |  4. ServerKeyExchange (if DHE/ECDHE)             |
    |    - DH parameters + signature                   |
    |<---------------------------------------------------|
    |                                                    |
    |  5. ServerHelloDone                              |
    |<---------------------------------------------------|
    |                                                    |
    |  6. ClientKeyExchange                            |
    |    - Pre-main secret (encrypted with             |
    |      server's public key) OR DH public value     |
    |--------------------------------------------------->|
    |                                                    |
    |  7. ChangeCipherSpec                             |
    |    - "Switching to encrypted mode"               |
    |--------------------------------------------------->|
    |                                                    |
    |  8. Finished (encrypted)                         |
    |    - Hash of all handshake messages              |
    |--------------------------------------------------->|
    |                                                    |
    |  9. ChangeCipherSpec                             |
    |<---------------------------------------------------|
    |                                                    |
    |  10. Finished (encrypted)                        |
    |<---------------------------------------------------|
    |                                                    |
    |  ======= Application Data (Encrypted) =======    |
    |         Using symmetric session key               |
    |         (derived from pre-main secret +          |
    |          client_random + server_random)            |

TLS 1.2 vs TLS 1.3

Feature TLS 1.2 TLS 1.3
Handshake RTTs 2 round trips 1 round trip (0-RTT for resumption)
Key Exchange RSA, DHE, ECDHE ECDHE only (forward secrecy mandatory)
Cipher Suites Many (some insecure: RC4, 3DES, SHA1) Only 5 secure suites (AEAD only)
Forward Secrecy Optional (only with DHE/ECDHE) Always (RSA key exchange removed)
Handshake Encryption Handshake in plaintext (except Finished) Most of handshake encrypted (after ServerHello)
0-RTT Resumption Not supported Supported (with replay protection caveats)
Removed N/A RSA key exchange, CBC, RC4, SHA-1, compression
Performance Slower (more round trips) Faster (~100ms saved per connection)

Forward Secrecy (Perfect Forward Secrecy - PFS)

With forward secrecy, even if the server's private key is compromised later, past recorded sessions cannot be decrypted. This is because each session uses a unique ephemeral key (ECDHE) that is discarded after use. Without PFS (RSA key exchange): if the private key leaks, ALL past sessions can be decrypted. TLS 1.3 mandates PFS.

3. Certificate Types

Type Validation Level Issuance Time Trust Indicator Use Case
DV (Domain Validated) Domain ownership only Minutes (automated) Padlock icon Blogs, APIs, internal services
OV (Organization Validated) Domain + organization identity Days (manual verification) Padlock + org name in details Business websites, SaaS
EV (Extended Validation) Domain + extensive org verification Weeks (strict process) Padlock (green bar removed) Banks, e-commerce (declining use)

Special Certificate Types

Wildcard Certificates

  • Covers *.example.com - all single-level subdomains
  • Does NOT cover *.sub.example.com (only one level)
  • Does NOT cover the bare domain example.com (need separate SAN)
  • Convenience vs security tradeoff (key compromise affects all subdomains)
  • Not available with EV certificates

SAN (Subject Alternative Names)

  • One certificate covering multiple domains
  • Example: example.com, www.example.com, api.example.com, example.org
  • Modern approach (CN field is deprecated for validation)
  • Can mix domains and wildcards in one cert
  • Browsers check SAN field (not CN) for validation

5. Let's Encrypt & ACME

ACME Protocol (Automatic Certificate Management Environment)

ACME automates the entire certificate lifecycle: domain verification, certificate issuance, and renewal. Let's Encrypt is the largest ACME-based CA, issuing free DV certificates with 90-day validity.

Challenge Types

Challenge How It Works Pros Cons Best For
HTTP-01 Place a file at http://domain/.well-known/acme-challenge/TOKEN Simple, works with any web server Requires port 80 open, no wildcards Single domain certs, simple setups
DNS-01 Create TXT record _acme-challenge.domain with token Supports wildcards, no port 80 needed Requires DNS API access, propagation delay Wildcard certs, private servers
TLS-ALPN-01 Respond on port 443 with special self-signed cert via ALPN Only needs port 443 Less widely supported When only 443 is available

7. mTLS (Mutual TLS)

What is mTLS?

In standard TLS, only the server presents a certificate (client verifies server identity). In mutual TLS (mTLS), both sides present certificates and verify each other. The client also proves its identity to the server. This provides strong authentication for service-to-service communication.

mTLS vs Standard TLS
  STANDARD TLS (one-way):            mTLS (two-way):

  Client         Server              Client         Server
    |               |                  |               |
    |  ClientHello  |                  |  ClientHello  |
    |-------------->|                  |-------------->|
    |               |                  |               |
    |  ServerHello  |                  |  ServerHello  |
    |  + Server Cert|                  |  + Server Cert|
    |<--------------|                  |  + CertRequest|  <-- Server requests
    |               |                  |<--------------|      client cert
    |  (verify      |                  |               |
    |   server)     |                  |  Client Cert  |  <-- Client sends
    |               |                  |-------------->|      its certificate
    |  Key Exchange |                  |               |
    |-------------->|                  |  (both verify |
    |               |                  |   each other) |
    |  === DATA === |                  |               |
    |               |                  |  Key Exchange |
    |               |                  |-------------->|
    | Server        |                  |               |
    | authenticated |                  |  === DATA === |
    | only          |                  |               |
    |               |                  | BOTH sides    |
    |               |                  | authenticated |

  USE CASES for mTLS:
  - Service-to-service in microservices (zero-trust)
  - Service mesh (Istio/Linkerd auto-inject mTLS)
  - API authentication (replace API keys)
  - IoT device authentication
  - Database connections

8. OpenSSL Commands

Generate Private Key

openssl genrsa -out server.key 2048
openssl ecparam -genkey -name prime256v1 -out server-ec.key
openssl genpkey -algorithm Ed25519 -out server-ed25519.key

Create Certificate Signing Request (CSR)

Generate CSR with SAN OpenSSL 
# Interactive CSR generation
openssl req -new -key server.key -out server.csr

# Non-interactive with subject
openssl req -new -key server.key -out server.csr \
  -subj "/C=US/ST=California/L=SF/O=MyOrg/CN=example.com"

# CSR with Subject Alternative Names (SAN)
openssl req -new -key server.key -out server.csr \
  -subj "/CN=example.com" \
  -addext "subjectAltName=DNS:example.com,DNS:www.example.com,DNS:api.example.com"

# View CSR contents
openssl req -in server.csr -text -noout

Self-Signed Certificate

Self-Signed Cert (testing/internal) OpenSSL 
# Generate key + self-signed cert in one command
openssl req -x509 -nodes -days 365 \
  -newkey rsa:2048 \
  -keyout tls.key \
  -out tls.crt \
  -subj "/CN=myapp.internal" \
  -addext "subjectAltName=DNS:myapp.internal,DNS:localhost,IP:127.0.0.1"

# Create CA, then sign server cert (proper chain)
# 1. Create CA key and cert
openssl req -x509 -nodes -days 3650 \
  -newkey rsa:4096 \
  -keyout ca.key -out ca.crt \
  -subj "/CN=My Internal CA"

# 2. Sign server CSR with CA
openssl x509 -req -days 365 \
  -in server.csr \
  -CA ca.crt -CAkey ca.key -CAcreateserial \
  -out server.crt \
  -extfile <(printf "subjectAltName=DNS:example.com,DNS:www.example.com")

Verify and Inspect Certificates

Certificate Inspection Commands OpenSSL 
# View certificate details
openssl x509 -in server.crt -text -noout

# Check certificate expiry date
openssl x509 -in server.crt -enddate -noout

# Verify cert chain
openssl verify -CAfile ca.crt server.crt

# Check remote server certificate
openssl s_client -connect example.com:443 -servername example.com /dev/null \
  | openssl x509 -text -noout

# Check expiry of remote server
openssl s_client -connect example.com:443 -servername example.com /dev/null \

# Check full chain from remote server
openssl s_client -connect example.com:443 -servername example.com -showcerts

9. Common Issues

Certificate Expired

  • Symptoms: ERR_CERT_DATE_INVALID, connection refused, HTTPS errors
  • Detection: openssl s_client -connect host:443 shows expiry
  • Fix: Renew certificate, update secret, restart/reload services
  • Prevention: cert-manager auto-renewal, monitoring expiry with alerts (30 days out)

Incomplete Certificate Chain

  • Symptoms: Works in Chrome (chain completion) but fails in curl/Python/Java
  • Cause: Server only sends leaf cert, missing intermediate(s)
  • Detection: openssl s_client -connect host:443 shows "verify error:num=21"
  • Fix: Concatenate leaf + intermediate(s) in server cert file (order matters: leaf first)

Name Mismatch

  • Symptoms: ERR_CERT_COMMON_NAME_INVALID
  • Cause: Requested hostname not in cert's SAN/CN
  • Example: Cert for www.example.com but accessing example.com
  • Fix: Reissue cert with all required SANs, or use wildcard

Mixed Content

  • Symptoms: Page loads but resources fail, browser warnings
  • Cause: HTTPS page loading HTTP resources (scripts, images, CSS)
  • Fix: Update all resource URLs to HTTPS or use protocol-relative URLs
  • Prevention: Content-Security-Policy header, upgrade-insecure-requests

TLS Version / Cipher Mismatch

  • Symptoms: "No common cipher suites" or "Protocol version not supported"
  • Cause: Server configured for TLS 1.3 only, client only supports TLS 1.2 (or vice versa)
  • Detection: openssl s_client -connect host:443 -tls1_2 (test specific version)
  • Fix: Enable compatible TLS versions and cipher suites on server. Recommended minimum: TLS 1.2

10. Interview Q&A

Q1: Walk through the TLS 1.3 handshake. How is it faster than TLS 1.2?
TLS 1.3 completes in 1 RTT (vs 2 for TLS 1.2). Client sends ClientHello with key_share (guesses server's preferred key exchange method, usually X25519). Server responds with ServerHello + key_share + encrypted extensions + certificate + CertificateVerify + Finished - all in one flight. Client sends Finished. The key insight: by including key_share speculatively, the key exchange and hello happen in parallel. Additionally, TLS 1.3 supports 0-RTT resumption for returning clients (sends encrypted data in the first message), though this has replay attack risks for non-idempotent requests.
Q2: What is the difference between symmetric and asymmetric encryption? Why does TLS use both?
Symmetric: Same key encrypts and decrypts. Fast (AES is hardware-accelerated, ~GB/s). Problem: how to share the key securely? Asymmetric: Public key encrypts, private key decrypts (or signs). Solves key distribution but is ~1000x slower. TLS uses both: Asymmetric (ECDHE) during handshake to securely derive a shared secret, then symmetric (AES-GCM) for bulk data transfer. This hybrid approach gives security of asymmetric key exchange with performance of symmetric encryption. The session key is ephemeral - new for each connection (forward secrecy).
Q3: What is mTLS and when would you use it?
mTLS (Mutual TLS) means both client and server authenticate each other with certificates. In standard TLS, only the server proves its identity. In mTLS, the server also requests and verifies a client certificate. Use cases: (1) Service-to-service authentication in microservices (zero-trust networking), (2) Service mesh (Istio/Linkerd automate mTLS between sidecars), (3) Strong API client authentication (replace API keys), (4) IoT device authentication, (5) Internal services where you control both ends. When NOT to use: Public-facing websites (users don't have client certs), situations where certificate management overhead isn't justified.
Q4: How does cert-manager work in Kubernetes? Walk through the certificate lifecycle.
cert-manager automates TLS certificate management in K8s. Lifecycle: (1) Create a ClusterIssuer (configures CA - Let's Encrypt ACME, Vault, self-signed). (2) Create a Certificate resource (specifies domains, secret name, issuer) OR annotate an Ingress for auto-creation. (3) cert-manager controller creates a CertificateRequest, then an Order (ACME). (4) For each domain, creates a Challenge (HTTP-01: temporary Ingress/pod, DNS-01: creates TXT record). (5) Once validated, submits CSR to CA, receives signed cert. (6) Stores cert+key in a Kubernetes Secret (type: kubernetes.io/tls). (7) Monitors expiry, auto-renews (default: 30 days before expiry). Pod/Ingress mounts the secret and uses the cert.
Q5: Explain the certificate chain of trust. What happens if the chain is incomplete?
Chain of trust: Root CA (in trust store) signs Intermediate CA, which signs Leaf certificate (your server's cert). The server must send the leaf + all intermediates (not the root - client already has it). Client verifies: leaf signed by intermediate? Intermediate signed by root? Root in trust store? All valid = trusted. If chain is incomplete: Browsers may still work (they can fetch missing intermediates via AIA extension), but programmatic clients (curl, Python, Java, mobile apps) will fail with "unable to verify certificate" errors. Fix: Concatenate certs in order: leaf cert first, then intermediate(s). Never include the root.
Q6: A service in production suddenly starts getting TLS errors. How do you diagnose?
Systematic approach: (1) Check certificate expiry: openssl s_client -connect host:443 | openssl x509 -enddate -noout. (2) Check certificate chain: openssl s_client -connect host:443 -showcerts - look for "verify return code". (3) Check domain match: does the SAN include the requested hostname? (4) Check TLS version: openssl s_client -connect host:443 -tls1_2 vs -tls1_3. (5) Check if cert was recently rotated (wrong cert mounted?). (6) For K8s: kubectl get secret tls-secret -o jsonpath='{.data.tls\.crt}' | base64 -d | openssl x509 -text. (7) Check cert-manager: kubectl describe certificate, kubectl get certificaterequest. (8) Monitor: set up alerts for certs expiring within 30 days.
Q7: What is forward secrecy and why does TLS 1.3 mandate it?
Forward secrecy (Perfect Forward Secrecy - PFS) means that compromising the server's long-term private key does NOT allow decryption of past recorded sessions. This is achieved by using ephemeral keys (ECDHE) for each session - the session key is derived from a one-time Diffie-Hellman exchange and the ephemeral keys are discarded. Without PFS (e.g., RSA key exchange in TLS 1.2): if an attacker records encrypted traffic AND later obtains the server's private key, they can decrypt ALL past sessions. TLS 1.3 removed RSA key exchange entirely, making forward secrecy mandatory, because mass surveillance capabilities make "record now, decrypt later" a real threat.
Q8: How does HTTPS work end-to-end? Explain what happens from URL to rendered page.
Steps: (1) DNS resolution: resolve hostname to IP address. (2) TCP handshake: SYN, SYN-ACK, ACK (establish connection). (3) TLS handshake: ClientHello (with SNI indicating hostname), ServerHello + certificate, key exchange (ECDHE), derive symmetric session key. Client verifies: cert chain valid? Domain matches? Not expired? Not revoked? (4) HTTP request: encrypted with session key, sent over TLS tunnel. (5) Server decrypts request, processes it, encrypts response. (6) Client decrypts response, renders page. Key detail: SNI (Server Name Indication) in ClientHello tells the server which certificate to present (one IP can host multiple HTTPS sites). SNI is sent in plaintext in TLS 1.2 (encrypted in TLS 1.3 with ECH).

11. Scenarios

1. Certificate Expired in Production

Symptom: Users getting ERR_CERT_DATE_INVALID. Monitoring alerts firing. Some clients returning 502.

1. Confirm expiry: openssl s_client -connect app.example.com:443 </dev/null 2>/dev/null | openssl x509 -enddate -noout
2. Check cert-manager status: kubectl describe certificate app-tls -n production - look at "Status" and "Events"
3. Check certificate request: kubectl get certificaterequest -n production - is there a pending/failed request?
4. Check challenge: kubectl get challenge -n production - is the ACME challenge failing? (DNS propagation, port 80 blocked?)
5. Emergency fix: If cert-manager is broken, manually issue cert with certbot and create the K8s secret: kubectl create secret tls app-tls-secret --cert=tls.crt --key=tls.key -n production --dry-run=client -o yaml | kubectl apply -f -
6. Force cert-manager renewal: kubectl delete secret app-tls-secret -n production (cert-manager will detect missing secret and re-issue)
7. Prevention: Set up monitoring for cert expiry (Prometheus cert_exporter, or CloudWatch). Alert at 30 days, page at 7 days. Test cert-manager renewal in staging regularly
2. cert-manager Not Issuing Certificates

Symptom: Certificate resource shows "Not Ready". New deployments have no TLS secret.

1. Check Certificate status: kubectl describe certificate <name> -n <ns> - look at Conditions and Events
2. Check Issuer/ClusterIssuer: kubectl describe clusterissuer letsencrypt-prod - is it Ready? ACME account registered?
3. Check CertificateRequest: kubectl get certificaterequest -n <ns> then describe it - look for approval/denial
4. Check Order and Challenge: kubectl get order,challenge -n <ns> - what state? For HTTP-01: is the solver pod/ingress created?
5. Common HTTP-01 issues: Ingress class mismatch, port 80 blocked by firewall/NLB, solver pod cannot be scheduled, rate limits exceeded
6. Common DNS-01 issues: IAM permissions for Route53 missing (IRSA not configured), wrong hosted zone ID, DNS propagation slow
7. Check cert-manager logs: kubectl logs -n cert-manager deploy/cert-manager -f - look for errors
8. Rate limits: Let's Encrypt has rate limits (50 certs/domain/week). Use staging issuer for testing. Check if you hit the limit
3. mTLS Breaking Legacy Service Communication

Symptom: After enabling Istio mTLS (STRICT mode), a legacy service that cannot present a client certificate starts getting connection refused errors.

1. Identify the issue: Legacy service has no Envoy sidecar (not part of mesh) trying to reach a mesh service with STRICT mTLS
2. Immediate fix - use PERMISSIVE mode for the target service (accepts both mTLS and plaintext):
peer-authentication-permissive.yaml Istio 
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: allow-legacy
  namespace: target-service-ns
spec:
  selector:
    matchLabels:
      app: target-service
  mtls:
    mode: PERMISSIVE  # Accept both mTLS and plain HTTP
3. Better fix: Add the legacy service to the mesh (inject sidecar). If it cannot have a sidecar, use a gateway or proxy in front of it
4. Long-term: Migrate all services into the mesh. Use PERMISSIVE during migration, then switch to STRICT once all services have sidecars
5. Alternative: Create a DestinationRule to disable mTLS for traffic TO the legacy service (mesh services connecting out to legacy)
