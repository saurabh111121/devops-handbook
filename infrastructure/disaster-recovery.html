Disaster Recovery

RPO/RTO, DR strategies, backups, Velero, multi-region architecture, chaos engineering, and runbooks

1. DR Fundamentals

RPO vs RTO
  Timeline of a disaster event:

  ───────────────────────────────────────────────────────────────────>
  │                    │                         │                   │
  Last                 │ DISASTER                │                   │
  Backup              OCCURS                  Recovery             Full
  Point                                       Begins              Service

  |<───── RPO ────────>|<──────────── RTO ──────────────────────────>|
  (Recovery Point       (Recovery Time Objective)
   Objective)

  RPO = How much data can you afford to LOSE?
  ┌──────────────────────────────────────────────────────────────────┐
  │  RPO = 0       : No data loss acceptable (sync replication)      │
  │  RPO = 5 min   : Lose at most 5 min of transactions             │
  │  RPO = 1 hour  : Hourly backups, lose up to 1 hour              │
  │  RPO = 24 hours: Daily backups, lose up to 1 day                │
  └──────────────────────────────────────────────────────────────────┘

  RTO = How long can you afford to be DOWN?
  ┌──────────────────────────────────────────────────────────────────┐
  │  RTO = 0       : No downtime (multi-active)                      │
  │  RTO = minutes : Warm standby, automated failover                │
  │  RTO = hours   : Pilot light, some provisioning needed           │
  │  RTO = days    : Backup/restore from scratch                     │
  └──────────────────────────────────────────────────────────────────┘

Business Impact Analysis (BIA)

Before defining DR strategy, understand the business impact:

1
Identify Critical Systems - Revenue-generating services, customer-facing apps, regulatory systems. Rank by business impact.
2
Define Acceptable Loss - How much data loss (RPO) and downtime (RTO) each system can tolerate. Different systems have different requirements.
3
Cost-Benefit Analysis - Lower RPO/RTO = higher cost. A payment system may justify multi-active ($$$), but an internal wiki may only need daily backups ($).
4
Dependencies Mapping - Identify all dependencies (databases, queues, external APIs). DR plan must cover the entire chain, not just individual services.

2. DR Strategies

Strategy RTO RPO Cost Description
Backup & Restore Hours Hours $ (lowest) Restore from backups in new region. Provision infrastructure from scratch.
Pilot Light 10-30 minutes Minutes $$ Core infrastructure running (DB replicas), compute scaled to zero. Scale up on failover.
Warm Standby Seconds-Minutes Seconds $$$ Scaled-down copy running in DR region. Scale up to full on failover.
Multi-Active Near-zero Near-zero $$$$ (highest) Full deployment in multiple regions. Traffic served from all. No failover needed.

Pilot Light Details

  • Database replicas continuously syncing to DR region
  • AMIs/container images pre-built and available
  • Infrastructure as Code ready to provision (Terraform)
  • Compute instances stopped or at minimal scale
  • DNS records pre-configured with low TTL
  • Failover: Start compute, promote DB replica, update DNS

Warm Standby Details

  • Full application stack running at reduced capacity
  • Handles a fraction of production traffic (canary)
  • Database in sync (multi-AZ or cross-region replica)
  • Auto-scaling configured to scale up rapidly
  • Health checks and monitoring active
  • Failover: Scale up compute, promote DB, shift DNS/LB

3. Backup Best Practices

The 3-2-1 Rule

  ┌─────────────────────────────────────────────────────────────┐
  │                                                             │
  │   3  copies of your data                                    │
  │   2  different storage media/types                          │
  │   1  copy offsite (different region/cloud)                  │
  │                                                             │
  │   Example:                                                  │
  │   - Copy 1: Production database (EBS volume)               │
  │   - Copy 2: Automated snapshots (same region S3)           │
  │   - Copy 3: Cross-region replication (different region S3)  │
  │                                                             │
  └─────────────────────────────────────────────────────────────┘

Backup Principles

1
Immutable Backups - Use S3 Object Lock (WORM) or Veeam immutability. Prevents ransomware from encrypting/deleting backups. Configure retention lock (governance or compliance mode).
2
Encryption - Encrypt at rest (AES-256, KMS) and in transit (TLS). Use separate encryption keys from production. Store keys in different account/region than backups.
3
Test Restores Regularly - A backup that cannot be restored is worthless. Schedule monthly restore tests. Automate restore validation. Measure actual RTO during tests.
4
Retention Policies - Define retention based on compliance and business needs. Example: daily for 30 days, weekly for 90 days, monthly for 1 year, yearly for 7 years. Automate lifecycle rules.
s3-backup-lifecycle.tf Terraform 
resource "aws_s3_bucket" "backups" {
  bucket = "company-db-backups-${var.environment}"

  # Enable versioning for additional protection
  versioning {
    enabled = true
  }
}

resource "aws_s3_bucket_lifecycle_configuration" "backup_lifecycle" {
  bucket = aws_s3_bucket.backups.id

  rule {
    id     = "backup-retention"
    status = "Enabled"

    transition {
      days          = 30
      storage_class = "STANDARD_IA"    # Cheaper after 30 days
    }

    transition {
      days          = 90
      storage_class = "GLACIER"        # Archive after 90 days
    }

    expiration {
      days = 365                       # Delete after 1 year
    }
  }
}

resource "aws_s3_bucket_object_lock_configuration" "immutable" {
  bucket = aws_s3_bucket.backups.id

  rule {
    default_retention {
      mode = "GOVERNANCE"
      days = 30                        # Cannot delete for 30 days
    }
  }
}

# Cross-region replication
resource "aws_s3_bucket_replication_configuration" "dr_replication" {
  bucket = aws_s3_bucket.backups.id
  role   = aws_iam_role.replication.arn

  rule {
    id     = "replicate-to-dr"
    status = "Enabled"

    destination {
      bucket        = aws_s3_bucket.backups_dr.arn
      storage_class = "STANDARD_IA"
    }
  }
}

4. Kubernetes DR

Velero (Backup & Restore)

What Velero Backs Up

  • Kubernetes resources (Deployments, Services, ConfigMaps, Secrets, etc.)
  • Persistent Volume data (via snapshots or Restic/Kopia file-level backup)
  • Cluster-scoped resources (Namespaces, CRDs, ClusterRoles)
  • Supports include/exclude by namespace, resource type, or label
Velero Installation Commands 
# Install Velero with AWS provider
velero install \
  --provider aws \
  --plugins velero/velero-plugin-for-aws:v1.8.0 \
  --bucket my-velero-backups \
  --backup-location-config region=us-east-1 \
  --snapshot-location-config region=us-east-1 \
  --secret-file ./credentials-velero \
  --use-node-agent \
  --default-volumes-to-fs-backup

# Verify installation
kubectl get pods -n velero
velero backup-location get
Velero Backup & Restore Commands Operations 
# Create a backup of entire cluster
velero backup create full-cluster-backup --wait

# Backup specific namespace
velero backup create prod-backup \
  --include-namespaces production \
  --include-resources deployments,services,configmaps,secrets,pvc \
  --wait

# Backup with label selector
velero backup create app-backup \
  --selector app=my-service \
  --wait

# Check backup status
velero backup describe full-cluster-backup --details
velero backup logs full-cluster-backup

# Restore entire backup
velero restore create --from-backup full-cluster-backup --wait

# Restore specific namespace to different name
velero restore create --from-backup prod-backup \
  --namespace-mappings production:production-restored \
  --wait

# Restore only specific resources
velero restore create --from-backup prod-backup \
  --include-resources deployments,services \
  --wait

# Check restore status
velero restore describe  --details

    - default

        backup-type: scheduled

etcd-backup-restore.sh Critical 
#!/bin/bash
# etcd backup (run on control plane node)

# Backup etcd
ETCDCTL_API=3 etcdctl snapshot save /backup/etcd-$(date +%Y%m%d-%H%M%S).db \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key

# Restore etcd (DESTRUCTIVE - stops cluster)
# 1. Stop kube-apiserver and etcd

sudo mv /etc/kubernetes/manifests/etcd.yaml /tmp/

# 2. Restore snapshot to new data directory
ETCDCTL_API=3 etcdctl snapshot restore /backup/etcd-latest.db \
  --data-dir=/var/lib/etcd-restored \
  --name=master-1 \
  --initial-cluster=master-1=https://10.0.1.10:2380 \
  --initial-advertise-peer-urls=https://10.0.1.10:2380

# 3. Update etcd manifest to use new data-dir
# 4. Move manifests back
sudo mv /tmp/etcd.yaml /etc/kubernetes/manifests/
sudo mv /tmp/kube-apiserver.yaml /etc/kubernetes/manifests/

# 5. Verify cluster health
kubectl get nodes
kubectl get pods --all-namespaces

RDS Multi-AZ and Cross-Region DR

  │                                                               │
  │   ┌── AZ-1a ──────────┐      ┌── AZ-1b ──────────┐          │

  │   │  └──────┬───────┘ │      │  └──────────────┘ │          │
  │   └─────────┼──────────┘      └────────────────────┘          │
  │             │                                                  │
  │             │ async replication                                 │
  └─────────────┼──────────────────────────────────────────────────┘
                │
                ▼
  ┌──────────────── Region: us-west-2 (DR) ───────────────────────┐
  │                                                                │
  │   ┌── AZ-2a ──────────┐                                       │
  │   │  ┌──────────────┐ │                                       │
  │   │  │ Cross-Region │ │  Read replica (async)                  │
  │   │  │ Read Replica │ │  Can be promoted to standalone         │
  │   │  └──────────────┘ │  in DR event                          │
  │   └────────────────────┘                                       │
  │                                                                │
  └────────────────────────────────────────────────────────────────┘

  Failover options:
  1. Multi-AZ (auto):  ~60-120s, same region, zero data loss
  2. Read Replica (manual): Promote cross-region replica, minutes, some data loss
  3. Aurora Global DB: < 1 min failover, ~1s RPO, managed by Aurora

RDS DR Features

Feature RPO RTO Scope Automatic?
Multi-AZ 0 (sync) 60-120s Same region, diff AZ Yes
Read Replica (same region) Seconds (async) Minutes Same region Manual promote
Cross-Region Read Replica Seconds-minutes Minutes Cross-region Manual promote
Aurora Global Database ~1 second < 1 minute Cross-region Managed failover
Automated Backups 5 min (transaction logs) Hours Same region Requires restore
Manual Snapshot + Copy Hours (snapshot interval) Hours Any region Fully manual
rds-dr-commands.sh AWS CLI 
# Create cross-region read replica
aws rds create-db-instance-read-replica \
  --db-instance-identifier mydb-dr-replica \
  --source-db-instance-identifier arn:aws:rds:us-east-1:123456789:db:mydb-primary \
  --region us-west-2 \
  --db-instance-class db.r6g.xlarge \
  --storage-encrypted \
  --kms-key-id alias/rds-dr-key

# Check replication lag
aws rds describe-db-instances \
  --db-instance-identifier mydb-dr-replica \
  --query 'DBInstances[0].StatusInfos'

# Promote replica to standalone (DR failover)
aws rds promote-read-replica \
  --db-instance-identifier mydb-dr-replica \
  --region us-west-2

# Copy snapshot cross-region (for backup/restore strategy)
aws rds copy-db-snapshot \
  --source-db-snapshot-identifier arn:aws:rds:us-east-1:123456789:snapshot:mydb-daily \
  --target-db-snapshot-identifier mydb-daily-dr-copy \
  --region us-west-2 \
  --kms-key-id alias/rds-dr-key

7. DR Runbooks

DR Runbook Template

Every DR runbook should include these sections:

dr-runbook-template.md Template 
# DR Runbook: [Service/System Name]

## Trigger Criteria
- When to activate this runbook
- Example: "Primary region health check failing for > 5 minutes"
- Example: "RDS Multi-AZ failover failed, manual intervention needed"

## Prerequisites
- [ ] DR region infrastructure verified (last test: YYYY-MM-DD)
- [ ] Credentials and access verified for DR region
- [ ] Communication channels established (Slack #incident, PagerDuty)

## Failover Steps
1. **Confirm the disaster** (not a false alarm)
   - Check CloudWatch alarms
   - Verify from multiple vantage points
   - Check AWS Health Dashboard

2. **Communicate**
   - Notify incident commander
   - Post in #incident channel
   - Update status page: "Investigating"

3. **Execute failover**
   - Step-by-step commands with exact syntax
   - Expected output at each step
   - Verification checks between steps

4. **Validate recovery**
   - Health check endpoints responding
   - Synthetic transactions passing
   - Data integrity checks
   - Performance within acceptable range

5. **Post-failover**
   - Update DNS TTLs back to normal
   - Monitor for 30 minutes
   - Update status page: "Resolved"

## Rollback (Failback)
- Steps to return to primary when ready
- Data synchronization procedure
- Validation before cutting back

## Contacts
| Role              | Name          | Phone/Slack       |
|-------------------|---------------|-------------------|
| Incident Commander| Jane Smith    | @jsmith           |
| DBA On-Call       | Bob Johnson   | +1-555-0123       |
| Platform Engineer | Alice Chen    | @achen            |

## Post-Incident
- Schedule post-mortem within 48 hours
- Update this runbook with lessons learned

Regular DR Drills

1
Tabletop Exercise (Monthly) - Walk through runbook verbally. Identify gaps in documentation. No actual failover performed. All stakeholders present.
2
Component Test (Quarterly) - Failover individual components (DB, cache, queue). Verify recovery of each subsystem independently. Measure actual RTO.
3
Full DR Test (Semi-annually) - Complete failover to DR region. Run full test suite against DR. Measure end-to-end RTO and RPO. Failback and verify.
4
Game Day (Annually) - Unannounced DR test during business hours. Tests team readiness and runbook accuracy. Closest to real disaster conditions.

8. Chaos Engineering for DR

Chaos Engineering Principles

  • Hypothesis: Define expected behavior under failure conditions
  • Steady state: Measure normal system behavior first
  • Inject failure: Introduce real-world failure conditions
  • Observe: Compare actual behavior to hypothesis
  • Minimize blast radius: Start small, increase gradually
  • Run in production: Only production reveals production problems (when ready)

Game Day Planning

Phase Activity Duration
Preparation Define scope, hypothesis, rollback plan. Notify stakeholders. Verify monitoring. 1-2 weeks before
Pre-game Verify steady state metrics. Confirm all participants ready. Check rollback works. 1 hour before
Execution Inject failure. Observe. Document findings in real-time. Abort if impact exceeds threshold. 1-4 hours
Recovery Remove failure injection. Verify full recovery. Confirm steady state restored. 30 min - 1 hour
Post-game Review findings. File improvement tickets. Update runbooks. Share learnings. Next day

Common Failure Injection Scenarios

Infrastructure Failures

  • Pod deletion (test self-healing)
  • Node drain/cordon (test rescheduling)
  • Network partition (test circuit breakers)
  • DNS failure (test fallback)
  • Disk fill (test storage alerts)
  • CPU/memory stress (test autoscaling)

Application Failures

  • Database connection failure (test retry logic)
  • Downstream service latency (test timeouts)
  • Cache failure (test cache-aside fallback)
  • Message queue unavailable (test buffering)
  • Certificate expiry (test TLS handling)
  • Config map deletion (test defaults)

9. Interview Q&A

Q1: What is the difference between RPO and RTO?
Answer: RPO (Recovery Point Objective) is the maximum acceptable data loss measured in time - "how much data can we afford to lose?" An RPO of 1 hour means we can lose up to 1 hour of transactions. RTO (Recovery Time Objective) is the maximum acceptable downtime - "how long can we be down?" An RTO of 15 minutes means the system must be restored within 15 minutes. They are independent: you might have RPO=0 (no data loss) but RTO=1 hour (can be down for an hour while restoring). The cost increases exponentially as both approach zero. RPO drives backup frequency, RTO drives infrastructure readiness.
Q2: Compare the four DR strategies and when to use each.
Answer: (1) Backup/Restore (RTO: hours, cost: $): Store backups off-site, rebuild from scratch. Use for non-critical systems, dev environments. (2) Pilot Light (RTO: 10-30 min, cost: $$): Keep data replicated, infrastructure as code ready, minimal compute running. Scale up on failover. Use for important but not mission-critical systems. (3) Warm Standby (RTO: seconds-minutes, cost: $$$): Scaled-down replica always running. Scale up on failover. Use for business-critical applications. (4) Multi-Active (RTO: near-zero, cost: $$$$): Full deployment in multiple regions, traffic split across all. Use for mission-critical, revenue-generating systems where any downtime is unacceptable. Choice depends on: cost of downtime vs cost of DR infrastructure.
Q3: Describe the Velero backup and restore workflow.
Answer: Backup workflow: (1) Velero calls Kubernetes API to get resources matching the backup spec (namespace, labels, resource types); (2) For PVs, it either takes CSI snapshots or uses file-level backup (Kopia/Restic) to copy data; (3) Resources are serialized to JSON and stored in object storage (S3/GCS); (4) Metadata (backup status, contents) tracked as Kubernetes CRDs. Restore workflow: (1) Velero reads backup from object storage; (2) Creates resources in target cluster (can remap namespaces); (3) Restores PV data from snapshots or file backups; (4) Handles resource conflicts (skip, patch, or overwrite). Key features: Scheduled backups via cron, TTL-based retention, hooks (pre/post backup commands), resource filtering.
Q4: What are the trade-offs of multi-region architecture?
Answer: Benefits: Near-zero RTO, improved latency (users hit closest region), isolation from regional failures, compliance (data sovereignty). Trade-offs: (1) Cost: 2x+ infrastructure, data transfer costs between regions; (2) Complexity: Data replication, conflict resolution, global state management; (3) Consistency: Cross-region replication is async (eventual consistency) or sync (high latency); (4) Operations: Deployments must coordinate across regions, monitoring is more complex; (5) Data conflicts: Active-active requires conflict resolution (LWW, CRDTs, or application logic); (6) Testing: Must test failover regularly, difficult to simulate real regional failures.
Q5: How do you test a DR plan effectively?
Answer: Progressive testing approach: (1) Tabletop exercises (monthly): Walk through runbooks verbally, identify gaps; (2) Component tests (quarterly): Failover individual services, measure actual RTO; (3) Full DR tests (semi-annually): Complete region failover, run full test suite; (4) Game days (annually): Unannounced tests during business hours. Key principles: document everything, measure actual RTO/RPO vs targets, treat test failures as learning opportunities, update runbooks after every test. Common mistakes: only testing during maintenance windows, not testing data integrity after restore, not involving application teams, not testing failback.
Q6: What should you backup in a Kubernetes cluster?
Answer: Critical backups: (1) etcd - Contains all cluster state (the most critical); (2) Kubernetes resources - Deployments, Services, ConfigMaps, Secrets, RBAC, CRDs (use Velero); (3) Persistent Volumes - Application data in PVCs (CSI snapshots or file-level); (4) Certificates and PKI - /etc/kubernetes/pki/ (CA certs, service account keys); (5) Infrastructure as Code - Terraform/Helm charts (in git, but verify); (6) External state - External DNS records, load balancer configs, IAM roles. What NOT to backup: pod logs (use log aggregation), metrics (use time-series DB), node state (immutable, rebuild from image). Strategy: etcd snapshot for cluster recovery, Velero for application-level recovery.
Q7: Explain the RDS failover process.
Answer: Multi-AZ failover: (1) AWS detects failure (instance crash, AZ outage, storage failure); (2) Standby in other AZ is promoted to primary (sync replicated, no data loss); (3) DNS endpoint (CNAME) is updated to point to new primary; (4) Process takes 60-120 seconds; (5) Application reconnects automatically via same endpoint. What triggers failover: AZ outage, primary instance failure, instance type change, software patching, manual reboot with failover. Application considerations: Connection strings use the cluster endpoint (not instance endpoint). Applications should handle brief disconnection (retry logic). Some connections may need to be re-established. Open transactions are lost and must be retried.
Q8: How does chaos engineering improve DR readiness?
Answer: Chaos engineering proactively finds weaknesses before real disasters expose them. For DR specifically: (1) Validates that failover mechanisms actually work under realistic conditions; (2) Reveals hidden dependencies not documented in runbooks; (3) Tests monitoring and alerting (do alarms fire correctly?); (4) Validates team response (do people know what to do?); (5) Measures actual RTO/RPO under failure (vs theoretical). Approach: Start with hypothesis ("system tolerates one pod failure"), inject failure (kill pod), observe (does service stay available?), learn (fix gaps found). Progress from dev to staging to production. Always have a kill switch to stop the experiment. Key insight: DR plans are theory until tested under realistic failure conditions.

10. Troubleshooting Scenarios

! Scenario: Complete AZ Failure

Symptom: Multiple services unreachable. CloudWatch showing widespread failures in us-east-1a. Health checks failing for resources in that AZ. AWS Health Dashboard confirms AZ impairment.

1. Assess impact: Identify all resources in affected AZ: EC2 instances, RDS (check if Multi-AZ), ElastiCache nodes, EBS volumes. Check which services have cross-AZ redundancy.
2. Automatic recovery: Multi-AZ RDS should auto-failover (60-120s). ECS/EKS should reschedule pods to healthy AZ nodes (if capacity exists). ALB should stop routing to unhealthy targets.
3. Manual actions: Scale up ASG in healthy AZs. Check if EKS nodes have capacity (may need to add nodes in healthy AZs). Verify ALB health checks are marking affected targets unhealthy. Check for services with AZ affinity that won't auto-recover.
4. Data services: Verify RDS failover completed. Check ElastiCache cluster mode (does it survive AZ loss?). Verify Kafka replicas in other AZs have caught up. Check EBS volumes - they are AZ-bound and inaccessible until AZ recovers.
5. Communication: Update status page. Inform customers of potential degradation. If recovery exceeds RTO, activate full DR plan (cross-region failover). Post-incident: review AZ distribution of all resources.
! Scenario: etcd Corruption Recovery

Symptom: kube-apiserver returning errors. etcd logs showing "database space exceeded" or corruption errors. kubectl commands failing. New pods not scheduling.

1. Immediate triage: Check etcd member health: etcdctl endpoint health. Check cluster status: etcdctl endpoint status. If one member corrupt but others healthy, remove and re-add the member.
2. If space exceeded: Compact etcd history: etcdctl compact $(etcdctl endpoint status -w json | jq '.[] .Status.header.revision'). Then defragment: etcdctl defrag. Increase --quota-backend-bytes.
3. If data corrupted: Stop kube-apiserver (move manifest). Identify latest valid snapshot. Restore from snapshot to new data directory. Update etcd manifest to use new data directory. Restart.
4. Validate recovery: Check all nodes rejoin: kubectl get nodes. Verify workloads: kubectl get pods --all-namespaces. Check CRDs and custom resources are intact. Verify RBAC still working.
5. Prevention: Regular etcd snapshots (every 30 min). Monitor etcd DB size and set alerts at 70% quota. Set appropriate --auto-compaction-retention. Run etcd on fast local SSD (not network-attached storage).
! Scenario: Accidental Namespace Deletion (Velero Restore)

Symptom: Someone ran kubectl delete namespace production. All deployments, services, configmaps, secrets, PVCs in the namespace are gone. Application completely down.

1. Confirm the damage: kubectl get namespace production returns NotFound. Check if any PVs have Retain reclaim policy (data may still exist on disk). Check Velero backups: velero backup get.
2. Find the right backup: velero backup describe daily-production-backup-YYYYMMDD --details. Verify it contains the production namespace. Check backup timestamp vs deletion time.
3. Restore namespace: velero restore create emergency-restore --from-backup daily-production-backup-20240615 --include-namespaces production --wait. This recreates the namespace and all resources.
4. Restore PV data: If PVs had Delete reclaim policy, data is lost. Velero's file-level backup (Kopia) can restore PV contents. Check: velero restore describe emergency-restore --details for PV restore status.
5. Validate and prevent: Verify all pods running, services accessible, data intact. Prevention: RBAC (restrict namespace deletion), ResourceQuota with finalizers, Velero scheduled backups, kubectl-safe plugin that confirms destructive commands, admission webhook that blocks namespace deletion in production.
! Scenario: RDS Failover During Deployment

Symptom: During a deployment that includes RDS instance class change, Multi-AZ failover triggered. Application experiencing connection errors. Some transactions lost. Deployment partially complete.

1. Assess RDS state: aws rds describe-db-instances - check status (modifying, available, failed). Check Events tab for failover confirmation. Verify new primary is accessible.
2. Application recovery: Connection pools may be holding stale connections to old primary. Restart application pods or trigger connection pool refresh. Verify application can connect to RDS endpoint (DNS should have updated).
3. Data integrity: Check for in-flight transactions that were lost during failover. Multi-AZ is synchronous so committed transactions are safe. Any uncommitted transactions on old primary are lost. Check application logs for transaction errors.
4. Deployment state: Check if the instance modification completed successfully after failover. If instance class change was the trigger, verify the new instance class is active. Determine if deployment pipeline needs to be re-run or if it can continue.
5. Prevention: Schedule RDS modifications during maintenance windows. Use apply-immediately=false for non-urgent changes. Ensure application has connection retry logic with exponential backoff. Set connection timeout and validation query in connection pool. Test failover behavior before production deployments.
