  1. Teams need to collaborate (finance + engineering + business)
  2. Everyone takes ownership for their cloud usage
  3. A centralized team drives FinOps best practices
  4. Reports should be accessible and timely
  5. Decisions are driven by business value of cloud
  6. Take advantage of the variable cost model of cloud

Culture of Cost Awareness

  • Shift-left cost: Engineers consider cost during design, not after the bill arrives
  • Unit economics: Measure cost per transaction/user/request, not just total spend
  • Cost as a metric: Display cost alongside performance metrics in dashboards
  • Team accountability: Each team owns their spend (tagged resources, namespace cost)
  • Gamification: Monthly cost reduction challenges, leaderboards for efficiency
  • Architecture reviews: Include cost analysis in design documents

Right-Sizing Strategy

  • AWS Compute Optimizer: Analyzes 14 days of CloudWatch metrics, recommends instance type changes
  • CloudWatch metrics to check: CPU average < 40%, memory < 60% (over-provisioned), CPU > 80% (under-provisioned)
  • Right-size before committing: Always right-size FIRST, then buy Reserved/Savings Plans for the optimized size
  • Graviton (ARM): 20-40% better price/performance for many workloads. Test compatibility first
  • Scheduled scaling: Scale down dev/staging environments outside business hours (save 60%+ on non-prod)

Data Transfer Cost (Often Overlooked)

  • Inbound: Free (data INTO AWS)
  • Outbound to internet: $0.09/GB (first 10TB/month) - use CloudFront ($0.085/GB with free tier)
  • Cross-AZ: $0.01/GB each direction ($0.02 round-trip) - significant for chatty microservices
  • Cross-Region: $0.02/GB - minimize cross-region data replication where possible
  • NAT Gateway: $0.045/GB processed + $0.045/hr - often the biggest surprise on bills
  • VPC endpoints: $0.01/GB (vs $0.045 for NAT) - use for S3, DynamoDB, ECR, etc.

  HIDDEN COSTS:
  1. Over-provisioned requests: Pod A requests 4 CPU but uses 0.5
     -> 3.5 CPU reserved but wasted (node cannot schedule other pods)
  2. Idle nodes: Node 3 has no pods but still costs $$$
     -> Cluster Autoscaler not configured or min_nodes too high
  3. Orphaned PVs: PVC deleted but PV retained (reclaimPolicy: Retain)
     -> 100GB EBS volume billing with no user
  4. Oversized nodes: Using m5.2xlarge when m5.xlarge would suffice
  5. No spot: All on-demand when workloads are fault-tolerant

Kubecost Overview

  • What: Open-source tool that provides real-time cost monitoring and allocation for K8s
  • Cost allocation: Breaks down cost by namespace, deployment, label, team, or custom grouping
  • Idle cost: Shows resources reserved but unused (the gap between requests and actual usage)
  • Recommendations: Suggests right-sized resource requests/limits based on actual usage
  • Efficiency score: Ratio of resources used vs. resources requested (target: 60-80%)
  • Network cost: Tracks cross-zone and internet egress cost per pod/service
  • Integration: Works with Prometheus metrics, can export to custom dashboards

4. Spot Instances

How Spot Instances Work

  • Pricing: Spare EC2 capacity at 60-90% discount. Price fluctuates based on supply/demand
  • Interruption: AWS can reclaim with 2-minute warning (via instance metadata or EventBridge)
  • Capacity pools: Each instance type + AZ is a separate pool with its own availability
  • No bidding (anymore): You pay the current spot price (max price = on-demand by default)
  • Rebalancing: New signal: "rebalance recommendation" gives earlier notice than 2-min interruption

Spot Best Practices

  • Diversify instance types: Use 10+ instance types across 3+ AZs to minimize interruption probability
  • Capacity-optimized allocation: AWS picks pools with most available capacity (reduces interruption)
  • Graceful shutdown: Handle SIGTERM, drain connections, checkpoint state within 2 minutes
  • Mix with on-demand: Run baseline on on-demand/reserved, burst with spot (e.g., 30% OD + 70% spot)
  • Karpenter: Automatically diversifies instance types and handles interruption/replacement
  • Workload suitability: Stateless services, batch processing, CI/CD workers, dev/test environments

5. Right-Sizing

VPA (Vertical Pod Autoscaler) Recommendations

  • Mode: Off - Only provides recommendations, does not change resources (safe for production)
  • Mode: Auto - Automatically updates resource requests (restarts pods - use with caution)
  • How it works: Observes actual CPU/memory usage over time, recommends optimal requests
  • Limitations: Cannot run simultaneously with HPA on CPU (conflicting decisions)
  • Target utilization: VPA targets ~90th percentile of actual usage as the recommended request
VPA Configuration K8s 
apiVersion: autoscaling.k8s.io/v1
kind: VerticalPodAutoscaler
metadata:
  name: my-app-vpa
  namespace: production
spec:
  targetRef:
    apiVersion: "apps/v1"
    kind: Deployment
    name: my-app
  updatePolicy:
    updateMode: "Off"   # Recommendation only (safest)
  resourcePolicy:
    containerPolicies:
      - containerName: "my-app"
        minAllowed:
          cpu: 50m
          memory: 64Mi
        maxAllowed:
          cpu: 2000m
          memory: 4Gi
        controlledResources: ["cpu", "memory"]
        controlledValues: RequestsOnly
---

Goldilocks Tool

  • What: Creates VPA objects for all deployments in a namespace, presents recommendations in a dashboard
  • Install: helm install goldilocks fairwinds-stable/goldilocks -n goldilocks
  • Enable per namespace: kubectl label ns production goldilocks.fairwinds.com/enabled=true
  • Dashboard: Shows current requests/limits vs recommended for every container
  • Savings estimate: Shows how much you would save by applying recommendations

CPU Throttling vs Over-Provisioning Trade-Off

  • Throttling: When a container hits its CPU limit, the kernel throttles it (CFS quota). Causes latency spikes
  • Detection: rate(container_cpu_cfs_throttled_periods_total[5m]) / rate(container_cpu_cfs_periods_total[5m]) > 0.25
  • Strategy 1 - No CPU limits: Set requests only (for scheduling), remove limits. Pods can burst. Risk: noisy neighbor
  • Strategy 2 - High limits: Set limits to 2-4x request. Allows burst without throttling in most cases
  • Strategy 3 - Burstable with monitoring: Remove limits but alert on sustained high CPU that could affect neighbors
  • Memory: ALWAYS set memory limits (OOM kill is better than node instability from memory pressure)

# ResourceQuota to cap total namespace usage
apiVersion: v1
kind: ResourceQuota
metadata:
  name: namespace-quota
  namespace: development
spec:
  hard:
    requests.cpu: "20"
    requests.memory: 40Gi
    limits.cpu: "40"
    limits.memory: 80Gi
    persistentvolumeclaims: "10"
    pods: "50"
1
Set requests based on P95 actual usage (from VPA/monitoring)

If your container typically uses 200m CPU, set request to 250m (not 1000m). This allows the scheduler to pack more pods per node.

2
Consider removing CPU limits for non-noisy-neighbor environments

CPU throttling causes latency. If cluster has good namespace isolation or dedicated node pools, CPU limits may cause more harm than good.

3
Always set memory limits (slightly above request)

Memory is incompressible - OOM kill is the only recourse. Set limit to 1.2-1.5x request to allow burst without risking node stability.

Common Storage Waste

  • Unused EBS volumes: Volumes detached from terminated instances still incur cost. Script to find: aws ec2 describe-volumes --filters "Name=status,Values=available"
  • Over-sized volumes: Provisioned 100GB but using 10GB. EBS charges for provisioned size, not used space
  • Old snapshots: Accumulate over time. Use lifecycle policies: aws dlm create-lifecycle-policy
  • gp2 -> gp3 migration: gp3 is 20% cheaper AND has better baseline performance (3000 IOPS vs 100 for small gp2). No downtime to migrate
  • Orphaned PVs in K8s: PVC deleted but PV retained (persistentVolumeReclaimPolicy: Retain). Script to audit: compare PV list with PVC bindings
aws ec2 describe-volumes --filters "Name=status,Values=available" --query "Volumes[*].{ID:VolumeId,Size:Size,Created:CreateTime}" --output table

S3 Cost Optimization

  • Intelligent-Tiering: Automatically moves objects between access tiers. Best when access patterns are unknown. Small monitoring fee per object
  • Lifecycle rules: Transition objects to cheaper classes based on age (e.g., IA after 30 days, Glacier after 90)
  • Multipart upload cleanup: Incomplete multipart uploads accumulate. Add lifecycle rule to abort after 7 days
  • Object versioning cost: Each version is a full copy. Use lifecycle rules to expire old versions after N days
  • Compression: Compress before upload (gzip logs before S3). Reduces storage AND transfer cost

NAT Gateway - The Silent Cost Killer

  • Hourly charge: $0.045/hr per NAT Gateway ($32/month per AZ just for existing)
  • Data processing: $0.045/GB through NAT - adds up fast with container image pulls, apt updates, external API calls
  • Common surprise: 3 AZs x $32/month + 1TB/month processing = $141/month just for NAT (before any useful traffic)
  • Mitigation:
    • VPC endpoints for AWS services (S3, ECR, DynamoDB, STS, SSM)
    • ECR pull-through cache (reduces internet pulls)
    • Private subnets only where needed (not every service needs internet egress)
    • Consider NAT instances for dev environments (cheaper, less reliable)

10. Cost Monitoring

AWS Cost Tools

  • Cost Explorer: Visualize, understand, and manage costs. Filter by service, tag, account. Forecast future spend
  • AWS Budgets: Set custom budgets with alerts at 50%, 80%, 100% thresholds. Alert via email/SNS/Slack
  • Cost Anomaly Detection: ML-based unusual spend detection. Alerts on unexpected spikes per service/account
  • Compute Optimizer: Recommends optimal instance types based on utilization metrics
  • Trusted Advisor: Identifies idle resources, underutilized instances, unused EBS/EIPs
  • CUR (Cost & Usage Report): Detailed line-item billing data to S3. Use Athena to query

K8s Cost Monitoring

  • Kubecost: Real-time cost per namespace/deployment/pod. Efficiency scoring, recommendations
  • OpenCost: CNCF project (open-source Kubecost core). Cost allocation API
  • Prometheus metrics:
    • kube_pod_container_resource_requests (what is reserved)
    • container_cpu_usage_seconds_total (what is used)
    • Gap between these = waste
  • Custom dashboard: Cost per namespace, idle resources, request vs actual efficiency

11. Interview Q&A

Q: Compare Spot vs Reserved Instances vs On-Demand. When would you use each?
A: On-Demand: Pay per hour, no commitment. Use for spiky/unpredictable workloads, initial development, anything you cannot predict. Reserved/Savings Plans: 1-3 year commitment for 30-72% discount. Use for steady-state baseline workloads (databases, core services that run 24/7). Buy AFTER right-sizing. Compute Savings Plans are most flexible (covers EC2+Fargate+Lambda). Spot: 60-90% discount but can be interrupted with 2-min notice. Use for fault-tolerant workloads: batch processing, CI/CD, stateless web workers, data pipelines. Optimal strategy: Cover baseline with Savings Plans (70-80% of steady usage), use On-Demand for predicted peaks, use Spot for fault-tolerant burst capacity.
Q: What are your top Kubernetes cost optimization techniques?
A: (1) Right-size requests: Use VPA recommendations to set requests based on actual usage (biggest single saving - often 40-60% over-provisioned). (2) Cluster Autoscaler / Karpenter: Scale nodes down when pods don't need them. Set appropriate scale-down thresholds. (3) Spot instances: Run 50-70% of fault-tolerant workloads on spot for 60-90% savings. (4) Namespace resource quotas: Prevent teams from over-requesting. (5) Schedule non-prod shutdown: Scale dev/staging to 0 replicas outside business hours (save ~65%). (6) Cleanup orphans: Script to find unused PVCs, old ConfigMaps/Secrets, completed Jobs. (7) Bin-packing: Choose node sizes that match workload patterns; avoid small pods on large nodes.
Q: How do you approach right-sizing an existing cluster?
A: (1) Measure first: Deploy VPA in recommendation-only mode + Kubecost for 2+ weeks to gather actual usage data. (2) Identify waste: Look for pods where request >> actual (e.g., 1 CPU requested, 100m used). Sort by absolute waste (biggest savings first). (3) Start with non-critical: Apply right-sizing to dev/staging first, validate no performance impact. (4) Gradual production rollout: Reduce requests by 20-30% at a time, monitor for throttling or OOM. (5) Verify with load testing: Ensure right-sized values handle peak traffic (not just average). (6) Automate: Set up Goldilocks dashboard, regular review cadence (monthly), alert on efficiency score dropping. (7) Then commit: Once stable, buy Savings Plans for the optimized capacity.
Q: How can you reduce data transfer costs in AWS?
A: (1) VPC Endpoints: Gateway endpoints (S3, DynamoDB) are free and eliminate NAT charges. Interface endpoints cost less than NAT for high-volume services (ECR, CloudWatch). (2) Same-AZ traffic: Use topology-aware routing in K8s (topology.kubernetes.io/zone) to keep pod-to-pod traffic within AZ ($0 vs $0.02 cross-AZ). (3) CloudFront: Serve static assets via CDN - cheaper than direct egress + reduces origin load. (4) Compression: Compress API responses, log streams, data in transit. (5) NAT reduction: Use ECR pull-through cache, S3 gateway endpoint, private link for frequently-accessed services. (6) Architecture: Co-locate communicating services in same AZ, use multi-region only where needed.
Q: What tagging strategy would you implement for a growing organization?
A: (1) Mandatory tags: Enforce team, environment, project, cost-center via SCP (prevent resource creation without them). (2) Enforcement layers: SCP for hard block, AWS Config for compliance reporting, Terraform variables for IaC resources. (3) Automation: Tag inheritance (from CloudFormation/Terraform), auto-tagging Lambda for resources that slip through. (4) K8s mapping: Map K8s namespace labels to AWS tags via Kubecost or custom tooling. (5) Governance: Monthly compliance report showing untagged resources, escalation process for teams not meeting tag requirements. (6) Evolution: Start simple (3-4 tags), add more as FinOps matures. Don't over-engineer initially.
Q: How do you build a FinOps culture in engineering teams?
A: (1) Visibility first: Show teams their spend weekly (Slack bot with cost summary, Grafana dashboard). People cannot optimize what they cannot see. (2) Make it easy: Provide self-service tools (Kubecost dashboard, right-sizing recommendations), not just reports. (3) Ownership: Each team owns their namespace/account budget. Include cost efficiency in sprint reviews. (4) Incentives: Celebrate cost savings in all-hands, gamify with cost-reduction challenges. (5) No blame: Frame as optimization opportunity, not punishment. Everyone over-provisions initially. (6) Architecture reviews: Include cost estimate in design docs ("this design costs ~$X/month"). (7) FinOps champion: Embedded engineer per team who drives cost awareness and training.

12. Troubleshooting Scenarios

!! Scenario: Monthly Bill Spike (30% Increase)

Symptoms: AWS bill jumped from $50K to $65K this month with no known infrastructure changes.

1. Open Cost Explorer: Filter by service, compare this month vs last month. Identify which service(s) increased
2. Drill down by tag: Use team and project tags to identify which team's resources grew
3. Check Cost Anomaly Detection: See if it flagged the spike and which resources caused it
4. Common causes: (a) Data transfer spike (video/logs), (b) New service deployed without cost review, (c) Auto-scaling over-reacted, (d) Forgotten resources from testing/experiments, (e) Spot interruption caused fallback to on-demand
5. Check for untagged resources: CUR query for resources without team tag in the spike period
6. Check NAT Gateway / Data Transfer: Often overlooked. Filter Cost Explorer by "EC2-Other" for data transfer line items
7. Immediate actions: Terminate idle resources, scale down over-provisioned services, set budget alerts at lower thresholds
8. Prevention: Set forecasted budget alerts (alert when forecast exceeds budget), weekly cost review meetings, anomaly detection with Slack notification
!! Scenario: Idle Cluster Resources

Symptoms: Kubecost shows 60% cluster idle cost. Nodes are running but pods are using fraction of allocated resources.

1. Identify idle: Check Kubecost "idle" metric. This is the gap between allocatable node resources and pod requests
2. Check Cluster Autoscaler: Is it configured? Are there scale-down delays too long? Check --scale-down-utilization-threshold (default 0.5)
3. Check for pods preventing scale-down: PodDisruptionBudgets, pods without controllers, pods with local storage, kube-system pods
4. Check minimum node count: minSize in node group may be too high for actual workload
5. Right-size pod requests: If pods request 1 CPU but use 100m, that reserved 900m blocks node scale-down
6. Fix: (a) Lower minSize to 1-2, (b) Right-size pod requests (VPA), (c) Use Karpenter for better bin-packing, (d) Set scale-down-delay to 5m not 30m, (e) Consolidate workloads to fewer, fuller nodes
7. Target: 60-80% node utilization is healthy. Below 40% means significant waste
!! Scenario: Over-Provisioned Dev Environments

Symptoms: Dev and staging environments running 24/7 identical to production. Costing 40% of total cloud spend for non-production use.

1. Audit: Compare prod vs non-prod spend in Cost Explorer (filter by environment tag)
2. Schedule shutdown: Use AWS Instance Scheduler or CronJobs to scale to 0 outside business hours (6pm-8am, weekends). Saves ~65%
3. Reduce replicas: Dev does not need 3 replicas or multi-AZ. Single replica, single AZ for dev
4. Smaller instances: Dev can use t3.medium where prod uses m5.2xlarge. Use separate node pools per environment
5. Spot for non-prod: Run all dev/staging on 100% spot instances (interruption is acceptable)
6. Ephemeral environments: Replace persistent dev environments with on-demand preview environments that spin up per-PR and tear down after merge
7. Resource quotas: Set namespace ResourceQuotas for dev (much lower than prod). Enforce with LimitRange
8. Target: Non-production should be <20% of total spend (ideally 10-15%)
!! Scenario: NAT Gateway Cost Surprise ($3K/month)

Symptoms: Monthly NAT Gateway charges unexpectedly high. "EC2-Other" line in Cost Explorer shows $3K in data processing fees.

1. Enable VPC Flow Logs: Identify which private subnet IPs are sending the most traffic through NAT
2. Check top talkers: Filter flow logs by NAT Gateway ENI. Group by source IP + destination to find heaviest flows
3. Common culprits: (a) ECR image pulls (large images, many nodes, frequent deployments), (b) S3 access without gateway endpoint, (c) External API calls from every pod, (d) Logging/metrics to external SaaS (Datadog, Splunk)
4. Fix - ECR: Add VPC endpoint for ECR (dkr + api). Use ECR pull-through cache. Reduce image sizes
5. Fix - S3: Add S3 Gateway endpoint (FREE, takes 30 seconds to create). Route table update, no code changes
6. Fix - External APIs: Cache responses where possible. Consider moving integration to services with direct internet access only where needed
7. Monitor: Alert on NATGateway BytesOutToDestination CloudWatch metric exceeding threshold. Set up weekly cost review for network line items
