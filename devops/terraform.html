Terraform Deep Dive

Infrastructure as Code - Declarative, version-controlled, collaborative infrastructure management

1. Core Concepts

What is Terraform?

Terraform is a declarative Infrastructure as Code (IaC) tool by HashiCorp. You define the desired state of your infrastructure in HCL (HashiCorp Configuration Language), and Terraform figures out what actions to take to reach that state.

Declarative vs Imperative

  • Declarative (Terraform): "I want 3 servers" - Terraform creates, removes, or modifies to reach 3
  • Imperative (scripts): "Create a server, then another, then another" - you manage the how

Declarative is idempotent: running it multiple times produces the same result.

Key Terminology

  • Provider: Plugin for interacting with APIs (AWS, Azure, GCP, K8s)
  • Resource: Infrastructure object to manage (EC2 instance, S3 bucket)
  • Data Source: Read-only query for existing infrastructure
  • State: Terraform's record of managed infrastructure
  • Module: Reusable container for multiple resources
  • Plan: Preview of changes before applying

2. Terraform Workflow

Terraform Core Workflow
  Write Code        Initialize         Plan              Apply            Destroy
 +-----------+    +-----------+    +-----------+    +-----------+    +-----------+
 |  .tf      |    | terraform |    | terraform |    | terraform |    | terraform |
 |  files    |--->|   init    |--->|   plan    |--->|   apply   |--->|  destroy  |
 |           |    |           |    |           |    |           |    |           |
 +-----------+    +-----------+    +-----------+    +-----------+    +-----------+
      |                |                |                |                |
      v                v                v                v                v
  Define HCL      Download         Compare state    Execute changes   Remove all
  resources       providers &      vs config,       via provider      managed
                  init backend     show diff         API calls        resources

  +------------------+         +-----------------+        +------------------+
  |  .terraform/     |         |  terraform.     |        |  terraform.      |
  |  (providers,     |         |  tfstate        |        |  tfstate         |
  |   modules)       |         |  (current)      |        |  (empty/removed) |
  +------------------+         +-----------------+        +------------------+

Commands

terraform init

Downloads providers, initializes backend, downloads modules. Run after adding new providers or modules.

terraform plan -out=tfplan

Shows what will change. Compares desired state (HCL) with current state (tfstate). Save plan for deterministic apply.

terraform apply tfplan

Executes the saved plan. Without a saved plan, runs plan interactively first.

terraform destroy

Destroys all managed resources. Use -target for selective destruction.

terraform fmt -recursive

Formats all .tf files to canonical style. Run in CI to enforce formatting.

terraform validate

Validates configuration syntax and internal consistency (does not access remote state or APIs).

3. State Management

Why State Matters

Terraform state is the source of truth for what Terraform manages. It maps real-world resources to your configuration, tracks metadata, and enables Terraform to determine what changes are needed.

  • Without state, Terraform cannot know which resources it created
  • State contains sensitive data (passwords, keys) - must be secured
  • State must be shared for team collaboration
  • State must be locked to prevent concurrent modifications

Remote State (S3 + DynamoDB)

terraform {
  backend "s3" {
    bucket         = "mycompany-terraform-state"
    key            = "production/networking/terraform.tfstate"
    region         = "us-east-1"
    encrypt        = true
    dynamodb_table = "terraform-state-lock"
    # Optional: use assume_role for cross-account access
    # role_arn     = "arn:aws:iam::123456789012:role/TerraformStateAccess"
  }
}

State Locking

How State Locking Works

When Terraform performs an operation that could modify state, it acquires a lock. This prevents concurrent operations from corrupting state.

  • S3 backend: Uses DynamoDB for locking (LockID = state file path)
  • GCS backend: Uses built-in object locking
  • Terraform Cloud: Built-in state locking per workspace
  • Force unlock: terraform force-unlock LOCK_ID (use with caution)

State Commands

terraform state list

List all resources in state.

terraform state show aws_instance.web

Show attributes of a specific resource in state.

terraform state mv aws_instance.old aws_instance.new

Rename a resource in state without destroying/recreating it.

terraform state rm aws_instance.legacy

Remove a resource from state (Terraform forgets about it, does not destroy it).

terraform import aws_instance.web i-1234567890abcdef0

Import an existing resource into Terraform state.

terraform state pull > backup.tfstate

Download remote state to a local file (for backup or inspection).

4. Modules

What is a Module?

A module is a container for multiple resources that are used together. Every Terraform configuration is a module (the root module). Child modules allow you to organize, encapsulate, and reuse infrastructure code.

count vs for_each

Feature count for_each
Input type Number (integer) Map or set of strings
Reference resource.name[0], resource.name[1] resource.name["key"]
Index type Numeric index String key
Removal behavior Removing middle item shifts all subsequent indices (recreates resources) Removing item only affects that key (no shift)
Best for Identical resources, conditional creation (count = 0 or 1) Resources with unique identifiers, maps of config
Conditional count = var.create ? 1 : 0 Use empty map: for_each = var.create ? toset(["this"]) : toset([])
count-vs-for-each.tf HCL - Comparison 
# ---- COUNT ----
# Problem: removing "staging" from the list recreates "production" (index shift)
variable "environments_list" {
  default = ["development", "staging", "production"]
}

resource "aws_s3_bucket" "with_count" {
  count  = length(var.environments_list)
  bucket = "myapp-${var.environments_list[count.index]}-assets"
}

# Conditional creation with count
resource "aws_cloudwatch_log_group" "optional" {
  count = var.enable_logging ? 1 : 0
  name  = "/app/${var.environment}"
}

# ---- FOR_EACH ----
# Better: removing "staging" only destroys that specific bucket
variable "environments_map" {
  default = {
    development = { region = "us-east-1", tier = "small" }
    staging     = { region = "us-east-1", tier = "medium" }
    production  = { region = "us-east-1", tier = "large" }
  }
}

resource "aws_s3_bucket" "with_for_each" {
  for_each = var.environments_map
  bucket   = "myapp-${each.key}-assets"

  tags = {
    Environment = each.key
    Tier        = each.value.tier
  }
}

# for_each with set of strings
resource "aws_iam_user" "developers" {
  for_each = toset(["alice", "bob", "carol"])
  name     = each.value
}

Lifecycle Rules

lifecycle.tf HCL - Lifecycle 
resource "aws_instance" "web" {
  ami           = data.aws_ami.ubuntu.id
  instance_type = var.instance_type

  lifecycle {
    # Create new instance before destroying old one (zero downtime)
    create_before_destroy = true

    # Prevent accidental destruction (must remove this block first)
    prevent_destroy = true

    # Ignore external changes to these attributes
    # (e.g., ASG modifies tags, or manual console changes)
    ignore_changes = [
      tags["LastModifiedBy"],
      ami,  # Don't recreate when new AMI is released
    ]

    # Replace when any of these change (force recreation)
    replace_triggered_by = [
      aws_security_group.web.id
    ]
  }
}

# Precondition and postcondition (Terraform 1.2+)
resource "aws_instance" "database" {
  ami           = var.ami_id
  instance_type = var.instance_type

  lifecycle {
    precondition {
      condition     = var.instance_type != "t3.micro"
      error_message = "Database instances must not use t3.micro."
    }

    postcondition {
      condition     = self.public_ip == null || !var.is_private
      error_message = "Database instance must not have a public IP in private mode."
    }
  }
}

depends_on (Implicit vs Explicit)

dependencies.tf HCL 
# IMPLICIT dependency: Terraform auto-detects via references
resource "aws_subnet" "main" {
  vpc_id     = aws_vpc.main.id  # <-- implicit dependency on aws_vpc.main
  cidr_block = "10.0.1.0/24"
}

# EXPLICIT dependency: when there is no reference but order matters
resource "aws_instance" "app" {
  ami           = data.aws_ami.ubuntu.id
  instance_type = "t3.micro"
  subnet_id     = aws_subnet.main.id

  # This instance needs the IAM role policy to be attached first,
  # but there's no direct reference to it in this resource
  depends_on = [
    aws_iam_role_policy_attachment.app_policy
  ]
}

# Use depends_on sparingly - prefer implicit dependencies via references
# Explicit depends_on creates hidden coupling and makes refactoring harder

Provisioners (and Why to Avoid Them)

Why Avoid Provisioners?

  • They break the declarative model - Terraform cannot track what they do
  • They add failure modes that Terraform cannot recover from
  • They run only on create/destroy, not on update
  • Better alternatives: user_data, cloud-init, Packer, Ansible, configuration management

HashiCorp official position: Provisioners are a last resort.

provisioners.tf HCL - Last Resort 
resource "aws_instance" "web" {
  ami           = data.aws_ami.ubuntu.id
  instance_type = "t3.micro"

  # PREFERRED: Use user_data instead of provisioners
  user_data = templatefile("${path.module}/init.sh", {
    environment = var.environment
  })

  # LAST RESORT: remote-exec provisioner
  provisioner "remote-exec" {
    inline = [
      "sudo apt-get update",
      "sudo apt-get install -y nginx",
    ]

    connection {
      type        = "ssh"
      user        = "ubuntu"
      private_key = file("~/.ssh/id_rsa")
      host        = self.public_ip
    }
  }

  # local-exec: runs on the machine running Terraform
  provisioner "local-exec" {
    command = "echo ${self.private_ip} >> inventory.txt"
  }

  # Destroy-time provisioner
  provisioner "local-exec" {
    when    = destroy
    command = "echo 'Instance ${self.id} is being destroyed' >> destroy.log"
  }
}

6. Workspaces

What are Workspaces?

Workspaces allow you to maintain multiple state files for the same configuration. Each workspace has its own state, enabling environment separation with a single codebase.

terraform workspace list
terraform workspace new staging
terraform workspace select production
terraform workspace show
Using workspace in config HCL 
locals {
  environment = terraform.workspace

  instance_type = {
    development = "t3.micro"
    staging     = "t3.small"
    production  = "m5.large"
  }
}

resource "aws_instance" "app" {
  instance_type = local.instance_type[terraform.workspace]

  tags = {
    Environment = terraform.workspace
  }
}

When to Use Workspaces

  • Quick prototyping of environments
  • Testing changes in isolation
  • Same infrastructure, different scale
  • Feature branch environments

When NOT to Use Workspaces

  • Environments with different providers or backends
  • Environments with significantly different resources
  • When teams need different access per environment
  • Alternative: Separate directories/repos per environment with shared modules

7. Drift Detection

What is Drift?

Drift occurs when the actual state of infrastructure diverges from the state Terraform expects. This happens when changes are made outside of Terraform (console, CLI, another tool).

How Terraform Detects Drift

1
terraform plan / terraform refresh
Terraform reads the real infrastructure state from the cloud API and compares it to the state file. Any differences appear in the plan output.
2
Plan Output Shows Drift
Changes made outside Terraform appear as "drift" - Terraform will propose to revert them to match your config.
3
Scheduled Drift Detection
Run terraform plan on a schedule (CI/CD) to detect drift early. Alert if plan shows unexpected changes.

Remediation Strategies

StrategyWhen to UseCommand
Revert driftExternal change was unauthorizedterraform apply (reverts to config)
Accept driftExternal change was intentionalUpdate .tf config to match, then terraform plan shows no changes
Ignore driftAttribute managed externallyAdd to lifecycle { ignore_changes = [...] }
ImportResource created outside TFterraform import resource.name id

State File Organization Patterns

  • By environment and component: env/production/networking/terraform.tfstate
  • By team: team-platform/production/terraform.tfstate
  • By region: us-east-1/production/terraform.tfstate

Rule of thumb: Split state when teams need independent deployment cycles or when blast radius of a mistake is too large.

Naming Conventions

  • Resources: Use underscores, descriptive names: aws_instance.web_server
  • Variables: Lowercase with underscores: var.instance_type
  • Outputs: Match the attribute name: output "vpc_id"
  • Modules: Short, descriptive: module "vpc", module "eks_cluster"
  • Files: main.tf, variables.tf, outputs.tf, versions.tf, data.tf
  • Tags: Consistent across all resources: Environment, Project, ManagedBy, Owner

11. Interview Q&A - Terraform

Q: What is Terraform state and why is it important?
Terraform state is a JSON file that maps your configuration to real-world resources. It stores resource IDs, attributes, and metadata. It's important because: (1) Terraform uses it to determine what changes are needed (plan), (2) it tracks dependencies between resources, (3) it improves performance by caching resource attributes, (4) it enables team collaboration when stored remotely. Without state, Terraform cannot know which resources it manages. State should be stored remotely (S3, GCS, Terraform Cloud) with encryption and locking enabled.
Q: What is the difference between count and for_each?
count takes a number and creates that many instances, referenced by numeric index (e.g., [0], [1]). for_each takes a map or set and creates one instance per element, referenced by key (e.g., ["web"], ["api"]). The critical difference: with count, removing an item from the middle shifts all indices, causing unnecessary recreation of resources. With for_each, removing an item only affects that specific key. Best practice: Use for_each for resources with unique identifiers; use count only for conditional creation (count = var.enabled ? 1 : 0).
Q: How do you handle secrets in Terraform?
Multiple layers of protection: (1) Never hardcode secrets in .tf files or commit .tfvars with secrets. (2) Mark variables as sensitive: sensitive = true hides values in plan output. (3) Use secret managers: reference AWS Secrets Manager, HashiCorp Vault, or SSM Parameter Store via data sources. (4) Encrypt state: state file contains sensitive values, so encrypt at rest (S3 SSE-KMS). (5) Environment variables: pass via TF_VAR_ prefix in CI/CD. (6) Restrict state access: IAM policies limiting who can read state. State is the weakest link - it always contains the actual values.
Q: What is the difference between terraform plan and terraform apply?
Plan is a dry run that shows what Terraform would do without making changes. It reads current state, compares with config, and outputs an execution plan showing creates, updates, and destroys. Apply executes those changes against real infrastructure. Best practice: always run plan first, save it (-out=tfplan), review, then apply the saved plan (terraform apply tfplan). This ensures what you reviewed is exactly what gets applied, preventing race conditions between plan and apply.
Q: What are module best practices?
Key practices: (1) Keep modules focused - one logical component per module (VPC, EKS cluster, RDS). (2) Version modules - use git tags or registry versions, never reference main branch. (3) Define clear interfaces - input variables with descriptions, validations, and sensible defaults; outputs for all values other modules need. (4) Don't nest too deeply - 2-3 levels max. (5) Document with examples. (6) Use consistent naming - main.tf, variables.tf, outputs.tf. (7) Test modules with Terratest or terraform test. (8) Avoid hardcoding - make modules configurable for reuse.
Q: How does state locking work and why is it important?
State locking prevents concurrent operations from corrupting state. When Terraform starts an operation (plan/apply), it acquires a lock on the state file. If another operation tries to run simultaneously, it fails with a lock error. For S3 backend, locking uses a DynamoDB table with a LockID key matching the state file path. It's important because: without locking, two people applying at the same time could write conflicting state, leading to duplicated or orphaned resources. If a lock gets stuck (crash during apply), use terraform force-unlock LOCK_ID carefully.
Q: How do you handle drift in Terraform?
Drift detection and remediation: (1) Detect: run terraform plan regularly (scheduled CI job). Any output means drift exists. (2) Revert drift: if unauthorized, terraform apply forces infrastructure back to match config. (3) Accept drift: if intentional, update .tf files to match the new reality. (4) Ignore drift: for attributes managed externally, use lifecycle { ignore_changes = [attribute] }. (5) Prevent drift: restrict console access, use Terraform for all changes, enable AWS Config rules to detect manual changes. The key insight: plan is your drift detection tool.
Q: Compare Terraform vs CloudFormation vs Pulumi.
Terraform: Multi-cloud, HCL language, large provider ecosystem, state management required, open-source core. CloudFormation: AWS-only, JSON/YAML, deeply integrated with AWS (no state file - AWS manages it), free, supports drift detection natively. Pulumi: Multi-cloud, uses real programming languages (Python, TypeScript, Go), built-in state management, supports testing with standard tools. Key trade-offs: Terraform wins on multi-cloud and ecosystem size; CloudFormation wins on AWS-native integration and zero state management; Pulumi wins on developer experience for teams preferring general-purpose languages over DSLs.
Q: How do you import existing resources into Terraform?
Two approaches: (1) terraform import (traditional): Write the resource block in .tf, then run terraform import aws_instance.web i-1234567890. This adds the resource to state but you must manually write matching config. (2) Import blocks (Terraform 1.5+): Add an import { to = aws_instance.web; id = "i-1234567890" } block, then run terraform plan -generate-config-out=generated.tf to auto-generate the config. The second approach is better for bulk imports and generates matching HCL automatically. After import, run plan to verify no changes are shown.
Q: When should you use workspaces vs separate state files?
Workspaces are good when: environments share the same configuration structure, same providers, same backend, and differ only in variable values (e.g., instance size, count). Separate state files (separate directories) are better when: environments differ significantly in resources, require different IAM permissions, use different providers/regions, or when teams need independent deployment cycles. Most production setups prefer separate directories with shared modules because it provides clearer separation of concerns, independent state, and easier access control per environment.
Q: What are Terraform workspaces and their limitations?
Workspaces provide multiple state files for the same configuration. Each workspace gets its own state stored under the same backend path. Use terraform.workspace to reference the current workspace in config. Limitations: (1) Same backend for all workspaces (cannot use different AWS accounts per workspace easily). (2) All workspaces share the same provider configuration. (3) Workspace switching is error-prone (forgetting which workspace you're in). (4) No built-in variable files per workspace. (5) Visibility is poor in CI/CD - easy to apply to the wrong workspace. For these reasons, separate directories with shared modules is the preferred pattern for production environments.

12. Common Mistakes and Gotchas

!! Gotcha: State File in Version Control

Mistake: Committing terraform.tfstate to Git.

Why it's bad: State contains secrets (passwords, tokens, private IPs). Multiple people pulling/pushing creates conflicts and state corruption.

Fix: Use remote backend (S3, GCS, Terraform Cloud). Add to .gitignore:

echo "*.tfstate*\n.terraform/\n*.tfvars" >> .gitignore
!! Gotcha: Using count with Maps/Lists That Change

Mistake: Using count with a list where items can be removed from the middle.

BAD - Index shift causes recreation Anti-pattern 
# If you remove "staging", production shifts from [2] to [1]
# Terraform DESTROYS production and recreates it!
variable "envs" { default = ["dev", "staging", "production"] }

resource "aws_s3_bucket" "env" {
  count  = length(var.envs)
  bucket = "app-${var.envs[count.index]}"
}

Fix: Use for_each with a set or map instead.

!! Gotcha: Hardcoding Provider Credentials

Mistake: Putting AWS keys directly in provider blocks or .tf files.

BAD - Never do this Security Risk 
provider "aws" {
  access_key = "AKIAIOSFODNN7EXAMPLE"  # NEVER DO THIS
  secret_key = "wJalrXUtnFEMI/K7MDENG" # NEVER DO THIS
  region     = "us-east-1"
}

Fix: Use environment variables, IAM roles, AWS profiles, or OIDC federation in CI/CD.

!! Gotcha: Not Using -out with Plan

Mistake: Running terraform plan then terraform apply separately without saving the plan.

Why it's bad: Between your plan and apply, someone else might push changes, or a dependent resource might change. The apply could do something different from what you reviewed.

Fix:

terraform plan -out=tfplan && terraform apply tfplan
!! Gotcha: Large Monolithic State

Mistake: Managing all infrastructure in a single state file (networking, compute, databases, K8s, DNS, IAM).

Why it's bad: Slow plan/apply, large blast radius (one mistake affects everything), lock contention in teams, hard to manage permissions.

Fix: Split by component/layer:

  • networking/ - VPC, subnets, route tables
  • compute/ - EKS, EC2, ASGs
  • data/ - RDS, ElastiCache, S3
  • dns/ - Route53 records

Use data sources or terraform_remote_state to reference between states.

!! Gotcha: Forgetting to Lock Provider Versions

Mistake: Not pinning provider versions in required_providers.

Why it's bad: A provider update could introduce breaking changes, causing different behavior between team members or CI runs.

Fix: Always pin versions and commit .terraform.lock.hcl:

versions.tf Best Practice 
terraform {
  required_version = ">= 1.5.0, < 2.0.0"

  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.30"  # Allows 5.30.x - 5.x, not 6.0
    }
  }
}
!! Gotcha: Destroying Resources by Removing Code

Mistake: Deleting a resource block from .tf and running apply, expecting Terraform to "forget" about it.

What happens: Terraform sees the resource in state but not in config, so it destroys the real resource.

If you want to keep the resource but stop managing it:

terraform state rm aws_instance.legacy

Then remove the code. The resource continues to exist but is no longer managed by Terraform.
