Git

Distributed version control - branching strategies, workflows, advanced operations, and CI/CD integration

1. Git Basics

What is Git?

Git is a distributed version control system that tracks changes in source code. Every developer has a full copy of the repository history, enabling offline work and fast operations.

Essential Commands

Repository Setup

git init

Initialize a new Git repository in the current directory.

git clone https://github.com/user/repo.git

Clone an existing remote repository locally.

Daily Workflow

git status

Show working tree status (modified, staged, untracked).

git diff

Show unstaged changes. Use --staged for staged changes.

Staging & Committing

git add file.txt

Stage a specific file for commit.

git add -A

Stage all changes (new, modified, deleted).

git commit -m "feat: add user auth"

Commit staged changes with a message.

Remote Operations

git push origin main

Push commits to remote repository.

git pull origin main

Fetch and merge remote changes (fetch + merge).

git fetch origin

Download remote changes without merging.

Working Tree / Staging / Repository Model

Git Three-Tree Architecture
 Working Directory          Staging Area (Index)         Local Repository         Remote Repository
 (Your files on disk)       (Next commit snapshot)       (.git directory)         (GitHub/GitLab)
+-------------------+      +-------------------+       +-------------------+    +-------------------+
|                   |      |                   |       |                   |    |                   |
|  modified files   | ---> |  staged changes   | --->  |  commit history   | -->|  shared commits   |
|                   | add  |                   | commit|                   |push|                   |
|                   |      |                   |       |                   |    |                   |
+-------------------+      +-------------------+       +-------------------+    +-------------------+
         ^                                                      |                        |
         |                                                      |                        |
         +---------------------- checkout -----------------------+                        |
         |                                                                               |
         +------------------------------- pull / fetch ----------------------------------+

File States:
  Untracked --> Staged --> Committed --> Pushed
  Modified  --> Staged --> Committed --> Pushed

2. Branching

Branch Operations

Branches are lightweight pointers to commits. They enable parallel development without affecting the main codebase.

Creating & Switching

git branch feature/login

Create a new branch.

git switch feature/login

Switch to branch (modern, preferred over checkout).

git switch -c feature/login

Create and switch in one command.

Merging & Deleting

git merge feature/login

Merge branch into current branch.

git branch -d feature/login

Delete branch (safe, only if merged).

git branch -D feature/login

Force delete branch (even if unmerged).

Branch Visualization

Branch and Merge Workflow
main:     A --- B --- C --------- G (merge commit)
                \               /
feature:         D --- E --- F

  A = Initial commit
  B = Second commit on main
  C = Third commit on main (branch point)
  D, E, F = Feature work
  G = Merge commit (combines C and F)

Fast-Forward Merge (when main hasn't diverged):
main:     A --- B --- C --- D --- E
                           ^
                      feature (fast-forward, no merge commit)

3. Branching Strategies

GitFlow

GitFlow Branch Model
          hotfix/fix-bug
             |
main:    ----*-----------*-----------*---------> (production releases)
             |          /|          /
release:     |    -----* |    -----*
             |   /       |   /
develop: ----*--*---*---*---*---*---*----------> (integration branch)
              \   /       \   /
feature:       --*         --*
           feature/auth  feature/payment

Branch Types:
  main      = Production-ready code, tagged with versions
  develop   = Integration branch for features
  feature/* = New features (branch from develop, merge to develop)
  release/* = Release preparation (branch from develop, merge to main + develop)
  hotfix/*  = Emergency fixes (branch from main, merge to main + develop)

Trunk-Based Development

Trunk-Based Development
main (trunk): ----*----*----*----*----*----*----*----*----> (continuous integration)
                   \  /      \  /      \  /
short-lived:        **        **        **
                  (hours)   (hours)   (hours)

Key Principles:
  - Single long-lived branch (main/trunk)
  - Short-lived feature branches (< 1-2 days)
  - Frequent integration (multiple times per day)
  - Feature flags for incomplete features
  - Continuous deployment from trunk

GitHub Flow

GitHub Flow is a simplified model: one main branch + feature branches + pull requests.

  1. Create a branch from main
  2. Add commits
  3. Open a Pull Request
  4. Discuss and review code
  5. Deploy for testing
  6. Merge to main

Strategy Comparison

Aspect GitFlow Trunk-Based GitHub Flow
Complexity High (5 branch types) Low (1 branch + short-lived) Medium (main + feature)
Release Cadence Scheduled releases Continuous deployment On-demand
Team Size Large teams, multiple versions Any size, high maturity Small to medium teams
CI/CD Requirement Optional Mandatory (feature flags) Recommended
Merge Conflicts More likely (long-lived branches) Rare (frequent integration) Moderate
Best For Versioned software, mobile apps SaaS, web apps, microservices Open source, web projects

4. Merge vs Rebase

Merge vs Rebase Visual Comparison
BEFORE (same starting point):
main:     A --- B --- C
                \
feature:         D --- E

AFTER MERGE (git merge feature from main):
main:     A --- B --- C --- M (merge commit)
                \         /
feature:         D --- E -

AFTER REBASE (git rebase main from feature, then fast-forward merge):
main:     A --- B --- C --- D' --- E' (linear history)

  D' and E' are NEW commits (different SHA) with same changes as D and E
  Original D and E are orphaned (eventually garbage collected)

When to Use Merge

  • Preserving complete branch history
  • Working on shared/public branches
  • Team collaboration on same branch
  • When traceability matters (audit trails)
  • Non-destructive operation (safe)

When to Use Rebase

  • Cleaning up local commits before sharing
  • Maintaining linear project history
  • Updating feature branch with latest main
  • Before creating a pull request
  • Squashing messy commits

The Golden Rule of Rebasing

NEVER rebase commits that have been pushed to a public/shared branch.

Rebasing rewrites commit history (new SHAs). If others have based work on the original commits, rebasing will cause divergence and confusion. Only rebase your own local, unpushed commits.

5. Conflict Resolution

How Conflicts Happen

Conflicts occur when two branches modify the same lines in the same file, or when one branch deletes a file that the other modified. Git cannot automatically determine which change to keep.

Resolving Step-by-Step

1
Identify conflicts
git status

Files marked "both modified" have conflicts.

2
Open conflicted file

Look for conflict markers:

file.txt (with conflict markers)
<<<<<<< HEAD
your changes on current branch
=======
their changes from incoming branch
>>>>>>> feature/login
3
Resolve the conflict

Edit the file to keep the desired content. Remove all conflict markers (<<<<<<<, =======, >>>>>>>).

4
Stage and commit
git add file.txt
git commit -m "resolve: merge conflict in file.txt"

Merge Tools

Tool Type Setup Command
VS Code GUI (built-in 3-way merge editor) git config --global merge.tool vscode
vimdiff Terminal git config --global merge.tool vimdiff
meld GUI (Linux) git config --global merge.tool meld
kdiff3 GUI (cross-platform) git config --global merge.tool kdiff3
git mergetool

Opens the configured merge tool for resolving conflicts interactively.

6. Advanced Git

Cherry-Pick

Apply a specific commit from another branch to your current branch without merging the entire branch.

git cherry-pick abc1234

Creates a new commit on the current branch with the same changes as abc1234.

git cherry-pick abc1234..def5678

Cherry-pick a range of commits (exclusive of the first, inclusive of the last).

Stash

Temporarily shelve uncommitted changes to work on something else.

git stash

Stash working directory and staged changes.

git stash list

List all stashed changes.

git stash pop

Apply most recent stash and remove it from the stash list.

git stash apply stash@{2}

Apply a specific stash without removing it.

git stash drop stash@{0}

Delete a specific stash entry.

Reset (Soft / Mixed / Hard)

git reset Modes Comparison
                     Working Dir    Staging Area    Commit History
                     -----------    ------------    --------------
git reset --soft      Unchanged      Unchanged      Moves HEAD back
git reset --mixed     Unchanged      Reset          Moves HEAD back  (DEFAULT)
git reset --hard      Reset          Reset          Moves HEAD back  (DANGEROUS)

Example: Undo last 2 commits
  git reset --soft HEAD~2   -> Changes back in staging (ready to re-commit)
  git reset --mixed HEAD~2  -> Changes in working dir (need to re-add)
  git reset --hard HEAD~2   -> Changes GONE (destructive!)

Revert

Revert creates a NEW commit that undoes a previous commit. Safe for public branches because it doesn't rewrite history.

git revert abc1234

Create a new commit that reverses the changes from abc1234.

git revert HEAD~3..HEAD

Revert the last 3 commits (creates 3 new revert commits).

Reflog

Reflog records every change to HEAD, even those not in the regular log. It's your safety net for recovering "lost" commits.

git reflog

Show reference log with all HEAD movements.

git reset --hard HEAD@{5}

Recover state from 5 operations ago.

Bisect

Bisect uses binary search to find the commit that introduced a bug.

git bisect start
git bisect bad

Mark current commit as bad (has the bug).

git bisect good v1.0

Mark a known good commit. Git will checkout the middle commit for testing.

git bisect good

or git bisect bad after testing each commit. Repeat until the culprit is found.

git bisect reset

End the bisect session.

Interactive Rebase & Squash

Rewrite commit history by combining, reordering, editing, or dropping commits.

git rebase -i HEAD~5

Opens editor with last 5 commits. Actions available:

Interactive rebase commands
pick   abc1234 feat: add login page       # keep as-is
squash def5678 fix: typo in login          # combine with previous
fixup  ghi9012 fix: another typo           # combine, discard message
reword jkl3456 feat: add dashboard         # edit commit message
edit   mno7890 feat: add settings          # pause to amend
drop   pqr1234 WIP: temporary commit       # remove entirely

7. Git Hooks

What are Git Hooks?

Scripts that run automatically at certain points in the Git workflow. Located in .git/hooks/ (local) or managed via tools like Husky (shared with team).

Hook Trigger Common Use
pre-commit Before commit is created Lint code, run formatters, check for secrets
commit-msg After commit message is entered Validate commit message format (Conventional Commits)
pre-push Before push to remote Run tests, check branch naming
prepare-commit-msg Before commit message editor opens Add ticket number from branch name
post-merge After successful merge Install dependencies if lock file changed

Hook Examples

.git/hooks/pre-commit Shell 
#!/bin/sh
# Run linter before allowing commit
echo "Running ESLint..."
npx eslint --ext .js,.ts src/

if [ $? -ne 0 ]; then
  echo "Lint errors found. Fix them before committing."
  exit 1
fi

# Check for secrets/credentials
if git diff --cached --name-only | xargs grep -l "AWS_SECRET\|password\|api_key" 2>/dev/null; then
  echo "Potential secrets detected! Review staged files."
  exit 1
fi
.git/hooks/commit-msg Shell 
#!/bin/sh
# Enforce Conventional Commits format
commit_msg=$(cat "$1")
pattern="^(feat|fix|docs|style|refactor|test|chore|ci|perf|build)(\(.+\))?: .{1,72}$"

if ! echo "$commit_msg" | grep -qE "$pattern"; then
  echo "Invalid commit message format!"
  echo "Expected: type(scope): description"
  echo "Examples: feat(auth): add OAuth2 login"
  echo "          fix: resolve null pointer in parser"
  exit 1
fi
package.json (Husky + lint-staged) JSON 
{
  "scripts": {
    "prepare": "husky install"
  },
  "lint-staged": {
    "*.{js,ts}": ["eslint --fix", "prettier --write"],
    "*.{css,md}": ["prettier --write"]
  }
}

8. .gitignore

Pattern Rules

  • *.log - Ignore all .log files
  • build/ - Ignore entire directory
  • !important.log - Negate (don't ignore this file)
  • **/temp - Match in any subdirectory
  • doc/*.pdf - Ignore PDFs only in doc/ (not subdirs)
  • doc/**/*.pdf - Ignore PDFs in doc/ and all subdirs
  • # - Comments

# Dependencies

# Environment & Secrets

# Coverage & Testing

Global .gitignore

Set a global gitignore for OS/IDE files that should never be committed from any repo:

git config --global core.excludesfile ~/.gitignore_global

Add patterns like .DS_Store, .idea/, *.swp to your global file.

Already Tracked Files

Adding a file to .gitignore won't remove it if it's already tracked. You must untrack it first:

git rm --cached file.txt

Removes from tracking but keeps the file on disk.

9. Git in CI/CD

Semantic Versioning

Semantic Versioning (SemVer)
  MAJOR.MINOR.PATCH
    |     |     |
    |     |     +-- Bug fixes, patches (backward compatible)
    |     +-------- New features (backward compatible)
    +-------------- Breaking changes (incompatible API changes)

  Examples:
    1.0.0 -> 1.0.1  (patch: bug fix)
    1.0.1 -> 1.1.0  (minor: new feature)
    1.1.0 -> 2.0.0  (major: breaking change)

  Pre-release: 1.0.0-alpha.1, 1.0.0-beta.2, 1.0.0-rc.1

Tags & Releases

git tag v1.2.0

Create a lightweight tag.

git tag -a v1.2.0 -m "Release 1.2.0: add OAuth support"

Create an annotated tag (recommended for releases).

git push origin v1.2.0

Push a specific tag to remote.

git push origin --tags

Push all tags to remote.

Release Workflow (CI/CD)

.github/workflows/release.yml YAML 
name: Release
on:
  push:
    tags:
      - 'v*'

jobs:
  release:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Build
        run: make build
      - name: Run Tests
        run: make test
      - name: Create GitHub Release
        uses: softprops/action-gh-release@v1
        with:
          generate_release_notes: true
          files: |
            dist/*.tar.gz
            dist/*.zip
.github/workflows/ci.yml (branch-based) YAML 
name: CI
on:
  pull_request:
    branches: [main]
  push:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0  # Full history for versioning
      - name: Lint
        run: npm run lint
      - name: Test
        run: npm test
      - name: Build
        run: npm run build

  deploy:
    needs: test
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Deploy to production
        run: ./deploy.sh

10. Interview Q&A

Q1: What is the difference between merge and rebase?

Merge creates a new merge commit that combines two branches, preserving the complete history and branch topology. Rebase replays commits from one branch onto another, creating a linear history but rewriting commit SHAs.

Use merge for shared branches (preserves context); use rebase for local cleanup (cleaner history). Never rebase public branches.

Q2: How do you resolve a merge conflict?

1) Run git status to identify conflicted files. 2) Open each file and look for conflict markers (<<<<<<<, =======, >>>>>>>). 3) Edit to keep desired changes and remove markers. 4) git add resolved files. 5) git commit (or git rebase --continue if rebasing). Optionally use git mergetool for a visual diff.

Q3: Compare GitFlow vs Trunk-Based Development. When would you use each?

GitFlow: Multiple long-lived branches (main, develop, feature, release, hotfix). Best for versioned releases, mobile apps, or teams needing strict release management.

Trunk-Based: Single main branch with very short-lived feature branches. Requires strong CI/CD, feature flags, and team discipline. Best for SaaS, microservices, and continuous deployment.

Choose GitFlow when you need multiple supported versions; choose Trunk-Based for maximum velocity and continuous delivery.

Q4: What is the difference between git reset and git revert?

git reset moves the HEAD pointer backward, potentially rewriting history. It's suitable for local/unpushed commits. Has three modes: --soft (keep staging), --mixed (unstage, keep working dir), --hard (discard everything).

git revert creates a new commit that undoes a previous commit's changes. It's safe for public branches because it doesn't rewrite history.

Q5: What are some use cases for cherry-pick?

1) Backporting a bug fix from develop to a release branch. 2) Pulling a specific feature commit into a hotfix branch. 3) Recovering a commit from a deleted branch. 4) Applying a fix across multiple release versions. Be cautious: cherry-picked commits have different SHAs, so merging later may cause duplicates.

Q6: How do you undo the last commit?

If not pushed: git reset --soft HEAD~1 (keeps changes staged) or git reset --mixed HEAD~1 (unstages changes).

If already pushed: git revert HEAD (creates a new undo commit, safe for shared branches).

If you want to discard entirely: git reset --hard HEAD~1 (destructive, use with caution).

Q7: How do you squash commits?

Use interactive rebase: git rebase -i HEAD~N where N is the number of commits to squash. In the editor, change pick to squash (or fixup) for commits to combine. Alternatively, when merging a PR, use "Squash and Merge" on GitHub/GitLab to combine all PR commits into one.

Q8: What is git reflog and when would you use it?

Reflog is a local log of all HEAD movements (commits, resets, checkouts, rebases). Use it to recover "lost" commits after a hard reset, find a previous state after a bad rebase, or identify when a branch was at a specific point. Example recovery: git reflog to find the SHA, then git reset --hard HEAD@{n}.

Q9: What is git bisect and how does it work?

Git bisect performs a binary search through commit history to find the exact commit that introduced a bug. You mark a "bad" commit (has the bug) and a "good" commit (before the bug). Git checks out the midpoint commit for you to test, then you mark it good or bad. This halves the search space each time, finding the culprit in O(log n) steps.

Q10: How do Git hooks help in a team workflow?

Git hooks automate quality gates: pre-commit runs linters/formatters (ensuring consistent code style), commit-msg enforces message conventions (Conventional Commits), pre-push runs tests (preventing broken pushes). Tools like Husky + lint-staged make hooks shareable via package.json. This shifts quality left, catching issues before code review.

11. Real-World Scenarios

⚠ Scenario: Accidentally Pushed Sensitive Data to Main

Situation: A developer accidentally committed and pushed a file containing API keys to the main branch.

1. Immediately rotate the exposed credentials - consider them compromised regardless of how quickly you act.
2. Remove from history using git filter-branch or BFG Repo Cleaner:
git filter-branch --force --index-filter 'git rm --cached --ignore-unmatch secrets.env' --prune-empty --tag-name-filter cat -- --all

Or use the faster BFG tool:

bfg --delete-files secrets.env
git reflog expire --expire=now --all && git gc --prune=now --aggressive
git push --force-with-lease --all
3. Add to .gitignore and set up pre-commit hooks to prevent future leaks.
4. Notify team to re-clone or force-pull since history was rewritten.
🔎 Scenario: Lost Commits After Bad Reset (Reflog Recovery)

Situation: A developer ran git reset --hard HEAD~5 and lost important commits.

1. Don't panic. The commits still exist in the reflog for ~90 days.
2. Find the lost commit:
git reflog
reflog output
abc1234 HEAD@{0}: reset: moving to HEAD~5
def5678 HEAD@{1}: commit: feat: add payment processing
ghi9012 HEAD@{2}: commit: feat: add cart functionality
jkl3456 HEAD@{3}: commit: fix: resolve auth bug
3. Recover by resetting to the desired state:
git reset --hard def5678

Or create a new branch from the lost commit:

git branch recovered-work def5678
🔧 Scenario: Merge Conflict in Pull Request

Situation: Your PR cannot be merged because of conflicts with the target branch.

1. Update your local main:
git switch main && git pull origin main
2. Switch to feature and merge/rebase main:
git switch feature/my-work
git rebase main

(or git merge main if you prefer merge commits)

3. Resolve conflicts in each file, then continue:
git add resolved-file.txt
git rebase --continue
4. Force push (only if you rebased, since SHAs changed):
git push --force-with-lease origin feature/my-work
📦 Scenario: Large File Accidentally Committed

Situation: A 500MB binary file was committed and push fails or bloats the repo.

1. If not yet pushed, remove from history:
git reset --soft HEAD~1
git rm --cached large-file.bin

Add to .gitignore, then recommit.

2. If already pushed, use BFG or filter-branch:
bfg --strip-blobs-bigger-than 100M
git reflog expire --expire=now --all && git gc --prune=now
git push --force-with-lease --all
3. For future large files, use Git LFS:
git lfs install
git lfs track "*.bin" "*.zip" "*.tar.gz"
git add .gitattributes
