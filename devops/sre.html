SRE Principles

Site Reliability Engineering - Reliability at scale through engineering discipline

1. SLIs, SLOs, and SLAs

SLI - Service Level Indicator

A carefully defined quantitative measure of some aspect of the level of service provided. It is the metric you measure.

Examples:

  • Request latency (p50, p95, p99)
  • Error rate (% of 5xx responses)
  • Availability (% of successful requests)
  • Throughput (requests per second)

SLO - Service Level Objective

A target value or range of values for a service level that is measured by an SLI. It is the goal you set.

Examples:

  • 99.9% of requests complete in < 300ms
  • 99.95% availability per month
  • Error rate < 0.1% over 30 days

SLA - Service Level Agreement

A contract with your users that includes consequences of meeting (or missing) the SLOs it contains. It is the promise with penalties.

Examples:

  • 99.9% uptime or service credits issued
  • P1 incidents resolved within 1 hour or penalty applies
  • SLA is always less strict than SLO (buffer)

Relationship

SLI measures the metric, SLO sets the target, SLA defines the contract.

Typical pattern: SLA < SLO < actual performance

Example: SLA promises 99.9%, SLO targets 99.95%, system actually delivers 99.99%

SLI Formulas

SLI Calculations Formulas 
Availability SLI = (Successful Requests / Total Requests) x 100%

Latency SLI = (Requests faster than threshold / Total Requests) x 100%

Error Rate SLI = (Failed Requests / Total Requests) x 100%

Throughput SLI = Successful Requests per unit time

Durability SLI = (Stored objects retrievable / Total stored objects) x 100%

2. Error Budgets

What is an Error Budget?

The error budget is the maximum amount of time or failures your service can have before violating its SLO. It represents the acceptable level of unreliability.

Formula: Error Budget = 1 - SLO

Error Budget Calculation

Error Budget Math Calculation 
SLO = 99.9% availability

Error Budget = 1 - 0.999 = 0.001 = 0.1%

Monthly (30 days):
  Total minutes = 30 x 24 x 60 = 43,200 minutes
  Allowed downtime = 43,200 x 0.001 = 43.2 minutes/month

Quarterly (90 days):
  Total minutes = 90 x 24 x 60 = 129,600 minutes
  Allowed downtime = 129,600 x 0.001 = 129.6 minutes/quarter

Common SLO Downtime Allowances (per month):
  99%    = 7h 12m
  99.9%  = 43.2m
  99.95% = 21.6m
  99.99% = 4.32m

Error Budget Policy

An error budget policy defines what happens when the budget is exhausted:

  • Budget remaining (>50%): Normal development velocity, feature work prioritized
  • Budget consumed (25-50%): Increased focus on reliability, extra review for risky changes
  • Budget nearly exhausted (<25%): Freeze non-critical deployments, prioritize reliability work
  • Budget exhausted (0%): Full feature freeze, all engineering effort on reliability

Burn Rate Alerting

Multi-Window, Multi-Burn-Rate Alerts

Burn rate measures how fast the error budget is being consumed relative to the SLO period.

  • Burn rate 1x: Budget consumed exactly over the SLO window (e.g., 30 days)
  • Burn rate 2x: Budget consumed in half the time (15 days)
  • Burn rate 14.4x: Budget consumed in ~2 days (page immediately)
  • Burn rate 36x: Budget consumed in ~20 hours (critical alert)
Burn Rate Alert Configuration (Prometheus) Alert Rules 
# Multi-window burn rate alert
# SLO: 99.9% over 30 days = error budget of 0.1%
# Burn rate 14.4x = budget exhausted in 2 days

groups:
  - name: slo-burn-rate
    rules:
      # Critical: 14.4x burn rate over 1h AND 5m windows
      - alert: HighErrorBudgetBurn
        expr: |
          (
            sum(rate(http_requests_total{code=~"5.."}[1h]))
            / sum(rate(http_requests_total[1h]))
          ) > (14.4 * 0.001)
          AND
          (
            sum(rate(http_requests_total{code=~"5.."}[5m]))
            / sum(rate(http_requests_total[5m]))
          ) > (14.4 * 0.001)
        labels:
          severity: critical
        annotations:
          summary: "Error budget burning at 14.4x rate"

      # Warning: 6x burn rate over 6h AND 30m windows
      - alert: ErrorBudgetBurnWarning
        expr: |
          (
            sum(rate(http_requests_total{code=~"5.."}[6h]))
            / sum(rate(http_requests_total[6h]))
          ) > (6 * 0.001)
          AND
          (
            sum(rate(http_requests_total{code=~"5.."}[30m]))
            / sum(rate(http_requests_total[30m]))
          ) > (6 * 0.001)
        labels:
          severity: warning

3. Toil

Definition

Toil is the kind of work tied to running a production service that tends to be manual, repetitive, automatable, tactical, devoid of enduring value, and scales linearly as the service grows.

Characteristics of Toil

1
Manual
Running a script manually that could be triggered automatically
2
Repetitive
The same task done over and over (e.g., restarting pods, clearing disk space)
3
Automatable
If a machine could accomplish the task just as well, it is toil
4
Tactical (Reactive)
Interrupt-driven, responding to alerts rather than proactive engineering
5
No Enduring Value
After the task is done, the service is in the same state as before (no improvement)
6
Scales Linearly with Service Growth
O(n) with service size - more traffic means more manual work

Toil Measurement

Google SRE targets: no more than 50% of an SRE's time should be spent on toil. The other 50% should be engineering work (automation, tooling, architecture).

Measurement approaches:

  • Track time spent on operational tasks vs engineering work
  • Count tickets/interrupts per on-call shift
  • Categorize work: toil, engineering, overhead
  • Survey team quarterly on perceived toil burden

Toil Reduction Strategies

StrategyExampleImpact
Automate repetitive tasksAuto-scaling instead of manual capacity addsHigh
Self-healing systemsK8s liveness probes auto-restart failed podsHigh
Better monitoringReduce false alerts to cut interrupt-driven workMedium
Eliminate root causesFix memory leaks instead of restarting weeklyHigh

Incident Roles

Incident Commander (IC)

  • Single point of coordination
  • Delegates tasks, does NOT debug directly
  • Makes decisions on mitigation strategy
  • Calls for escalation when needed
  • Declares incident resolved
  • Ensures postmortem is scheduled

Communications Lead

  • Updates status page
  • Communicates with stakeholders
  • Sends periodic updates (every 15-30 min)
  • Shields responders from external interrupts
  • Maintains incident timeline
  • Coordinates customer communication

Operations Lead

  • Hands-on debugging and mitigation
  • Applies fixes to production systems
  • Reports findings to IC
  • Proposes mitigation options

Subject Matter Expert (SME)

  • Called in for specific domain knowledge
  • Advises on system behavior
  • May provide historical context
  • Rotated in/out as needed

Communication During Incidents

Key principles:

  • Use a dedicated incident channel (e.g., #inc-20260115-payment-failure)
  • Regular cadence updates even if no new info: "Still investigating, no change"
  • Clear, factual language - avoid blame or speculation
  • Announce all major actions before executing them
  • Timestamp everything for the postmortem timeline

5. Blameless Postmortems

Core Philosophy

A blameless postmortem assumes that everyone involved in an incident had good intentions and made the best decisions they could given the information available at the time. The goal is to understand what happened and how to prevent it, not who to blame.

5 Whys Technique

A root cause analysis technique that asks "why" repeatedly to drill past symptoms to the underlying cause:

1
Why did the service go down?
Because the database ran out of connections.
2
Why did it run out of connections?
Because a new deployment opened connections but never closed them.
3
Why were connections not closed?
Because the code path for error handling skipped the cleanup function.
4
Why was this not caught in testing?
Because integration tests only test the happy path, not error scenarios.
5
Why do integration tests only cover happy paths?
Because there is no requirement or checklist for error-path test coverage.

Root Cause: Missing error-path test coverage requirement. Action: Add error-scenario testing to the Definition of Done and CI pipeline.

7. Chaos Engineering

Definition

Chaos Engineering is the discipline of experimenting on a system to build confidence in the system's capability to withstand turbulent conditions in production.

Principles of Chaos Engineering

1
Build a Hypothesis Around Steady State
Define what "normal" looks like in terms of measurable output (throughput, error rate, latency)
2
Vary Real-World Events
Inject failures that reflect real-world turbulence: server crashes, network partitions, disk full, clock skew
3
Run Experiments in Production
Systems behave differently in production vs staging due to traffic patterns, data volume, and configuration
4
Automate Experiments to Run Continuously
One-off tests provide less confidence than continuous validation
5
Minimize Blast Radius
Start small, contain experiments, have abort mechanisms ready

Running a Game Day

  1. Plan: Define hypothesis, scope, abort criteria, participants
  2. Communicate: Notify stakeholders, ensure on-call awareness

  4. Observe: Monitor dashboards, check alerts fire correctly, verify auto-healing

  6. Remediate: Fix gaps discovered, improve runbooks, update alerts

8. On-Call Best Practices

Rotation Design

Rotation Structure

  • Primary + secondary on-call (backup)
  • Rotation duration: 1 week typical
  • Minimum team size: 8 people for sustainable rotation
  • Follow-the-sun for global teams
  • Fair compensation for out-of-hours work
  • Handoff meeting at rotation change

On-Call Load Targets

  • < 2 pages per shift (target)
  • < 25% of on-call time on incidents
  • Maximum 1 week on-call per month
  • No more than 2 consecutive on-call shifts
  • Post-incident recovery time (day off after P1)

Runbooks

What Makes a Good Runbook

  • Actionable: Step-by-step commands, not just theory
  • Linked from alerts: Every alert should link to its runbook
  • Up to date: Review and update after every incident
  • Scoped: One runbook per alert/scenario, not monolithic
  • Includes rollback: How to undo any action taken
  • Has decision trees: "If X then do Y, else do Z"
  • Contact list: Who to escalate to for specific subsystems

10. USE and RED Methods

USE Method (Infrastructure/Resources)

Created by Brendan Gregg. For every resource (CPU, memory, disk, network):

  • Utilization: % of resource busy (e.g., CPU at 85%)
  • Saturation: Queue depth / work waiting (e.g., run queue length)
  • Errors: Error count (e.g., disk I/O errors, network CRC errors)

Best for: Infrastructure monitoring, hardware resources, system-level troubleshooting

RED Method (Services/Requests)

Created by Tom Wilkie. For every service:

  • Rate: Requests per second
  • Errors: Number of failed requests per second
  • Duration: Distribution of request latency (histogram)

Best for: Microservices monitoring, request-driven systems, user-facing metrics

When to Use Each

ScenarioMethodWhy
Database server running slowUSECheck CPU, memory, disk I/O saturation
API response times increasingREDTrack rate, errors, latency distribution
Kubernetes node pressureUSECheck node resource utilization and saturation
Microservice error spikeREDMeasure request rate, error rate, duration
Network throughput issuesUSECheck bandwidth utilization, packet queuing, errors
User-facing latency SLO breachREDFocus on request duration distribution

12. Interview Q&A - SRE Concepts

Q: What is the difference between SLI, SLO, and SLA?
SLI is the metric you measure (e.g., request latency). SLO is the target you set for that metric (e.g., p99 latency < 200ms). SLA is the contractual agreement with customers that includes consequences for not meeting the SLO. The SLA threshold should always be less strict than the SLO to provide a safety buffer.
Q: How do you calculate and use error budgets?
Error Budget = 1 - SLO. For a 99.9% SLO, the error budget is 0.1% (about 43 minutes/month). It represents the acceptable unreliability. Teams use it to balance velocity and reliability: when budget is healthy, ship features faster; when exhausted, freeze deployments and fix reliability. Burn rate alerts warn when budget is being consumed too quickly.
Q: What is toil and how do you reduce it?
Toil is operational work that is manual, repetitive, automatable, tactical, devoid of enduring value, and scales linearly with service size. Reduce it by: automating repetitive tasks, building self-healing systems, eliminating root causes rather than treating symptoms, improving monitoring to reduce false alerts, and creating self-service platforms. Google targets <50% toil for SRE teams.
Q: Explain blameless postmortems. Why are they important?
Blameless postmortems focus on systemic causes rather than individual mistakes. They assume everyone made the best decisions with available information. This is critical because: (1) it encourages honest reporting, (2) people won't hide mistakes, (3) you find real systemic improvements, (4) it builds psychological safety. The output should be actionable items that prevent recurrence through system improvements, not human process changes.
Q: What are the DORA metrics and why do they matter?
DORA metrics are four measures of software delivery performance: Deployment Frequency, Lead Time for Changes, Change Failure Rate, and Mean Time to Restore. They matter because research shows they predict both organizational performance and team wellbeing. Elite performers deploy multiple times/day with <1h lead time, 0-15% failure rate, and <1h MTTR. These metrics help teams identify bottlenecks and measure improvement.
Q: How would you implement SLOs for a new service?
Step-by-step: (1) Identify critical user journeys - what matters most to users. (2) Choose SLIs - typically availability, latency, and error rate. (3) Set initial SLO targets based on current performance (slightly below actual). (4) Implement measurement - instrument the service, build dashboards. (5) Define error budget policy - what happens when budget is consumed. (6) Iterate - review SLOs quarterly, tighten as system matures. Start conservative and get tighter over time.
Q: What is the difference between USE and RED methods?
USE (Utilization, Saturation, Errors) is for infrastructure resources - CPU, memory, disk, network. RED (Rate, Errors, Duration) is for services/requests - HTTP endpoints, microservices, APIs. Use USE for system-level troubleshooting (why is the node slow?). Use RED for application-level monitoring (why are users seeing errors?). They complement each other - USE finds resource bottlenecks, RED finds service-level issues.
Q: How do you approach on-call sustainably?
Key practices: (1) Team size of at least 8 for weekly rotation. (2) Target <2 pages per shift; more means systems need reliability work. (3) Every alert must be actionable - if it doesn't require action, delete it. (4) Runbooks for every alert - reduce cognitive load at 3am. (5) Post-incident recovery time after P1s. (6) Toil budget - invest in automation to reduce page frequency. (7) Fair compensation for out-of-hours work. (8) Regular review of alert frequency and on-call load.
Q: Describe the incident management lifecycle.
Five phases: (1) Detection - monitoring/alerts detect anomaly. (2) Triage - classify severity, determine if it's a real incident. (3) Response - IC assigned, war room opened, roles delegated (ops lead, comms lead, SMEs). (4) Resolution - identify root cause, implement fix, verify recovery. (5) Post-incident - blameless postmortem within 48h, action items tracked to completion. Communication is critical throughout: regular status updates, clear language, timestamped actions.
Q: What is chaos engineering and when should you use it?
Chaos engineering is proactively injecting failures to discover weaknesses before they cause outages. Use it when: (1) your system is complex enough that failures are unpredictable, (2) you need confidence in auto-healing capabilities, (3) you want to validate runbooks/alerts work correctly. Start small (kill one pod), build confidence, then escalate (network partition, AZ failure). Always have a hypothesis, measure SLIs, and have abort criteria. Tools: Litmus Chaos (K8s), Chaos Mesh, Gremlin (enterprise).
Q: How do you prevent alert fatigue?
Strategies: (1) Every alert must be actionable - if no action needed, remove it. (2) Use burn rate alerts instead of threshold alerts for SLOs. (3) Group related alerts - one page for correlated issues, not five. (4) Tune thresholds - review false positive rate monthly. (5) Route by severity - page for P1/P2, ticket for P3/P4. (6) Deduplicate - suppress repeat notifications for same incident. (7) Regular hygiene - delete alerts nobody acts on. Target: <2 pages per on-call shift.
Q: Explain the 12-Factor App methodology and its relevance to cloud-native.
The 12-Factor App defines principles for building scalable, maintainable SaaS: single codebase, explicit dependencies, config in environment, backing services as resources, separate build/release/run, stateless processes, port binding, scale via processes, fast startup/graceful shutdown, dev/prod parity, logs as streams, admin as one-off processes. It maps directly to Kubernetes: ConfigMaps for config, Deployments for stateless processes, HPA for concurrency, preStop hooks for disposability, DaemonSet log collectors for logs-as-streams.

13. Real-World Scenarios

!! Scenario: Error Budget Exhausted

Situation: Your payment service has a 99.95% availability SLO (21.6 minutes/month budget). It's day 15 of the month and you've already used 20 minutes due to two incidents.

Response Plan:

1. Declare feature freeze immediately - no new deployments unless they improve reliability
2. Review both incidents - identify common patterns or systemic issues
3. Prioritize reliability improvements: add circuit breakers, improve health checks, reduce blast radius
4. Tighten deployment process: require additional review, canary all changes, reduce deployment batch size
5. Communicate to stakeholders: product team knows feature delivery is paused, leadership understands why
6. Set criteria for resuming normal velocity: specific reliability improvements completed, budget recovery in next period

Key insight: Error budgets create alignment between development and operations. When the budget is exhausted, everyone agrees: reliability work takes priority. This removes the "move fast vs. be reliable" conflict.

!! Scenario: Cascading Failure

Situation: Service A calls Service B, which calls Service C. Service C's database becomes slow, causing timeouts in B, which causes retries from A, which amplifies the load on B and C, eventually bringing down all three services.

Immediate Response:

1. Identify the cascade: look at error rates propagating upstream in distributed traces
2. Break the cascade: enable circuit breakers on Service A to stop retries to B
3. Shed load: rate-limit incoming traffic to Service A to reduce overall system pressure
4. Fix the root cause: address Service C's database issue (slow query, connection exhaustion, etc.)
5. Gradually restore: remove rate limits, re-enable circuit breakers in controlled fashion

Prevention Strategies:

  • Circuit breakers: Stop calling failing services (Hystrix pattern, Istio)
  • Timeouts: Always set timeouts shorter than the caller's timeout
  • Bulkheads: Isolate failure domains (separate thread pools, connection pools)
  • Retry with backoff: Exponential backoff + jitter, capped retries
  • Graceful degradation: Return cached/default data when downstream is down
  • Load shedding: Reject excess traffic early rather than processing slowly

!! Scenario: Noisy Alert Fatigue

Situation: Your team receives 50+ alerts per on-call shift. Engineers are ignoring alerts, real issues get buried, and morale is dropping. Last week a P1 was missed because it looked like another false positive.

Diagnosis:

1. Audit all alerts from the past 30 days: categorize as actionable, non-actionable, duplicate, or false positive
2. Measure signal-to-noise ratio: what % of alerts required human action?
3. Identify top offenders: which alerts fire most frequently?

Remediation Plan:

1. Delete non-actionable alerts: If no one does anything when it fires, remove it or convert to a dashboard metric
2. Fix the root cause: If an alert fires daily for the same issue, fix the issue instead of alerting on it
3. Tune thresholds: Increase thresholds to reduce false positives (accept slightly later detection for much fewer pages)
4. Use burn-rate alerts: Replace threshold alerts with SLO-based multi-window burn-rate alerts
5. Group correlated alerts: One incident = one page, not 15 alerts from the same root cause
6. Route appropriately: P3/P4 to tickets, not pages. Only P1/P2 should page.
7. Set targets: Aim for <2 pages per shift. Review weekly until achieved.

Success criteria: Every alert that pages results in an engineer taking a meaningful action. If you achieve this, you will never miss a real incident because trust in the alerting system is restored.
