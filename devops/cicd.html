CI/CD & GitOps

Continuous Integration, Continuous Delivery, and GitOps workflows for modern software delivery

1. Pipeline Design

Core Pipeline Principles

  • Fail-Fast: Run cheapest/fastest checks first (lint before integration tests)
  • Immutable Artifacts: Build once, deploy the same artifact to every environment
  • Environment Parity: Dev, staging, and production should be as identical as possible
  • Reproducibility: Given the same inputs, the pipeline should produce the same outputs
  • Traceability: Every artifact should be traceable to a specific commit

FAIL-FAST PRINCIPLE:
  Lint (seconds) --> Unit Tests (minutes) --> Integration (minutes) --> Security Scan (minutes)

  If lint fails, skip everything else = save time + resources

Pipeline Anti-Patterns

Anti-Patterns

  • Building different artifacts per environment
  • Manual steps in the pipeline
  • No artifact versioning
  • Running all tests regardless of what changed
  • Deploying directly from feature branches
  • Secrets hardcoded in pipeline configs

Best Practices

  • Build once, promote same artifact
  • Automate everything
  • Tag artifacts with commit SHA + semantic version
  • Use path-based triggers for monorepos
  • Deploy from main/release branches only
  • Use external secret managers

2. GitOps

What is GitOps?

GitOps is a paradigm where Git is the single source of truth for declarative infrastructure and applications. All changes are made through Git commits, and an operator ensures the live state matches the desired state in Git.

Push vs Pull Model

Aspect Push-Based (Traditional CI/CD) Pull-Based (GitOps)
Trigger CI system pushes changes to cluster Agent in cluster pulls desired state from Git
Access CI needs cluster credentials Agent has cluster access; CI never touches cluster
Security Credentials stored in CI (attack surface) No external access to cluster needed
Drift Detection Manual or none Automatic - agent continuously reconciles
Rollback Re-run previous pipeline or manual git revert - automatic reconciliation
Audit Trail CI logs (may expire) Full Git history (permanent)
Examples Jenkins, GitHub Actions deploy step ArgoCD, Flux

How ArgoCD Works (5 Key Points)

  1. Application Definition: You create an ArgoCD Application CR that points to a Git repo path and a target cluster/namespace
  2. Continuous Monitoring: The Repo Server clones the repo and generates manifests (plain YAML, Helm, Kustomize). The Application Controller compares desired state vs live state
  3. Drift Detection: If live state differs from Git (manual kubectl edit, etc.), ArgoCD marks the app as "OutOfSync"
  4. Sync (Reconciliation): Auto-sync or manual sync applies the Git-defined state to the cluster using kubectl apply semantics
  5. Health Assessment: ArgoCD monitors resource health (Healthy, Degraded, Progressing, Missing) and reports status back
argocd-application.yaml ArgoCD 
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: my-app
  namespace: argocd
  finalizers:
    - resources-finalizer.argocd.argoproj.io
spec:
  project: default

  source:
    repoURL: https://github.com/org/k8s-manifests.git
    targetRevision: main
    path: apps/my-app/overlays/production
    # For Helm charts:
    # helm:
    #   valueFiles:
    #     - values-prod.yaml

  destination:
    server: https://kubernetes.default.svc
    namespace: production

  syncPolicy:
    automated:
      prune: true        # Delete resources removed from Git
      selfHeal: true     # Revert manual changes in cluster
      allowEmpty: false  # Don't sync if manifests are empty
    syncOptions:
      - CreateNamespace=true
      - PrunePropagationPolicy=foreground
      - PruneLast=true
    retry:
      limit: 5
      backoff:
        duration: 5s
        factor: 2
        maxDuration: 3m

Flux Overview

Flux v2

Flux is a set of continuous delivery solutions for Kubernetes that are open and extensible. It uses GitOps Toolkit components:

  • Source Controller: Manages Git repos, Helm repos, S3 buckets as artifact sources
  • Kustomize Controller: Reconciles Kustomization resources, applies manifests
  • Helm Controller: Manages Helm chart releases via HelmRelease CRDs
  • Notification Controller: Handles inbound/outbound events (webhooks, alerts)
  • Image Automation: Scans container registries, updates Git with new image tags

Drift Detection & Auto-Sync

Drift Detection

GitOps operators continuously compare:

  • Desired State: Manifests in Git repository
  • Live State: Actual resources in the cluster

Drift occurs when someone runs kubectl edit, kubectl scale, or any manual change outside Git.

Auto-Sync Behavior

  • selfHeal: true - Reverts manual changes back to Git state
  • prune: true - Deletes resources removed from Git
  • Default poll: Every 3 minutes (or webhook for instant)
  • Sync Windows: Restrict when auto-sync can happen

3. Helm

What is Helm?

Helm is the package manager for Kubernetes. It packages K8s manifests into reusable charts, allowing templating, versioning, and dependency management of complex applications.

Key concepts:

  • Chart: A package of pre-configured Kubernetes resources
  • Release: An instance of a chart running in a cluster
  • Repository: A place to store and share charts
  • Values: Configuration that customizes a chart at install time

4. GitHub Actions

GitHub Actions Concepts

  • Workflow: Automated process defined in YAML, triggered by events
  • Job: A set of steps that run on the same runner
  • Step: Individual task (run command or use an action)
  • Action: Reusable unit of code (marketplace or custom)
  • Runner: Server that executes workflows (GitHub-hosted or self-hosted)
.github/workflows/ci.yaml GitHub Actions 
name: CI/CD Pipeline

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

env:
  REGISTRY: ghcr.io
  IMAGE_NAME: ${{ github.repository }}

jobs:
  # ===========================
  # Job 1: Lint & Test
  # ===========================
  test:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      - name: Run linter
        run: npm run lint

      - name: Run unit tests
        run: npm test -- --coverage

      - name: Upload coverage
        uses: actions/upload-artifact@v4
        with:
          name: coverage-report
          path: coverage/

  # ===========================
  # Job 2: Security Scan
  # ===========================
  security:
    runs-on: ubuntu-latest
    needs: test
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Run Trivy vulnerability scanner
        uses: aquasecurity/trivy-action@master
        with:
          scan-type: 'fs'
          scan-ref: '.'
          severity: 'CRITICAL,HIGH'
          exit-code: '1'

      - name: Run Gitleaks (secret detection)
        uses: gitleaks/gitleaks-action@v2
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}

  # ===========================
  # Job 3: Build & Push Docker
  # ===========================
  build:
    runs-on: ubuntu-latest
    needs: [test, security]
    if: github.event_name == 'push' && github.ref == 'refs/heads/main'
    permissions:
      contents: read
      packages: write
    outputs:
      image-tag: ${{ steps.meta.outputs.tags }}
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Setup Docker Buildx
        uses: docker/setup-buildx-action@v3

      - name: Login to GitHub Container Registry
        uses: docker/login-action@v3
        with:
          registry: ${{ env.REGISTRY }}
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Extract metadata
        id: meta
        uses: docker/metadata-action@v5
        with:
          images: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}
          tags: |
            type=sha,prefix=
            type=semver,pattern={{version}}
            type=raw,value=latest,enable={{is_default_branch}}

      - name: Build and push Docker image
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: ${{ steps.meta.outputs.tags }}
          labels: ${{ steps.meta.outputs.labels }}
          cache-from: type=gha
          cache-to: type=gha,mode=max

  # ===========================
  # Job 4: Deploy to Kubernetes
  # ===========================
  deploy:
    runs-on: ubuntu-latest
    needs: build
    environment: production
    steps:
      - name: Checkout manifests repo
        uses: actions/checkout@v4
        with:
          repository: org/k8s-manifests
          token: ${{ secrets.DEPLOY_TOKEN }}
          path: manifests

      - name: Update image tag in manifests
        run: |
          cd manifests/apps/my-app/overlays/production
          kustomize edit set image my-app=${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ github.sha }}

      - name: Commit and push manifest change
        run: |
          cd manifests
          git config user.name "github-actions[bot]"
          git config user.email "github-actions[bot]@users.noreply.github.com"
          git add .
          git commit -m "deploy: update my-app to ${{ github.sha }}"
          git push

      # ArgoCD will detect the change and sync automatically
      - name: Wait for ArgoCD sync
        run: |
          argocd app wait my-app --timeout 300 --health

5. Artifact Management

Immutable Tags

Never overwrite image tags. Use unique identifiers:

  • myapp:abc1234 (commit SHA)
  • myapp:1.2.3 (semantic version)
  • myapp:1.2.3-abc1234 (version + SHA)

Never use: :latest in production (mutable, non-deterministic)

Semantic Versioning

MAJOR.MINOR.PATCH

  • MAJOR: Breaking changes
  • MINOR: New features (backwards-compatible)
  • PATCH: Bug fixes (backwards-compatible)

Example: 2.1.0 -> 2.2.0 (new feature)

Vulnerability Scanning

Scan at multiple stages:

  • Source code: SAST (Static Application Security Testing) - SonarQube, Semgrep
  • Dependencies: SCA (Software Composition Analysis) - Snyk, Dependabot
  • Container images: Trivy, Grype, Clair
  • Registry: Continuous scanning of stored images
  • Runtime: Falco, Sysdig for runtime behavior

Lifecycle Policies & Image Signing

  • Retention policies: Keep last N tags, delete untagged after X days
  • Image signing: Use Cosign/Sigstore to cryptographically sign images
  • Admission control: Only allow signed images (Kyverno, OPA Gatekeeper)
  • SBOM: Generate Software Bill of Materials for supply chain security

6. Deployment Strategies in CI/CD

Canary with ArgoCD Rollouts

canary-rollout.yaml Argo Rollouts 
apiVersion: argoproj.io/v1alpha1
kind: Rollout
metadata:
  name: my-app
spec:
  replicas: 10
  revisionHistoryLimit: 3
  selector:
    matchLabels:
      app: my-app
  template:
    metadata:
      labels:
        app: my-app
    spec:
      containers:
        - name: my-app
          image: myregistry.io/my-app:1.2.0
          ports:
            - containerPort: 8080
  strategy:
    canary:
      # Traffic splitting
      canaryService: my-app-canary
      stableService: my-app-stable
      trafficRouting:
        nginx:
          stableIngress: my-app-ingress

      # Step-by-step rollout
      steps:
        - setWeight: 5           # Send 5% traffic to canary
        - pause: { duration: 5m } # Wait 5 minutes

        - setWeight: 20          # Increase to 20%
        - pause: { duration: 5m }

        - setWeight: 50          # Increase to 50%
        - pause: { duration: 10m }

        - setWeight: 80          # Increase to 80%
        - pause: { duration: 5m }
        # Automatically promotes to 100% if no issues

      # Automated analysis
      analysis:
        templates:
          - templateName: success-rate
        startingStep: 2  # Start analysis at step 2
        args:
          - name: service-name
            value: my-app-canary

      # Auto-rollback on failure
      abortScaleDownDelaySeconds: 30
---
apiVersion: argoproj.io/v1alpha1
kind: AnalysisTemplate
metadata:
  name: success-rate
spec:
  args:
    - name: service-name
  metrics:
    - name: success-rate
      interval: 1m
      count: 5
      successCondition: result[0] >= 0.95
      failureLimit: 3
      provider:
        prometheus:
          address: http://prometheus:9090
          query: |
            sum(rate(http_requests_total{service="{{args.service-name}}",
              status=~"2.."}[5m])) /
            sum(rate(http_requests_total{service="{{args.service-name}}"}[5m]))

Feature Flags

Feature Flags in CI/CD

Decouple deployment from release. Deploy code with features behind flags, enable gradually.

  • Tools: LaunchDarkly, Unleash, Flagsmith, ConfigCat
  • Types: Boolean (on/off), Percentage (gradual rollout), User-based (beta users)
  • Benefits: Instant rollback (disable flag), A/B testing, trunk-based development
  • Risk: Technical debt if flags not cleaned up, testing complexity

7. Interview Q&A

Q1: How would you design a CI/CD pipeline for a microservices architecture?

Answer: I would implement a pipeline per service with shared pipeline templates. Each service pipeline includes: lint, unit test, build container image with commit SHA tag, security scan (SAST + container scan), push to registry, update GitOps repo with new image tag. For the deployment side, I would use ArgoCD watching the GitOps repo. Shared libraries and base images have their own pipelines that trigger downstream rebuilds. Monorepo? Use path-based triggers. Key: each service is independently deployable.

Q2: What is the difference between push-based and pull-based GitOps?

Answer: Push-based (traditional): CI system has cluster credentials and pushes changes directly (kubectl apply). Pull-based (true GitOps): An agent running inside the cluster (ArgoCD, Flux) continuously polls a Git repo and reconciles the cluster state. Pull-based is more secure because cluster credentials never leave the cluster, provides automatic drift detection, and offers a complete audit trail through Git history. Push-based is simpler to set up but has a larger attack surface.

Q3: When would you choose Helm over Kustomize (or vice versa)?

Answer: Helm when: you need to distribute charts to external users, have complex templating logic, need release management (rollback by revision), or want a package ecosystem. Kustomize when: you prefer overlay-based customization without templating, want native kubectl integration, need to patch third-party manifests without forking, or have simpler environments (base + overlays). Many teams use both: Helm for third-party charts, Kustomize for in-house apps. ArgoCD supports both natively.

Q4: How do you handle rollbacks in a GitOps workflow?

Answer: In GitOps, rollback = git revert. Revert the commit that introduced the bad change, push to main, and the GitOps operator (ArgoCD/Flux) automatically reconciles to the previous good state. For Argo Rollouts: automated rollback via AnalysisTemplate (metrics-driven). For Helm: helm rollback release-name revision. Key principle: never manually fix the cluster -- always go through Git so the audit trail remains intact and drift detection stays clean.

Q5: How do you manage secrets in a CI/CD pipeline?

Answer: Never store secrets in Git or pipeline YAML. Use: (1) External Secrets Operator to sync from AWS Secrets Manager/Vault into K8s Secrets, (2) Sealed Secrets for encrypted secrets in Git (only the controller can decrypt), (3) SOPS with age/KMS for encrypted values files, (4) GitHub Actions secrets/OIDC for CI runtime secrets. Pipeline credentials should use short-lived tokens (OIDC federation) rather than static credentials.

Q6: How do you implement progressive delivery?

Answer: Progressive delivery extends CI/CD by gradually rolling out changes and validating with real traffic. Implementation: (1) Argo Rollouts with canary strategy - send 5% -> 20% -> 50% -> 100% traffic to new version, (2) At each step, run automated analysis (success rate, latency, error rate via Prometheus), (3) Auto-rollback if metrics degrade, (4) Combine with feature flags for additional control. This reduces blast radius: a bad deploy only affects 5% of users initially.

Q7: How would you set up CI/CD for a monorepo?

Answer: Use path-based triggers: only build/test services whose files changed. GitHub Actions: paths: ['services/api/**']. Implement shared workflow templates (reusable workflows). Dependency graph: if shared-lib changes, rebuild all dependent services. Challenges: build time (use caching aggressively), test isolation (each service has its own tests), and versioning (independent semver per service or repo-wide tags).

Q8: What happens when your CI/CD pipeline becomes the bottleneck?

Answer: Identify where time is spent: (1) Build speed - use multi-stage Docker builds, layer caching, BuildKit. (2) Test speed - parallelize tests, use test splitting, skip unchanged service tests. (3) Queue time - add more runners, use self-hosted runners with auto-scaling. (4) Deploy speed - use parallel deployments for independent services. (5) Approval bottlenecks - auto-approve for staging, require approval only for production. Metrics to track: lead time, deployment frequency, mean time to recovery.

8. Scenarios

Scenario 1: Pipeline Failing on Security Scan

Situation: A developer's PR is blocked because Trivy found a CRITICAL vulnerability in a base image dependency.

Investigation:

  1. Check the scan output: which CVE? Which package? Is it in the base image or app dependency?
  2. Check if a patched version exists: trivy image --severity CRITICAL myimage:tag
  3. If in base image: update FROM to latest patched version
  4. If in app dependency: update package version in package.json/requirements.txt
  5. If no fix available: document risk, add to allowlist with expiry, create tracking ticket

Prevention: Schedule weekly base image rebuilds, enable Dependabot/Renovate for automated dependency updates, have a process for accepting risk on unfixed CVEs.

Scenario 2: Rollback a Bad Canary Deployment

Situation: Canary deployment at 20% traffic. Error rate spiked from 0.1% to 5% for canary pods. AnalysisTemplate marked it as Failed.

What happens:

  1. Argo Rollouts detects AnalysisRun failure (success rate < 95% threshold)
  2. Rollout status changes to "Degraded"
  3. Traffic automatically shifts back to 100% stable (old) version
  4. Canary pods are scaled down after abortScaleDownDelaySeconds
  5. Team is notified via webhook (Slack/PagerDuty)

Post-mortem: Check canary pod logs, compare Prometheus metrics between stable and canary, identify the specific commit that caused the regression, fix, and re-deploy.

Scenario 3: ArgoCD Detects Drift

Situation: ArgoCD shows application as "OutOfSync". Someone ran kubectl scale deployment my-app --replicas=1 directly.

Resolution:

  1. If selfHeal: true is configured: ArgoCD automatically reverts the change back to Git-defined state (3 replicas). No action needed.
  2. If selfHeal is disabled: ArgoCD shows diff in UI. Team decides whether to sync (revert manual change) or update Git (if the change was intentional).
  3. Investigate who made the change and why (audit logs, kubectl get events).
  4. If the change was needed: update Git repo with the new replica count, commit, push. ArgoCD syncs and shows "Synced + Healthy".

Prevention: Enable selfHeal, restrict kubectl access with RBAC, train team on GitOps workflow (all changes through Git PRs).
