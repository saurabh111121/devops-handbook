Ansible

Agentless automation - configuration management, application deployment, and orchestration

1. What is Ansible

Core Characteristics

  • Agentless: No software to install on managed nodes. Uses SSH (Linux) or WinRM (Windows).
  • Push-Based: Control node pushes configuration to managed nodes (vs. pull-based like Puppet/Chef).
  • Idempotent: Running the same playbook multiple times produces the same result without side effects.
  • Declarative: You describe the desired state; Ansible figures out how to get there.
  • YAML-Based: Human-readable configuration (no custom DSL to learn).

Architecture

Ansible Architecture
+---------------------------+
|      Control Node         |
|  (Your workstation/CI)    |
|                           |
|  +---------------------+  |
|  | Ansible Engine      |  |
|  |   - Playbooks       |  |
|  |   - Inventory       |  |
|  |   - Modules         |  |
|  |   - Plugins         |  |
|  +---------------------+  |
+------------||--------------+
             ||
             || SSH / WinRM (no agent required)
             ||
    +--------||--------+--------+--------+
    |        ||        |        |        |
    v        v         v        v        v
+-------+ +-------+ +-------+ +-------+ +-------+
| Web 1 | | Web 2 | | DB 1  | | DB 2  | | Cache |
+-------+ +-------+ +-------+ +-------+ +-------+
  Managed Nodes (Linux/Windows servers, network devices, cloud resources)

Flow:
  1. Control node reads inventory (which hosts to manage)
  2. Reads playbook (what to do)
  3. Connects via SSH to managed nodes
  4. Executes modules on managed nodes
  5. Reports results back to control node
  6. Modules are copied, executed, then removed (no residue)

Control Node Requirements

  • Python 3.9+ installed
  • Ansible package installed (pip install ansible)
  • SSH access to managed nodes
  • Can be your laptop, a CI server, or a dedicated Ansible controller

Managed Node Requirements

  • Python 2.7+ or 3.5+ (most modules)
  • SSH server running (sshd)
  • No Ansible installation needed
  • User with appropriate privileges (sudo)

2. Inventory

What is Inventory?

The inventory defines which hosts Ansible manages. It can be a static file (INI or YAML) or dynamically generated from cloud providers, CMDBs, or scripts.

Static Inventory (INI Format)

inventory/hosts.ini INI 
[webservers]
web1.example.com ansible_host=10.0.1.10
web2.example.com ansible_host=10.0.1.11

[dbservers]
db1.example.com ansible_host=10.0.2.10 ansible_user=dbadmin
db2.example.com ansible_host=10.0.2.11 ansible_user=dbadmin

[loadbalancers]
lb1.example.com ansible_host=10.0.0.10

[production:children]
webservers
dbservers
loadbalancers

[production:vars]
ansible_ssh_private_key_file=~/.ssh/prod_key
ansible_python_interpreter=/usr/bin/python3

Static Inventory (YAML Format)

inventory/hosts.yml YAML 
all:
  children:

        db1.example.com:

        db2.example.com:
          ansible_host: 10.0.2.11

      children:
        webservers:

        env: production

3. Playbooks

Playbook Structure

A playbook is a YAML file containing one or more plays. Each play targets a group of hosts and defines a list of tasks to execute in order.

Playbook Anatomy
Playbook (YAML file)
  |
  +-- Play 1 (hosts: webservers)
  |     |-- vars:
  |     |-- tasks:
  |     |     |-- Task 1 (module: apt)
  |     |     |-- Task 2 (module: template)
  |     |     +-- Task 3 (module: service)
  |     +-- handlers:
  |           +-- Handler 1 (restart nginx)
  |
  +-- Play 2 (hosts: dbservers)
        |-- vars:
        |-- tasks:
        |     |-- Task 1 (module: apt)
        |     +-- Task 2 (module: postgresql_db)
        +-- handlers:
              +-- Handler 1 (restart postgresql)

Complete Playbook Example

playbooks/deploy-nginx.yml YAML 
---
- name: Deploy and configure Nginx web server
  hosts: webservers
  become: true
  vars:
    nginx_port: 80
    nginx_server_name: "{{ inventory_hostname }}"
    app_root: /var/www/app
    nginx_worker_processes: "{{ ansible_processor_vcpus }}"

  tasks:
    - name: Update apt cache
      ansible.builtin.apt:
        update_cache: true
        cache_valid_time: 3600

    - name: Install Nginx
      ansible.builtin.apt:
        name: nginx
        state: present
      notify: Start Nginx

    - name: Install required packages
      ansible.builtin.apt:
        name:
          - curl
          - vim
          - htop
          - certbot
        state: present

    - name: Create application directory
      ansible.builtin.file:
        path: "{{ app_root }}"
        state: directory
        owner: www-data
        group: www-data
        mode: '0755'

    - name: Deploy Nginx configuration
      ansible.builtin.template:
        src: templates/nginx.conf.j2
        dest: /etc/nginx/sites-available/default
        owner: root
        group: root
        mode: '0644'
      notify: Reload Nginx
      when: nginx_port is defined

    - name: Deploy application index page
      ansible.builtin.copy:
        content: |
          <html>
            <body>
              <h1>Welcome to {{ nginx_server_name }}</h1>
              <p>Deployed by Ansible</p>
            </body>
          </html>
        dest: "{{ app_root }}/index.html"
        owner: www-data
        group: www-data
        mode: '0644'

    - name: Ensure Nginx is enabled and started
      ansible.builtin.service:
        name: nginx
        state: started
        enabled: true

    - name: Open firewall port for HTTP
      ansible.builtin.ufw:
        rule: allow
        port: "{{ nginx_port }}"
        proto: tcp
      when: ansible_os_family == "Debian"

    - name: Verify Nginx is responding
      ansible.builtin.uri:
        url: "http://localhost:{{ nginx_port }}"
        status_code: 200
      register: result
      retries: 3
      delay: 5
      until: result.status == 200

  handlers:
    - name: Start Nginx
      ansible.builtin.service:
        name: nginx
        state: started

    - name: Reload Nginx
      ansible.builtin.service:
        name: nginx
        state: reloaded

    - name: Restart Nginx
      ansible.builtin.service:
        name: nginx
        state: restarted

Variables, Conditionals & Loops

playbooks/users.yml (conditionals and loops) YAML 
---
- name: Manage system users
  hosts: all
  become: true
  vars:
    users:
      - name: deploy
        groups: [sudo, docker]
        shell: /bin/bash
        state: present
      - name: monitor
        groups: [docker]
        shell: /bin/bash
        state: present
      - name: olduser
        state: absent

  tasks:
    - name: Create or remove users
      ansible.builtin.user:
        name: "{{ item.name }}"
        groups: "{{ item.groups | default([]) }}"
        shell: "{{ item.shell | default('/bin/bash') }}"
        state: "{{ item.state }}"
        create_home: "{{ true if item.state == 'present' else omit }}"
      loop: "{{ users }}"
      when: item.name != "root"

    - name: Add SSH keys for present users
      ansible.posix.authorized_key:
        user: "{{ item.name }}"
        key: "{{ lookup('file', 'files/ssh_keys/' + item.name + '.pub') }}"
        state: present
      loop: "{{ users | selectattr('state', 'equalto', 'present') | list }}"
      ignore_errors: true

    - name: Install packages based on OS family
      ansible.builtin.package:
        name: "{{ item }}"
        state: present
      loop:
        - git
        - curl
        - wget
      when: ansible_os_family in ['Debian', 'RedHat']

4. Roles

What are Roles?

Roles are a way to organize playbooks into reusable, shareable components. They provide a standardized directory structure that automatically loads vars, tasks, handlers, templates, and files.

Role Directory Structure

Standard Role Layout
roles/
  nginx/
  +-- defaults/
  |     +-- main.yml          # Default variables (lowest priority)
  +-- vars/
  |     +-- main.yml          # Role variables (higher priority)
  +-- tasks/
  |     +-- main.yml          # Main task list
  |     +-- install.yml       # Included task file
  |     +-- configure.yml     # Included task file
  +-- handlers/
  |     +-- main.yml          # Handler definitions
  +-- templates/
  |     +-- nginx.conf.j2     # Jinja2 templates
  |     +-- vhost.conf.j2
  +-- files/
  |     +-- custom-502.html   # Static files to copy
  +-- meta/
  |     +-- main.yml          # Role metadata & dependencies
  +-- tests/
  |     +-- test.yml          # Test playbook
  +-- molecule/               # Molecule test scenarios
        +-- default/
              +-- molecule.yml
              +-- converge.yml
              +-- verify.yml

Example Role

roles/nginx/defaults/main.yml YAML 
---
nginx_port: 80
nginx_worker_processes: auto
nginx_worker_connections: 1024
nginx_server_name: "_"
nginx_root: /var/www/html
nginx_access_log: /var/log/nginx/access.log
nginx_error_log: /var/log/nginx/error.log
nginx_ssl_enabled: false
nginx_ssl_certificate: ""
nginx_ssl_certificate_key: ""
roles/nginx/tasks/main.yml YAML 
---
- name: Include OS-specific variables
  ansible.builtin.include_vars: "{{ ansible_os_family }}.yml"

- name: Install Nginx
  ansible.builtin.include_tasks: install.yml

- name: Configure Nginx
  ansible.builtin.include_tasks: configure.yml
  tags: [nginx, configure]

- name: Ensure Nginx is running
  ansible.builtin.service:
    name: nginx
    state: started
    enabled: true
  tags: [nginx, service]
roles/nginx/meta/main.yml YAML 
---
galaxy_info:
  author: DevOps Team
  description: Install and configure Nginx
  license: MIT
  min_ansible_version: "2.14"
  platforms:
    - name: Ubuntu
      versions: [focal, jammy]
    - name: Debian
      versions: [bullseye, bookworm]

dependencies:
  - role: common
  - role: firewall
    vars:
      firewall_allowed_ports:
        - "{{ nginx_port }}/tcp"

Using Roles in a Playbook

site.yml YAML 
---
- name: Configure web servers
  hosts: webservers
  become: true
  roles:
    - common
    - role: nginx
      vars:
        nginx_port: 443
        nginx_ssl_enabled: true
    - role: app
      tags: [app, deploy]

- name: Configure database servers
  hosts: dbservers
  become: true
  roles:
    - common
    - postgresql
    - monitoring

5. Modules

What are Modules?

Modules are the units of work in Ansible. Each module handles a specific task (install a package, copy a file, manage a service). Ansible ships with thousands of modules organized into collections.

Module Purpose Example Usage
apt Manage Debian/Ubuntu packages apt: name=nginx state=present
yum / dnf Manage RHEL/CentOS packages dnf: name=httpd state=latest
copy Copy files to remote hosts copy: src=app.conf dest=/etc/app/
template Render Jinja2 templates to remote template: src=nginx.conf.j2 dest=/etc/nginx/nginx.conf
service Manage system services service: name=nginx state=restarted
file Manage files, directories, symlinks file: path=/opt/app state=directory mode='0755'
user Manage user accounts user: name=deploy groups=sudo
shell Run shell commands (not idempotent) shell: /opt/app/migrate.sh
command Run commands (no shell features) command: /usr/bin/make install
lineinfile Ensure a line in a file lineinfile: path=/etc/hosts line="10.0.0.1 app"
docker_container Manage Docker containers docker_container: name=app image=myapp:latest
k8s Manage Kubernetes resources k8s: state=present src=deployment.yml

6. Templates (Jinja2)

Jinja2 Basics

  • {{ variable }} - Output a variable value
  • {% for item in list %} - Loop construct
  • {% if condition %} - Conditional
  • {# comment #} - Comment (not rendered)
  • {{ var | default('fallback') }} - Filters
templates/nginx.conf.j2 Jinja2 
{# Nginx configuration managed by Ansible - DO NOT EDIT MANUALLY #}

error_log {{ nginx_error_log }};

    worker_connections {{ nginx_worker_connections }};

http {
    include /etc/nginx/mime.types;
    default_type application/octet-stream;

    log_format main '$remote_addr - $remote_user [$time_local] '
                    '"$request" $status $body_bytes_sent '
                    '"$http_referer" "$http_user_agent"';

    access_log {{ nginx_access_log }} main;

    sendfile on;
    keepalive_timeout 65;

{% if nginx_ssl_enabled %}
    # SSL Configuration
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers HIGH:!aNULL:!MD5;
    ssl_prefer_server_ciphers on;
    ssl_session_cache shared:SSL:10m;
{% endif %}

{% for vhost in nginx_vhosts | default([]) %}
    server {
        listen {{ vhost.port | default(80) }};
        server_name {{ vhost.server_name }};
        root {{ vhost.root | default(nginx_root) }};

{% if vhost.ssl | default(false) %}
        listen 443 ssl;
        ssl_certificate {{ vhost.ssl_cert }};
        ssl_certificate_key {{ vhost.ssl_key }};
{% endif %}

{% if vhost.locations is defined %}
{% for location in vhost.locations %}
        location {{ location.path }} {
            proxy_pass {{ location.proxy_pass }};
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
{% if location.websocket | default(false) %}
            proxy_http_version 1.1;
            proxy_set_header Upgrade $http_upgrade;
            proxy_set_header Connection "upgrade";
{% endif %}
        }
{% endfor %}
{% endif %}
    }
{% endfor %}

{% if nginx_vhosts is not defined or nginx_vhosts | length == 0 %}
    server {
        listen {{ nginx_port }};
        server_name {{ nginx_server_name }};
        root {{ nginx_root }};

        location / {
            try_files $uri $uri/ =404;
        }
    }
{% endif %}
}

7. Ansible Vault

Never commit plain-text secrets!

Ansible Vault encrypts sensitive data (passwords, API keys, certificates) so they can be safely stored in version control alongside your playbooks.

Vault Commands

ansible-vault create secrets.yml

Create a new encrypted file (opens editor).

ansible-vault encrypt existing-file.yml

Encrypt an existing file in-place.

ansible-vault decrypt secrets.yml

Decrypt a file (use carefully, re-encrypt after editing).

ansible-vault edit secrets.yml

Edit encrypted file in-place (decrypts, opens editor, re-encrypts).

ansible-vault view secrets.yml

View encrypted file contents without decrypting on disk.

encrypt_string (Inline Encryption)

Encrypt a single variable Shell 
# Encrypt a string for use in a YAML file
ansible-vault encrypt_string 'SuperSecretPassword123' --name 'db_password'

# Output (paste into your vars file):
db_password: !vault |
  $ANSIBLE_VAULT;1.1;AES256
  613436386535383739363...
  363438343064623535363...

Using Vault in Playbooks

Running playbook with vault Shell 
# Prompt for vault password
ansible-playbook site.yml --ask-vault-pass

# Use a password file (for CI/CD)
ansible-playbook site.yml --vault-password-file ~/.vault_pass

# Multiple vault IDs (for different environments)
ansible-playbook site.yml \
  --vault-id prod@~/.vault_pass_prod \
  --vault-id dev@~/.vault_pass_dev
group_vars/production/vault.yml (encrypted) YAML 
---
vault_db_password: "SuperSecretPassword123"
vault_api_key: "ak_live_abc123def456"
vault_ssl_private_key: |
  -----BEGIN PRIVATE KEY-----
  MIIEvgIBADANBgkqhkiG9...
  -----END PRIVATE KEY-----
group_vars/production/vars.yml (references vault vars) YAML 
---
# Convention: prefix vault variables with vault_
# Then reference them in regular vars for clarity

ssl_private_key: "{{ vault_ssl_private_key }}"

8. Ansible vs Terraform

Aspect Ansible Terraform
Primary Purpose Configuration management & app deployment Infrastructure provisioning
Approach Procedural (tasks run in order) Declarative (desired state, dependency graph)
Infrastructure Type Mutable (modify in place) Immutable (destroy and recreate)
State Management Stateless (no state file) Stateful (terraform.tfstate tracks resources)
Agent Agentless (SSH/WinRM) Agentless (API calls)
Language YAML (playbooks) HCL (HashiCorp Configuration Language)
Idempotency Module-dependent (most are idempotent) Built-in (plan/apply lifecycle)
Best For OS config, app deploy, patching, orchestration Cloud infra, networking, managed services
Complementary Use Terraform provisions infrastructure, Ansible configures it

Common Pattern: Terraform + Ansible Together

Use Terraform to provision VMs/networking/cloud resources, then use Ansible to configure the OS, deploy applications, and manage ongoing configuration. Terraform outputs (IP addresses, DNS names) feed into Ansible's dynamic inventory.

9. Best Practices

Idempotency

  • Use modules instead of shell/command
  • Add creates: or removes: to shell tasks
  • Use changed_when and failed_when for custom logic
  • Test with --check mode (dry run)

Tags & Organization

  • Tag tasks for selective execution
  • Use ansible-playbook site.yml --tags deploy
  • Group related tasks into roles
  • Use include_tasks for logical separation

Handlers

  • Use handlers for service restarts (only trigger on change)
  • Handlers run at end of play (or on flush_handlers)
  • Name handlers clearly: "Restart Nginx"
  • Use listen for multiple triggers on one handler

Testing with Molecule

  • Use Molecule for role testing
  • Test against multiple OS versions
  • Use Docker or Vagrant as test drivers
  • Include lint, syntax, converge, verify, idempotence steps

10. Interview Q&A

Q1: What is Ansible and how does it differ from other configuration management tools?

Ansible is an agentless, push-based automation tool that uses SSH to configure systems. Unlike Chef and Puppet (which require agents on managed nodes and use a pull model), Ansible has zero footprint on target systems. It uses simple YAML playbooks instead of Ruby (Chef) or a custom DSL (Puppet), making it easier to learn. It's also procedural but idempotent, meaning tasks execute in order but repeated runs don't cause unintended changes.

Q2: What does idempotency mean in the context of Ansible?

Idempotency means running a playbook multiple times produces the same end state without side effects. For example, apt: name=nginx state=present will install Nginx on the first run but do nothing on subsequent runs if it's already installed. This is critical for safe re-runs - you can run a playbook after a failure without worrying about duplicate actions. Non-idempotent modules like shell require guards (creates:, when:) to achieve idempotency.

Q3: What is the difference between a playbook and a role?

A playbook is a YAML file that defines plays (target hosts + tasks). A role is a reusable, structured package of tasks, variables, templates, handlers, and files organized in a standard directory layout. Roles promote reuse and separation of concerns - a playbook orchestrates which roles to apply to which hosts. Think of a playbook as the "recipe" and roles as the "ingredients."

Q4: How does Ansible compare to Terraform?

They serve different but complementary purposes. Terraform excels at provisioning cloud infrastructure (VMs, networks, databases) using a declarative approach with state tracking. Ansible excels at configuring those resources (installing software, deploying apps, managing configs). In practice, teams use both: Terraform creates the infrastructure, then Ansible configures it. Terraform is immutable (destroy/recreate); Ansible is mutable (modify in place).

Q5: How do you handle secrets in Ansible?

Use Ansible Vault to encrypt sensitive data. You can encrypt entire files (ansible-vault encrypt) or individual variables (ansible-vault encrypt_string). Best practice: keep vault-encrypted vars in separate files (e.g., vault.yml) and reference them from regular var files. For CI/CD, use --vault-password-file pointing to a file loaded from a secrets manager. Never commit vault passwords to git.

Q6: What is dynamic inventory and when would you use it?

Dynamic inventory automatically discovers hosts from external sources (AWS EC2, GCP, Azure, Kubernetes, LDAP) instead of maintaining static host lists. Use it when your infrastructure is dynamic (auto-scaling groups, ephemeral containers), when host counts change frequently, or when you want a single source of truth. Ansible provides inventory plugins (amazon.aws.aws_ec2, azure.azcollection.azure_rm) that query cloud APIs and group hosts by tags, regions, or other attributes.

Q7: What are handlers and how do they differ from regular tasks?

Handlers are tasks that only run when notified by another task that reports a change. They're typically used for service restarts/reloads. Key differences: 1) Handlers run at the end of a play (not immediately when notified). 2) They run only once even if notified multiple times. 3) They only run if at least one notifying task made a change. This prevents unnecessary service restarts when config hasn't actually changed.

Q8: How do you test Ansible roles?

Use Molecule, the standard testing framework for Ansible roles. It provides a full test lifecycle: lint (yamllint, ansible-lint), syntax check, create (spin up test instances via Docker/Vagrant/cloud), converge (run the role), idempotence (run again to verify no changes), verify (run assertions with testinfra/goss), and destroy (cleanup). This enables TDD for infrastructure code and CI/CD integration.

11. Real-World Scenarios

⚠ Scenario: Playbook Failing on One Host

Situation: A playbook runs successfully on 9 of 10 hosts but fails on one with a package installation error.

1. Run with verbose output to get detailed error messages:
ansible-playbook site.yml --limit failed-host -vvv
2. Check the host's facts (OS version might differ):
ansible failed-host -m ansible.builtin.setup -a "filter=ansible_distribution*"
3. Common causes: Different OS version, stale package cache, disk full, different Python version, network connectivity to package repos.
4. Fix: Add a conditional or pre-task to handle the edge case:
Conditional task
- name: Update apt cache (Debian/Ubuntu)
  ansible.builtin.apt:
    update_cache: true
    cache_valid_time: 3600
  when: ansible_os_family == "Debian"

- name: Update yum cache (RHEL/CentOS)
  ansible.builtin.yum:
    update_cache: true
  when: ansible_os_family == "RedHat"
5. Retry just the failed host:
ansible-playbook site.yml --limit @site.retry
🔧 Scenario: Handling Different Operating Systems

Situation: Your fleet has both Ubuntu and CentOS servers. You need a role that works on both.

1. Use OS-specific variable files:
roles/webserver/vars/Debian.yml
---
package_name: nginx
service_name: nginx
config_path: /etc/nginx/sites-available/default
roles/webserver/vars/RedHat.yml
---
package_name: nginx
service_name: nginx
config_path: /etc/nginx/conf.d/default.conf
2. Load variables dynamically based on OS facts:
roles/webserver/tasks/main.yml
- name: Load OS-specific variables
  ansible.builtin.include_vars: "{{ ansible_os_family }}.yml"

- name: Install web server package
  ansible.builtin.package:
    name: "{{ package_name }}"
    state: present
3. Use the generic package module instead of apt/yum when possible for cross-platform compatibility.
🔒 Scenario: Managing Secrets Across Environments

Situation: You need different secrets for dev, staging, and production, all stored securely in Git.

1. Structure vault files per environment: 
inventory/
  production/
    group_vars/
      all/
        vars.yml         # db_password: "{{ vault_db_password }}"
        vault.yml        # vault_db_password: (encrypted)
  staging/
    group_vars/
      all/
        vars.yml
        vault.yml
2. Use different vault passwords per environment:
ansible-playbook -i inventory/production site.yml --vault-id prod@vault_pass_prod
ansible-playbook -i inventory/staging site.yml --vault-id staging@vault_pass_staging
3. For CI/CD, store vault passwords in your CI secrets store (GitHub Secrets, GitLab CI variables, AWS Secrets Manager) and write to a temp file during pipeline execution.
4. Alternative: External lookup plugins - retrieve secrets at runtime from HashiCorp Vault, AWS Secrets Manager, or Azure Key Vault using lookup plugins:
Using HashiCorp Vault lookup
- name: Get database password from HashiCorp Vault
  set_fact:
    db_password: "{{ lookup('hashi_vault', 'secret/data/production/db:password') }}"
