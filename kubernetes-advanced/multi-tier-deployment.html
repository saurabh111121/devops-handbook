Multi-Tier Application Deployment Patterns

Complete, production-ready patterns for deploying multi-tier applications on Kubernetes. Includes full YAML manifests, networking, scaling strategies, and real-world troubleshooting.

1. What is Multi-Tier Architecture?

Definition

Multi-tier architecture separates an application into distinct layers (tiers), each responsible for a specific concern. Each tier can be developed, deployed, scaled, and maintained independently.

Classic 3-Tier Architecture

+-------------------+       +-------------------+       +-------------------+
|                   |       |                   |       |                   |
|   PRESENTATION    |       |    APPLICATION    |       |       DATA        |
|    (Frontend)     | ----> |   (Backend/API)   | ----> |    (Database)     |
|                   |       |                   |       |                   |
|  React / Angular  |       |  Node.js / Java   |       | PostgreSQL/MySQL  |
|  Served by Nginx  |       |  REST/gRPC API    |       | Redis (Cache)     |
|                   |       |                   |       |                   |
+-------------------+       +-------------------+       +-------------------+
      Port 80/443                Port 3000                  Port 5432
     (User-facing)             (Internal only)            (Internal only)

Why Separate Tiers?

1
Independent Scaling

Scale the frontend to handle traffic spikes without scaling the database. Scale the backend independently based on API load.

2
Fault Isolation

A crash in the frontend tier does not take down the database. Each tier can fail and recover independently.

3
Team Ownership

Frontend team owns the presentation tier, backend team owns the API, and DBAs own the data layer. Clear boundaries reduce conflicts.

4
Security Boundaries

Only the frontend is exposed to the internet. The backend is only reachable from the frontend. The database is only reachable from the backend. Network policies enforce this.

Concrete Example

Tier Technology K8s Resource Exposed To
Frontend React app served by Nginx Deployment + Service + Ingress Internet (via Ingress)
Backend Node.js Express API Deployment + Service + HPA Frontend only (ClusterIP)
Cache Redis Deployment + Service Backend only (ClusterIP)
Database PostgreSQL 15 StatefulSet + Headless Service + PVC Backend only (ClusterIP)

Overview

Below is a complete, production-ready set of Kubernetes manifests for a 3-tier application with a Redis cache layer. All resources are deployed in the myapp namespace. Copy-paste these YAMLs and apply them with kubectl apply -f.

Frontend Tier (React/Nginx)

Frontend ConfigMap (nginx.conf)

frontend-nginx-config.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: frontend-nginx-config
  namespace: myapp
  labels:
    app: frontend
    tier: presentation
data:
  nginx.conf: |
    user nginx;
    worker_processes auto;
    error_log /var/log/nginx/error.log warn;
    pid /tmp/nginx.pid;

    events {
        worker_connections 1024;
    }

    http {
        include /etc/nginx/mime.types;
        default_type application/octet-stream;

        log_format main '$remote_addr - $remote_user [$time_local] "$request" '
                        '$status $body_bytes_sent "$http_referer" '
                        '"$http_x_request_id"';

        access_log /var/log/nginx/access.log main;
        sendfile on;
        keepalive_timeout 65;
        gzip on;
        gzip_types text/plain text/css application/json application/javascript;

        # Temp paths for non-root
        client_body_temp_path /tmp/client_temp;
        proxy_temp_path /tmp/proxy_temp;
        fastcgi_temp_path /tmp/fastcgi_temp;
        uwsgi_temp_path /tmp/uwsgi_temp;
        scgi_temp_path /tmp/scgi_temp;

        server {
            listen 8080;
            server_name _;
            root /usr/share/nginx/html;
            index index.html;

            # Health check endpoint
            location /health {
                access_log off;
                return 200 'healthy';
                add_header Content-Type text/plain;
            }

            # Proxy API requests to backend service
            location /api/ {
                proxy_pass http://backend-svc.myapp.svc.cluster.local:3000/api/;
                proxy_set_header Host $host;
                proxy_set_header X-Real-IP $remote_addr;
                proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
                proxy_set_header X-Request-ID $http_x_request_id;
                proxy_connect_timeout 5s;
                proxy_read_timeout 30s;
            }

            # SPA fallback - serve index.html for all routes
            location / {
                try_files $uri $uri/ /index.html;
            }

            # Cache static assets
            location ~* \.(js|css|png|jpg|jpeg|gif|ico|svg|woff2)$ {
                expires 1y;
                add_header Cache-Control "public, immutable";
            }
        }
    }

Frontend Deployment

frontend-deployment.yaml

kind: Deployment
metadata:

      securityContext:
        runAsNonRoot: true
        runAsUser: 101
        runAsGroup: 101
        fsGroup: 101

      containers:
        - name: nginx
          image: myregistry.io/myapp-frontend:1.2.0

            - name: http
              containerPort: 8080
              protocol: TCP

            requests:
              cpu: 50m
              memory: 64Mi
            limits:
              cpu: 200m
              memory: 128Mi
          livenessProbe:
            httpGet:
              path: /health
              port: http
            initialDelaySeconds: 5
            periodSeconds: 15
            timeoutSeconds: 3
            failureThreshold: 3
          readinessProbe:
            httpGet:
              path: /health
              port: http
            initialDelaySeconds: 3
            periodSeconds: 5
            timeoutSeconds: 2
            failureThreshold: 2
          securityContext:
            allowPrivilegeEscalation: false
            readOnlyRootFilesystem: true

PostgreSQL Headless Service

postgres-service.yaml
apiVersion: v1
kind: Service
metadata:
  name: postgres-svc
  namespace: myapp
  labels:
    app: postgres
    tier: data
spec:
  type: ClusterIP
  clusterIP: None  # Headless service for StatefulSet
  selector:
    app: postgres
    tier: data
  ports:
    - name: postgresql
      port: 5432
      targetPort: postgresql
      protocol: TCP

Why Headless Service for StatefulSets?

A headless service (clusterIP: None) creates DNS records for each pod individually: postgres-0.postgres-svc.myapp.svc.cluster.local. This provides stable network identity for stateful workloads, allowing clients to connect to specific pods (important for primary/replica topologies).

4. Networking Between Tiers

Service DNS Resolution

From To DNS Name Port
Frontend (nginx proxy_pass) Backend backend-svc.myapp.svc.cluster.local 3000
Backend PostgreSQL postgres-svc.myapp.svc.cluster.local 5432
Backend PostgreSQL (direct pod) postgres-0.postgres-svc.myapp.svc.cluster.local 5432
Backend Redis redis-svc.myapp.svc.cluster.local 6379

DNS Short Names

Within the same namespace, you can use the short service name: backend-svc instead of backend-svc.myapp.svc.cluster.local. However, using the FQDN is recommended in production for clarity and to avoid namespace confusion.

Network Policies for Tier Isolation

Network Policies enforce that each tier can only communicate with its adjacent tier. This prevents a compromised frontend pod from directly accessing the database.

Default Deny All (Baseline)

netpol-default-deny.yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-all
  namespace: myapp
spec:
  podSelector: {}  # Applies to ALL pods in namespace
  policyTypes:
    - Ingress
    - Egress

Allow DNS Egress (Required for all pods)

netpol-allow-dns.yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-dns-egress
  namespace: myapp
spec:
  podSelector: {}  # All pods
  policyTypes:
    - Egress
  egress:
    - to:
        - namespaceSelector:
            matchLabels:
              kubernetes.io/metadata.name: kube-system
          podSelector:
            matchLabels:
              k8s-app: kube-dns
      ports:
        - protocol: UDP
          port: 53
        - protocol: TCP
          port: 53

Frontend Network Policy

netpol-frontend.yaml
apiVersion: networking.k8s.io/v1

6. Scaling Strategies

Scaling Strategy per Tier:

+------------------+     +------------------+     +------------------+     +------------------+
|    FRONTEND      |     |     BACKEND      |     |     DATABASE     |     |      CACHE       |
+------------------+     +------------------+     +------------------+     +------------------+
|                  |     |                  |     |                  |     |                  |
| HPA: CPU 60%    |     | HPA: CPU 70%    |     | VPA: Vertical    |     | Single instance  |
| Min: 3, Max: 20 |     | Min: 3, Max: 20 |     | Scale UP memory  |     | (fast enough)    |
|                  |     | + Custom metric: |     | and CPU          |     |                  |
| Stateless =     |     |   req/sec > 100  |     |                  |     | Or Redis Cluster |
| easy horizontal  |     |                  |     | Read replicas    |     | for >100K ops/s  |
|                  |     | Scale fast,      |     | for read-heavy   |     |                  |
| Scale in seconds |     | cool down slow   |     | Scale in minutes |     | maxmemory-policy |
+------------------+     +------------------+     +------------------+     +------------------+
       |                        |                        |                        |
       v                        v                        v                        v
  3 ──> 20 pods           3 ──> 20 pods           1 ──> bigger pod          1 instance
  (horizontal)            (horizontal)            (vertical)               (or 6-node cluster)

Frontend Scaling (Stateless)

1
Horizontal Pod Autoscaler on CPU

Frontend pods serve static files via Nginx. They are completely stateless and can scale horizontally with zero session concerns. Scale on CPU utilization at 60% threshold with aggressive scale-up and conservative scale-down.

Backend Scaling (Stateless with connections)

2
HPA on CPU + Custom Metrics

Backend pods are stateless but maintain DB connection pools. Scale on CPU (70%) and custom metrics (requests per second). Use connection pooling (PgBouncer) to prevent database connection exhaustion as pods scale. Scale-down window should be longer (5 min) to avoid flapping.

Database Scaling (Stateful)

3
Vertical Scaling + Read Replicas

Databases cannot simply add replicas like stateless services. Scale vertically first (VPA or manual resource increase). For read-heavy workloads, add read replicas with a separate read-only service. Consider managed databases (RDS, Cloud SQL) for production.

Cache Scaling

4
Single Instance or Cluster Mode

A single Redis instance handles 100K+ operations/second. Only scale to Redis Cluster if you need more memory than a single node provides or need >100K ops/sec. Use maxmemory-policy (allkeys-lru) to handle memory pressure gracefully.

Connection Pool Exhaustion

When the backend scales from 3 to 20 pods, each with a pool of 10 connections, you go from 30 to 200 database connections. PostgreSQL's default max_connections is 100. Always use a connection pooler like PgBouncer between backend and database to multiplex connections.

7. Health Checks Across Tiers

Tier Liveness Probe Readiness Probe What It Checks
Frontend HTTP GET /health HTTP GET /health Nginx is serving (does NOT check backend)
Backend HTTP GET /api/health/live HTTP GET /api/health/ready Liveness: process alive. Readiness: DB + Redis connected
Database exec: pg_isready exec: pg_isready PostgreSQL accepting connections
Redis exec: redis-cli ping exec: redis-cli ping Redis responding to commands

Cascading Failure Scenario

What happens when PostgreSQL goes down:

1. pg_isready fails on postgres-0
   └──> PostgreSQL readiness probe fails
        └──> postgres-0 removed from postgres-svc endpoints

2. Backend's /api/health/ready checks DB connection
   └──> DB connection fails
        └──> Backend readiness probe fails
             └──> backend pods removed from backend-svc endpoints

3. Frontend nginx proxy_pass to backend-svc gets no endpoints
   └──> Returns 502/503 to user
        └──> BUT frontend stays "Ready" (its own health check passes)
             └──> This is CORRECT: nginx is healthy, backend is not

RECOVERY: When DB comes back, readiness cascades UP:
   DB ready ──> backend ready ──> traffic flows again

Backend Health Check Implementation

health-check-example.js (Node.js)
// Liveness: Is the process alive? (lightweight, no dependencies)
app.get('/api/health/live', (req, res) => {
  res.status(200).json({ status: 'alive', uptime: process.uptime() });
});

// Readiness: Can this pod serve traffic? (checks dependencies)
app.get('/api/health/ready', async (req, res) => {
  try {
    // Check database connection (with timeout)
    await Promise.race([
      db.query('SELECT 1'),
      new Promise((_, reject) =>
        setTimeout(() => reject(new Error('DB timeout')), 2000)
      )
    ]);

    // Check Redis connection
    await Promise.race([
      redis.ping(),
      new Promise((_, reject) =>
        setTimeout(() => reject(new Error('Redis timeout')), 1000)
      )
    ]);

    res.status(200).json({ status: 'ready', db: 'connected', redis: 'connected' });
  } catch (error) {
    res.status(503).json({ status: 'not ready', error: error.message });
  }
});

Critical: Separate Liveness from Readiness

Never check external dependencies in your liveness probe. If the database is down, your backend pod is still alive -- it just cannot serve traffic. If you check DB in liveness, Kubernetes will restart your pods, causing a cascade of restarts that makes recovery harder. Use liveness to detect process hangs only; use readiness to signal traffic-serving ability.

8. Deployment Strategies for Multi-Tier

Deployment Order Matters

Correct deployment order for breaking changes:

  Step 1: Database Migration (Job)
    └── Run schema changes FIRST (add columns, create tables)
    └── Must be backward-compatible with current backend version

  Step 2: Backend Deployment (Rolling Update)
    └── New backend version uses new schema
    └── Wait until all pods are ready

  Step 3: Frontend Deployment (Rolling Update)
    └── New frontend uses new API endpoints
    └── Old frontend continues working during rollout

  Step 4: Cleanup Migration (Optional Job)
    └── Remove deprecated columns/tables
    └── Only AFTER all old backend pods are gone

Database Migration Job

db-migration-job.yaml

kind: Job
metadata:
  name: db-migrate-v1-2-0
  namespace: myapp
  labels:
    app: backend
    tier: application
    purpose: migration
spec:
  backoffLimit: 3
  activeDeadlineSeconds: 300
  template:

      labels:
        app: backend
        purpose: migration

      restartPolicy: Never
      securityContext:
        runAsNonRoot: true
        runAsUser: 1000

      containers:

10. Interview Q&A

Q1: How would you design a 3-tier application on Kubernetes?

I would create three logical layers in a dedicated namespace:

Frontend: Deployment with Nginx serving static React files, exposed via Ingress with TLS. The nginx.conf includes a proxy_pass directive to route /api/ requests to the backend service.

Backend: Deployment with the API server (Node.js/Java), using ConfigMaps for configuration and Secrets for credentials. Exposed via ClusterIP service (not accessible from outside). HPA for auto-scaling.

Database: StatefulSet with a PersistentVolumeClaim for data durability. Exposed via a headless service for stable DNS. No external access.

Network Policies enforce that each tier can only communicate with its adjacent tier. This provides defense in depth -- a compromised frontend pod cannot directly access the database.

Q2: How do tiers communicate with each other in Kubernetes?

Tiers communicate via Kubernetes Services, which provide stable DNS names and load balancing:

DNS resolution: service-name.namespace.svc.cluster.local resolves to the service's ClusterIP. Within the same namespace, just service-name works.

Frontend to Backend: Nginx uses proxy_pass http://backend-svc:3000/api/ to forward API requests.

Backend to Database: Connection string uses the headless service DNS: postgres-0.postgres-svc.myapp.svc.cluster.local:5432 for StatefulSet pods, or just postgres-svc:5432 for the service.

Service types: All internal communication uses ClusterIP (default). Only the frontend is exposed externally via Ingress.

Q3: How do you handle database schema migrations in Kubernetes?

Use a Kubernetes Job that runs before the backend deployment:

1. Create a Job with the same image as the backend but with a migration command (e.g., node scripts/migrate.js --up).

2. The Job uses the same ConfigMap/Secret as the backend for DB credentials.

3. In CI/CD, wait for the Job to complete: kubectl wait --for=condition=complete job/db-migrate.

4. Only then deploy the new backend version.

Key principle: Migrations must be backward-compatible. Add new columns as nullable first, deploy new backend, then add NOT NULL constraint in a later migration. This allows rollback without breaking the old backend version.

Q4: How do you implement network isolation between tiers?

Using Kubernetes NetworkPolicies:

1. Start with a default-deny-all policy for both ingress and egress in the namespace.

2. Explicitly allow DNS egress for all pods (port 53 to kube-dns).

3. Frontend: Allow ingress from ingress-nginx namespace, allow egress only to backend pods on port 3000.

4. Backend: Allow ingress from frontend pods only, allow egress to postgres on 5432 and redis on 6379.

5. Database: Allow ingress from backend pods only, deny all egress.

This means even if an attacker compromises a frontend pod, they cannot reach the database directly. The CNI plugin must support NetworkPolicies (Calico, Cilium, etc.).

Q5: What is the scaling strategy for each tier?

Frontend: HPA on CPU (60%). Completely stateless, can scale from 3 to 20 pods in seconds. No session state, no connection concerns.

Backend: HPA on CPU (70%) + custom metrics (requests/sec). Scale-up quickly (30s stabilization), scale-down slowly (5min stabilization) to handle traffic bursts. Watch database connection pool -- use PgBouncer to avoid exhausting connections.

Database: Vertical scaling (VPA) or manual resource increase. For read-heavy workloads, add streaming replicas with a read-only service. Cannot scale horizontally without sharding. Consider managed DB (RDS) for production.

Cache: Single Redis handles 100K ops/sec. Only scale to Redis Cluster if memory exceeds single node capacity. Use maxmemory-policy=allkeys-lru to handle pressure gracefully.

Q6: What happens when the database goes down in a multi-tier app?

A cascading failure occurs through the readiness probe chain:

1. PostgreSQL's pg_isready fails, pod marked NotReady, removed from service endpoints.

2. Backend's readiness probe (which checks DB connection) fails after timeout. Backend pods marked NotReady.

3. Frontend's nginx proxy_pass gets no healthy endpoints, returns 502/503 to users.

4. Frontend remains Ready (its own health check passes) -- this is correct behavior.

Recovery: When DB comes back, readiness cascades upward: DB ready, backend ready, traffic flows. No pod restarts needed.

Key: Never check DB in liveness probe. If you do, Kubernetes restarts backend pods when DB is down, causing a restart storm that makes recovery harder.

Q7: How do you achieve zero-downtime deployment for a multi-tier app?

Several practices together ensure zero downtime:

1. Rolling update with maxSurge=1, maxUnavailable=0. New pod starts before old pod terminates.

2. Readiness probes ensure traffic only goes to ready pods. New pod must pass readiness before old pod receives SIGTERM.

3. preStop hook with sleep (5-10s) to allow in-flight requests to drain before container stops.

4. Backward-compatible migrations: New schema works with both old and new code.

5. Deploy order: DB migration first, then backend (handles both old and new schema), then frontend.

6. PodDisruptionBudget: Ensure at least N pods remain available during voluntary disruptions.

Q8: Blue/green vs canary for multi-tier applications?

Blue/Green: Deploy entire new stack (all tiers) alongside existing one. Switch traffic at ingress level. Pros: instant rollback (switch back), full testing of new version. Cons: 2x resources, database schema must be shared or migrated carefully.

Canary: Route a percentage of traffic (e.g., 10%) to new version of a single tier (usually backend). Pros: less resource usage, gradual validation, can catch issues before full rollout. Cons: two versions running simultaneously must be compatible, harder to implement for stateful tiers.

In practice for multi-tier: Use canary at the backend level (where business logic changes) and rolling update for frontend (less risky). Database changes must be backward-compatible regardless of strategy -- you cannot canary a schema change.

Q9: How do you manage secrets across tiers?

Multiple approaches in order of maturity:

1. Kubernetes Secrets (basic): Create Secret objects, reference via envFrom/secretRef. Enable encryption at rest in etcd. RBAC restricts who can read secrets.

2. Sealed Secrets (GitOps-friendly): Encrypt secrets with a cluster-specific key. Encrypted version is safe to commit to git. Controller decrypts in-cluster.

3. External Secrets Operator (production): Syncs secrets from external stores (AWS Secrets Manager, HashiCorp Vault, Azure Key Vault) into Kubernetes Secrets automatically. Supports rotation.

4. CSI Secrets Store: Mount secrets directly as volumes from external providers. No Kubernetes Secret object created.

Cross-tier sharing: Shared credentials (like DB password used by both the migration Job and backend) should reference the same Secret object. Use RBAC to restrict which ServiceAccounts can access which Secrets.

Q10: How do you debug latency in a multi-tier application on Kubernetes?

Systematic approach using observability data:

1. Identify which tier: Check the distributed trace (Jaeger/Zipkin) to see where time is spent. Is it frontend, backend processing, DB query, or network?

2. Per-tier investigation:

  - Frontend slow: Check nginx access logs for upstream_response_time. If high, issue is backend.

  - Backend slow: Check application metrics -- is it CPU-bound (scale up) or waiting on DB (check queries)?

  - DB slow: Check pg_stat_statements for slow queries. Look at lock waits, table bloat, missing indexes.

  - Network slow: Check inter-node latency. Are pods on different AZs? Is the CNI saturated?

3. Resource pressure: kubectl top pods to check CPU/memory. Pods hitting limits get throttled (CPU) or OOMKilled (memory).

4. Connection pooling: Check if backend is waiting for connections (pool exhaustion). pg_stat_activity shows active connections.

5. DNS resolution: Rarely, CoreDNS can be a bottleneck. Check if ndots:5 in resolv.conf causes excessive DNS queries.

11. Real-World Scenarios

Scenario 1: Backend Cannot Connect to Database After Deployment

Symptoms: Backend pods are in Running state but readiness probe fails. Logs show ECONNREFUSED or authentication failed for user.

Root Causes:

Cause Diagnosis Fix
Secret mismatch (new deploy references wrong secret name) kubectl describe pod shows warning about missing secret Fix secret name in deployment YAML
Network Policy blocking traffic kubectl exec backend-pod -- nc -zv postgres-svc 5432 times out Update NetworkPolicy to allow backend->postgres
DB not ready yet (StatefulSet slower to start) kubectl get pods -l app=postgres shows 0/1 Ready Add initContainer or startup probe with longer timeout
Wrong DB_HOST in ConfigMap after namespace change kubectl get configmap backend-config -o yaml shows old namespace Update ConfigMap with correct FQDN for new namespace
Debugging Commands
# Check if backend can reach postgres
kubectl exec -it deploy/backend -n myapp -- \
  sh -c "nc -zv postgres-svc 5432"

# Check what environment variables the pod sees
kubectl exec -it deploy/backend -n myapp -- env | grep DB_

# Check network policy effects
kubectl get networkpolicies -n myapp -o yaml

# Check postgres pod is ready
kubectl get pods -l app=postgres -n myapp

# Check events for clues

Scenario 2: Frontend Showing Stale Data

Symptoms: Users see old content. Refreshing does not help. API returns correct data when called directly.

Root Causes:

Cause Diagnosis Fix
Nginx caching responses from backend Check nginx.conf for proxy_cache directives or aggressive Cache-Control headers Add proxy_cache_bypass or set Cache-Control: no-cache for API routes
Frontend deployment not rolled (old image) kubectl get deploy frontend -o jsonpath='{.spec.template.spec.containers[0].image}' Update image tag, ensure imagePullPolicy is not IfNotPresent with :latest
CDN or browser cache serving old static files Check response headers for age, x-cache-hit Use content-hashed filenames (app.abc123.js) or invalidate CDN
Redis cache not invalidated after DB write GET key from Redis shows old value, DB has new value Implement cache invalidation on write or reduce TTL

Scenario 3: Database Migration Failed Mid-Deploy

Symptoms: Migration Job failed (status: Failed). Backend is partially deployed. Some users getting errors.

Recovery Steps:

1
Stop the deployment pipeline immediately

Cancel any in-progress backend rollout: kubectl rollout undo deployment/backend -n myapp

2
Assess migration state

Check migration logs: kubectl logs job/db-migrate-v1-2-0 -n myapp. Determine which migrations ran successfully and which failed. Check the migration tracking table.

3
Rollback the failed migration

Run a rollback Job: node scripts/migrate.js --down --to=last-known-good. Ensure the rollback is idempotent -- it should be safe to run multiple times.

4
Fix and retry

Fix the migration script, test in staging, then re-run the full deployment pipeline from the beginning.

Prevention: Use Transactional Migrations

Wrap each migration in a transaction (where possible). If a statement fails, the entire migration rolls back atomically. Note: DDL statements in MySQL are not transactional, but PostgreSQL supports transactional DDL. Always test migrations against a copy of production data before deploying.

rollback-migration-job.yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: db-rollback-v1-2-0
  namespace: myapp
spec:
  backoffLimit: 1
  activeDeadlineSeconds: 120
  template:
    spec:
      restartPolicy: Never
      containers:
        - name: rollback
          image: myregistry.io/myapp-backend:1.1.0  # Previous known-good version
          command: ["node", "scripts/migrate.js", "--down", "--to", "20240101_001"]
          envFrom:
            - configMapRef:
                name: backend-config
            - secretRef:
                name: backend-secrets
Scenario 4: One Tier Scaling Causes Resource Pressure on Node

Symptoms: Backend HPA scales to 15 pods. Other pods on same nodes start getting OOMKilled or throttled. Database latency spikes.

Root Causes and Solutions:

Issue Solution
Backend pods not setting resource requests/limits Always set requests (for scheduling) and limits (for enforcement). Without requests, scheduler cannot make good decisions.
Backend pods scheduled on same node as database Use pod anti-affinity or taints/tolerations to keep database on dedicated nodes.
HPA maxReplicas too high for cluster capacity Set maxReplicas based on actual cluster capacity. Use Cluster Autoscaler to add nodes when needed.
No PriorityClass -- all pods equally preemptible Set higher PriorityClass for database and lower for frontend. Ensures database is never evicted for frontend pods.
priority-classes.yaml
# Database gets highest priority -- never evict
apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: database-critical
value: 1000000
globalDefault: false
description: "Reserved for database workloads. Never preempted."
---
# Backend gets medium priority
apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: backend-high
value: 500000
globalDefault: false
description: "Backend API tier."
---
# Frontend gets standard priority
apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: frontend-standard
value: 100000
globalDefault: false
description: "Frontend tier. Can be preempted if needed."
Database node affinity (keep DB on dedicated nodes)
# Add to PostgreSQL StatefulSet spec.template.spec:
affinity:
  nodeAffinity:
    requiredDuringSchedulingIgnoredDuringExecution:
      nodeSelectorTerms:
        - matchExpressions:
            - key: node-role
              operator: In
              values:
                - database
tolerations:
  - key: "dedicated"
    operator: "Equal"
    value: "database"
    effect: "NoSchedule"
