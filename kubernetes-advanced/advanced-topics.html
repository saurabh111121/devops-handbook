1. Deployment Strategies

Rolling Update (Default)

Gradually replaces old pods with new ones, ensuring zero downtime. Controlled by maxSurge (extra pods during update) and maxUnavailable (pods that can be down).

Rolling Update Process (maxSurge=1, maxUnavailable=1, replicas=4)
 Time 0: All pods running v1
 +------+------+------+------+
 | v1   | v1   | v1   | v1   |   Desired: 4, Available: 4
 +------+------+------+------+

 Time 1: Create new pod, terminate one old
 +------+------+------+------+------+
 | v1   | v1   | v1   | (v1) | v2   |   maxSurge=1 (5 total allowed)
 +------+------+------+------+------+   maxUnavailable=1 (3 min available)

 Time 2: v2 ready, continue rolling
 +------+------+------+------+
 | v1   | v1   | v2   | v2   |   2 old, 2 new
 +------+------+------+------+

 Time 3: Continue...
 +------+------+------+------+
 | v1   | v2   | v2   | v2   |   1 old, 3 new
 +------+------+------+------+

 Time 4: Complete
 +------+------+------+------+
 | v2   | v2   | v2   | v2   |   All pods running v2
 +------+------+------+------+
rolling-update.yaml Strategy 
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-app
spec:
  replicas: 4
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1           # Can have 5 pods during update
      maxUnavailable: 1     # At least 3 must be available
  selector:
    matchLabels:
      app: web
  template:
    metadata:
      labels:
        app: web
    spec:
      containers:
      - name: web
        image: web-app:v2
        readinessProbe:
          httpGet:
            path: /ready
            port: 8080
          initialDelaySeconds: 5
          periodSeconds: 5
      minReadySeconds: 10     # Wait 10s after ready before continuing
      progressDeadlineSeconds: 600  # Fail if not complete in 10min

A/B Testing

A/B Testing routes traffic based on user attributes (headers, cookies, geographic location) rather than random percentages. It is used for feature experiments, not deployment safety.

  • Route by header: x-user-group: beta-testers
  • Route by cookie: ab-test=variant-b
  • Route by geography or user segment
  • Typically implemented via Istio VirtualService match rules or Ingress annotations

2. CNI Plugins

Container Network Interface (CNI) plugins provide pod networking - assigning IPs, creating network namespaces, and configuring routes. The choice of CNI affects performance, features, and network policy capabilities.

PluginNetworking ModelNetwork PolicyEncryptionBest For
Calico BGP (Layer 3), VXLAN, IPIP Full (own + K8s standard) WireGuard Large clusters, fine-grained policy, hybrid cloud
Cilium eBPF (Layer 3/4/7) Full (L3-L7, DNS-aware) WireGuard, IPsec High performance, L7 visibility, service mesh replacement
Flannel VXLAN overlay None (needs Calico for policy) None Simple clusters, learning, minimal requirements
Weave Net VXLAN mesh overlay Basic K8s NetworkPolicy IPsec (sleeve) Simple setup, multicast support
AWS VPC CNI Native VPC (ENI-based) Via Calico add-on VPC-level EKS clusters (pods get VPC IPs)
Azure CNI Native VNET Via Calico/Cilium VNET-level AKS clusters

When choosing a CNI:

  • Cilium if you need high performance (eBPF avoids iptables overhead), L7 policy, or plan to replace your service mesh
  • Calico if you need mature network policies, BGP peering with physical network, or wide ecosystem support
  • Flannel if you just need basic networking with no policy requirements (simplest to set up)
  • Cloud-native CNI (AWS VPC, Azure CNI) if you need native cloud integration and pod IPs routable outside the cluster

3. CSI Drivers

Container Storage Interface (CSI) is a standard for exposing storage systems to containerized workloads. CSI drivers run as pods in the cluster and handle volume lifecycle operations.

CSI Architecture

CSI Driver Architecture
 +--Control Plane------------------------------------------+
 |                                                          |
 |  +--CSI Controller (Deployment)--+                       |
 |  |                               |                       |
 |  |  external-provisioner         |  Creates/deletes      |
 |  |  external-attacher            |  volumes via cloud API |
 |  |  external-snapshotter         |                       |
 |  |  external-resizer             |                       |
 |  |  csi-driver-plugin            |                       |
 |  +-------------------------------+                       |
 |                                                          |
 +----------------------------------------------------------+

 +--Worker Nodes-------------------------------------------+
 |                                                          |
 |  +--CSI Node Plugin (DaemonSet)--+                       |
 |  |                               |                       |
 |  |  node-driver-registrar        |  Mounts volumes       |
 |  |  csi-driver-plugin            |  into pods on         |
 |  |                               |  this node            |
 |  +-------------------------------+                       |
 |                                                          |
 +----------------------------------------------------------+

 Flow:
 1. PVC created -> external-provisioner calls CreateVolume
 2. Pod scheduled -> external-attacher calls ControllerPublish
 3. kubelet -> node plugin calls NodeStage + NodePublish
 4. Volume mounted into pod container

4. Custom Resource Definitions (CRDs)

CRDs extend the Kubernetes API with custom resource types. Once a CRD is created, users can create instances of that resource using kubectl just like native resources. CRDs are the foundation for Operators.

crd-example.yaml apiextensions.k8s.io/v1 
apiVersion: apiextensions.k8s.io/v1
kind: CustomResourceDefinition
metadata:
  name: databases.mycompany.io
spec:
  group: mycompany.io
  versions:
  - name: v1
    served: true
    storage: true
    schema:
      openAPIV3Schema:
        type: object
        properties:
          spec:
            type: object
            required: ["engine", "version", "storage"]
            properties:
              engine:
                type: string
                enum: ["postgres", "mysql", "redis"]
              version:
                type: string
              replicas:
                type: integer
                minimum: 1
                maximum: 5
                default: 1
              storage:
                type: string
                pattern: "^[0-9]+(Gi|Ti)$"
              backup:
                type: object
                properties:
                  enabled:
                    type: boolean
                    default: true
                  schedule:
                    type: string
                    default: "0 2 * * *"
          status:
            type: object
            properties:
              phase:
                type: string
              endpoint:
                type: string
              readyReplicas:
                type: integer
    subresources:
      status: {}
    additionalPrinterColumns:
    - name: Engine
      type: string
      jsonPath: .spec.engine
    - name: Version
      type: string
      jsonPath: .spec.version
    - name: Status
      type: string
      jsonPath: .status.phase
    - name: Age
      type: date
      jsonPath: .metadata.creationTimestamp
  scope: Namespaced
  names:
    plural: databases
    singular: database
    kind: Database
    shortNames:
    - db
---
# Using the custom resource
apiVersion: mycompany.io/v1
kind: Database
metadata:
  name: orders-db
  namespace: production
spec:
  engine: postgres
  version: "15.4"
  replicas: 3
  storage: 100Gi
  backup:
    enabled: true
    schedule: "0 */6 * * *"

5. Operators

An Operator is a method of packaging, deploying, and managing a Kubernetes application using custom resources and custom controllers. It encodes operational knowledge (day-2 operations) into software.

Operator Pattern

Operator Reconciliation Loop
                   +--User/Admin--+
                   |              |
                   | kubectl apply|
                   | Database CR  |
                   +------+-------+
                          |
                          v
                   +------+-------+
                   |  Kubernetes   |
                   |  API Server   |
                   +------+-------+
                          |
               Watch for Database CRs
                          |
                          v
            +-------------+---------------+
            |      Database Operator       |
            |      (Custom Controller)     |
            |                             |
            |  Reconcile Loop:            |
            |  1. Observe current state   |
            |  2. Compare to desired spec |
            |  3. Take action to converge |
            +--+--------+--------+--------+
               |        |        |
               v        v        v
         +-----+--+ +---+----+ +-+--------+
         |Create  | |Scale   | |Backup    |
         |StatefulSet|Replicas| |to S3    |
         |+ PVCs  | |        | |          |
         +---------+ +--------+ +----------+

 The Operator automates:
 - Provisioning (create DB, users, schemas)
 - Scaling (add replicas, rebalance)
 - Upgrades (rolling version updates)
 - Backup/Restore (scheduled snapshots)
 - Failover (promote replica to primary)
 - Monitoring (expose metrics, health checks)

6. Cluster Administration

Upgrade Process

Rules for Kubernetes upgrades:

  • Only upgrade one minor version at a time (1.27 -> 1.28, not 1.27 -> 1.29)
  • Upgrade control plane first, then worker nodes
  • Check deprecation notices and API changes before upgrading
  • Test in staging environment first
1
Pre-upgrade - Review changelog, check deprecated APIs (kubectl get --raw /metrics | grep apiserver_requested_deprecated_apis), backup etcd
2
Control plane upgrade - Upgrade kubeadm, then kubeadm upgrade plan / kubeadm upgrade apply v1.28.0
3
Worker nodes - Cordon, drain, upgrade kubelet and kubeadm, uncordon (one node at a time)

etcd Backup & Restore

ETCDCTL_API=3 etcdctl snapshot save /backup/etcd-snapshot-$(date +%Y%m%d).db --endpoints=https://127.0.0.1:2379 --cacert=/etc/kubernetes/pki/etcd/ca.crt --cert=/etc/kubernetes/pki/etcd/server.crt --key=/etc/kubernetes/pki/etcd/server.key
ETCDCTL_API=3 etcdctl snapshot status /backup/etcd-snapshot.db --write-out=table
etcd-restore.sh Restore procedure 
# Stop kube-apiserver (move static pod manifest)
mv /etc/kubernetes/manifests/kube-apiserver.yaml /tmp/

# Restore the snapshot
ETCDCTL_API=3 etcdctl snapshot restore /backup/etcd-snapshot.db \
  --data-dir=/var/lib/etcd-restored \
  --name=control-plane-1 \
  --initial-cluster=control-plane-1=https://10.0.0.10:2380 \
  --initial-advertise-peer-urls=https://10.0.0.10:2380

# Update etcd static pod to use new data directory
# Edit /etc/kubernetes/manifests/etcd.yaml:
#   --data-dir=/var/lib/etcd-restored
#   volumeMounts and volumes updated

# Restart kube-apiserver
mv /tmp/kube-apiserver.yaml /etc/kubernetes/manifests/

Node Maintenance

kubectl cordon node-3 # Mark unschedulable (no new pods)
kubectl drain node-3 --ignore-daemonsets --delete-emptydir-data --grace-period=60
kubectl uncordon node-3 # Mark schedulable again

Drain Flags Explained

  • --ignore-daemonsets - Skip DaemonSet pods (they will be recreated)
  • --delete-emptydir-data - Allow deletion of pods using emptyDir volumes
  • --force - Delete pods not managed by a controller (bare pods)
  • --grace-period=N - Override pod's terminationGracePeriodSeconds
  • --pod-selector='app=batch' - Only drain matching pods

8. Init Containers & Sidecars

Init Containers

Init containers run before app containers start. They run sequentially (one at a time), must complete successfully, and are useful for setup tasks.

init-containers.yaml Sequential setup 
apiVersion: v1
kind: Pod
metadata:
  name: web-app
spec:
  initContainers:
  # Wait for database to be ready
  - name: wait-for-db
    image: busybox:1.36
    command: ['sh', '-c', 'until nc -z postgres-svc 5432; do echo waiting for db; sleep 2; done']
  # Run database migrations
  - name: run-migrations
    image: myapp:v2
    command: ['./migrate', '--direction=up']
    env:
    - name: DB_URL
      valueFrom:
        secretKeyRef:
          name: db-credentials
          key: url
  # Download configuration from remote
  - name: fetch-config
    image: curlimages/curl:8.4.0
    command: ['sh', '-c', 'curl -o /config/app.yaml https://config-server/app.yaml']
    volumeMounts:
    - name: config-volume
      mountPath: /config
  containers:
  - name: web
    image: myapp:v2
    volumeMounts:
    - name: config-volume
      mountPath: /app/config
  volumes:
  - name: config-volume
    emptyDir: {}

Sidecar Containers

Sidecar containers run alongside the main container for the lifetime of the pod. In Kubernetes 1.28+, native sidecar support was added via restartPolicy: Always in initContainers (ensures sidecars start before and stop after main containers).

sidecar-patterns.yaml Common patterns 
apiVersion: v1
kind: Pod
metadata:
  name: app-with-sidecars
spec:
  # Native sidecar (K8s 1.28+) - starts before main, stops after main
  initContainers:
  - name: log-shipper
    image: fluent-bit:2.2
    restartPolicy: Always      # Makes it a native sidecar
    volumeMounts:
    - name: log-volume
      mountPath: /var/log/app
  containers:
  # Main application
  - name: app
    image: myapp:v2
    ports:
    - containerPort: 8080
    volumeMounts:
    - name: log-volume
      mountPath: /var/log/app
  # Traditional sidecar - runs as regular container
  - name: envoy-proxy
    image: envoyproxy/envoy:v1.28
    ports:
    - containerPort: 9901
    volumeMounts:
    - name: envoy-config
      mountPath: /etc/envoy
  # Monitoring sidecar
  - name: metrics-exporter
    image: prom/statsd-exporter:v0.26
    ports:
    - containerPort: 9102
  volumes:
  - name: log-volume
    emptyDir: {}
  - name: envoy-config
    configMap:
      name: envoy-config

Common Sidecar Patterns

Ambassador / Proxy

Proxy that simplifies access to external services (Envoy, nginx, HAProxy). Handles TLS, retries, circuit breaking.

Adapter

Transforms output of the main container (log format conversion, metrics format conversion for Prometheus).

Log Shipper

Reads logs from shared volume and ships to centralized logging (Fluent Bit, Filebeat).

Config Watcher

Watches for ConfigMap changes and signals the main container to reload (via shared file or HTTP endpoint).

9. Pod Disruption Budgets (PDB)

PDBs limit the number of pods that can be voluntarily disrupted at any time. They protect availability during node drains, cluster upgrades, and autoscaler scale-downs.

pdb.yaml policy/v1 
# Option 1: Minimum available pods
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: web-app-pdb
  namespace: production
spec:
  minAvailable: 2            # At least 2 pods must always be running
  selector:
    matchLabels:
      app: web-app
---
# Option 2: Maximum unavailable pods
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: api-pdb
  namespace: production
spec:
  maxUnavailable: 1          # At most 1 pod can be down at a time
  selector:
    matchLabels:
      app: api-server
---
# Option 3: Percentage-based
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: workers-pdb
  namespace: production
spec:
  maxUnavailable: "25%"      # Allow 25% of pods to be disrupted
  selector:
    matchLabels:
      app: worker
  # K8s 1.27+: Unhealthy pod eviction policy
  unhealthyPodEvictionPolicy: AlwaysAllow  # Allow evicting unhealthy pods

How PDBs interact with drains:

  • kubectl drain respects PDBs - it will wait if evicting a pod would violate the budget
  • If PDB cannot be satisfied, drain hangs (use --timeout to limit waiting)
  • PDBs do NOT protect against involuntary disruptions (node crash, OOM kill)
  • Cluster Autoscaler also respects PDBs before removing a node
  • Common mistake: setting minAvailable equal to replicas - makes draining impossible

10. Finalizers

Finalizers are keys on resources that signal pre-delete hooks. When a resource with finalizers is deleted, it enters a "Terminating" state and waits until all finalizers are removed by their controllers before actual deletion.

finalizer-example.yaml Cleanup hook 
apiVersion: v1
kind: Namespace
metadata:
  name: production
  finalizers:
  - kubernetes    # Standard namespace finalizer - cleans up all resources
---
# Custom finalizer on a CRD instance
apiVersion: mycompany.io/v1
kind: Database
metadata:
  name: orders-db
  finalizers:
  - databases.mycompany.io/cleanup   # Operator removes after cleanup
spec:
  engine: postgres
  storage: 100Gi

Fixing Stuck Deletion (Finalizer Issues)

Symptom: Resource stuck in "Terminating" state indefinitely. Usually because the controller responsible for removing the finalizer is not running or has a bug.

kubectl get namespace stuck-namespace -o json | jq '.spec.finalizers'
kubectl get namespace stuck-namespace -o json | jq '.spec.finalizers = []' | kubectl replace --raw "/api/v1/namespaces/stuck-namespace/finalize" -f -
remove-finalizer.sh Emergency fix 
# For non-namespace resources stuck in Terminating
kubectl patch database orders-db \
  --type='json' \
  -p='[{"op": "remove", "path": "/metadata/finalizers"}]'

# For namespaces stuck in Terminating
kubectl get namespace stuck-ns -o json > /tmp/ns.json
# Edit /tmp/ns.json - remove finalizers from spec.finalizers[]
kubectl replace --raw "/api/v1/namespaces/stuck-ns/finalize" -f /tmp/ns.json

Warning: Removing finalizers bypasses cleanup logic. Only do this after verifying the associated resources have been manually cleaned up, or the controller responsible is permanently gone.

13. Interview Q&A

Q: Compare rolling update, blue/green, and canary deployment strategies. When would you use each?

Rolling Update: Default K8s strategy. Gradually replaces pods. Pros: zero downtime, simple, built-in. Cons: both versions serve traffic simultaneously, slow rollback (must re-roll).

Blue/Green: Two full environments, switch traffic instantly. Pros: instant rollback (switch back), no mixed versions. Cons: 2x resources during deployment, requires external switching mechanism.

Canary: Route small % of traffic to new version, gradually increase. Pros: catch issues with minimal blast radius, data-driven promotion. Cons: complex (needs traffic splitting), longer deployment time.

When to use:

  • Rolling update: most stateless services, simple apps
  • Blue/Green: critical services needing instant rollback, database migrations that are hard to roll back
  • Canary: high-traffic services where you need to validate with real traffic before full rollout

Q: What is a CRD and how does it differ from an Operator?

CRD (Custom Resource Definition): Extends the Kubernetes API with a new resource type. It only defines the schema - what the resource looks like. By itself, a CRD does nothing when you create instances of it.

Operator: A CRD + a custom controller that watches for instances of that CRD and takes action. The controller implements the reconciliation logic (the "operational knowledge").

Analogy: A CRD is like a form template. An Operator is the form template + the person who processes submitted forms and takes action.

Q: How would you back up and restore an etcd cluster?

Backup:

  1. Use etcdctl snapshot save with TLS certificates
  2. Schedule regular backups (CronJob or external tool)
  3. Store snapshots in remote storage (S3, GCS) with retention
  4. Verify snapshots with etcdctl snapshot status

Restore:

  1. Stop API server (move static pod manifest)
  2. etcdctl snapshot restore to a new data directory
  3. Update etcd pod to use new data directory
  4. Restart API server
  5. Verify cluster health

Important: For multi-node etcd, all members must restore from the same snapshot simultaneously.

Q: What is a Pod Disruption Budget and why is it important?

A PDB limits voluntary disruptions to ensure minimum availability during:

  • Node drains (maintenance, upgrades)
  • Cluster autoscaler removing underutilized nodes
  • Voluntary evictions by admission controllers

Key points:

  • Use minAvailable or maxUnavailable (not both)
  • PDBs do NOT protect against involuntary disruptions (node crash, OOM)
  • Setting minAvailable = replicas makes draining impossible - avoid this
  • Best practice: maxUnavailable: 1 for most services

Q: Explain the difference between init containers and sidecar containers.

Init containers:

  • Run before app containers start
  • Run sequentially (one at a time)
  • Must complete successfully (exit 0) before the next starts
  • Use for: setup, migrations, waiting for dependencies, downloading configs

Sidecar containers:

  • Run alongside app containers for the pod's lifetime
  • Run in parallel with the main container
  • Use for: proxies, log shippers, config watchers, metrics exporters
  • K8s 1.28+ native sidecars: init containers with restartPolicy: Always

Q: What is the Gateway API and why is it replacing Ingress?

Gateway API is the next-generation Kubernetes traffic routing API that addresses Ingress limitations:

  • Role separation: GatewayClass (infra provider) -> Gateway (cluster operator) -> HTTPRoute (developer). Each role manages their own resources.
  • Expressiveness: Native header matching, weight-based splitting, query parameter routing - no vendor-specific annotations.
  • Protocol support: HTTP, gRPC, TCP, UDP, TLS - not just HTTP.
  • Portability: Standard API means switching implementations (Istio, Envoy Gateway, nginx) without rewriting manifests.
  • Cross-namespace: Routes in app namespaces can attach to Gateways in infra namespaces.

Q: When would you choose Calico vs Cilium as a CNI?

Choose Calico when:

  • You need BGP peering with physical network infrastructure
  • You need mature, battle-tested network policies
  • You run on older kernels (Cilium requires 4.19+)
  • You need integration with non-K8s workloads (VMs, bare metal)

Choose Cilium when:

  • You need high performance (eBPF bypasses iptables overhead)
  • You need L7 network policies (HTTP method, path, gRPC)
  • You want to replace your service mesh (Cilium has mTLS, L7 LB)
  • You need transparent encryption (WireGuard)
  • You want deep network observability (Hubble)

Q: What is a Helm chart and how do you handle secrets in Helm?

Helm chart: A package of pre-configured Kubernetes resources with templating (Go templates). Charts have a values.yaml for customization, enabling reuse across environments.

Handling secrets:

  • helm-secrets plugin - Encrypts values files with SOPS/age/GPG; decrypts at install time
  • External Secrets Operator - Chart deploys ExternalSecret CRD; secrets fetched from Vault/AWS SM at runtime
  • Sealed Secrets - Encrypt secrets client-side; only the cluster can decrypt
  • Never: Store plaintext secrets in values.yaml committed to Git

Q: What are finalizers and how do you debug a resource stuck in Terminating state?

Finalizers are keys in metadata.finalizers that prevent resource deletion until the responsible controller performs cleanup and removes its finalizer.

Debugging stuck Terminating:

  1. Check finalizers: kubectl get <resource> -o jsonpath='{.metadata.finalizers}'
  2. Identify which controller owns each finalizer
  3. Check if that controller is running and healthy
  4. Check controller logs for errors during cleanup
  5. Last resort: Patch to remove finalizers manually (after verifying cleanup is done or not needed)

Common causes: Deleted CRD before deleting instances, operator uninstalled while resources exist, webhook failure blocking finalizer removal.
