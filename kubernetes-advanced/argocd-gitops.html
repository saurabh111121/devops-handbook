ArgoCD & GitOps

Declarative, Git-centric continuous delivery for Kubernetes with automated reconciliation

1. GitOps Principles

The Four GitOps Principles

  1. Declarative: The entire system is described declaratively (YAML manifests, Helm charts, Kustomize).
  2. Versioned and Immutable: The desired state is stored in Git, providing a complete audit trail and single source of truth.
  3. Pulled Automatically: Approved changes are automatically applied to the cluster (no manual kubectl apply).
  4. Continuously Reconciled: Software agents (ArgoCD/Flux) continuously observe and correct drift between desired and actual state.

Push vs Pull Deployment

Push-Based vs Pull-Based Deployment
PUSH-BASED (Traditional CI/CD):
+--------+     +--------+     +---------+     +------------+
| Commit | --> | CI/CD  | --> | kubectl | --> | Kubernetes |
| (Git)  |     | Server |     |  apply  |     |  Cluster   |
+--------+     +--------+     +---------+     +------------+
                    |
                    +-- CI server has cluster credentials (security risk)
                    +-- No drift detection
                    +-- Manual or webhook-triggered

PULL-BASED (GitOps with ArgoCD):
+--------+                              +------------+
| Commit | <--- periodically polls ---  | ArgoCD     |
| (Git)  |                              | Controller |
+--------+                              +------+-----+
                                               |
                                               | reconciles
                                               v
                                        +------------+
                                        | Kubernetes |
                                        |  Cluster   |
                                        +------------+

Advantages of Pull-Based:
  - Cluster credentials never leave the cluster
  - Automatic drift detection and correction
  - Git is the single source of truth
  - Easy rollback (git revert)
  - Audit trail built into git history

GitOps Benefits

  • Complete audit trail in Git history
  • Easy rollback via git revert
  • Consistent environments (same manifests = same state)
  • Self-healing (automatic drift correction)
  • Developer-friendly (PRs for infra changes)
  • Reduced blast radius (PR review before deploy)

GitOps Challenges

  • Secret management (secrets can't live in Git plaintext)
  • Managing multiple environments
  • Handling stateful workloads
  • Initial learning curve
  • Git repo proliferation
  • Debugging sync failures

2. ArgoCD Architecture

ArgoCD Internal Architecture
+------------------------------------------------------------------+
|                        ArgoCD Namespace                            |
|                                                                    |
|  +------------------+    +------------------+    +--------------+ |
|  |   API Server     |    | Repo Server      |    |    Redis     | |
|  |                  |    |                  |    |              | |
|  | - REST/gRPC API  |    | - Clones Git     |    | - Cache for  | |
|  | - Web UI serving |    | - Generates      |    |   manifests  | |
|  | - Authentication |    |   manifests      |    | - App state  | |
|  | - RBAC           |    | - Helm template  |    |              | |
|  | - SSO (Dex/OIDC) |    | - Kustomize      |    +--------------+ |
|  +-------+----------+    | - Jsonnet        |                     |
|          |               +--------+---------+                     |
|          |                        |                               |
|  +-------v------------------------v------+                        |
|  |      Application Controller          |                        |
|  |                                       |                        |
|  | - Watches Applications (CRD)         |                        |
|  | - Compares desired vs live state     |                        |
|  | - Syncs changes to cluster           |                        |
|  | - Runs health checks                 |                        |
|  | - Manages sync waves & hooks         |                        |
|  +---------------------------------------+                        |
|                                                                    |
|  +------------------+                                              |
|  |   Dex Server     |   (Optional - for SSO)                      |
|  | - OIDC provider  |                                              |
|  | - LDAP/SAML/     |                                              |
|  |   GitHub/GitLab  |                                              |
|  +------------------+                                              |
+------------------------------------------------------------------+
         |                        |
         v                        v
+------------------+     +------------------+
|   Git Repos      |     | Target Clusters  |
| (Source of truth)|     | (Managed K8s)    |
+------------------+     +------------------+

How It Works (5 Steps):
  1. Developer pushes manifest changes to Git
  2. Repo Server detects changes (polls every 3min or webhook)
  3. Repo Server generates final manifests (Helm/Kustomize/plain)
  4. Application Controller compares desired state vs live cluster state
  5. If out-of-sync: auto-sync (if enabled) or manual sync via UI/CLI

API Server

  • Exposes REST and gRPC API
  • Serves Web UI
  • Handles authentication (local users, SSO via Dex)
  • Enforces RBAC policies
  • Manages Application CRD lifecycle

Application Controller

  • Core reconciliation engine
  • Continuously monitors Application resources
  • Computes diff between desired and live state
  • Executes sync operations
  • Manages sync waves, hooks, and health assessment

3. Installation

Quick Install (Non-HA)

kubectl create namespace argocd
kubectl apply -n argocd -f https://raw.githubusercontent.com/argoproj/argo-cd/stable/manifests/install.yaml

HA Install (Production)

kubectl create namespace argocd
kubectl apply -n argocd -f https://raw.githubusercontent.com/argoproj/argo-cd/stable/manifests/ha/install.yaml

CLI Setup

Install ArgoCD CLI Shell 
# macOS
brew install argocd

# Linux
curl -sSL -o argocd-linux-amd64 \
  https://github.com/argoproj/argo-cd/releases/latest/download/argocd-linux-amd64
sudo install -m 555 argocd-linux-amd64 /usr/local/bin/argocd

# Get initial admin password
argocd admin initial-password -n argocd

# Login
argocd login localhost:8080 --username admin --password <password>

# Change default password immediately
argocd account update-password

Accessing the UI

kubectl port-forward svc/argocd-server -n argocd 8080:443

Access at https://localhost:8080. Default username: admin.

For production, expose via Ingress with TLS termination.

4. Application Manifest

application.yaml (complete example) YAML 
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: my-app
  namespace: argocd
  labels:
    team: backend
    env: production
  finalizers:
    - resources-finalizer.argocd.argoproj.io  # Cascade delete on app removal
  annotations:
    notifications.argoproj.io/subscribe.on-sync-succeeded.slack: deployments
spec:
  project: default   # ArgoCD project (RBAC boundary)

  # Source: where to find manifests
  source:
    repoURL: https://github.com/myorg/k8s-manifests.git
    targetRevision: main          # Branch, tag, or commit SHA
    path: apps/my-app/overlays/production  # Path within repo

    # For Helm charts:
    # chart: my-chart
    # helm:
    #   releaseName: my-app
    #   valueFiles:
    #     - values-production.yaml
    #   parameters:
    #     - name: image.tag
    #       value: "v1.2.3"

    # For Kustomize:
    kustomize:
      images:
        - myregistry.io/my-app:v1.2.3
      namePrefix: prod-

  # Destination: where to deploy
  destination:
    server: https://kubernetes.default.svc  # In-cluster
    # server: https://remote-cluster-api:6443  # External cluster
    namespace: my-app

  # Sync Policy: automation settings
  syncPolicy:
    automated:
      prune: true        # Delete resources removed from Git
      selfHeal: true     # Auto-correct manual changes (drift)
      allowEmpty: false  # Don't sync if source produces empty manifests
    syncOptions:
      - CreateNamespace=true     # Create namespace if it doesn't exist
      - PrunePropagationPolicy=foreground
      - PruneLast=true           # Prune after all other syncs
      - ApplyOutOfSyncOnly=true  # Only sync changed resources
      - ServerSideApply=true     # Use server-side apply
    retry:
      limit: 5
      backoff:
        duration: 5s
        factor: 2
        maxDuration: 3m

  # Health checks & status
  ignoreDifferences:
    - group: apps
      kind: Deployment
      jsonPointers:
        - /spec/replicas  # Ignore replica count (managed by HPA)
    - group: ""
      kind: Service
      jqPathExpressions:
        - .spec.clusterIP

Key Fields Explained

Field Purpose Common Values
source.repoURL Git repository URL HTTPS or SSH URL
source.targetRevision Branch, tag, or commit to track main, HEAD, v1.2.3, SHA
source.path Directory path within the repo apps/my-app/production
syncPolicy.automated.prune Delete resources not in Git true / false
syncPolicy.automated.selfHeal Revert manual cluster changes true / false
ignoreDifferences Ignore specific field diffs HPA-managed replicas, cluster IPs

5. App of Apps Pattern

What is App of Apps?

A pattern where a "root" ArgoCD Application manages other Application manifests. The root app points to a Git directory containing Application YAML files, enabling declarative management of the entire application fleet from a single entry point.

App of Apps Structure
                    +-------------------+
                    |   Root App        |
                    | (app-of-apps)     |
                    | path: apps/       |
                    +---------+---------+
                              |
              +---------------+---------------+
              |               |               |
    +---------v---+  +--------v----+  +-------v-----+
    | App: frontend|  | App: backend|  | App: infra  |
    | path: frontend/| | path: backend/| | path: infra/ |
    +-------------+  +-------------+  +-------------+
         |                  |                 |
    +----v----+       +----v----+       +----v----+
    | Deploy  |       | Deploy  |       | Ingress |
    | Service |       | Service |       | CertMgr |
    | Ingress |       | HPA     |       | Monitor |
    +---------+       +---------+       +---------+

Git Repository Layout:
  k8s-manifests/
  +-- apps/                     # Root app points here
  |     +-- frontend.yaml       # Application manifest for frontend
  |     +-- backend.yaml        # Application manifest for backend
  |     +-- infra.yaml          # Application manifest for infra
  +-- frontend/                 # Actual frontend manifests
  |     +-- deployment.yaml
  |     +-- service.yaml
  +-- backend/                  # Actual backend manifests
  |     +-- deployment.yaml
  |     +-- service.yaml
  +-- infra/                    # Infrastructure manifests
        +-- ingress.yaml
        +-- cert-manager.yaml
apps/root-app.yaml YAML 
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: app-of-apps
  namespace: argocd
spec:
  project: default
  source:
    repoURL: https://github.com/myorg/k8s-manifests.git
    targetRevision: main
    path: apps    # Directory containing Application YAMLs
  destination:
    server: https://kubernetes.default.svc
    namespace: argocd
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
apps/frontend.yaml (child application) YAML 
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: frontend
  namespace: argocd
  finalizers:
    - resources-finalizer.argocd.argoproj.io
spec:
  project: default
  source:
    repoURL: https://github.com/myorg/k8s-manifests.git
    targetRevision: main
    path: frontend
  destination:
    server: https://kubernetes.default.svc
    namespace: frontend
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
    syncOptions:
      - CreateNamespace=true

6. Sync Waves & Hooks

Ordering Deployments

Sync waves allow you to control the order in which resources are applied. Lower wave numbers sync first. Resources within the same wave are synced together. Hooks allow running Jobs at specific phases.

Sync Wave Execution Order
Wave -1: PreSync Hooks
  |  (e.g., database migration Job)
  v
Wave 0: Core Resources (default)
  |  (e.g., ConfigMaps, Secrets, Deployments)
  v
Wave 1: Dependent Resources
  |  (e.g., Services, Ingress)
  v
Wave 2: Post-deployment
  |  (e.g., HPA, monitoring configs)
  v
PostSync Hooks
  |  (e.g., smoke test Job, notification)
  v
SyncFail Hooks
     (e.g., rollback notification)

Hook Phases:
  PreSync  -> Before any sync operation
  Sync     -> During sync (with wave ordering)
  PostSync -> After successful sync
  SyncFail -> If sync fails
  Skip     -> Skip this resource during sync
Sync wave annotations YAML 
# Wave -1: Namespace (created first)
apiVersion: v1
kind: Namespace
metadata:
  name: my-app
  annotations:
    argocd.argoproj.io/sync-wave: "-1"
---
# Wave 0: ConfigMap and Secret (default wave)
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
  annotations:
    argocd.argoproj.io/sync-wave: "0"
data:
  DATABASE_URL: "postgres://db:5432/app"
---
# Wave 1: Deployment (after config is ready)
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
  annotations:
    argocd.argoproj.io/sync-wave: "1"
spec:
  replicas: 3
  selector:
    matchLabels:
      app: my-app
  template:
    metadata:
      labels:
        app: my-app
    spec:
      containers:
        - name: app
          image: myregistry.io/my-app:v1.2.3
---
# Wave 2: Service and Ingress (after deployment)
apiVersion: v1
kind: Service
metadata:
  name: my-app
  annotations:
    argocd.argoproj.io/sync-wave: "2"
spec:
  selector:
    app: my-app
  ports:
    - port: 80
      targetPort: 8080

Hooks Example

db-migration-hook.yaml (PreSync) YAML 
apiVersion: batch/v1
kind: Job
metadata:
  name: db-migrate
  annotations:
    argocd.argoproj.io/hook: PreSync
    argocd.argoproj.io/hook-delete-policy: HookSucceeded
spec:
  template:
    spec:
      containers:
        - name: migrate
          image: myregistry.io/my-app:v1.2.3
          command: ["./migrate", "--direction", "up"]
          env:
            - name: DATABASE_URL
              valueFrom:
                secretKeyRef:
                  name: db-credentials
                  key: url
      restartPolicy: Never
  backoffLimit: 3
---
apiVersion: batch/v1
kind: Job
metadata:
  name: smoke-test
  annotations:
    argocd.argoproj.io/hook: PostSync
    argocd.argoproj.io/hook-delete-policy: HookSucceeded
spec:
  template:
    spec:
      containers:
        - name: test
          image: curlimages/curl:latest
          command:
            - sh
            - -c
            - |
              sleep 10
              curl -sf http://my-app.my-app.svc/health || exit 1
              echo "Smoke test passed!"
      restartPolicy: Never
  backoffLimit: 1

7. Multi-Cluster Management

Adding External Clusters

ArgoCD can manage multiple Kubernetes clusters from a single control plane. Cluster credentials are stored as Secrets in the ArgoCD namespace.

argocd cluster add my-staging-context --name staging
argocd cluster add my-production-context --name production
argocd cluster list
Cluster Secret (declarative) YAML 
apiVersion: v1
kind: Secret
metadata:
  name: production-cluster
  namespace: argocd
  labels:
    argocd.argoproj.io/secret-type: cluster
type: Opaque
stringData:
  name: production
  server: https://production-api.example.com:6443
  config: |
    {
      "bearerToken": "<service-account-token>",
      "tlsClientConfig": {
        "insecure": false,
        "caData": "<base64-encoded-ca-cert>"
      }
    }
Multi-Cluster Deployment Pattern
                   +------------------+
                   |  Management      |
                   |  Cluster         |
                   |  (ArgoCD lives   |
                   |   here)          |
                   +--------+---------+
                            |
            +---------------+---------------+
            |               |               |
   +--------v-------+ +----v---------+ +---v----------+
   | Dev Cluster    | | Staging      | | Production   |
   | (auto-sync)   | | (auto-sync)  | | (manual sync)|
   +----------------+ +--------------+ +--------------+

Strategy:
  - ArgoCD in dedicated management cluster
  - Different sync policies per environment
  - Production requires manual approval
  - Same Git repo, different paths/overlays per env

8. ApplicationSets

What are ApplicationSets?

ApplicationSets automate the generation of ArgoCD Applications using templates and generators. Instead of writing one Application YAML per service/cluster/environment, you define a template that generates many Applications dynamically.

Generator Types

Generator Purpose Use Case
list Static list of parameters Known set of clusters or environments
cluster Generate per registered cluster Deploy same app to all clusters
git Generate from Git directories or files Monorepo with one dir per service
matrix Combine two generators (cross-product) All apps x all clusters
merge Merge multiple generators Override specific params per cluster
pullRequest Generate for open PRs Preview environments per PR

List Generator

applicationset-list.yaml YAML 
apiVersion: argoproj.io/v1alpha1
kind: ApplicationSet
metadata:
  name: my-app-environments
  namespace: argocd
spec:
  generators:
    - list:
        elements:
          - cluster: dev
            url: https://dev-api.example.com:6443
            namespace: my-app-dev
            replicas: "1"
          - cluster: staging
            url: https://staging-api.example.com:6443
            namespace: my-app-staging
            replicas: "2"
          - cluster: production
            url: https://prod-api.example.com:6443
            namespace: my-app-prod
            replicas: "5"
  template:
    metadata:
      name: 'my-app-{{cluster}}'
    spec:
      project: default
      source:
        repoURL: https://github.com/myorg/k8s-manifests.git
        targetRevision: main
        path: 'apps/my-app/overlays/{{cluster}}'
        kustomize:
          images:
            - 'myregistry.io/my-app:v1.2.3'
      destination:
        server: '{{url}}'
        namespace: '{{namespace}}'
      syncPolicy:
        automated:
          prune: true
          selfHeal: true
        syncOptions:
          - CreateNamespace=true

9. RBAC in ArgoCD

Projects as RBAC Boundaries

ArgoCD Projects restrict what applications can do: which repos they can access, which clusters/namespaces they can deploy to, and which resource kinds are allowed.

AppProject definition YAML 
apiVersion: argoproj.io/v1alpha1
kind: AppProject
metadata:
  name: team-backend
  namespace: argocd
spec:
  description: "Backend team project"

  # Allowed source repositories
  sourceRepos:
    - 'https://github.com/myorg/backend-*'
    - 'https://charts.helm.sh/stable'

  # Allowed destination clusters and namespaces
  destinations:
    - server: https://kubernetes.default.svc
      namespace: 'backend-*'
    - server: https://staging-api.example.com:6443
      namespace: 'backend-*'

  # Deny deploying to these namespaces
  namespaceResourceBlacklist:
    - group: ''
      kind: ResourceQuota
    - group: ''
      kind: LimitRange

  # Only allow these resource types
  namespaceResourceWhitelist:
    - group: 'apps'
      kind: Deployment
    - group: 'apps'
      kind: StatefulSet
    - group: ''
      kind: Service
    - group: ''
      kind: ConfigMap
    - group: ''
      kind: Secret
    - group: 'networking.k8s.io'
      kind: Ingress

  # Deny cluster-scoped resources
  clusterResourceWhitelist: []

  # Role definitions
  roles:
    - name: developer
      description: "Can sync and view applications"
      policies:
        - p, proj:team-backend:developer, applications, get, team-backend/*, allow
        - p, proj:team-backend:developer, applications, sync, team-backend/*, allow
      groups:
        - backend-developers   # SSO group mapping

    - name: admin
      description: "Full access to project"
      policies:
        - p, proj:team-backend:admin, applications, *, team-backend/*, allow
      groups:
        - backend-leads

RBAC Policy ConfigMap

argocd-rbac-cm ConfigMap YAML 
apiVersion: v1
kind: ConfigMap
metadata:
  name: argocd-rbac-cm
  namespace: argocd
data:
  policy.default: role:readonly
  policy.csv: |
    # Roles
    p, role:org-admin, applications, *, */*, allow
    p, role:org-admin, clusters, *, *, allow
    p, role:org-admin, repositories, *, *, allow
    p, role:org-admin, projects, *, *, allow

    p, role:developer, applications, get, */*, allow
    p, role:developer, applications, sync, */*, allow
    p, role:developer, logs, get, */*, allow

    p, role:readonly, applications, get, */*, allow
    p, role:readonly, logs, get, */*, allow

    # Group mappings (SSO groups -> ArgoCD roles)
    g, platform-team, role:org-admin
    g, developers, role:developer
    g, viewers, role:readonly
  scopes: '[groups]'

10. Notifications

ArgoCD Notifications

ArgoCD Notifications monitors applications and sends alerts on state changes. Supports Slack, email, webhook, Grafana, GitHub, Teams, and more.

argocd-notifications-cm ConfigMap YAML 
apiVersion: v1
kind: ConfigMap
metadata:
  name: argocd-notifications-cm
  namespace: argocd
data:
  # Slack service configuration
  service.slack: |
    token: $slack-token

  # Triggers - when to send notifications
  trigger.on-sync-succeeded: |
    - when: app.status.operationState.phase in ['Succeeded']
      send: [app-sync-succeeded]

  trigger.on-sync-failed: |
    - when: app.status.operationState.phase in ['Error', 'Failed']
      send: [app-sync-failed]

  trigger.on-health-degraded: |
    - when: app.status.health.status == 'Degraded'
      send: [app-health-degraded]

  # Templates - what to send
  template.app-sync-succeeded: |
    slack:
      attachments: |
        [{
          "color": "#18be52",
          "title": "{{.app.metadata.name}} synced successfully",
          "fields": [
            {"title": "Application", "value": "{{.app.metadata.name}}", "short": true},
            {"title": "Revision", "value": "{{.app.status.sync.revision | trunc 7}}", "short": true},
            {"title": "Environment", "value": "{{.app.spec.destination.namespace}}", "short": true}
          ]
        }]

  template.app-sync-failed: |
    slack:
      attachments: |
        [{
          "color": "#E96D76",
          "title": "{{.app.metadata.name}} sync FAILED",
          "fields": [
            {"title": "Application", "value": "{{.app.metadata.name}}", "short": true},
            {"title": "Error", "value": "{{.app.status.operationState.message}}", "short": false}
          ]
        }]

  template.app-health-degraded: |
    slack:
      attachments: |
        [{
          "color": "#f4c030",
          "title": "{{.app.metadata.name}} health DEGRADED",
          "text": "Application health status is degraded. Check pod status.",
          "fields": [
            {"title": "Health Status", "value": "{{.app.status.health.status}}", "short": true}
          ]
        }]
Subscribe application to notifications YAML 
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: my-app
  namespace: argocd
  annotations:
    # Subscribe to triggers and route to Slack channels
    notifications.argoproj.io/subscribe.on-sync-succeeded.slack: deployments
    notifications.argoproj.io/subscribe.on-sync-failed.slack: deployments-alerts
    notifications.argoproj.io/subscribe.on-health-degraded.slack: oncall-alerts
spec:
  # ... application spec

11. ArgoCD vs Flux

Aspect ArgoCD Flux
UI Rich built-in Web UI with visualization No built-in UI (Weave GitOps optional)
Architecture Centralized (single ArgoCD instance) Distributed (per-cluster controllers)
Multi-tenancy Projects + RBAC + SSO Namespace-scoped controllers
Multi-cluster Single control plane manages many clusters Flux instance per cluster (or remote)
Sync Method Application CRD (centralized) GitRepository + Kustomization CRDs
Helm Support Native (renders in repo server) HelmRelease CRD (controller-based)

12. Interview Q&A

Q1: What are the benefits of GitOps over traditional CI/CD?

GitOps provides: 1) Audit trail - every change is a git commit with author, timestamp, and reason. 2) Easy rollback - just revert the git commit. 3) Drift detection - continuous reconciliation catches unauthorized manual changes. 4) Security - cluster credentials don't need to leave the cluster (pull model). 5) Consistency - same manifests guarantee same state. 6) Developer experience - deploy via PR, not kubectl.

Q2: Explain the ArgoCD sync process from commit to deployment.

1) Developer pushes to Git (or webhook triggers). 2) ArgoCD Repo Server clones the repo and generates manifests (Helm template, Kustomize build, or plain YAML). 3) Application Controller compares the generated "desired state" against the live cluster state via Kubernetes API. 4) If differences exist, the app is marked "OutOfSync." 5) If auto-sync is enabled (or manual sync triggered), the controller applies the manifests to the cluster. 6) Health checks run to verify the deployment succeeded. 7) Status transitions to "Synced" and "Healthy" (or "Degraded" on failure).

Q3: How does ArgoCD detect drift, and what happens when it does?

ArgoCD continuously (every ~3 minutes by default) compares the live cluster state against the desired state from Git. It computes a structured diff of all managed resources. When drift is detected: 1) The application status changes to "OutOfSync." 2) The UI/CLI shows exactly which resources differ and what changed. 3) If selfHeal: true is configured, ArgoCD automatically reverts the drift by reapplying the Git state. 4) If self-heal is disabled, it alerts and waits for manual sync. The ignoreDifferences field allows excluding expected diffs (like HPA-managed replica counts).

Q4: What is the App of Apps pattern and when would you use it?

App of Apps is a pattern where a "root" Application manages other Application manifests stored in Git. The root app points to a directory containing Application YAMLs, and ArgoCD recursively manages all child apps. Use it when: 1) You have many microservices to manage declaratively. 2) You want to bootstrap an entire cluster from a single Application. 3) You need consistent sync policies across apps. 4) You want to add/remove apps by simply adding/removing YAML files in Git. Alternative: ApplicationSets for more dynamic generation.

Q5: How do sync waves work and why are they important?

Sync waves control the order of resource application using annotations (argocd.argoproj.io/sync-wave: "N"). Resources with lower wave numbers deploy first. Within a wave, ArgoCD waits for all resources to be healthy before proceeding. This is critical for: 1) Database migrations before app deployment. 2) CRDs before CRs. 3) Namespaces before workloads. 4) Secrets/ConfigMaps before Deployments that reference them. Combined with hooks (PreSync, PostSync), you can orchestrate complex multi-step deployments.

Q6: How would you manage multiple clusters with ArgoCD?

Options: 1) Single ArgoCD + cluster secrets: One ArgoCD instance registers multiple clusters via argocd cluster add. Applications target different destination.server URLs. 2) ApplicationSets with cluster generator: Automatically deploy to all registered clusters matching a label selector. 3) Separate ArgoCD per cluster: Each cluster has its own ArgoCD (more isolation, more operational overhead). Best practice: dedicated management cluster running ArgoCD, deploying to workload clusters. Use different sync policies per environment (auto-sync for dev/staging, manual for production).

Q7: How do you handle secrets in a GitOps workflow?

Since secrets can't be stored as plaintext in Git, common approaches: 1) Sealed Secrets: Encrypt secrets with a cluster-specific public key; only the cluster can decrypt. 2) SOPS + age/KMS: Encrypt YAML values in-place; decrypt during ArgoCD sync via a plugin. 3) External Secrets Operator: CRD syncs secrets from external stores (AWS Secrets Manager, HashiCorp Vault, GCP Secret Manager) into Kubernetes Secrets. 4) Vault Agent Injector: Sidecar injects secrets from HashiCorp Vault at runtime. Best practice: External Secrets Operator for production (secrets stay in a dedicated secrets manager, not in Git at all).

Q8: What is the difference between ArgoCD Applications and ApplicationSets?

Application is a single deployment unit: one source repo/path deployed to one destination cluster/namespace. ApplicationSet is a template that generates multiple Applications using generators (list, cluster, git, matrix). Use ApplicationSets when you need to deploy the same app to multiple clusters/environments, or generate one Application per directory in a monorepo. ApplicationSets reduce boilerplate and enable dynamic fleet management. The generated Applications are regular Application resources that ArgoCD manages normally.

13. Real-World Scenarios

⚠ Scenario: Application Stuck OutOfSync

Situation: An application shows "OutOfSync" in ArgoCD even after syncing, and keeps reverting.

1. Check the diff in the ArgoCD UI to see what's different:
argocd app diff my-app
2. Common causes:
  • HPA modifying replicas: Add to ignoreDifferences
  • Mutating webhooks: Cluster adds fields (annotations, labels) not in Git
  • Default values: Kubernetes API server adds defaults not in your manifest
  • Resource conflicts: Another controller (Istio, cert-manager) modifying resources
3. Fix with ignoreDifferences:
Ignore HPA-managed replicas
spec:
  ignoreDifferences:
    - group: apps
      kind: Deployment
      jsonPointers:
        - /spec/replicas
    - group: admissionregistration.k8s.io
      kind: MutatingWebhookConfiguration
      jqPathExpressions:
        - '.webhooks[]?.clientConfig.caBundle'
4. If caused by server-side defaults, use ServerSideApply=true in syncOptions to only track fields you explicitly set.
🔧 Scenario: Sync Failed - Health Check Timeout

Situation: Sync completes but application stays "Progressing" then transitions to "Degraded." Pods are in CrashLoopBackOff.

1. Check application health details:
argocd app get my-app
argocd app resources my-app
2. Check pod logs for the failing deployment:
kubectl logs -n my-app deployment/my-app --previous
kubectl describe pod -n my-app -l app=my-app
3. Common causes: Image tag doesn't exist, misconfigured environment variables, failed health/readiness probes, insufficient resources, missing secrets/configmaps.
4. Rollback via Git:
git revert HEAD
git push origin main

ArgoCD will automatically sync to the reverted state (if auto-sync enabled).

5. Or rollback via ArgoCD CLI:
argocd app history my-app
argocd app rollback my-app <revision-id>

Note: ArgoCD rollback is temporary - next sync will override. Fix in Git for permanent rollback.

🔒 Scenario: Managing Secrets in GitOps

Situation: You need to deploy applications that require database passwords, API keys, and TLS certificates, but cannot store secrets in plaintext in Git.

Option 1: Sealed Secrets

1. Install Sealed Secrets controller in the cluster.
kubectl apply -f https://github.com/bitnami-labs/sealed-secrets/releases/download/v0.24.0/controller.yaml
2. Encrypt the secret locally (only the cluster can decrypt):
kubeseal --format yaml < secret.yaml > sealed-secret.yaml
3. Commit sealed-secret.yaml to Git. ArgoCD deploys it; the controller decrypts it in-cluster.

Option 2: External Secrets Operator (Recommended for Production)

external-secret.yamlYAML
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: app-secrets
  namespace: my-app
spec:
  refreshInterval: 1h
  secretStoreRef:
    name: aws-secrets-manager
    kind: ClusterSecretStore
  target:
    name: app-secrets           # Kubernetes Secret name to create
    creationPolicy: Owner
  data:
    - secretKey: db-password    # Key in K8s Secret
      remoteRef:
        key: production/my-app  # Path in AWS Secrets Manager
        property: db_password   # JSON key within the secret
    - secretKey: api-key
      remoteRef:
        key: production/my-app
        property: api_key

Option 3: SOPS with ArgoCD

1. Configure ArgoCD with a SOPS decryption plugin (via kustomize-sops or Helm Secrets).
2. Encrypt secrets in-place with SOPS before committing:
sops --encrypt --age age1... secret.yaml > secret.enc.yaml
3. ArgoCD decrypts during manifest generation before applying to the cluster.
