Service Mesh

Istio, Linkerd, traffic management, mTLS, observability, and production deployment patterns

1. What is a Service Mesh

Problems a Service Mesh Solves

  • Observability: Automatic metrics, distributed tracing, and access logs for all service-to-service traffic without code changes
  • Security: Automatic mTLS encryption and identity-based authorization between services
  • Traffic Management: Fine-grained routing, retries, timeouts, circuit breaking, and traffic splitting
  • Resilience: Automatic retries, failover, rate limiting, and fault injection for chaos testing

All of this happens at the infrastructure layer - no application code changes needed.

                                          | +-----+ +-------+ |    | +-------+ +-----+ |

  - Every service implements retry,                    |

  - Consistent behavior across ALL services (any language)

3. Istio Traffic Management

VirtualService

virtualservice-full.yaml Istio 
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: reviews-vs
  namespace: production
spec:
  hosts:
  - reviews  # K8s service name (short or FQDN)
  http:
  # Route 1: Header-based routing (canary for internal testers)
  - match:
    - headers:
        x-canary:
          exact: "true"
    route:
    - destination:
        host: reviews
        subset: v3  # References DestinationRule subset
      weight: 100

  # Route 2: Traffic splitting (90/10 canary)
  - route:
    - destination:
        host: reviews
        subset: v2
      weight: 90
    - destination:
        host: reviews
        subset: v3
      weight: 10

    # Retry policy
    retries:
      attempts: 3
      perTryTimeout: 2s
      retryOn: 5xx,reset,connect-failure,retriable-4xx

    # Timeout
    timeout: 10s

    # Fault injection (chaos testing)
    fault:
      delay:
        percentage:
          value: 5.0    # 5% of requests
        fixedDelay: 3s  # Add 3s latency
      abort:
        percentage:
          value: 1.0    # 1% of requests
        httpStatus: 503 # Return 503

    # Mirror traffic (shadow testing)
    mirror:
      host: reviews
      subset: v3
    mirrorPercentage:
      value: 10.0  # Mirror 10% of traffic

DestinationRule

destinationrule.yaml Istio 
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata:
  name: reviews-dr
  namespace: production
spec:
  host: reviews
  trafficPolicy:
    # Load balancing
    loadBalancer:
      simple: LEAST_REQUEST  # ROUND_ROBIN, RANDOM, LEAST_REQUEST, PASSTHROUGH

    # Connection pool settings
    connectionPool:
      tcp:
        maxConnections: 100
        connectTimeout: 5s
      http:
        h2UpgradePolicy: DEFAULT
        http1MaxPendingRequests: 100
        http2MaxRequests: 1000
        maxRequestsPerConnection: 10
        maxRetries: 3

    # Circuit breaker (outlier detection)
    outlierDetection:
      consecutive5xxErrors: 5       # Trip after 5 consecutive 5xx
      interval: 10s                 # Check every 10s
      baseEjectionTime: 30s         # Eject for 30s minimum
      maxEjectionPercent: 50        # Max 50% of hosts ejected
      minHealthPercent: 50          # Require 50% healthy hosts

  # Subsets (versions) - referenced by VirtualService
  subsets:
  - name: v1
    labels:
      version: v1
  - name: v2
    labels:
      version: v2
    trafficPolicy:
      loadBalancer:
        simple: ROUND_ROBIN
  - name: v3
    labels:
      version: v3
    trafficPolicy:
      connectionPool:
        http:
          h2UpgradePolicy: UPGRADE  # Force HTTP/2 for v3

4. Istio Security

PeerAuthentication (mTLS)

peer-authentication.yaml Security 
# Mesh-wide STRICT mTLS (all services must use mTLS)
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: default
  namespace: istio-system  # Mesh-wide when in istio-system
spec:
  mtls:
    mode: STRICT  # STRICT, PERMISSIVE, DISABLE
---
# Namespace-level override (allow legacy services)
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: legacy-ns-policy
  namespace: legacy-services
spec:
  mtls:
    mode: PERMISSIVE  # Accept both mTLS and plaintext
---
# Workload-level: specific port exemption
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: metrics-port-exception
  namespace: production
spec:
  selector:
    matchLabels:
      app: my-service
  mtls:
    mode: STRICT
  portLevelMtls:
    9090:  # Prometheus scrape port
      mode: PERMISSIVE  # Allow plain HTTP for metrics scraping

AuthorizationPolicy

authorization-policy.yaml Security 
# Deny all traffic by default (zero-trust)
apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata:
  name: deny-all
  namespace: production
spec:
  {}  # Empty spec = deny all
---
# Allow specific service-to-service communication
apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata:
  name: allow-frontend-to-api
  namespace: production
spec:
  selector:
    matchLabels:
      app: api-service
  action: ALLOW
  rules:
  - from:
    - source:
        principals:
        - "cluster.local/ns/production/sa/frontend-sa"
        namespaces:
        - "production"
    to:
    - operation:
        methods: ["GET", "POST"]
        paths: ["/api/v1/*"]
    when:
    - key: request.headers[x-request-id]
      notValues: [""]
---
# Deny traffic from outside namespace
apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata:
  name: deny-external
  namespace: production
spec:
  selector:
    matchLabels:
      app: internal-service
  action: DENY
  rules:
  - from:
    - source:
        notNamespaces: ["production"]

5. Istio Observability

Automatic Telemetry (Zero Code Changes)

Envoy sidecars automatically generate metrics, traces, and access logs for all traffic. No instrumentation libraries needed in your application code.

Automatic Metrics

  • Request count: istio_requests_total (by source, dest, response code)
  • Request duration: istio_request_duration_milliseconds
  • Request size: istio_request_bytes
  • Response size: istio_response_bytes
  • TCP connections: istio_tcp_connections_opened_total
  • Golden signals (latency, traffic, errors, saturation) out of the box

Observability Tools

  • Kiali: Service mesh topology visualization, traffic flow, health status, configuration validation
  • Jaeger/Zipkin: Distributed tracing - see request path across services, identify latency bottlenecks
  • Grafana: Dashboards for mesh metrics (pre-built Istio dashboards)
  • Prometheus: Scrapes Envoy metrics, powers alerting
  • Access Logs: Configurable Envoy access logs (JSON format for parsing)

Enabling Telemetry

telemetry.yaml Observability 
# Enable access logging mesh-wide
apiVersion: telemetry.istio.io/v1alpha1
kind: Telemetry
metadata:
  name: mesh-logging
  namespace: istio-system
spec:
  accessLogging:
  - providers:
    - name: envoy
    filter:
      expression: "response.code >= 400"  # Only log errors

---
# Enable tracing with sampling
apiVersion: telemetry.istio.io/v1alpha1
kind: Telemetry
metadata:
  name: mesh-tracing
  namespace: istio-system
spec:
  tracing:
  - providers:
    - name: jaeger
    randomSamplingPercentage: 10.0  # Sample 10% of traces
    customTags:
      environment:
        literal:
          value: "production"

Header-Based Routing (Internal Testing)

header-based-routing.yaml Routing 
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: my-app
  namespace: production
spec:
  hosts:
  - my-app
  http:
  # Route internal testers to v2
  - match:
    - headers:
        x-canary-version:
          exact: "v2"
    - headers:
        cookie:
          regex: ".*canary=true.*"
    route:
    - destination:
        host: my-app
        subset: v2

  # Route by user identity (e.g., beta users)
  - match:
    - headers:
        x-user-group:
          exact: "beta"
    route:
    - destination:
        host: my-app
        subset: v2

  # All other traffic to v1 (stable)
  - route:
    - destination:
        host: my-app
        subset: v1
Canary Deployment Progress
  Time    v1 Weight    v2 Weight    Action
  -----   ---------    ---------    ------
  T+0     100%         0%           Deploy v2 pods (no traffic)
  T+5m    95%          5%           Initial canary (watch metrics)
  T+30m   75%          25%          Increase if error rate OK
  T+1h    50%          50%          Half traffic (validate perf)
  T+2h    25%          75%          Almost full rollout
  T+3h    0%           100%         Complete rollout
  T+24h   -            -            Remove v1 Deployment

  ROLLBACK at any step: set v1=100%, v2=0%
  Decision criteria:
  - Error rate (5xx) increase > 0.1% --> ROLLBACK
  - P99 latency increase > 20% --> ROLLBACK
  - Key business metric degradation --> ROLLBACK

7. Linkerd

Istio vs Linkerd Comparison

Feature Istio Linkerd
Proxy Envoy (C++, feature-rich) linkerd2-proxy (Rust, ultra-lightweight)
Resource Usage Higher (~50-100MB per sidecar) Lower (~20-30MB per sidecar, ~1ms p99 latency)
Complexity Complex (many CRDs, extensive config) Simple (minimal config, sane defaults)
mTLS Yes (configurable STRICT/PERMISSIVE) Yes (on by default, automatic)
Traffic Splitting VirtualService + DestinationRule TrafficSplit (SMI spec) or HTTPRoute
Protocol Support HTTP, gRPC, TCP, WebSocket, MongoDB, etc. HTTP, gRPC, TCP (fewer L7 protocols)
Multi-cluster Yes (complex setup) Yes (simpler model)
Extensibility Wasm plugins, EnvoyFilter Policy extensions (limited)
CNCF Status Graduated Graduated
Best For Complex routing, multi-protocol, extensibility Simplicity, low overhead, quick adoption

8. When to Use a Service Mesh

Criteria Service Mesh Recommended Probably Overkill
Number of Services 10+ microservices with complex communication Fewer than 5 services (monolith or simple arch)
Team Size Multiple teams owning different services Single team managing everything
Security Requirements Zero-trust, compliance (PCI, HIPAA), audit needs Internal-only services with trusted network
Traffic Management Canary releases, A/B testing, traffic mirroring Simple deployments, no advanced routing needed
Observability Need consistent metrics/tracing across all services Already have good instrumentation in code
Language Diversity Polyglot environment (Go, Java, Python, Node) Single language with good libraries
Operational Maturity Team experienced with K8s, ready for complexity Still learning Kubernetes basics

When NOT to Use a Service Mesh

  • You have fewer than 5-10 services (overhead not justified)
  • Team is still learning Kubernetes fundamentals (adds complexity)
  • Low-latency requirements where sidecar overhead (~1-3ms) is unacceptable
  • Simple architecture with one team and one language (library-based approach works)
  • You don't need mTLS, advanced routing, or mesh-level observability
  • Resource-constrained clusters (sidecars add memory/CPU per pod)

9. Service Mesh vs Ingress vs API Gateway

Feature Ingress Controller API Gateway Service Mesh
Scope North-South (external to cluster) North-South (external to services) East-West (service-to-service) + North-South
Position Edge of cluster Edge of platform/API surface Between every service pair
TLS Terminates at edge Terminates at edge mTLS between all services
Routing Host/path based routing Advanced routing, transformation L7 routing, traffic splitting, mirroring
Auth Basic (cert, OAuth via annotation) API keys, OAuth, JWT, rate limiting mTLS identity, JWT, RBAC between services
Observability Access logs, basic metrics API analytics, developer portal Full distributed tracing, golden signals
Circuit Breaking Limited Yes (per API) Yes (between all services)
Examples Nginx Ingress, Traefik, HAProxy Kong, Apigee, AWS API Gateway Istio, Linkerd, Consul Connect
Best For Simple external access to services API management, developer experience Microservice networking, security, observability

They Complement Each Other

In a production architecture, you often use all three together:

  • API Gateway at the edge: authentication, rate limiting, API versioning for external consumers
  • Ingress Controller (or Istio Gateway): route external traffic into the cluster
  • Service Mesh internally: secure and observe all service-to-service communication

10. Interview Q&A

Q1: What is a service mesh and what problems does it solve?
A service mesh is a dedicated infrastructure layer for handling service-to-service communication. It deploys a sidecar proxy (like Envoy) alongside each service, intercepting all network traffic. It solves: (1) Observability: automatic metrics, distributed tracing, access logs without code changes. (2) Security: automatic mTLS encryption, identity-based auth between services. (3) Traffic Management: retries, timeouts, circuit breaking, canary deployments, traffic mirroring. (4) Reliability: consistent retry/timeout behavior across all services regardless of language. The key value: these cross-cutting concerns are moved from application code to infrastructure, applied consistently, and managed centrally.
Q2: Compare Istio and Linkerd. When would you choose one over the other?
Istio: Uses Envoy proxy (C++), very feature-rich (Wasm extensions, multiple protocols, complex routing rules). Higher resource usage (~50-100MB per sidecar). Many CRDs and configuration options. Linkerd: Uses linkerd2-proxy (Rust), ultra-lightweight (~20MB, ~1ms p99 latency overhead). Simpler to operate with sane defaults. Less extensible but covers 90% of use cases. Choose Istio when: you need complex routing, Wasm extensibility, multi-protocol support (MongoDB, Redis), or your team has Envoy expertise. Choose Linkerd when: you want simplicity, minimal overhead, quick time-to-value, or your team is smaller and wants less operational burden. Both are CNCF graduated projects.
Q3: How does mTLS work in a service mesh? How are certificates managed?
In a service mesh, mTLS is automatic and transparent to applications. The control plane (Istiod/Linkerd identity) acts as a Certificate Authority. Each sidecar proxy gets a short-lived certificate (SPIFFE identity: spiffe://cluster/ns/namespace/sa/service-account). When Service A calls Service B: (1) A's sidecar initiates mTLS to B's sidecar, (2) both present their certificates, (3) both verify the other's cert was signed by the mesh CA, (4) encrypted tunnel established. The application sends plain HTTP to localhost; the sidecar handles all TLS. Certificates are automatically rotated (typically 24h in Istio). No manual cert management, no expiry incidents.
Q4: How would you implement a canary deployment using a service mesh?
With Istio: (1) Deploy v2 alongside v1 (separate Deployment with version label). (2) Create a DestinationRule defining subsets (v1, v2 based on labels). (3) Create a VirtualService with weight-based routing: start at 95/5 (v1/v2). (4) Monitor error rates and latency for v2 traffic using mesh metrics (istio_requests_total with destination_version label). (5) Gradually shift traffic: 90/10, 75/25, 50/50, 0/100. (6) If errors spike, rollback by setting v1=100, v2=0. Advanced: Use header-based routing to test v2 with internal team first (match header x-canary: true). Use traffic mirroring to send shadow traffic to v2 without affecting users. Automate with Flagger or Argo Rollouts for progressive delivery.
Q5: How does sidecar injection work in Istio?
Istio uses a MutatingAdmissionWebhook to automatically inject sidecars. When a pod is created: (1) The API server sends the pod spec to Istio's webhook. (2) If the namespace has label istio-injection=enabled (or pod has annotation sidecar.istio.io/inject=true), (3) The webhook mutates the pod spec to add: an init container (istio-init) that sets up iptables rules to intercept traffic, and the Envoy sidecar container (istio-proxy). (4) The iptables rules redirect all inbound/outbound traffic through the sidecar on port 15001/15006. The application is unaware - it sends traffic to original destinations but iptables redirects it through Envoy. You can exclude specific ports/IPs with annotations.
Q6: When would you NOT use a service mesh?
Avoid a service mesh when: (1) Few services (<5-10): Overhead of running/maintaining the mesh isn't justified. Use library-based approaches instead. (2) Early K8s adoption: Adding a mesh before mastering K8s basics compounds complexity. (3) Ultra-low latency requirements: The sidecar adds 1-5ms per hop. For high-frequency trading or real-time systems, this matters. (4) Resource-constrained: Each sidecar uses 50-100MB RAM (Istio). For 100 pods, that's 5-10GB extra. (5) Simple communication patterns: If services just do synchronous REST calls with no need for traffic splitting, circuit breaking, or mTLS, a mesh is overkill. (6) Single language/framework: If all services use the same language, library-based solutions (like gRPC interceptors) may suffice.
Q7: What is the difference between a service mesh, an ingress controller, and an API gateway?
Ingress Controller (Nginx, Traefik): Handles North-South traffic at cluster edge. Simple host/path routing, TLS termination. One point of entry. API Gateway (Kong, Apigee): Also North-South but with API management features: rate limiting, API keys, developer portal, request transformation, analytics. Aimed at API consumers. Service Mesh (Istio, Linkerd): Primarily East-West (service-to-service) traffic. mTLS everywhere, distributed tracing, fine-grained traffic control between all services. In practice, they're complementary: API Gateway for external API management, Ingress for routing into the cluster, Service Mesh for internal communication security and observability. Istio Gateway can also serve as an ingress controller.
Q8: How do you troubleshoot connectivity issues in a service mesh?
Systematic approach: (1) Check if sidecar is injected: kubectl get pod -o jsonpath='{.spec.containers[*].name}' - should see istio-proxy. (2) Check proxy status: istioctl proxy-status - are all proxies synced? Look for "STALE" or "NOT SENT". (3) Check proxy config: istioctl proxy-config routes <pod> - verify routes exist for the target service. (4) Check mTLS: istioctl authn tls-check <pod> <service> - confirm both sides agree on mTLS mode. (5) Check AuthorizationPolicy: Is there a deny-all without matching allow? (6) Check Envoy logs: kubectl logs <pod> -c istio-proxy - look for "403 RBAC: access denied" or connection errors. (7) Analyze with Kiali: visualize traffic flow, see which connections are failing. (8) Check DestinationRule: wrong subset labels can cause "no healthy upstream".

11. Scenarios

1. Sidecar Not Being Injected

Symptom: New pods in a namespace don't have the istio-proxy container. Service mesh features not working.

1. Check namespace label: kubectl get namespace production --show-labels - must have istio-injection=enabled
2. Check if pod has opt-out annotation: sidecar.istio.io/inject: "false" on pod or deployment template
3. Check webhook: kubectl get mutatingwebhookconfigurations - istio-sidecar-injector should exist
4. Check istiod is running: kubectl get pods -n istio-system -l app=istiod - webhook calls istiod for injection
5. Check if pod was created before label was added: Existing pods are NOT retroactively injected. Must restart: kubectl rollout restart deployment/my-app -n production
6. Check for host network: Pods with hostNetwork: true skip injection. Init containers and Jobs may also be excluded by default
7. Manual injection test: kubectl get deploy my-app -o yaml | istioctl kube-inject -f - | kubectl apply -f -
2. mTLS Breaking Legacy Service Communication

Symptom: After enabling STRICT mTLS mesh-wide, a legacy service (no sidecar) cannot communicate with mesh services. Getting "connection reset" errors.

1. Confirm the issue: Legacy service has no sidecar, cannot present a client certificate when calling a STRICT mTLS service
2. Check current mTLS mode: istioctl authn tls-check <legacy-pod> <target-service>
3. Immediate fix: Set PERMISSIVE mode for the target service (accepts both mTLS and plaintext)
permissive-for-legacy.yaml Fix 
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: allow-legacy-clients
  namespace: target-namespace
spec:
  selector:
    matchLabels:
      app: target-service
  mtls:
    mode: PERMISSIVE
4. Better solution: Add the legacy service to the mesh (inject sidecar). If not possible (e.g., VM, external service), use a mesh gateway as an intermediary
5. For mesh services calling OUT to legacy: Create a DestinationRule disabling mTLS for that destination
disable-mtls-to-legacy.yaml DestinationRule 
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata:
  name: legacy-service-dr
  namespace: production
spec:
  host: legacy-service.legacy-ns.svc.cluster.local
  trafficPolicy:
    tls:
      mode: DISABLE  # Don't use mTLS when calling this service
6. Migration plan: Keep PERMISSIVE during migration, track plaintext connections in Kiali, switch to STRICT only after all services are in the mesh
3. Traffic Split for Gradual Rollout

Requirement: Roll out v2 of payment service gradually. Zero downtime. Automatic rollback if error rate exceeds threshold.

1. Deploy v2 alongside v1 (separate Deployment, same Service selector but with version label)
2. Create DestinationRule with subsets for v1 and v2
3. Start with header-based routing: send internal team traffic to v2 via x-canary: true header. Validate functionality
4. Begin weight-based split: 95% v1, 5% v2. Monitor for 30 minutes
5. Monitor key metrics via Prometheus/Grafana: rate(istio_requests_total{destination_version="v2",response_code=~"5.."}[5m])
6. If error rate OK: progress 5% -> 25% -> 50% -> 100%. Each step: wait, validate, proceed
7. Automate with Flagger: Define canary analysis (metrics query + threshold), Flagger automatically progresses or rolls back
flagger-canary.yaml Flagger 
apiVersion: flagger.app/v1beta1
kind: Canary
metadata:
  name: payment-service
  namespace: production
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: payment-service
  service:
    port: 8080
  analysis:
    interval: 1m           # Check every minute
    threshold: 5           # Max failed checks before rollback
    maxWeight: 50          # Max canary weight
    stepWeight: 10         # Increase by 10% each step
    metrics:
    - name: request-success-rate
      thresholdRange:
        min: 99            # Must maintain 99% success
      interval: 1m
    - name: request-duration
      thresholdRange:
        max: 500           # P99 must stay under 500ms
      interval: 1m
8. Rollback: If automated - Flagger detects metric breach and sets v1=100%. If manual: kubectl apply VirtualService with v1 weight=100
