Docker

Complete reference for Docker containers, images, networking, storage, security, Docker Compose, and a detailed Docker vs Kubernetes comparison for interview preparation and production use.

1. What is Docker?

Docker is a platform for developing, shipping, and running applications inside containers. Containers package an application with all its dependencies, libraries, and configuration into a single portable unit that runs consistently across any environment.

Containers vs Virtual Machines

Virtual Machines vs Containers
     VIRTUAL MACHINES                           CONTAINERS
 ________________________               ________________________
|  App A  |  App B  | App C |           |  App A  |  App B  | App C |
|---------|---------|-------|           |---------|---------|-------|
| Bins/   | Bins/   | Bins/ |           | Bins/   | Bins/   | Bins/ |
| Libs    | Libs    | Libs  |           | Libs    | Libs    | Libs  |
|---------|---------|-------|           |_________|_________|_______|
| Guest   | Guest   | Guest |           |                           |
| OS      | OS      | OS    |           |     Container Runtime     |
| (full)  | (full)  | (full)|           |      (Docker Engine)      |
|_________|_________|_______|           |___________________________|
|                           |           |                           |
|       Hypervisor          |           |        Host OS            |
|   (VMware/KVM/Hyper-V)   |           |   (Linux/macOS/Windows)   |
|___________________________|           |___________________________|
|                           |           |                           |
|     Host OS / Hardware    |           |        Hardware           |
|___________________________|           |___________________________|

 Each VM: Full OS (GBs)                 Each Container: Shared kernel (MBs)
 Boot time: Minutes                     Start time: Seconds
 Heavy resource usage                   Lightweight, efficient
 Strong isolation (hardware)            Process-level isolation (namespaces)

Why Containers?

BenefitExplanation
LightweightShare host kernel, no guest OS overhead. Images are MBs not GBs.
Fast StartupContainers start in seconds (processes, not full OS boot).
Consistent Environments"Works on my machine" solved. Same image runs everywhere.
IsolationProcess, network, and filesystem isolation via Linux namespaces and cgroups.
PortabilityBuild once, run anywhere: laptop, CI/CD, cloud, bare metal.
Efficient ResourcesHigher density than VMs. Run many more containers per host.
Version ControlImages are immutable and versioned. Easy rollback.
MicroservicesNatural fit for microservice architectures. One service per container.

2. Docker Images

What is an Image?

A Docker image is a read-only template containing a layered filesystem with everything needed to run an application: OS libraries, runtime, application code, and configuration. Each instruction in a Dockerfile creates a new layer. Layers are cached and shared between images for efficiency.

Image Layers

Docker Image Layers (Read-Only) + Container Layer (Read-Write)
┌─────────────────────────────────────────────┐
│        Container Layer (Read-Write)          │  <-- Writable, ephemeral
│   (runtime changes, logs, temp files)        │
├─────────────────────────────────────────────┤
│  Layer 5: CMD ["node", "server.js"]          │  <-- Metadata (0 bytes)
├─────────────────────────────────────────────┤
│  Layer 4: COPY . /app                        │  <-- Application code (~5MB)
├─────────────────────────────────────────────┤
│  Layer 3: RUN npm install                    │  <-- Dependencies (~150MB)
├─────────────────────────────────────────────┤
│  Layer 2: RUN apt-get install -y curl        │  <-- System packages (~20MB)
├─────────────────────────────────────────────┤
│  Layer 1: FROM node:20-slim                  │  <-- Base OS + Node.js (~180MB)
└─────────────────────────────────────────────┘

All layers are READ-ONLY and CACHED.
Changing one layer invalidates all layers ABOVE it.
Multiple containers share the same image layers (copy-on-write).

Complete Dockerfile Examples

Node.js App (Multi-Stage Build)

Dockerfile (Node.js Multi-Stage)
# ============ Stage 1: Build ============
FROM node:20-alpine AS builder

WORKDIR /app

# Copy dependency files first (cache optimization)
COPY package.json package-lock.json ./

# Install ALL dependencies (including devDependencies for build)
RUN npm ci

# Copy source code
COPY src/ ./src/
COPY tsconfig.json ./

# Build the TypeScript application
RUN npm run build

# Prune devDependencies
RUN npm prune --production

# ============ Stage 2: Production ============
FROM node:20-alpine AS production

# Security: run as non-root
RUN addgroup -S appgroup && adduser -S appuser -G appgroup

WORKDIR /app

# Copy only production node_modules from builder
COPY --from=builder /app/node_modules ./node_modules

# Copy built application
COPY --from=builder /app/dist ./dist

# Copy package.json for metadata
COPY package.json ./

# Set environment
ENV NODE_ENV=production
ENV PORT=3000

# Expose port
EXPOSE 3000

# Switch to non-root user
USER appuser

# Health check
HEALTHCHECK --interval=30s --timeout=3s --start-period=5s --retries=3 \
    CMD wget --no-verbose --tries=1 --spider http://localhost:3000/health || exit 1

# Start application
CMD ["node", "dist/server.js"]

Python App (with Virtual Environment)

Dockerfile (Python with venv)
# ============ Stage 1: Build ============
FROM python:3.12-slim AS builder

WORKDIR /app

# Install build dependencies
RUN apt-get update && \
    apt-get install -y --no-install-recommends gcc libpq-dev && \
    rm -rf /var/lib/apt/lists/*

# Create virtual environment
RUN python -m venv /opt/venv
ENV PATH="/opt/venv/bin:$PATH"

# Install Python dependencies
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# ============ Stage 2: Production ============
FROM python:3.12-slim AS production

# Install runtime dependencies only
RUN apt-get update && \
    apt-get install -y --no-install-recommends libpq5 curl && \
    rm -rf /var/lib/apt/lists/*

# Create non-root user
RUN useradd --create-home --shell /bin/bash appuser

WORKDIR /app

# Copy virtual environment from builder
COPY --from=builder /opt/venv /opt/venv
ENV PATH="/opt/venv/bin:$PATH"

# Copy application code
COPY --chown=appuser:appuser . .

# Switch to non-root user
USER appuser

EXPOSE 8000

HEALTHCHECK --interval=30s --timeout=5s --start-period=10s --retries=3 \
    CMD curl -f http://localhost:8000/health || exit 1

CMD ["gunicorn", "--bind", "0.0.0.0:8000", "--workers", "4", "app:create_app()"]

Go App (Scratch Final Image)

Dockerfile (Go with scratch)
# ============ Stage 1: Build ============
FROM golang:1.22-alpine AS builder

# Install certificates (needed for HTTPS calls in final image)
RUN apk add --no-cache ca-certificates

WORKDIR /app

# Download dependencies first (cache optimization)
COPY go.mod go.sum ./
RUN go mod download

# Copy source code
COPY . .

# Build static binary (CGO disabled for scratch compatibility)
RUN CGO_ENABLED=0 GOOS=linux GOARCH=amd64 \
    go build -ldflags="-w -s" -o /app/server ./cmd/server

# ============ Stage 2: Minimal Production Image ============
FROM scratch

# Copy CA certificates for HTTPS
COPY --from=builder /etc/ssl/certs/ca-certificates.crt /etc/ssl/certs/

# Copy the static binary
COPY --from=builder /app/server /server

# Expose port
EXPOSE 8080

# Run the binary
ENTRYPOINT ["/server"]

3. Dockerfile Best Practices

Multi-Stage Builds (Reduce Image Size)

Before vs After Multi-Stage
# BAD: Single stage (image includes build tools = 1.2GB)
FROM node:20
WORKDIR /app
COPY . .
RUN npm install
RUN npm run build
CMD ["node", "dist/server.js"]
# Result: 1.2GB image with devDependencies, source, build tools

# GOOD: Multi-stage (final image has only runtime = 150MB)
FROM node:20 AS build
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

FROM node:20-alpine
WORKDIR /app
COPY --from=build /app/dist ./dist
COPY --from=build /app/node_modules ./node_modules
CMD ["node", "dist/server.js"]
# Result: 150MB image with only production artifacts

Best Practices Checklist

PracticeWhyExample
Use multi-stage builds Reduces final image size by 5-10x Separate build and runtime stages
Minimize layers Fewer layers = smaller image, faster pulls Combine RUN commands with &&
Use specific base image tags Reproducible builds, no surprise breakage FROM node:20.11-alpine3.19 not FROM node:latest
Order by change frequency Maximize cache hits during rebuilds COPY package.json before COPY source code
Run as non-root Security: limit container escape damage USER appuser (create user first)
Use COPY not ADD COPY is explicit; ADD has hidden behavior COPY . /app (use ADD only for tar extraction)
Never store secrets in images Images may be shared; layers are inspectable Use runtime env vars, secrets managers, or BuildKit secrets
Use .dockerignore Smaller build context, faster builds, no secrets leaked Exclude .git, node_modules, .env, tests
Pin package versions Reproducible builds across time RUN apt-get install -y curl=7.88.1-10
Add HEALTHCHECK Docker can monitor container health HEALTHCHECK CMD curl -f http://localhost/ || exit 1
Clean up in same RUN layer Deleted files still exist in prior layers RUN apt-get install -y pkg && rm -rf /var/lib/apt/lists/*
Use alpine or slim variants Smaller base = smaller final image FROM python:3.12-slim (not python:3.12)

5. Docker Compose

Docker Compose defines and runs multi-container applications using a single YAML file. It manages the complete lifecycle of your application stack: building images, creating networks, starting services in dependency order, and tearing everything down.

Complete docker-compose.yml Example

docker-compose.yml (Full-Stack Application)
version: '3.8'

services:
  # ============ Application Service ============
  app:
    build:
      context: .
      dockerfile: Dockerfile
      args:
        NODE_ENV: production
    container_name: myapp
    ports:
      - "3000:3000"
    environment:
      - NODE_ENV=production
      - DB_HOST=db
      - DB_PORT=5432
      - DB_NAME=appdb
      - DB_USER=appuser
      - DB_PASSWORD=${DB_PASSWORD}  # from .env file
      - REDIS_URL=redis://redis:6379
    depends_on:
      db:
        condition: service_healthy
      redis:
        condition: service_started
    volumes:
      - app-logs:/app/logs
      - ./uploads:/app/uploads  # bind mount for uploaded files
    networks:
      - frontend
      - backend
    healthcheck:
      test: ["CMD", "wget", "--no-verbose", "--tries=1", "--spider", "http://localhost:3000/health"]
      interval: 30s
      timeout: 5s
      retries: 3
      start_period: 10s
    restart: unless-stopped
    deploy:
      resources:
        limits:
          cpus: '1.0'
          memory: 512M
        reservations:
          cpus: '0.25'
          memory: 128M

  # ============ Database Service ============
  db:
    image: postgres:15-alpine
    container_name: myapp-db
    volumes:
      - postgres-data:/var/lib/postgresql/data
      - ./init-scripts:/docker-entrypoint-initdb.d  # auto-run SQL on first start
    environment:
      POSTGRES_DB: appdb
      POSTGRES_USER: appuser
      POSTGRES_PASSWORD: ${DB_PASSWORD}
    ports:
      - "5432:5432"  # expose for local dev tools
    networks:
      - backend
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U appuser -d appdb"]
      interval: 10s
      timeout: 5s
      retries: 5
    restart: unless-stopped

  # ============ Redis Cache ============
  redis:
    image: redis:7-alpine
    container_name: myapp-redis
    command: redis-server --appendonly yes --maxmemory 256mb --maxmemory-policy allkeys-lru
    volumes:
      - redis-data:/data
    ports:
      - "6379:6379"
    networks:
      - backend
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 10s
      timeout: 3s
      retries: 3
    restart: unless-stopped

  # ============ Nginx Reverse Proxy ============
  nginx:
    image: nginx:1.25-alpine
    container_name: myapp-nginx
    volumes:
      - ./nginx/nginx.conf:/etc/nginx/nginx.conf:ro
      - ./nginx/conf.d:/etc/nginx/conf.d:ro
      - ./certbot/conf:/etc/letsencrypt:ro
      - ./certbot/www:/var/www/certbot:ro
    ports:
      - "80:80"
      - "443:443"
    depends_on:
      - app
    networks:
      - frontend
    restart: unless-stopped

# ============ Named Volumes ============
volumes:
  postgres-data:
    driver: local
  redis-data:
    driver: local
  app-logs:
    driver: local

# ============ Networks ============
networks:
  frontend:
    driver: bridge
  backend:
    driver: bridge
    internal: true  # no external access, only inter-service

# docker compose --profile dev --profile monitoring up  # all except debug

8. Docker Security

Security Best Practices

PracticeImplementationWhy
Run as non-root RUN adduser -D appuser
USER appuser 		Limits damage if container is compromised. Root in container = root on host (without user namespaces).
Read-only filesystem docker run --read-only --tmpfs /tmp myapp Prevents attackers from writing malicious files. Use tmpfs for necessary write paths.
Drop capabilities --cap-drop ALL --cap-add NET_BIND_SERVICE Containers get subset of root capabilities by default. Drop all, add only what is needed.
Resource limits --memory 512m --cpus 1.0 Prevent DoS from runaway processes. Contain resource exhaustion.
No privileged mode Never use --privileged in production Privileged gives ALL host capabilities. Effectively removes all container isolation.
Image scanning trivy image myapp:v1 in CI/CD Detect known CVEs in base images and dependencies before deployment.
Content trust export DOCKER_CONTENT_TRUST=1 Only pull signed images. Prevents tampered images from running.
Secrets management Docker secrets, Vault, AWS Secrets Manager Never put secrets in ENV (visible in inspect), ARG (visible in history), or image layers.
seccomp profiles --security-opt seccomp=profile.json Restrict which system calls a container can make. Default profile blocks ~44 dangerous syscalls.
AppArmor / SELinux --security-opt apparmor=docker-custom Mandatory access control. Further restricts container processes beyond capabilities.
No new privileges --security-opt no-new-privileges Prevent privilege escalation via setuid/setgid binaries inside container.
Minimal base images Use alpine, distroless, or scratch Fewer packages = fewer vulnerabilities. No shell in distroless = harder to exploit.

10. Interview Questions & Answers

Q: What is Docker and how does it differ from Virtual Machines?

A: Docker is a containerization platform that packages applications with their dependencies into isolated units called containers. Unlike VMs which run a full guest OS on a hypervisor (taking minutes to boot and GBs of space), containers share the host kernel and use Linux namespaces/cgroups for isolation. Containers start in seconds, use MBs of memory, and achieve near-native performance. VMs provide stronger isolation (hardware-level) while containers provide process-level isolation.

Q: Explain Docker image layers. Why do they matter?

A: Docker images are built as a stack of read-only layers. Each Dockerfile instruction (RUN, COPY, ADD) creates a new layer. Layers are cached and shared between images -- if two images use the same base layer, it is stored only once on disk. This makes builds faster (unchanged layers are cached) and pulls efficient (only download layers you do not have). The downside: deleting a file in a later layer does not reduce image size because the file still exists in its original layer. This is why you must clean up within the same RUN instruction.

Q: What is the difference between CMD and ENTRYPOINT?

CMD vs ENTRYPOINT Behavior
# CMD: Default command, easily overridden at runtime
FROM ubuntu
CMD ["echo", "hello world"]
# docker run myimage              → "hello world"
# docker run myimage echo bye     → "bye" (CMD completely replaced)

# ENTRYPOINT: Fixed executable, CMD becomes default arguments
FROM ubuntu
ENTRYPOINT ["echo"]
CMD ["hello world"]
# docker run myimage              → "hello world"
# docker run myimage goodbye      → "goodbye" (only args replaced)
# docker run --entrypoint bash myimage  → override entrypoint

# Common pattern: ENTRYPOINT for the binary, CMD for default flags
ENTRYPOINT ["python", "app.py"]
CMD ["--port", "8000"]
# docker run myimage                    → python app.py --port 8000
# docker run myimage --port 9000        → python app.py --port 9000

Key Rule: CMD is the whole command that can be replaced. ENTRYPOINT is the fixed executable; CMD provides its default arguments.

Q: COPY vs ADD -- when to use which?

A: Always prefer COPY unless you specifically need ADD's extra features. COPY simply copies files from build context to image. ADD does the same but also: (1) auto-extracts tar archives (.tar, .tar.gz), and (2) can download from URLs (though this is discouraged -- use RUN curl instead for cache control). Using COPY makes your Dockerfile more explicit and predictable.

Q: Explain multi-stage builds. Why use them?

A: Multi-stage builds use multiple FROM statements in one Dockerfile. You build in one stage (with compilers, dev dependencies) and copy only the final artifacts to a minimal production stage. Benefits: (1) Final image is 5-10x smaller (no build tools), (2) Smaller attack surface, (3) Faster deploys and pulls, (4) Single Dockerfile for build and run. Example: build a Go binary in golang:1.22 image, then COPY --from=builder the binary to a scratch image (just the binary, no OS = ~5MB).

Q: Explain Docker networking types and when to use each.

A: Bridge (default): containers get isolated network, communicate via IP or DNS (custom bridge only). Use for most containers. Host: container uses host's network stack directly. Use when you need maximum network performance or host port access without mapping. None: no networking at all. Use for batch processing that needs no network. Overlay: spans multiple Docker hosts (Swarm). Use for multi-host communication. Macvlan: assigns MAC address, container appears on physical LAN. Use for legacy apps requiring LAN presence.

Q: What is the difference between volumes and bind mounts?

A: Volumes are managed by Docker (stored in /var/lib/docker/volumes/), portable, work on all platforms, support volume drivers for cloud storage, and are the recommended approach for persistent data. Bind mounts map a specific host path into the container, are host-dependent (not portable), but allow real-time file sharing between host and container. Use volumes for production data (databases), bind mounts for development (live code reload).

Q: How do you reduce Docker image size? Name 5 techniques.

A: (1) Multi-stage builds -- only copy artifacts, not build tools. (2) Use alpine/slim/distroless base images -- alpine is ~5MB vs ~1GB for full Ubuntu. (3) Combine RUN commands -- clean up in same layer (apt-get install && rm -rf /var/lib/apt/lists/*). (4) Use .dockerignore -- exclude .git, node_modules, tests from context. (5) Remove unnecessary files -- docs, man pages, locale files, package caches. Bonus: (6) Use scratch for static binaries (Go, Rust).

Q: What are Docker security best practices?

A: (1) Run as non-root user (USER instruction). (2) Use read-only filesystems (--read-only). (3) Drop all capabilities, add back only needed ones (--cap-drop ALL). (4) Set resource limits (--memory, --cpus). (5) Never use --privileged in production. (6) Scan images for vulnerabilities (Trivy, Snyk). (7) Use minimal base images. (8) Never store secrets in images. (9) Enable Docker Content Trust for signed images. (10) Use seccomp/AppArmor profiles.

Q: Docker vs Kubernetes -- when to use which?

A: Use Docker Compose for: local development, single-server apps, small teams, prototypes, CI/CD pipelines, budget-constrained projects. Use Kubernetes for: production microservices needing high availability, auto-scaling, multi-team environments with RBAC, zero-downtime deployments, compliance requirements, and when running across multiple nodes/regions. Docker is your build tool regardless -- it creates the images that K8s runs.

Q: What happened to Docker in Kubernetes (dockershim removal)?

A: In Kubernetes 1.24 (May 2022), dockershim was removed. Previously, K8s used Docker as its container runtime via a compatibility layer called dockershim. This was replaced by direct containerd/CRI-O integration via the CRI (Container Runtime Interface). Impact: Docker-built images still work perfectly (OCI standard), but the Docker daemon is no longer needed on K8s nodes. K8s nodes now run containerd or CRI-O directly, which is lighter and more efficient.

Q: Explain the container lifecycle.

A: Created (docker create): filesystem layers assembled, config set, but process not started. Running (docker start): main process (PID 1) executing. Paused (docker pause): process frozen via cgroups freezer, still in memory. Stopped (docker stop): SIGTERM sent, then SIGKILL after grace period (default 10s). Deleted (docker rm): container and writable layer removed. docker run combines create + start in one command.

Q: What is a dangling image?

A: A dangling image is a layer that has no tag and is not referenced by any other image. They are typically created when you rebuild an image with the same tag -- the old image loses its tag and becomes <none>:<none>. They waste disk space. Remove them with: docker image prune (dangling only) or docker image prune -a (all unused images, including those without containers).

Q: Docker Compose vs Docker Swarm vs Kubernetes?

A: Docker Compose: Multi-container apps on a single host. Development and simple deployments. Not an orchestrator. Docker Swarm: Docker's native orchestrator. Multi-host, built into Docker, simpler than K8s but fewer features. Good for small clusters. Largely deprecated in favor of K8s. Kubernetes: Industry-standard orchestrator. Multi-host, auto-scaling, self-healing, extensive ecosystem. Complex but most powerful. Production standard for microservices at scale. Most organizations choose K8s over Swarm today.

11. Real-World Scenarios & Troubleshooting

ProblemCauseSolution
Container exits immediately No foreground process. CMD runs and exits (e.g., bash with no -it, or script finishes). Ensure CMD runs a long-lived foreground process. Check: docker logs container_name. For debugging: docker run -it image bash.
Image build: COPY file not found File is outside build context, or excluded by .dockerignore. Build context is the directory passed to docker build. Ensure file is within it and not in .dockerignore. Check: docker build -f Dockerfile .
Container cannot reach internet DNS misconfiguration, or bridge network without NAT. Check DNS: docker exec container cat /etc/resolv.conf. Try --dns 8.8.8.8. Verify host iptables/firewall allows forwarding.
Port already in use Another process (or container) is bound to the same host port. Find process: lsof -i :8080 or docker ps. Change port mapping or stop conflicting service.
Disk space full Dangling images, stopped containers, unused volumes, build cache. docker system df (show usage), then docker system prune -a --volumes (removes everything unused).
Permission denied on volume UID/GID mismatch between host and container user. Match UIDs: RUN chown 1000:1000 /app/data in Dockerfile. Or run container with --user $(id -u):$(id -g).
Slow builds Large build context, cache invalidation, missing .dockerignore. Add .dockerignore (exclude .git, node_modules). Order Dockerfile: dependencies before source code. Use BuildKit: DOCKER_BUILDKIT=1.
Container OOM killed Application exceeds memory limit set by --memory flag. Check: docker inspect container | grep OOMKilled. Increase limit or optimize app memory usage.

12. Common Dockerfile Mistakes (Wrong vs Correct)

Mistake 1: Using :latest Tag

WRONG: Unpinned base image
# WRONG - "latest" changes without warning, breaks builds unpredictably
FROM node:latest
FROM python:latest
FROM nginx:latest
CORRECT: Pinned version and variant
# CORRECT - specific version, reproducible builds
FROM node:20.11-alpine3.19
FROM python:3.12-slim-bookworm
FROM nginx:1.25-alpine

Mistake 2: Running as Root

WRONG: Container runs as root (default)
# WRONG - if container is compromised, attacker has root access
FROM node:20-alpine
WORKDIR /app
COPY . .
RUN npm install
CMD ["node", "server.js"]
# Process runs as root (UID 0)
CORRECT: Non-root user
# CORRECT - minimal privileges
FROM node:20-alpine
WORKDIR /app

# Create non-root user
RUN addgroup -S appgroup && adduser -S appuser -G appgroup

# Install dependencies as root (needs write to /app)
COPY package*.json ./
RUN npm ci --production

# Copy app and change ownership
COPY --chown=appuser:appgroup . .

# Switch to non-root user
USER appuser

CMD ["node", "server.js"]
# Process runs as appuser (non-root)

Mistake 3: Not Using Multi-Stage Builds

WRONG: Single stage includes build tools (image: ~1.2GB)
# WRONG - final image contains gcc, make, devDependencies, source
FROM node:20
WORKDIR /app
COPY . .
RUN npm install       # includes devDependencies
RUN npm run build     # TypeScript compiler included
CMD ["node", "dist/server.js"]
# Image size: ~1.2GB
CORRECT: Multi-stage build (image: ~150MB)
# CORRECT - build stage discarded, only runtime artifacts kept
FROM node:20-alpine AS build
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build
RUN npm prune --production

FROM node:20-alpine
WORKDIR /app
COPY --from=build /app/dist ./dist
COPY --from=build /app/node_modules ./node_modules
USER node
CMD ["node", "dist/server.js"]
# Image size: ~150MB

Mistake 4: COPY Before Dependency Install (Cache Bust)

WRONG: Any source change invalidates npm install cache
# WRONG - copying all source first means ANY file change
# invalidates the npm install cache (slow rebuilds!)
FROM node:20-alpine
WORKDIR /app
COPY . .                  # <-- source changes bust ALL cache below
RUN npm install           # Re-runs every time any file changes!
CMD ["node", "server.js"]
CORRECT: Copy dependency manifest first, then source
# CORRECT - dependency install is cached until package.json changes
FROM node:20-alpine
WORKDIR /app
COPY package.json package-lock.json ./   # <-- changes rarely
RUN npm ci                               # Cached until package.json changes
COPY . .                                 # <-- source changes only affect this layer
CMD ["node", "server.js"]

Mistake 5: Using ADD When COPY Suffices

WRONG: ADD has hidden behavior
# WRONG - ADD auto-extracts tars and can fetch URLs (surprising behavior)
FROM node:20-alpine
ADD . /app                    # Might accidentally extract tar files
ADD https://example.com/file /app/file  # Downloaded at build time, not cached well
CORRECT: Explicit COPY (use ADD only for tar extraction)
# CORRECT - COPY is explicit and predictable
FROM node:20-alpine
COPY . /app

# Only use ADD when you specifically want tar extraction:
ADD rootfs.tar.gz /

Mistake 6: Not Cleaning Up in Same RUN Layer

WRONG: Cleanup in separate layer (files still in previous layer)
# WRONG - apt cache is in layer 1, removing in layer 2 does NOT reduce size
RUN apt-get update
RUN apt-get install -y curl wget git
RUN rm -rf /var/lib/apt/lists/*   # This layer adds to size, doesn't reduce!
CORRECT: Install and clean in same RUN (single layer)
# CORRECT - single layer, cleanup happens before layer is committed
RUN apt-get update && \
    apt-get install -y --no-install-recommends \
        curl \
        wget \
    && rm -rf /var/lib/apt/lists/* \
    && apt-get clean

Mistake 7: Storing Secrets in ENV or ARG

WRONG: Secrets visible in image history and inspect
# WRONG - secrets are baked into image layers forever
# Visible via: docker history, docker inspect, or pulling the image
ENV API_KEY=sk-1234567890abcdef
ARG DB_PASSWORD=supersecret
RUN echo "machine github.com login token password ${GITHUB_TOKEN}" > ~/.netrc
CORRECT: Use BuildKit secrets or runtime injection
# CORRECT Option 1: BuildKit secret mounts (never stored in layers)
# syntax=docker/dockerfile:1
RUN --mount=type=secret,id=github_token \
    GITHUB_TOKEN=$(cat /run/secrets/github_token) && \
    git clone https://${GITHUB_TOKEN}@github.com/org/repo.git
# Build with: docker build --secret id=github_token,src=./token.txt .

# CORRECT Option 2: Runtime environment variables (not in image)
# Pass at runtime only:
# docker run -e API_KEY=$API_KEY myapp
# docker run --env-file .env myapp

# CORRECT Option 3: External secrets manager
# Application fetches secrets from Vault/AWS Secrets Manager at startup
