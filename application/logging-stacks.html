K8s Logging Fundamentals

  • stdout/stderr: Container runtime captures stdout/stderr and writes to JSON log files on the node
  • Log rotation: kubelet manages log rotation (default: 10MB per file, 5 files per container)
  • Node path: /var/log/containers/ contains symlinks to /var/log/pods/
  • kubectl logs: Reads from the container runtime API (only shows current/previous container logs)
  • Log lifecycle: Logs are lost when a pod is deleted (unless collected by an external system)
  • Multi-line: Stack traces appear as multiple log entries unless the application uses structured logging

2. Structured Logging

Unstructured (Bad)

2024-03-15 10:23:45 ERROR Failed to process payment for user 12345: connection timeout to payment-gateway:443 after 30s
2024-03-15 10:23:45 WARN Retrying payment for order #98765 (attempt 2/3)

Hard to parse, search, and aggregate. Requires regex/grok patterns.

Structured JSON (Good)

{"timestamp":"2024-03-15T10:23:45.123Z","level":"ERROR","service":"payment-api","trace_id":"abc123def456","span_id":"789xyz","message":"Failed to process payment","user_id":"12345","order_id":"98765","error":"connection timeout","target":"payment-gateway:443","timeout_ms":30000,"attempt":1,"max_attempts":3}

Machine-parseable, searchable by any field, easy to aggregate and correlate.

Essential Structured Log Fields

levelSeverity for filteringERROR, WARN, INFO
serviceSource service namepayment-api
trace_idDistributed trace correlationabc123def456
span_idSpecific operation within trace789xyz
messageHuman-readable descriptionFailed to process payment
errorError type/message if applicableconnection timeout
environmentDeployment environmentproduction
versionApplication version/commitv2.3.1, abc123
request_idUnique request identifierreq-uuid-here

Elasticsearch Concepts

  • Index: Collection of documents (like a database table). Typically one index per day: logs-2024.03.15
  • Shard: Subdivision of an index. Each shard is a Lucene instance. Primary + Replica shards
  • Replica: Copy of a primary shard on a different node for HA and read scaling
  • Mapping: Schema defining field types (text, keyword, date, integer). Dynamic mapping auto-detects
  • Inverted index: Core data structure - maps terms to documents containing them (enables fast full-text search)
  • ILM (Index Lifecycle Management): Automates rollover, shrink, delete based on age/size
logstash.conf - Pipeline Configuration Logstash 
input {
  beats {
    port => 5044
    ssl => true
    ssl_certificate => "/etc/logstash/certs/logstash.crt"
    ssl_key => "/etc/logstash/certs/logstash.key"
  }
}

filter {
  # Parse JSON logs
  if [message] =~ /^\{/ {
    json {
      source => "message"
      target => "app"
    }
  } else {
    # Parse unstructured logs with Grok
    grok {
      match => {
        "message" => "%{TIMESTAMP_ISO8601:timestamp} %{LOGLEVEL:level} %{GREEDYDATA:msg}"
      }
    }
  }

  # Add Kubernetes metadata enrichment
  if [kubernetes][namespace] {
    mutate {
      add_field => {
        "environment" => "%{[kubernetes][namespace]}"
        "service" => "%{[kubernetes][labels][app]}"
      }
    }
  }

  # Parse timestamps
  date {
    match => ["timestamp", "ISO8601", "yyyy-MM-dd HH:mm:ss"]
    target => "@timestamp"
  }

  # Remove verbose/unnecessary fields
  mutate {
    remove_field => ["agent", "ecs", "host", "input"]
  }

  # GeoIP enrichment for access logs
  if [client_ip] {
    geoip {
      source => "client_ip"
      target => "geo"
    }
  }
}

output {
  elasticsearch {
    hosts => ["https://es-cluster:9200"]
    index => "logs-%{[kubernetes][namespace]}-%{+YYYY.MM.dd}"
    user => "logstash_writer"
    password => "${ES_PASSWORD}"
    ssl => true
    cacert => "/etc/logstash/certs/ca.crt"
  }

  # Dead letter queue for failed events
  # (configured in logstash.yml: dead_letter_queue.enable: true)
}

5. Loki + Promtail + Grafana

Loki Architecture (Like Prometheus, But for Logs)
  +----------+     +----------+     +----------+
  | Promtail |     | Promtail |     | Promtail |     (DaemonSet on each node)
  | (Agent)  |     | (Agent)  |     | (Agent)  |     Tails log files, adds labels
  +----+-----+     +----+-----+     +----+-----+
       |                |                |
       +----------------+----------------+
                        |
                        v (Push logs with labels)
          +-------------+-------------+
          |            LOKI            |
          |  +---------------------+  |
          |  |    Distributor      |  |   Validates, routes to ingesters
          |  +----------+----------+  |
          |             |             |
          |  +----------v----------+  |
          |  |     Ingester        |  |   Builds chunks in memory
          |  +----------+----------+  |
          |             |             |
          |  +----------v----------+  |
          |  |   Chunk Storage     |  |   S3/GCS/filesystem
          |  |   (Only labels are  |  |
          |  |    indexed, NOT     |  |
          |  |    log content!)    |  |
          |  +---------------------+  |
          |  |   Index (BoltDB/    |  |
          |  |   TSDB)             |  |
          |  +---------------------+  |
          |  |      Querier        |  |   Executes LogQL queries
          |  +---------------------+  |
          +---------------------------+
                        |
                        v
          +-------------+-------------+
          |          GRAFANA           |
          |  - Explore (LogQL)         |
          |  - Log panels in           |
          |    dashboards              |
          |  - Correlate with          |
          |    Prometheus metrics      |
          +---------------------------+

  KEY INSIGHT: Loki only indexes LABELS (like Prometheus)
  not the full log content. This makes it:
  - Much cheaper to store (no full-text index)
  - Much cheaper to operate (less CPU/memory)
  - Slower for grep-style searches across all logs
  - Fast for targeted queries with known labels

LogQL Query Examples

LogQL Queries LogQL

6. ELK vs Loki Comparison

AspectELK/EFK (Elasticsearch)Loki + Grafana
IndexingFull-text inverted index (indexes ALL content)Only indexes labels (not log content)
Storage costHigh (index can be 1-2x raw data size)Low (compressed chunks, no content index)
Query speed (known fields)Fast (already indexed)Fast (label-based selection)
Query speed (grep/search)Fast (full-text search is native)Slower (must scan chunks with matching labels)
Resource usageHigh (JVM heap, CPU for indexing)Low (2-10x less memory/CPU)
ScalabilityComplex (shard management, split-brain)Simpler (microservices, object storage)
Operational complexityHigh (cluster management, JVM tuning)Low-Medium (especially with object storage)
Multi-tenancyIndex/alias basedNative tenant ID support
Alerting on logsElasticsearch Watcher / Kibana AlertsLoki Ruler (PromQL-style alerting on log metrics)
IntegrationElastic ecosystem (APM, SIEM)Grafana ecosystem (Prometheus, Tempo, Mimir)

Cost at scale (100TB/month)$$$$ (many data nodes + large disks)$$ (object storage is cheap)

When to Choose Which

  • Choose Loki when: Already using Grafana/Prometheus, cost-sensitive, K8s-native, logs are well-labeled, grep-style search is acceptable
  • Choose ELK when: Need full-text search (security/SIEM), complex log analytics, existing Elastic expertise, need to search unknown patterns across all logs

7. Log Collection in Kubernetes

DaemonSet Pattern

  • How: One collector pod per node, reads all container logs from host filesystem
  • Pros: Low resource overhead, single config for all pods, no app changes needed
  • Cons: Only captures stdout/stderr, shared config limits per-app customization
  • Use when: Standard logging setup, most applications write to stdout
  • Tools: Fluent Bit (lightweight), Fluentd, Promtail, Filebeat

Sidecar Pattern

  • How: Dedicated logging container in each pod, shares volume with app container
  • Pros: Per-app config, can read file-based logs (not just stdout), isolation
  • Cons: More resource usage (collector per pod), more config to manage
  • Use when: App writes to files (not stdout), needs custom parsing, multi-tenant isolation
  • Tools: Fluentd sidecar, custom collectors

8. Log Levels

LevelWhen to UseProduction BehaviorExample
DEBUG Detailed diagnostic info for developers OFF in production (too verbose, fills storage) Variable values, function entry/exit, SQL queries
INFO Normal operation milestones ON - primary production log level Server started, request processed, job completed
WARN Unexpected but recoverable situations ON - monitor for patterns Retry succeeded, deprecated API used, slow query
ERROR Operation failed but service continues ON - alert on rate increase Payment failed, DB connection lost, API timeout
FATAL Unrecoverable error, service will stop ON - immediate alert Cannot bind port, config file missing, OOM

Production Log Level Best Practices

  • Default to INFO: Production services should log at INFO level minimum
  • Dynamic log levels: Implement ability to change log level at runtime (for debugging without redeploy)
  • Rate-limit ERROR logs: If an error happens 1000x/sec, log it once with count, not 1000 times
  • Never log sensitive data: Passwords, tokens, PII regardless of level
  • Avoid logging in hot paths: Per-request logging should be INFO; avoid DEBUG in loops
  • Context over volume: One rich structured log entry beats 10 sparse ones

Trace ID Propagation

Cost Optimization Strategies

  • Drop debug logs in prod: Filter at collector level (save 50-80% volume)
  • Sample verbose logs: Keep 1/10 of health check logs, all error logs
  • Compress aggressively: Both Loki and ES support compression (zstd, deflate)
  • Reduce replicas for old data: Warm/cold tiers need 0 replicas if using snapshots for backup
  • Use object storage: S3 is 10-50x cheaper than SSD for archival
  • Index only needed fields: Disable indexing for message body in ES (use keyword for structured fields)
  • Monitor log volume: Alert on sum(bytes_rate({namespace="production"}[1h])) by (app) spikes
  • Set per-namespace retention: Dev logs: 3 days, staging: 7 days, production: 30-90 days

11. Interview Q&A

Q: When would you choose ELK over Loki (or vice versa)?
A: Choose ELK when: (1) Need powerful full-text search across unknown patterns (security/SIEM use cases), (2) Complex log analytics with aggregations and visualizations, (3) Team already has Elastic expertise, (4) Regulatory requirements mandate searchable audit logs. Choose Loki when: (1) Already invested in Grafana/Prometheus ecosystem, (2) Cost is a primary concern (Loki is 5-10x cheaper at scale), (3) Logs are well-structured with good labels, (4) K8s-native deployment preferred, (5) Acceptable to query by known labels rather than arbitrary text search. Key insight: Loki trades query flexibility for dramatic cost reduction by not indexing log content.
Q: What are the benefits of structured logging?
A: (1) Machine-parseable: No regex/grok needed - JSON fields are immediately queryable, (2) Consistent schema: Every log has same fields enabling aggregation (count errors by service), (3) Correlation: trace_id field enables linking logs to traces to metrics, (4) Reduced storage: Parsers don't need to run on every query (Loki can filter on parsed fields), (5) Better alerting: Can alert on specific field values (| json | latency_ms > 1000), (6) Developer experience: No need to write grok patterns - just access fields. Trade-off: slightly more work at application level (use a structured logging library).
Q: DaemonSet vs Sidecar for log collection - when would you use each?
A: DaemonSet (default choice): One collector per node tails /var/log/containers/. Pros: minimal resource overhead (one pod per node vs per application pod), simpler management, no app changes needed. Works when apps write to stdout/stderr. Sidecar: Collector container in each pod sharing a volume. Use when: (1) App writes logs to files not stdout (legacy apps), (2) Need per-app custom parsing that varies significantly between apps, (3) Multi-tenant environment where teams need isolated log configs, (4) Need to process logs before they leave the pod (sensitive data scrubbing). Rule: start with DaemonSet, add sidecars only for specific apps that need it.
Q: What log level strategy do you recommend for production?
A: Production default should be INFO. Strategy: (1) INFO for request lifecycle (started, completed), service events (connected, disconnected), business events (order placed), (2) WARN for recoverable issues (retry succeeded, circuit breaker opened, slow dependency), (3) ERROR for failed operations that need investigation (payment failed, data inconsistency), (4) Never DEBUG in production by default. Critical additions: implement dynamic log levels (change at runtime via config map or API for debugging), rate-limit repetitive errors, and always include context (request_id, user_id, trace_id) so a single log line is enough to begin investigation.
Q: How do you implement correlation IDs across microservices?
A: (1) Generate at ingress: API gateway/load balancer generates a unique trace_id (or use W3C TraceContext from client), (2) Propagate via headers: Every service extracts trace_id from incoming request headers and passes it to downstream calls, (3) Include in all logs: Structured logging middleware adds trace_id to every log statement automatically, (4) Include in metrics: Use Prometheus exemplars to link metric samples to trace_ids, (5) Implementation: Use OpenTelemetry SDK for automatic propagation, or middleware that extracts/injects traceparent header. This enables: searching all logs for one request across all services, linking a metric spike to the specific requests that caused it, viewing end-to-end trace timeline.
Q: How do you determine log retention strategy?
A: Consider: (1) Compliance: Regulatory requirements (PCI: 1 year, SOX: 7 years, GDPR: minimize). These override all other factors. (2) Cost vs value: Most debugging happens within 7 days. After 30 days, logs rarely accessed. Use tiered storage. (3) Environment-based: Dev: 3 days, Staging: 7 days, Production: 30-90 days hot, 1 year cold/archived. (4) Log level based: ERROR logs retained longer (90 days), INFO shorter (14 days), DEBUG never in prod. (5) Implementation: Elasticsearch ILM with hot/warm/cold tiers, Loki retention policies per tenant, S3 lifecycle rules for archived data. Always monitor storage costs and log volume growth.
Q: How would you scale Elasticsearch for a high-volume logging cluster?
A: (1) Separate node roles: Dedicated master (3, small), hot data (SSD, high CPU), warm data (HDD, large), coordinating (query routing). (2) Shard sizing: Target 30-50GB per shard, avoid too many shards (1000s per node causes overhead). Use ILM rollover by size. (3) Index per day + rollover: Don't keep growing one index. (4) Reduce replicas in warm/cold: 1 replica in hot, 0 in warm (use snapshots for durability). (5) Tune indexing: Increase refresh_interval (30s vs 1s), bulk indexing, reduce field mappings. (6) Memory: 50% of RAM to JVM heap (max 31GB), rest for filesystem cache. (7) Horizontal scaling: Add data nodes for throughput, use cross-cluster search for multi-cluster.
Q: A team reports they cannot find logs for their pods in Kibana. How do you troubleshoot?
A: Systematic approach: (1) Verify pod is logging: kubectl logs <pod> - confirm stdout has output. (2) Check log collector: Is Fluentd/Fluent Bit DaemonSet running on that node? Check its logs for errors. (3) Check node log files: /var/log/containers/ - are log files being written? (4) Check collector config: Is the namespace/label excluded by a filter? Check exclude directives. (5) Check pipeline: Are logs reaching Elasticsearch? Check Logstash/Fluentd output metrics for rejected/errored events. (6) Check ES index: Is the index pattern in Kibana matching the actual index name? (7) Check time range: Is Kibana query time range correct? Timezone mismatch? (8) Check mapping conflicts: A field type conflict can cause documents to be rejected.

12. Troubleshooting Scenarios

!! Scenario: Elasticsearch Cluster Red (Unassigned Shards)

Symptoms: Cluster health is RED, some indices unavailable, Kibana shows "shard allocation failed", new logs not being indexed.

1. Check cluster health: GET _cluster/health - shows number of unassigned shards
2. Identify unassigned shards: GET _cat/shards?v&h=index,shard,prirep,state,unassigned.reason&s=state
3. Check allocation explanation: GET _cluster/allocation/explain - tells WHY shard cannot be allocated
4. Common causes: (a) Disk full (>85% watermark blocks allocation), (b) Node left cluster, (c) Allocation filtering rules, (d) Too many shards per node
5. Disk full fix: Delete old indices, increase disk, or lower watermark: PUT _cluster/settings {"transient":{"cluster.routing.allocation.disk.watermark.high":"90%"}}
6. Node down fix: If node returns, shards auto-recover. If node permanently gone, reduce replica count or wait for timeout
7. Force allocate (last resort): POST _cluster/reroute {"commands":[{"allocate_stale_primary":{"index":"...","shard":0,"node":"...","accept_data_loss":true}}]}
8. Prevention: Monitor disk usage, set ILM policies, alert on elasticsearch_cluster_health_status{color="red"}
!! Scenario: Fluentd Losing Logs (Backpressure)

Symptoms: Gaps in logs, Fluentd logs show "buffer overflow" or "chunk bytes limit exceeded", memory usage growing.

1. Check Fluentd logs: kubectl logs -n logging daemonset/fluentd | grep -i "overflow\|error\|retry"
2. Identify bottleneck: Is the output (ES/Loki) slow to ingest? Check output latency metrics
3. Check buffer status: Fluentd exposes /api/plugins.json - check buffer queue length and retry count
4. Immediate fix: Increase buffer limits: chunk_limit_size 16MB, queue_limit_length 64
5. Use file buffer: Switch from memory buffer to file buffer: @type file with path /var/log/fluentd-buffers/
6. Set overflow_action: overflow_action block (blocks input, preserves logs) vs drop_oldest_chunk (loses old data)
7. Scale output: Increase flush_thread_count, add more ES data nodes, or reduce log volume at source
8. Long-term: Add aggregation layer (Fluent Bit -> Fluentd aggregator -> ES), filter unnecessary logs earlier
!! Scenario: Loki Query Timeout

Symptoms: Grafana shows "query timeout" or "context deadline exceeded" when searching logs in Loki.

1. Check query: Broad queries without label selectors scan too much data. {job=~".+"} |= "error" scans EVERYTHING
2. Fix query: Always start with specific labels: {namespace="production", app="my-service"} |= "error"
3. Reduce time range: Query last 1 hour instead of 7 days. Loki scans linearly through chunks
4. Check cardinality: Too many label values (like pod names) creates too many streams. Check /loki/api/v1/series
5. Increase limits: query_timeout: 5m in Loki config, max_query_length: 720h
6. Check querier resources: Loki queriers need CPU/memory to decompress and scan chunks. Scale horizontally
7. Long-term: Add better labels at ingestion, use recording rules for frequently-needed log metrics, consider bloom filters (Loki 3.0+)
!! Scenario: Disk Full From Logs

Symptoms: Node disk at 100%, pods evicted with "DiskPressure", kubelet taint applied, new pods cannot schedule.

1. Identify largest log files: du -sh /var/log/containers/* | sort -rh | head -20
2. Find the culprit pod: File name format is <pod>_<namespace>_<container>-<id>.log
3. Immediate fix: Truncate the log file (not delete - container runtime still has handle): truncate -s 0 /var/log/containers/culprit-pod*.log
4. Check kubelet config: containerLogMaxSize (default 10Mi) and containerLogMaxFiles (default 5). If not set, logs grow unbounded
5. Fix the application: Investigate why it is logging so much. Stack trace in a loop? Debug logging enabled? Infinite retry with logging?
6. Prevention: Set kubelet log rotation, use resource quotas with ephemeral storage limits, alert on disk usage > 70%
7. Add ephemeral storage limit to pod spec: resources.limits.ephemeral-storage: 1Gi (evicts pod if exceeded)
