Microservices Architecture

Design patterns, decomposition strategies, inter-service communication, resilience patterns, and the 12-factor app methodology for cloud-native applications.

1. Monolith vs Microservices

Aspect Monolith Microservices
Deployment Single unit, all-or-nothing Independent per service
Scaling Scale entire application Scale individual services
Technology Single stack Polyglot (multiple languages/frameworks)
Data Shared database Database per service
Team Large team, shared codebase Small autonomous teams per service
Complexity Simple at start, grows exponential Higher initial complexity, manageable growth
Failure Impact Entire application affected Isolated to single service (if resilient)
Testing Easier end-to-end testing Requires contract + integration testing
Latency In-process calls (fast) Network calls (higher latency)
DevOps Maturity Low bar Requires CI/CD, monitoring, orchestration

When to Choose Each

Choose Monolith When

  • Small team (fewer than 10 developers)
  • Early-stage startup / MVP
  • Simple domain with few bounded contexts
  • Limited DevOps expertise
  • Low traffic with predictable scale

Choose Microservices When

  • Large teams needing autonomous delivery
  • Different parts scale independently
  • Multiple technology stacks needed
  • High availability requirements
  • Rapid, independent deployment cycles

2. Microservice Design Principles

1
Single Responsibility Principle

Each service does one thing well. A service should be small enough for a single team to own (two-pizza team rule). If you cannot describe its purpose in one sentence, it is too large.

2
Bounded Contexts (DDD)

Domain-Driven Design bounded contexts define clear boundaries around a domain model. Each microservice owns exactly one bounded context. A "Customer" in the billing context may differ from a "Customer" in the shipping context -- this is expected.

3
API-First Design

Define contracts (OpenAPI, protobuf) before implementation. Consumers and producers agree on the interface, enabling parallel development. APIs are versioned and backward-compatible.

4
Database Per Service

Each service owns its data store. No shared databases. Other services access data only through the service API. This enables polyglot persistence -- choose the best database for each use case.

5
Loose Coupling / High Cohesion

Services should be independently deployable with minimal dependencies. Changes to one service should not require changes to others. Communicate through well-defined APIs and events, not shared libraries or databases.

4. API Gateway

What an API Gateway Does

  • Request Routing -- routes incoming requests to appropriate backend services
  • Authentication/Authorization -- validates tokens, API keys before requests reach services
  • Rate Limiting -- throttles requests to prevent abuse
  • Request/Response Transformation -- modifies headers, body format, protocol translation
  • Load Balancing -- distributes traffic across service instances
  • Caching -- caches responses to reduce backend load
  • Observability -- centralized logging, metrics, tracing
  • Circuit Breaking -- prevents cascade failures to unhealthy backends

5. Service Discovery

Client-Side Discovery

Client queries a service registry to find available instances, then load-balances directly.

  • Client knows all instances
  • Client-side load balancing (round-robin, random)
  • Example: Netflix Eureka + Ribbon
  • Pro: no single point of failure for routing
  • Con: coupling discovery logic into every client

Server-Side Discovery

Client sends request to a load balancer/router which queries the registry and forwards the request.

  • Client only knows the router endpoint
  • Router handles discovery + load balancing
  • Example: AWS ALB, Kubernetes Services
  • Pro: simpler clients, centralized logic
  • Con: additional network hop, single point of failure

Kubernetes Native Service Discovery

Kubernetes provides built-in service discovery via DNS and Service objects:

  • my-service.my-namespace.svc.cluster.local resolves to ClusterIP
  • CoreDNS watches the Kubernetes API for Service/Endpoint changes
  • kube-proxy programs iptables/IPVS rules for load balancing
  • No external registry needed (etcd backs everything)
  • Headless Services (clusterIP: None) return pod IPs directly for client-side balancing
service-discovery.yamlK8s
# Standard Service (server-side discovery via kube-proxy)

  name: order-service
  namespace: production
spec:
  selector:
    app: order-service
  ports:
    - port: 80
      targetPort: 8080

# Headless Service (client-side discovery, returns pod IPs)
apiVersion: v1

  name: order-service-headless

  clusterIP: None

6. Data Patterns

Database Per Service

Each microservice owns its data exclusively. Benefits:

  • Services can choose the best database technology (polyglot persistence)
  • Schema changes do not affect other services
  • Independent scaling of data tier
  • Clear ownership and access control

Challenge: Cross-service queries require API composition or CQRS pattern.

Saga Pattern

A saga is a sequence of local transactions where each step publishes an event/command to trigger the next. If a step fails, compensating transactions undo preceding steps.

Saga: Choreography (Event-Driven)
 +----------+    OrderCreated    +-----------+    PaymentProcessed    +-----------+
 |  Order   | -----------------> |  Payment  | ----------------------> | Inventory |
 |  Service |                    |  Service  |                        |  Service  |
 +----+-----+                    +-----+-----+                        +-----+-----+
      ^                                |                                    |
      |         PaymentFailed          |         InventoryReserved          |
      +--------------------------------+                                    |
      |                                                                     |
      +---------------------------------------------------------------------+
                              OrderCompleted

  Each service listens to events and decides what to do next.
  No central coordinator. Services are independent.
  Compensation: if Payment fails, Order publishes OrderCancelled.
Saga: Orchestration (Command-Driven)
                    +------------------+
                    |  Saga            |
                    |  Orchestrator    |
                    +--+------+----+--+
                       |      |    |
            Command    |      |    |    Command
          +------------+      |    +-----------+
          |                   |                |
          v                   v                v
  +-------+----+    +---------+--+    +--------+---+

  Orchestrator sends commands and listens for replies.
  Central coordination. Easier to understand flow.

  Read side: optimized for queries, denormalized, eventually consistent
  Sync via domain events or change data capture (CDC)

Event Sourcing

Instead of storing current state, store a sequence of events that led to current state. The current state is derived by replaying events.

  • Event Store: Append-only log of all domain events
  • Rebuild state: Replay events from beginning (or snapshot + events)
  • Audit trail: Complete history of every change
  • Temporal queries: "What was the account balance on March 1st?"
  • Tools: EventStoreDB, Apache Kafka (as event log), Axon Framework

Eventual Consistency

In distributed systems, strong consistency across services is expensive (2PC) and fragile. Eventual consistency accepts that:

  • Data will be consistent eventually (milliseconds to seconds typically)
  • Reads may return stale data briefly
  • Use idempotent operations to handle duplicate messages
  • Design UIs to handle intermediate states gracefully
  • CAP Theorem: in a partition, choose Availability (AP) or Consistency (CP)

7. Decomposition Strategies

1
Strangler Fig Pattern

Incrementally migrate from monolith to microservices. Route new functionality to new services while gradually extracting existing functionality. The monolith "shrinks" over time until it disappears.

  • Use an API gateway/proxy to route between monolith and new services
  • Migrate one bounded context at a time
  • Lowest risk approach -- can stop at any point
  • Named after strangler fig trees that grow around host trees
2
Domain-Driven Decomposition

Use DDD strategic patterns to identify bounded contexts. Each bounded context becomes a microservice. Conduct Event Storming workshops to discover domains, aggregates, and context boundaries.

3
By Business Capability

Align services to business capabilities: Order Management, Inventory, Payments, Shipping, Notifications. Each capability is relatively stable even as implementation changes.

4
By Subdomain

DDD identifies core, supporting, and generic subdomains. Core subdomains (competitive advantage) get the most investment. Generic subdomains (authentication, email) can use off-the-shelf solutions.

8. Resilience Patterns

Circuit Breaker

Prevents cascading failures by stopping calls to a failing service. Three states:

  • Closed: Requests flow normally. Failures are counted.
  • Open: All requests fail immediately (fast fail). No calls to downstream.
  • Half-Open: Limited requests allowed to test if service recovered.

Tools: Resilience4j (Java), Polly (.NET), Hystrix (deprecated), Istio (mesh-level)

Retry with Exponential Backoff

Automatically retry failed requests with increasing delays to handle transient failures.

  • Retry 1: Wait 100ms
  • Retry 2: Wait 200ms
  • Retry 3: Wait 400ms
  • Retry 4: Wait 800ms + jitter (random 0-100ms)

Jitter prevents thundering herd when many clients retry simultaneously.

Max retries must be bounded to prevent infinite loops.

Timeout

Every external call must have a timeout. Without timeouts, threads/connections can be held indefinitely by slow/dead services.

  • Connection timeout: time to establish TCP connection (short: 1-5s)
  • Read timeout: time to receive response (depends on operation)
  • Cascading timeouts: upstream timeout > downstream timeout

Chaos Testing

Deliberately inject failures to test system resilience:

  • Pod kill: Randomly terminate pods to test restart/failover
  • Network partition: Block traffic between services
  • Latency injection: Add artificial delays
  • Resource exhaustion: CPU/memory stress tests
  • DNS failure: Simulate service discovery failures

11. Interview Q&A

Q1: What are the key trade-offs between monolith and microservices?

Monolith advantages: Simpler development, easier testing (in-process), single deployment, lower operational overhead, better for small teams.

Microservices advantages: Independent scaling, technology freedom, fault isolation, team autonomy, faster deployment cycles.

Microservices costs: Network complexity, distributed tracing needed, data consistency challenges (no ACID across services), operational overhead (CI/CD per service, monitoring), debugging distributed systems is harder.

The decision depends on team size, domain complexity, and operational maturity. Start monolithic and decompose when pain points emerge (Conway's Law applies).

Q2: Explain the Saga pattern and when you would use it.

A Saga manages distributed transactions across multiple services without 2PC (two-phase commit). It consists of a sequence of local transactions, each publishing events/commands to trigger the next step. If any step fails, compensating transactions undo the previous steps.

Two approaches:

  • Choreography: Services listen to events and react independently. Good for simple flows (2-4 steps). Harder to trace.
  • Orchestration: A central orchestrator coordinates the saga steps. Better for complex flows. Easier to understand but introduces a coordinator dependency.

Example: E-commerce order: Create Order -> Reserve Inventory -> Process Payment -> Confirm Shipment. If payment fails: Release Inventory -> Cancel Order.

Q3: How does service discovery work in Kubernetes?

Kubernetes provides built-in server-side service discovery through:

  • DNS: CoreDNS resolves service names to ClusterIPs. Format: service-name.namespace.svc.cluster.local
  • Service objects: Abstract away pod IPs. kube-proxy programs iptables/IPVS rules to load-balance across healthy pods.
  • Endpoints: Automatically updated as pods start/stop (based on readiness probes).
  • Environment variables: Kubernetes injects service addresses as env vars into pods (legacy approach).

For client-side discovery: use Headless Services (clusterIP: None) which return all pod IPs, letting the client decide which pod to call.

Q4: How do you handle distributed transactions across microservices?

Avoid distributed transactions (2PC) when possible. Instead:

  • Saga pattern: Sequence of local transactions with compensating actions
  • Eventual consistency: Accept that data will be consistent eventually, not immediately
  • Outbox pattern: Write events to an outbox table in the same local transaction, then publish asynchronously (guarantees at-least-once delivery)
  • Idempotent operations: Design all operations to be safely retried
  • Event sourcing: Derive state from events, enabling replay and recovery

The key principle: design for failure. Every network call can fail, so each step must be reversible or idempotent.

Q5: How would you approach decomposing a monolith into microservices?

Step-by-step approach:

  1. Identify bounded contexts: Use Event Storming or domain analysis to find natural boundaries
  2. Prioritize extraction: Start with services that change frequently, need independent scaling, or have clear domain boundaries
  3. Apply Strangler Fig: Route traffic through a proxy; gradually redirect to new services
  4. Extract data: Move from shared DB to database-per-service (hardest step)
  5. Add anti-corruption layer: Translate between old and new models
  6. Iterate: Extract one service at a time, validate, then continue

Do NOT: Rewrite everything at once ("big bang"). This almost always fails.

Q6: What are the challenges of database-per-service?
  • Cross-service queries: Cannot JOIN across databases. Must use API Composition or CQRS read models.
  • Data consistency: No ACID transactions across services. Must use sagas + eventual consistency.
  • Data duplication: Some data may be duplicated across services (denormalization).
  • Reporting: Centralized reporting requires data aggregation (ETL, event streaming to data warehouse).
  • Migration complexity: Splitting a shared database into per-service databases requires careful data migration and dual-write period.
Q7: Explain the three states of a circuit breaker.
  • Closed (normal): Requests pass through. A failure counter tracks errors. If failures exceed a threshold (e.g., 5 failures in 10 seconds), transitions to Open.
  • Open (blocking): All requests fail immediately without calling the downstream service. Returns a fallback response or error. After a timeout period (e.g., 30 seconds), transitions to Half-Open.
  • Half-Open (testing): Allows a limited number of test requests through. If they succeed, transitions back to Closed. If they fail, transitions back to Open.

This prevents wasting resources on a service that is clearly down and gives it time to recover without being overwhelmed by retries.

Q8: What is your API versioning strategy for microservices?

Strategies (pick one):

  • URL path: /api/v1/users, /api/v2/users -- simplest, most visible
  • Header: Accept: application/vnd.api+json;version=2 -- cleaner URLs
  • Query param: /api/users?version=2 -- easy but less standard

Best practices:

  • Prefer backward-compatible (additive) changes -- adding fields is safe
  • Only create new versions for breaking changes
  • Support N-1 version minimum (deprecation period)
  • Use semantic versioning for internal APIs
  • Document breaking changes and provide migration guides
Q9: What is your testing strategy for microservices?

Follow the testing pyramid:

  • Many unit tests: Fast, isolated, test business logic
  • Integration tests: Test with real dependencies using Testcontainers (Docker)
  • Contract tests (Pact): Verify API contracts between services independently -- prevents integration surprises
  • Few E2E tests: Critical happy paths only (expensive, flaky)
  • Chaos testing: Production-like environments -- kill pods, inject latency, partition networks

Key principles: Each service has its own test pipeline. Contract tests replace the need for running all services together. Test in production with feature flags and canary deployments.

Q10: How do you handle eventual consistency from a user experience perspective?

Strategies:

  • Optimistic UI: Show success immediately, reconcile if operation fails later
  • Status polling: Return 202 Accepted with a status URL. Client polls until complete.
  • WebSocket/SSE: Push notifications when async operation completes
  • Read-your-writes: After a write, route subsequent reads to the primary (not replica) for that session
  • Compensating UX: If a saga fails, notify the user and offer alternatives (e.g., "Payment failed, try another card")
  • Idempotency keys: Allow safe retries without double-processing

12. Scenarios

⚠ Scenario 1: Distributed Transaction Failure

Situation: An order is placed. Order Service creates the order, Payment Service charges the card successfully, but Inventory Service fails to reserve stock (item out of stock).

1. Inventory Service publishes InventoryReservationFailed event
2. Payment Service listens for this event and initiates a refund (compensating transaction)
3. Order Service listens for PaymentRefunded and updates order status to CANCELLED
4. Notification Service sends email to customer: "Your order could not be fulfilled"
5. All compensating transactions are idempotent -- safe to retry if any step fails

Key design decisions:

  • Use outbox pattern to guarantee event publishing (same DB transaction)
  • Dead letter queue for events that repeatedly fail processing
  • Correlation ID tracks the entire saga across services
  • Monitoring alerts if compensation takes too long
⚠ Scenario 2: Cascading Service Failure

Situation: The Recommendation Service becomes slow (high latency). Product Service calls Recommendation Service synchronously. All Product Service threads become blocked waiting. Product Service stops responding. Frontend (which calls Product Service) also times out.

1. Immediate: Circuit breaker on Product Service trips after N failures to Recommendation Service
2. Fallback: Return products without recommendations (degraded but functional)
3. Bulkhead: Separate thread pool for Recommendation calls so other Product Service endpoints are unaffected
4. Timeout: Set aggressive timeout (500ms) for Recommendation calls -- better to skip than block
5. Alerting: Circuit breaker state change triggers PagerDuty alert to Recommendation team

Prevention:

  • All synchronous calls MUST have timeouts
  • Consider making non-critical calls asynchronous
  • Load shedding: reject excess requests early (HTTP 503)
  • Rate limiting protects downstream services
⚠ Scenario 3: Data Consistency Across Services

Situation: A user updates their email address. User Service, Order Service, and Notification Service all store user email. How do you keep them consistent?

1. User Service is the source of truth for user data
2. User Service publishes UserEmailUpdated event to message broker (Kafka)
3. Order Service and Notification Service consume the event and update their local copies
4. If a consumer is down, the message stays in the queue until processed (at-least-once delivery)
5. Consumers are idempotent -- processing the same event twice has no adverse effect

Design principles:

  • Single writer principle: only one service writes a piece of data
  • Other services maintain read-only copies (materialized views)
  • Accept brief inconsistency window (seconds typically)
  • Use Change Data Capture (CDC) with Debezium as alternative to application-level events
⚠ Scenario 4: Migrating from Monolith to Microservices

Situation: Your team has a large Django monolith. Deployments take 2 hours, the test suite takes 45 minutes, and teams step on each other's code. Leadership wants to move to microservices.

1. Do NOT rewrite from scratch. Use Strangler Fig pattern.
2. Place an API Gateway (Nginx/Kong) in front of the monolith. Route ALL traffic through it.
3. Identify the first bounded context to extract (choose one with clear boundaries and high change frequency).
4. Build the new service alongside the monolith. Dual-write data during migration period.
5. Update API Gateway routing to send traffic to new service. Keep monolith as fallback.
6. Validate data consistency. Remove old code from monolith once stable (weeks/months).
7. Repeat for next bounded context. Each extraction learns from the previous.

Timeline reality: A large monolith migration takes 1-3 years for a team of ~20 engineers. Set incremental milestones and demonstrate value early.
