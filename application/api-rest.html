3. REST Principles

1
Stateless

Each request contains ALL information needed to process it. No server-side session state. Authentication via tokens (JWT, API keys) sent with every request. Enables horizontal scaling -- any server can handle any request.

2
Resource-Based URLs

URLs represent resources (nouns), not actions (verbs). HTTP methods express the action.

  • GET /users -- list users (not GET /getUsers)
  • POST /users -- create user (not POST /createUser)
  • GET /users/123/orders -- list orders for user 123
3
HATEOAS (Hypermedia As The Engine Of Application State)

Responses include links to related resources and available actions. Clients discover APIs by following links rather than hardcoding URLs.

HATEOAS Response Example
{
  "id": "123",
  "name": "John Doe",
  "status": "active",
  "_links": {
    "self": { "href": "/users/123" },
    "orders": { "href": "/users/123/orders" },
    "deactivate": { "href": "/users/123/deactivate", "method": "POST" }
  }
}
4
Richardson Maturity Model
LevelNameDescription
0Swamp of POXSingle endpoint, single HTTP method (e.g., SOAP over POST)
1ResourcesMultiple endpoints representing resources
2HTTP VerbsProper use of GET, POST, PUT, DELETE + status codes
3Hypermedia (HATEOAS)Responses include links for discoverability

Most production APIs are Level 2. Level 3 is aspirational but rarely fully implemented.

4. API Design Best Practices

Naming Conventions

  • Use plural nouns for collections: /users, /orders
  • Use kebab-case for multi-word: /order-items
  • Nest for relationships: /users/123/orders (max 2 levels deep)
  • Use query params for filtering: /orders?status=pending&created_after=2026-01-01
  • Avoid verbs in URLs: POST /orders not POST /createOrder
  • Exception for actions: POST /orders/123/cancel (non-CRUD operations)

Versioning Strategies

Strategy Example Pros Cons
URL Path /api/v1/users Clear, easy to route, cacheable URL pollution, hard to deprecate
Header Accept: application/vnd.api.v2+json Clean URLs, content negotiation Hidden from URL, harder to test in browser
Query Param /users?version=2 Simple, optional Non-standard, caching issues

Pagination

Offset-based Pagination
GET /api/v1/users?page=2&per_page=25

Response:
{
  "data": [...],
  "pagination": {
    "page": 2,
    "per_page": 25,
    "total": 150,
    "total_pages": 6,
    "next": "/api/v1/users?page=3&per_page=25",
    "prev": "/api/v1/users?page=1&per_page=25"
  }
}
Cursor-based Pagination (preferred for large datasets)
GET /api/v1/users?limit=25&cursor=eyJpZCI6MTAwfQ==

Response:
{
  "data": [...],
  "pagination": {
    "next_cursor": "eyJpZCI6MTI1fQ==",
    "has_more": true
  }
}

Cursor-based is preferred for large or frequently-changing datasets because it avoids the "missing items" problem of offset pagination when records are inserted/deleted between pages.

JWT Structure

JWT: header.payload.signatureJWT
eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9.    <-- Header (Base64)
eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6Ik... <-- Payload (Base64)

--- DECODED ---

Header:
{
  "alg": "RS256",     // Algorithm: RSA + SHA-256
  "typ": "JWT",       // Token type
  "kid": "key-id-1"   // Key ID for rotation
}

Payload (Claims):
{
  "sub": "user-123",         // Subject (user ID)
  "iss": "auth.example.com", // Issuer
  "aud": "api.example.com",  // Audience
  "exp": 1717200000,         // Expiration (Unix timestamp)
  "iat": 1717196400,         // Issued at
  "scope": "read:users write:orders",  // Permissions
  "roles": ["admin"]
}

Signature:
  RSASHA256(base64(header) + "." + base64(payload), private_key)

Token Refresh Flow

  • Access Token: Short-lived (15 min to 1 hour). Sent with every API request.
  • Refresh Token: Long-lived (days to weeks). Used only to get new access tokens.
  • When access token expires, client sends refresh token to auth server for a new access token.
  • Refresh tokens should be stored securely (HTTP-only cookie, secure storage).
  • Refresh token rotation: Issue new refresh token with each use; invalidate old one (prevents replay).

 Advantages:
 - Allows bursts up to bucket capacity
 - Smooth rate over time
 - Simple to implement (Redis INCR + EXPIRE)
Algorithm How It Works Best For
Token Bucket Fixed capacity, refills at constant rate Allowing bursts within limit
Leaky Bucket Requests processed at fixed rate, excess queued/dropped Smoothing traffic
Fixed Window Count requests per time window (e.g., 100/min) Simple implementation
Sliding Window Weighted count across current and previous window Avoiding boundary spikes

CORS (Cross-Origin Resource Sharing)

Browsers enforce Same-Origin Policy. CORS headers allow controlled cross-origin access.

CORS Response Headers
Access-Control-Allow-Origin: https://app.example.com
Access-Control-Allow-Methods: GET, POST, PUT, DELETE
Access-Control-Allow-Headers: Authorization, Content-Type
Access-Control-Max-Age: 86400
Access-Control-Allow-Credentials: true

Preflight request: Browser sends OPTIONS request first for non-simple requests (custom headers, PUT/DELETE). Server responds with allowed methods/headers.

Security: Never use Access-Control-Allow-Origin: * with credentials. Allowlist specific origins.

Input Validation

  • Validate ALL input on the server (never trust client-side validation)
  • Validate data types, ranges, lengths, and patterns
  • Sanitize against injection (SQL, NoSQL, LDAP, OS command)
  • Use parameterized queries (never string concatenation for SQL)
  • Limit request body size (e.g., 1MB max)
  • Validate Content-Type header matches actual body

7. gRPC

Protocol Buffers (.proto)

user_service.protoprotobuf
syntax = "proto3";

package user;

option go_package = "github.com/example/user/pb";

// Service definition
service UserService {
  // Unary RPC
  rpc GetUser (GetUserRequest) returns (User);

  // Server streaming
  rpc ListUsers (ListUsersRequest) returns (stream User);

  // Client streaming
  rpc UploadUsers (stream User) returns (UploadResponse);

  // Bidirectional streaming
  rpc Chat (stream ChatMessage) returns (stream ChatMessage);
}

message GetUserRequest {
  string id = 1;
}

message ListUsersRequest {
  int32 page_size = 1;
  string page_token = 2;
  string filter = 3;  // e.g., "status=active"
}

message User {
  string id = 1;
  string name = 2;
  string email = 3;
  UserRole role = 4;
  google.protobuf.Timestamp created_at = 5;
}

enum UserRole {
  USER_ROLE_UNSPECIFIED = 0;
  USER_ROLE_ADMIN = 1;
  USER_ROLE_MEMBER = 2;
}

message UploadResponse {
  int32 count = 1;
}

message ChatMessage {
  string sender = 1;
  string content = 2;
}

10. Interview Q&A

Q1: What are the core principles of REST?

Six constraints of REST (Roy Fielding, 2000):

  1. Client-Server: Separation of concerns between UI and data storage
  2. Stateless: Each request contains all necessary information (no server-side sessions)
  3. Cacheable: Responses must indicate if they are cacheable or not
  4. Uniform Interface: Resource identification via URLs, manipulation through representations, self-descriptive messages, HATEOAS
  5. Layered System: Client cannot tell if connected directly to server or through intermediary
  6. Code on Demand (optional): Server can extend client functionality by transferring code
Q2: Explain idempotency in HTTP methods. Why does it matter?

Idempotent means making the same request N times produces the same result as making it once.

  • GET: Idempotent -- reading data N times does not change it
  • PUT: Idempotent -- replacing a resource with the same data N times gives same result
  • DELETE: Idempotent -- deleting something already deleted is still "deleted"
  • POST: NOT idempotent -- creating a resource N times creates N resources
  • PATCH: Depends on implementation (JSON Merge Patch is idempotent; "increment" operations are not)

Why it matters: In distributed systems, network failures cause retries. If an operation is idempotent, retries are safe. This is why payment APIs use idempotency keys -- submitting the same payment request twice should only charge once.

Q3: What is the difference between PUT and PATCH?
  • PUT: Replaces the ENTIRE resource. Client must send the complete representation. Missing fields are set to null/default. Idempotent.
  • PATCH: Partial update. Client sends only the fields to change. Other fields remain unchanged. May or may not be idempotent.

Example: User has name, email, phone. To update just email:

  • PUT /users/123 -- must send {name, email, phone} (otherwise name and phone become null)
  • PATCH /users/123 -- send only {email: "new@email.com"} (name and phone preserved)

When to use PUT: Replace entire configuration, upload a file. When to use PATCH: Update a single field on a large resource.

Q4: What status code would you return for these scenarios?
  • User tries to create an account with an existing email: 409 Conflict
  • Request body JSON is malformed: 400 Bad Request
  • Valid JSON but email field is invalid: 422 Unprocessable Entity
  • User authenticated but tries to access admin endpoint: 403 Forbidden
  • Request accepted for background processing: 202 Accepted
  • Successful DELETE with no response body: 204 No Content
  • Rate limit exceeded: 429 Too Many Requests
  • Server's database is down: 503 Service Unavailable
Q5: Walk through the OAuth 2.0 Authorization Code flow.
  1. User clicks "Login with Google" -- app redirects browser to Google's auth server with client_id, redirect_uri, scope, state, and code_challenge (PKCE)
  2. User authenticates with Google and grants consent for requested scopes
  3. Google redirects back to app's redirect_uri with an authorization code (short-lived, single-use)
  4. App's backend exchanges the code + client_secret + code_verifier for tokens (access_token + refresh_token)
  5. App uses access_token to call APIs on behalf of the user
  6. When access_token expires, app uses refresh_token to get a new one without user interaction

PKCE (Proof Key for Code Exchange): Prevents authorization code interception attacks. Required for public clients (SPAs, mobile apps). Recommended for all clients.

Q6: What are the pros and cons of JWT?

Pros:

  • Stateless -- no database lookup needed to validate (verify signature locally)
  • Self-contained -- carries user info and permissions in the payload
  • Scalable -- any server with the public key can validate
  • Cross-domain -- works across microservices without shared session store

Cons:

  • Cannot be revoked individually (until expiration) without a denylist
  • Payload is only Base64-encoded, NOT encrypted (do not store secrets in it)
  • Token size grows with claims (larger than session cookies)
  • Clock skew between servers can cause validation issues
  • If signing key is compromised, ALL tokens are compromised

Mitigations: Short expiry (15 min), refresh tokens, token denylist for critical revocations, use RS256 (asymmetric) so only auth server has private key.

Q7: Explain the common rate limiting algorithms.
  • Token Bucket: Bucket holds N tokens, refills at rate R/sec. Each request takes a token. Allows bursts up to N. Most common (AWS, Stripe).
  • Leaky Bucket: Requests enter a queue, processed at constant rate. Excess dropped. Smooths traffic but adds latency.
  • Fixed Window: Counter resets every window (e.g., 100 requests per minute). Simple but allows 2x burst at window boundary.
  • Sliding Window Log: Store timestamp of each request. Count requests in last N seconds. Accurate but memory-intensive.
  • Sliding Window Counter: Hybrid: weighted average of current and previous window. Good balance of accuracy and efficiency.

Implementation: Typically Redis-based (INCR + TTL for fixed window, sorted sets for sliding window). Return X-RateLimit-Remaining and Retry-After headers.

Q8: When would you choose gRPC over REST?

Choose gRPC when:

  • Internal service-to-service communication (not public-facing)
  • High performance requirements (binary serialization is ~10x faster)
  • Streaming needed (real-time data, large datasets)
  • Strong typing is important (proto files enforce schema)
  • Polyglot environment (protoc generates clients in any language)

Choose REST when:

  • Public-facing API (browser support, curl-friendly)
  • Simple CRUD operations
  • HTTP caching is important (CDN, browser cache)
  • Team is familiar with REST and JSON
  • Debugging ease (human-readable payloads)

Common pattern: gRPC for internal services, REST/GraphQL for BFF (Backend for Frontend).

Q9: How would you version a REST API?

Recommended approach: URL path versioning (/api/v1/) for simplicity and visibility.

Strategy:

  • Avoid breaking changes whenever possible (additive changes are backward-compatible)
  • Adding new fields, new endpoints, new optional parameters = non-breaking
  • Removing/renaming fields, changing types, changing behavior = breaking
  • When breaking change is unavoidable: increment version, support v(N-1) for deprecation period (6-12 months)
  • Use Sunset header and Deprecation header to signal upcoming removal
  • Provide migration guide and changelog
Q10: Explain CORS and why it exists.

CORS (Cross-Origin Resource Sharing) is a browser security mechanism that restricts web pages from making requests to a different origin (domain, port, or protocol) than the one serving the page.

Why it exists: Without CORS, a malicious site could make authenticated requests to your bank's API using your cookies (CSRF attack). The Same-Origin Policy prevents this by default.

How it works:

  • Simple requests (GET, POST with standard headers): Browser adds Origin header; server responds with Access-Control-Allow-Origin
  • Preflight requests (PUT, DELETE, custom headers): Browser sends OPTIONS request first to check if the actual request is allowed
  • Server responds with allowed origins, methods, and headers
  • Access-Control-Max-Age caches preflight results to reduce OPTIONS requests

Common mistake: Setting Access-Control-Allow-Origin: * -- this disables credentials and may expose your API. Always allowlist specific trusted origins.

11. Scenarios

⚠ Scenario 1: 401 vs 403 -- When to Return Which

Situation: Your API has public endpoints, authenticated endpoints, and admin-only endpoints. When should you return 401 vs 403?

401 Unauthorized -- "I do not know who you are"
  • No Authorization header provided
  • Token is expired or malformed
  • API key is invalid
  • Include WWW-Authenticate header to indicate expected auth scheme
403 Forbidden -- "I know who you are, but you cannot do this"
  • Valid token, but user lacks required role/permission
  • Regular user accessing admin endpoint
  • User trying to access another user's private data
  • IP not in allowlist

Security note: Some APIs return 404 instead of 403 to avoid revealing that a resource exists (information leakage). Example: GET /users/123/secret-document -- if the document exists but user has no access, returning 403 confirms its existence. Returning 404 hides this.

⚠ Scenario 2: Designing a Paginated API

Situation: You need to build an API endpoint that returns a list of orders. The table has 10 million rows. How do you design it?

1. Choose cursor-based pagination for large datasets (offset-based breaks at scale -- scanning millions of rows for OFFSET 1000000 is slow)
2. Encode cursor as opaque string (Base64 of last seen ID + sort key)
3. Use WHERE id > :last_id ORDER BY id LIMIT :page_size (uses index, constant performance)
4. Return has_more: true/false and next_cursor in response
5. Set maximum page_size (e.g., 100) to prevent abuse
6. Add filtering (?status=pending&created_after=2026-01-01) with indexed columns
7. Include total count only if explicitly requested (?include_count=true) as COUNT(*) is expensive

Trade-off: Cursor-based cannot "jump to page 5" -- only forward/backward navigation. If random page access is required, use keyset pagination with offset as fallback for small datasets.

⚠ Scenario 3: Handling Breaking Changes

Situation: You need to rename a field from userName to display_name in your public API response. Thousands of clients depend on the current format.

1. Add new field display_name alongside old userName (non-breaking)
2. Add Deprecation: true header and document the change
3. Notify API consumers via changelog, email, dashboard warning
4. Set a sunset date (e.g., 6 months). Add Sunset: Sat, 01 Nov 2026 00:00:00 GMT header
5. Monitor usage of deprecated field (log when clients still read it)
6. After sunset date: stop returning userName (or create v2 if many breaking changes)

Principles:

  • Never remove without deprecation period
  • Additive changes first (return both fields during transition)
  • Communicate early and repeatedly
  • Provide SDKs/migration tools if change is complex
⚠ Scenario 4: API Gateway Throttling

Situation: Your API gateway is returning 429 errors to legitimate customers during peak hours. How do you resolve this?

1. Analyze: Identify which customers/endpoints are hitting limits. Check if limits are per-customer or global.
2. Tiered rate limits: Different limits per plan (free: 100/min, pro: 1000/min, enterprise: 10000/min)
3. Burst allowance: Use token bucket algorithm that allows short bursts above steady-state rate
4. Response headers: Always include X-RateLimit-Limit, X-RateLimit-Remaining, X-RateLimit-Reset
5. Client guidance: Return Retry-After header with 429 so clients know when to retry
6. Backend optimization: If backend is the bottleneck, add caching (CDN for GET), optimize queries, scale horizontally
7. Priority queuing: Use request priority (paying customers > free tier) during overload
