Kubernetes Networking

Complete reference covering the Kubernetes networking model, Services, Ingress, Network Policies, DNS, and CNI plugins -- everything needed for production clusters and interviews.

Kubernetes Networking Model

Kubernetes imposes a set of fundamental requirements on the cluster networking implementation. These guarantees simplify the mental model for developers and ensure portability across different environments.

The Three Guarantees

1. Every Pod gets its own unique IP address (no manual allocation).

2. Pods on any node can communicate with all pods on any other node without NAT.

3. Agents on a node (kubelet, system daemons) can communicate with all pods on that node.

1
Flat Network Model: Kubernetes requires a flat, routable network where every pod can reach every other pod directly by IP, regardless of which node it runs on. There is no NAT between pods. This dramatically simplifies application networking -- containers behave as if they are on the same LAN.
2
Pod IP Assignment: Each pod receives a unique IP address from the node's pod CIDR range. Containers within the same pod share the network namespace (same IP, same port space) and communicate via localhost. The IP is ephemeral -- it changes when a pod is recreated.
3
No NAT Requirement: The source IP a pod sees from incoming traffic is the actual sender pod's IP address. This means pods can rely on IP-based authentication and logging without translation confusion. Network policies also operate on real pod IPs.
4
CNI Plugins: Kubernetes does not implement networking itself. It delegates to Container Network Interface (CNI) plugins. The CNI specification defines how the container runtime invokes the plugin to set up or tear down a pod's network interface. Popular plugins: Calico, Cilium, Flannel, Weave Net.
5
Network Namespaces: Each pod runs in its own Linux network namespace, created by the pause container. The CNI plugin attaches a virtual ethernet (veth) pair -- one end in the pod namespace, the other on the host -- and configures routes so traffic flows correctly between pods and the node.
6
Node-to-Node Communication: For cross-node pod traffic, CNI plugins use various strategies: VXLAN overlay tunnels (Flannel default), BGP-advertised routes (Calico), eBPF programs (Cilium), or cloud-native VPC routing (AWS VPC CNI, Azure CNI). Each trades off simplicity, performance, and feature set.
7
Pod CIDR Allocation: The cluster allocates a large CIDR (e.g., 10.244.0.0/16) for pods. Each node receives a slice (e.g., /24 giving 254 pod IPs per node). The controller-manager's --cluster-cidr and --node-cidr-mask-size flags control this allocation.

Services

Services provide a stable networking abstraction over a set of pods. Since pod IPs are ephemeral, Services offer a permanent virtual IP (ClusterIP) and DNS name that routes traffic to healthy pods matching a label selector.

ClusterIP (Default)

1
Internal-Only VIP: ClusterIP services expose the application on an internal cluster IP. This IP is only reachable from within the cluster. It provides a stable endpoint for inter-service communication.
2
Virtual IP Allocation: The service CIDR (e.g., 10.96.0.0/12) is configured at cluster creation via --service-cluster-ip-range. IPs are allocated sequentially or can be specified manually. This range must NOT overlap with the pod CIDR or node networks.
3
Load Balancing: kube-proxy programs iptables/IPVS rules to distribute traffic across all healthy endpoints (pods matching the selector). In iptables mode, the distribution is random probability-based; in IPVS mode, you can choose round-robin, least-connections, etc.
4
Session Affinity: Optionally configure sessionAffinity: ClientIP to route all requests from the same client IP to the same pod. The timeout is configurable via sessionAffinityConfig.clientIP.timeoutSeconds (default 10800s / 3 hours).
5
Multi-Port Services: A single service can expose multiple ports, each with a name, protocol, port number, and targetPort. Useful for applications that serve HTTP and gRPC on different ports from the same pods.
clusterip-service.yaml ClusterIP 
# ClusterIP Service -- Internal-only access within the cluster
# Provides stable IP and DNS name for a set of backend pods
apiVersion: v1
kind: Service
metadata:
  name: backend-api
  namespace: production
  labels:
    app: backend-api
    tier: backend
spec:
  type: ClusterIP          # Default type -- internal access only
  selector:
    app: backend-api       # Routes to pods with this label
    version: v2            # Can use multiple labels for precise targeting
  ports:
    - name: http
      protocol: TCP
      port: 80             # Port the service listens on (ClusterIP:80)
      targetPort: 8080     # Port on the pod container
    - name: grpc
      protocol: TCP
      port: 9090
      targetPort: 9090
  sessionAffinity: None    # Options: None, ClientIP

NodePort

1
External Access via Node IP: NodePort extends ClusterIP by additionally opening a static port (30000-32767) on every node in the cluster. External traffic can reach the service at <NodeIP>:<NodePort> from outside the cluster.
2
Port Range: The default NodePort range is 30000-32767, configurable via the API server's --service-node-port-range flag. You can specify a port or let Kubernetes auto-allocate one.
3
All Nodes Listen: The chosen NodePort is opened on ALL nodes, even if no matching pod runs on that node. kube-proxy handles routing the request to a node that has a backend pod.
4
Source IP Caveat: By default, NodePort traffic arriving at a node without a local pod is forwarded to another node, causing SNAT (the original source IP is lost). Set externalTrafficPolicy: Local to preserve source IP (but lose even load distribution).
5
Use Cases: Development/testing environments, on-premises clusters without cloud load balancers, or as a building block under a manual external load balancer. Not recommended for production external traffic -- use LoadBalancer or Ingress instead.
nodeport-service.yaml NodePort 
# NodePort Service -- Exposes service on a static port on each node
# Accessible externally at <NodeIP>:<NodePort>
apiVersion: v1
kind: Service
metadata:
  name: frontend-web
  namespace: production
  labels:
    app: frontend
    tier: frontend
spec:
  type: NodePort
  selector:
    app: frontend
    tier: frontend
  ports:
    - name: http
      protocol: TCP
      port: 80              # ClusterIP port (internal)
      targetPort: 3000      # Container port
      nodePort: 30080       # Static port on all nodes (30000-32767)
    - name: https
      protocol: TCP
      port: 443
      targetPort: 3443
      nodePort: 30443
  externalTrafficPolicy: Local   # Preserve source IP (skip extra hop)

LoadBalancer

1
Cloud-Native External LB: LoadBalancer type extends NodePort by provisioning an external load balancer from the cloud provider (AWS ELB/NLB, GCP LB, Azure LB). The cloud controller-manager handles the provisioning and assigns an external IP.
2
Automatic Provisioning: When you create a LoadBalancer service, the cloud-controller-manager calls the cloud API to provision an LB, configure health checks, and point it at all nodes' NodePorts. The external IP appears in status.loadBalancer.ingress.
3
Cost Implications: Each LoadBalancer service creates a separate cloud load balancer, which costs money. For many services, use an Ingress controller instead to share a single LB across multiple services via L7 routing rules.
4
Health Checks: Cloud LBs health-check nodes on the NodePort. With externalTrafficPolicy: Local, only nodes with local pods pass health checks, giving direct routing and source IP preservation.
5
Annotations for Configuration: Cloud-specific behavior is configured via annotations (e.g., service.beta.kubernetes.io/aws-load-balancer-type: nlb for AWS NLB, or service.beta.kubernetes.io/aws-load-balancer-internal: "true" for internal-only LBs).
loadbalancer-service.yaml LoadBalancer 
# LoadBalancer Service -- Provisions external cloud load balancer
# Suitable for production external traffic on cloud providers
apiVersion: v1
kind: Service
metadata:
  name: api-gateway
  namespace: production
  labels:
    app: api-gateway
  annotations:
    # AWS-specific: use Network Load Balancer instead of Classic
    service.beta.kubernetes.io/aws-load-balancer-type: "nlb"
    # Enable cross-zone load balancing
    service.beta.kubernetes.io/aws-load-balancer-cross-zone-load-balancing-enabled: "true"
    # Attach SSL certificate from ACM
    service.beta.kubernetes.io/aws-load-balancer-ssl-cert: "arn:aws:acm:us-east-1:123456789:certificate/abc-123"
    # SSL negotiation policy
    service.beta.kubernetes.io/aws-load-balancer-ssl-negotiation-policy: "ELBSecurityPolicy-TLS13-1-2-2021-06"
spec:
  type: LoadBalancer
  selector:
    app: api-gateway
  ports:
    - name: https
      protocol: TCP
      port: 443              # External LB port
      targetPort: 8443       # Container port
  externalTrafficPolicy: Local   # Preserve client source IP
  # Restrict access to specific CIDR ranges
  loadBalancerSourceRanges:
    - 10.0.0.0/8
    - 203.0.113.0/24

ExternalName

1
DNS CNAME Alias: ExternalName services map a service DNS name to an external DNS name (CNAME record). No proxying occurs -- it is purely a DNS-level redirect. Useful for referencing external services (databases, SaaS APIs) via a Kubernetes-native DNS name.
2
No Selector or Endpoints: ExternalName services do not have selectors, ClusterIPs, or EndpointSlices. They simply return a CNAME record when queried via DNS. This means kube-proxy is not involved at all.
3
Migration Use Case: Lets you migrate external services into the cluster gradually. Start with ExternalName pointing to the external DB, then switch to a ClusterIP service when the DB moves in-cluster -- application code references the same service name throughout.
4
Limitations: Does not work with IP addresses (only DNS names). Some applications that do not follow CNAME redirects may have issues. No port remapping -- the target must use the same port the client connects to.
5
TLS Considerations: Since it returns a CNAME, TLS certificate validation must match the external hostname, not the Kubernetes service name. This can cause issues if clients validate the SNI against the service name.
externalname-service.yaml ExternalName 
# ExternalName Service -- DNS CNAME redirect to external service
# No proxying, no ClusterIP -- purely DNS-level abstraction
apiVersion: v1
kind: Service
metadata:
  name: external-database
  namespace: production
spec:
  type: ExternalName
  # Querying external-database.production.svc.cluster.local
  # returns a CNAME record pointing to this external DNS name
  externalName: prod-db.c9xk2rgh.us-east-1.rds.amazonaws.com
  # No selector, no ports, no ClusterIP

Headless Service

1
No ClusterIP (clusterIP: None): A headless service skips the virtual IP allocation. DNS queries return the individual pod IPs directly instead of a single VIP. This enables client-side load balancing and direct pod discovery.
2
StatefulSet Integration: Headless services are essential for StatefulSets. They create individual DNS records like pod-0.my-svc.ns.svc.cluster.local for each pod, enabling stable network identities critical for databases and clustered applications.
3
DNS A Records: Instead of a single A record pointing to the ClusterIP, a headless service returns multiple A records -- one for each ready pod endpoint. Clients receive all IPs and can implement their own selection logic.
4
No kube-proxy Involvement: Since there is no VIP, kube-proxy does not create any iptables/IPVS rules for headless services. Traffic goes directly from client to pod without kernel-level load balancing.
5
Use Cases: StatefulSets (Kafka, Cassandra, etcd), service meshes that handle their own load balancing, applications that need peer discovery (each pod must know about all other pods), and gRPC services that benefit from client-side balancing.
headless-service.yaml Headless 
# Headless Service -- No ClusterIP, returns individual pod IPs
# Essential for StatefulSets and peer-discovery use cases
apiVersion: v1
kind: Service
metadata:
  name: cassandra
  namespace: database
  labels:
    app: cassandra
spec:
  clusterIP: None          # This makes it headless
  selector:
    app: cassandra
  ports:
    - name: cql
      port: 9042
      targetPort: 9042
    - name: inter-node
      port: 7000
      targetPort: 7000
  # With a StatefulSet named "cassandra", DNS records become:
  # cassandra-0.cassandra.database.svc.cluster.local
  # cassandra-1.cassandra.database.svc.cluster.local
  # cassandra-2.cassandra.database.svc.cluster.local

Endpoints and EndpointSlices

Endpoints (legacy): A single Endpoints object per Service that lists all pod IPs. Becomes a bottleneck with many pods because any change requires updating the entire object.

EndpointSlices (preferred, default since 1.21): Splits endpoints into multiple smaller objects (max 100 endpoints per slice by default). Enables efficient updates -- only the changed slice is transmitted. Supports dual-stack (IPv4 + IPv6) and topology hints for zone-aware routing.

How they work: The EndpointSlice controller watches pods matching the service selector. When a pod becomes Ready (passes readiness probe), it is added to an EndpointSlice. When NotReady or deleted, it is removed. kube-proxy watches EndpointSlices to update its routing rules.

Ingress

Ingress provides Layer 7 (HTTP/HTTPS) routing to services within the cluster. It consolidates routing rules into a single resource, allowing you to expose multiple services behind one external IP with path-based and host-based routing.

1
L7 Load Balancing: Unlike Services (L4), Ingress operates at Layer 7. It can inspect HTTP headers, paths, and hostnames to make routing decisions. This enables virtual hosting, URL rewrites, rate limiting, and content-based routing.
2
Ingress Controller Required: The Ingress resource is just a specification -- it does nothing without a controller. Popular controllers: nginx-ingress-controller, Traefik, HAProxy, AWS ALB Ingress Controller, Istio Gateway. Each interprets Ingress rules and configures the underlying proxy.
3
TLS Termination: Ingress can terminate TLS at the edge, decrypting HTTPS traffic before forwarding plain HTTP to backend services. Certificates are stored as Kubernetes Secrets of type kubernetes.io/tls.
4
Host-Based Routing: Route traffic to different services based on the Host header (virtual hosting). For example, api.example.com goes to the API service, app.example.com goes to the frontend service -- all through a single IP.
5
Path-Based Routing: Route traffic based on URL path prefixes or exact matches. For example, /api/* to the API backend, /static/* to the CDN backend, and / to the frontend -- shared under one domain.
6
IngressClass: Since Kubernetes 1.18, IngressClass resources specify which controller should handle a given Ingress resource. This allows multiple ingress controllers to coexist (e.g., internal nginx + external ALB) without conflicts.
7
Annotations for Advanced Config: Since the Ingress spec is intentionally limited, controllers expose advanced features via annotations (rate limiting, OAuth proxy, custom timeouts, rewrite rules, CORS). This makes Ingress resources somewhat controller-specific.
ingress-host-path-routing.yaml Ingress 
# Ingress with host-based and path-based routing
# Routes traffic to different services based on hostname and URL path
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: app-ingress
  namespace: production
  annotations:
    # NGINX Ingress Controller annotations
    nginx.ingress.kubernetes.io/rewrite-target: /
    nginx.ingress.kubernetes.io/proxy-body-size: "50m"
    nginx.ingress.kubernetes.io/proxy-read-timeout: "60"
    nginx.ingress.kubernetes.io/proxy-send-timeout: "60"
    nginx.ingress.kubernetes.io/rate-limit: "100"
    nginx.ingress.kubernetes.io/rate-limit-window: "1m"
    # Enable CORS
    nginx.ingress.kubernetes.io/enable-cors: "true"
    nginx.ingress.kubernetes.io/cors-allow-origin: "https://app.example.com"
spec:
  ingressClassName: nginx    # Which ingress controller handles this
  rules:
    # Host-based routing: api.example.com
    - host: api.example.com
      http:
        paths:
          - path: /v1
            pathType: Prefix        # Matches /v1, /v1/, /v1/users, etc.
            backend:
              service:
                name: api-v1
                port:
                  number: 80
          - path: /v2
            pathType: Prefix
            backend:
              service:
                name: api-v2
                port:
                  number: 80
    # Host-based routing: app.example.com
    - host: app.example.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: frontend
                port:
                  number: 80
          - path: /static
            pathType: Prefix
            backend:
              service:
                name: static-assets
                port:
                  number: 80
  # Default backend for unmatched requests
  defaultBackend:
    service:
      name: default-backend
      port:
        number: 80

Network Policies

Network Policies are Kubernetes resources that control traffic flow at the IP/port level (L3/L4) between pods, namespaces, and external endpoints. They act as a firewall for pod communication within the cluster.

Important: Network Policies require a CNI plugin that supports them (Calico, Cilium, Weave). Flannel alone does NOT enforce network policies. If your CNI does not support them, the NetworkPolicy resources are silently ignored.

1
Default Allow-All: By default, Kubernetes has NO network restrictions. All pods can communicate with all other pods across all namespaces. Network Policies add restrictions -- they are additive and whitelist-based (deny by default, then allow specific traffic).
2
Pod Selector Targeting: Each NetworkPolicy targets a set of pods via spec.podSelector. Once a pod is selected by ANY NetworkPolicy, all traffic not explicitly allowed by a policy is denied. Unselected pods remain fully open.
3
Ingress and Egress Rules: Policies can control incoming traffic (ingress), outgoing traffic (egress), or both. Each rule specifies allowed sources (for ingress) or destinations (for egress) using pod selectors, namespace selectors, or IP blocks.
4
Namespace Isolation: Use namespaceSelector to allow traffic only from specific namespaces. This is the foundation of multi-tenant isolation -- e.g., only allow traffic from the same namespace or from the monitoring namespace.
5
IP Block (CIDR): The ipBlock field allows specifying CIDR ranges for external traffic sources/destinations. Useful for allowing traffic from specific external networks or blocking traffic to known bad ranges. Can include except to exclude sub-ranges.
6
Port-Level Control: Rules can specify allowed ports (protocol + port number). If no ports are specified in a rule, all ports are allowed for that source/destination. Named ports are supported for readability.
default-deny-all.yaml Default Deny 
# Default deny all ingress and egress traffic in a namespace
# Apply this first, then create allow policies for specific traffic
# This is the zero-trust starting point for network security
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-all
  namespace: production
spec:
  podSelector: {}          # Empty selector = applies to ALL pods in namespace
  policyTypes:
    - Ingress              # Deny all incoming traffic
    - Egress               # Deny all outgoing traffic
  # No ingress or egress rules = nothing is allowed
  # Pods can still reach the Kubernetes API server (usually)
  # DNS will be blocked -- add a separate policy to allow it

DNS (CoreDNS)

Kubernetes clusters run a DNS server (CoreDNS) that automatically creates DNS records for Services and Pods. This enables service discovery by name rather than IP address.

1
CoreDNS: The default cluster DNS since Kubernetes 1.13 (replaced kube-dns). Runs as a Deployment in kube-system namespace, exposed via a ClusterIP Service (usually 10.96.0.10). All pods are configured to use this as their DNS resolver via /etc/resolv.conf.
2
Service DNS Format: <service-name>.<namespace>.svc.cluster.local. Within the same namespace, you can use just <service-name>. The search domains in resolv.conf handle the expansion (e.g., search production.svc.cluster.local svc.cluster.local cluster.local).
3
Pod DNS Format: <pod-ip-dashed>.<namespace>.pod.cluster.local (e.g., 10-244-1-5.production.pod.cluster.local). For pods with a hostname and subdomain field (used with StatefulSets), the format is <hostname>.<subdomain>.<namespace>.svc.cluster.local.
4
SRV Records: For named ports in a service, CoreDNS creates SRV records: _<port-name>._<protocol>.<service>.<namespace>.svc.cluster.local. These are used by applications that need to discover both the IP and port dynamically.
5
Corefile Configuration: CoreDNS is configured via a ConfigMap named coredns in kube-system. The Corefile supports plugins for caching, forwarding to upstream DNS, custom zone files, and health checking. Changes to the ConfigMap are auto-reloaded.
6
DNS Policies: Pod dnsPolicy controls resolution behavior: ClusterFirst (default -- cluster DNS first, then upstream), Default (inherit node's resolv.conf), None (custom -- must specify dnsConfig), ClusterFirstWithHostNet (for hostNetwork pods).
7
External DNS Resolution: Queries for non-cluster domains (e.g., google.com) are forwarded by CoreDNS to upstream nameservers (configured via the forward plugin, typically inheriting from the node's /etc/resolv.conf). The cache plugin reduces upstream query load.

Gateway API Overview

Gateway API -- The Future of Kubernetes Ingress

Gateway API is the successor to Ingress, providing more expressive, extensible, and role-oriented routing. It reached GA (v1.0) in October 2023 and is the recommended path for new deployments.

1
Role-Oriented Design: Gateway API separates concerns into three resource types: GatewayClass (infra provider, like StorageClass), Gateway (cluster operator configures listeners/ports), and HTTPRoute/TCPRoute (app developer defines routing rules). This matches organizational boundaries.
2
Multi-Protocol Support: Unlike Ingress (HTTP/HTTPS only), Gateway API supports HTTP, HTTPS, TCP, UDP, TLS passthrough, and gRPC natively. Different route types (HTTPRoute, TCPRoute, TLSRoute, GRPCRoute) handle each protocol.
3
Advanced Features: Header-based routing, request mirroring, traffic splitting (canary), URL rewrites, request/response header modification -- all expressible in the core API without implementation-specific annotations.
4
Cross-Namespace References: HTTPRoutes can reference Services in other namespaces (with ReferenceGrant permission), enabling shared infrastructure gateways while maintaining namespace isolation.
5
Implementation Support: Supported by all major implementations: Istio, Cilium, Contour, Envoy Gateway, nginx-gateway-fabric, Traefik, HAProxy, AWS Gateway API Controller, and GKE.

Interview Questions and Answers

Q1: How does Kubernetes Service routing actually work at the kernel level?
Through iptables DNAT rules or IPVS virtual servers programmed by kube-proxy. When a pod sends traffic to a Service ClusterIP, the packet hits iptables rules before leaving the node. These rules perform DNAT (Destination NAT), rewriting the destination IP from the ClusterIP to a randomly selected backend pod IP. The response traffic is automatically un-DNAT'd. In IPVS mode, the kernel's built-in load balancer handles this more efficiently using hash tables. The kube-proxy process watches the API server for Service/EndpointSlice changes and reprograms these rules accordingly.
Q2: What is the difference between Ingress and a LoadBalancer Service? When would you use each?
LoadBalancer = L4 (one service per LB), Ingress = L7 (many services per LB). A LoadBalancer service provisions one cloud load balancer per service -- expensive at scale but simple and supports any TCP/UDP protocol. Ingress provides L7 routing (host/path-based) allowing many services to share a single load balancer -- cost-effective but limited to HTTP/HTTPS. Use LoadBalancer for non-HTTP protocols (databases, gRPC without HTTP/2, MQTT), and Ingress for HTTP-based microservices. In practice, the Ingress controller itself runs behind a LoadBalancer service.
Q3: How do you implement zero-trust networking in Kubernetes with Network Policies?
Start with default-deny, then explicitly allow required traffic paths. Step 1: Apply a default-deny NetworkPolicy (empty podSelector, both Ingress and Egress policyTypes, no rules) in every namespace. Step 2: Allow DNS egress to kube-dns (otherwise nothing works). Step 3: For each service, create policies allowing only the specific ingress from known callers and egress to known dependencies. Step 4: Use namespace labels to restrict cross-namespace traffic. Step 5: Use ipBlock with exceptions for external traffic. This ensures every allowed path is explicitly documented and auditable.
Q4: How does DNS resolution work inside a Kubernetes pod, step by step?
Pod resolv.conf points to CoreDNS, which resolves cluster names or forwards to upstream. Step 1: Application calls getaddrinfo("backend-api"). Step 2: Libc reads /etc/resolv.conf (nameserver: 10.96.0.10, ndots:5). Step 3: Since "backend-api" has fewer than 5 dots, the resolver appends search domains and tries: backend-api.production.svc.cluster.local, then backend-api.svc.cluster.local, then backend-api.cluster.local, then bare backend-api. Step 4: CoreDNS receives the query, checks its cache, looks up the Service ClusterIP. Step 5: CoreDNS returns the A record with the ClusterIP. For external names, CoreDNS forwards to upstream DNS (usually the node's configured nameserver).

Q6: What is a headless service and when would you use one?
A service with clusterIP: None that returns pod IPs directly via DNS. Instead of load-balancing behind a VIP, DNS queries return A records for each endpoint pod. Use cases: (1) StatefulSets -- each pod gets a stable DNS name (pod-0.svc.ns.svc.cluster.local) for peer discovery in databases like Cassandra or Kafka. (2) Client-side load balancing -- gRPC clients can discover all backends and balance across them. (3) Service meshes that do their own routing. (4) Leader election -- clients can query all members and determine the leader themselves.
Q7: How does externalTrafficPolicy affect service behavior?
It controls whether traffic is routed only to local pods or to any pod in the cluster. Cluster (default): Traffic arriving at any node is load-balanced to all pods cluster-wide. Extra hop may occur (node-to-node), source IP is SNAT'd. All nodes pass health checks. Local: Traffic only goes to pods on the receiving node. No extra hops, source IP preserved, but uneven load if pods are not evenly distributed. Nodes without local pods fail health checks, so the external LB stops sending them traffic. Use Local when you need source IP (rate limiting, geo-routing, logging) or want to minimize latency.
Q8: What is the ndots setting and why does it matter for performance?
ndots:5 means any name with fewer than 5 dots triggers search domain expansion. By default, Kubernetes sets ndots:5 in pod resolv.conf. When you query "api.external.com" (2 dots, less than 5), the resolver first tries: api.external.com.production.svc.cluster.local, then api.external.com.svc.cluster.local, then api.external.com.cluster.local, and finally api.external.com. This means 4 DNS queries for every external name! Mitigations: (1) Use FQDNs with trailing dot: "api.external.com." (2) Lower ndots in pod dnsConfig. (3) Enable CoreDNS autopath plugin. (4) Accept the overhead if external lookups are rare.
Q9: How does Cilium replace kube-proxy and why would you want that?
Cilium uses eBPF programs to handle service routing directly in the kernel, bypassing iptables entirely. Benefits: (1) O(1) lookup performance regardless of service count (vs. O(n) for iptables). (2) Socket-level load balancing -- decisions are made at connect() time, not per-packet, reducing overhead. (3) No iptables rule explosion -- clusters with 10,000+ services become manageable. (4) Advanced load balancing (Maglev consistent hashing for DSR). (5) Built-in observability (Hubble flow logs without tcpdump). You would want this for large clusters, performance-sensitive workloads, or when you need L7 visibility without a full service mesh.
Q10: Explain the complete network path of an external request reaching a pod through Ingress.
DNS → Cloud LB → Ingress Controller Pod → Backend Service → Backend Pod. Step 1: Client resolves app.example.com to the external LB IP (via external-dns or manual DNS record). Step 2: Request arrives at the cloud load balancer (provisioned by the Ingress controller's Service of type LoadBalancer). Step 3: LB forwards to one of the Ingress controller pods (via NodePort or direct pod routing). Step 4: Ingress controller matches Host header and path against Ingress rules. Step 5: Controller resolves the backend service's endpoints (either via kube-proxy rules or direct endpoint lookup). Step 6: Controller forwards the request to a selected backend pod IP. Step 7: Backend pod processes and responds. Response follows the reverse path.

Troubleshooting Scenarios

⚠ Scenario: Service Has No Endpoints

Symptom: Requests to a Service return connection refused or time out. kubectl get endpoints <svc> shows <none>.

Root Causes and Diagnostics:

1. Verify selector matches pods: kubectl get svc <svc> -o jsonpath='{.spec.selector}' then kubectl get pods -l app=<value>. The most common cause is a label mismatch between service selector and pod labels (typo, wrong version label, etc.).
2. Check pods are Ready: kubectl get pods -l <selector>. Only pods in Ready state (passing readiness probes) are added to EndpointSlices. If pods are running but not ready, fix the readiness probe or the underlying health issue.
3. Verify namespace: kubectl get svc -n <ns> and kubectl get pods -n <ns> -l <selector>. Service and pods must be in the same namespace for the selector to match.
4. Check targetPort: kubectl get svc -o yaml -- verify targetPort matches what the container is actually listening on. A mismatch means traffic reaches the pod but hits a closed port.
5. Examine EndpointSlices: kubectl get endpointslices -l kubernetes.io/service-name=<svc>. Check if endpoints exist but have conditions.ready: false.
⚠ Scenario: DNS Resolution Failing Inside Pods

Symptom: Pods cannot resolve service names or external hostnames. Applications fail with "Name or service not known" or "Temporary failure in name resolution".

Root Causes and Diagnostics:

1. Check CoreDNS pods: kubectl get pods -n kube-system -l k8s-app=kube-dns. If CoreDNS pods are CrashLoopBackOff or not running, all DNS resolution fails. Check logs: kubectl logs -n kube-system -l k8s-app=kube-dns.
2. Verify CoreDNS Service: kubectl get svc kube-dns -n kube-system. Confirm the ClusterIP matches what is in the pod's /etc/resolv.conf. If the service is missing or has no endpoints, recreate it.
3. Test from the pod: kubectl exec <pod> -- cat /etc/resolv.conf and kubectl exec <pod> -- nslookup kubernetes.default. If resolv.conf is wrong, check the pod's dnsPolicy and dnsConfig settings.
4. Check NetworkPolicy blocking DNS: If you have default-deny egress policies, ensure DNS egress (UDP/TCP port 53 to kube-dns pods) is explicitly allowed. This is the most common oversight with network policies.
5. CoreDNS ConfigMap issues: kubectl get cm coredns -n kube-system -o yaml. Check for syntax errors in the Corefile, incorrect forward directives, or custom plugins causing crashes. Validate with kubectl logs.
⚠ Scenario: Ingress Returning 502 Bad Gateway

Symptom: Requests through Ingress return HTTP 502 Bad Gateway. Direct access to the service (via port-forward) works fine.

Root Causes and Diagnostics:

1. Check backend service endpoints: kubectl get endpoints <backend-svc>. A 502 from the ingress controller means it cannot connect to the backend. If endpoints are empty, fix the service selector (see scenario above).
2. Verify port configuration: The Ingress backend port must match the Service port (not targetPort). Check kubectl get svc <backend> -o yaml and compare with the Ingress spec's backend.service.port.number.
3. Check ingress controller logs: kubectl logs -n ingress-nginx -l app.kubernetes.io/name=ingress-nginx. Look for "upstream connect error" or "connection refused". The logs reveal which backend IP the controller tried to connect to.
4. Network policies blocking ingress controller: If network policies exist, ensure the ingress controller namespace/pods are allowed to reach backend pods. The ingress controller needs egress to backend pods on the target port.
5. Backend pod readiness: kubectl get pods -l <selector>. Even if pods exist, they may be failing readiness probes (503 from app). Check pod logs and increase proxy-read-timeout annotation if backends are slow to respond.
⚠ Scenario: Pod Cannot Reach the Internet

Symptom: Pods can communicate with other pods and services within the cluster, but cannot reach external endpoints (APIs, package registries, etc.).

Root Causes and Diagnostics:

1. Check egress NetworkPolicy: kubectl get networkpolicies -n <namespace>. If a default-deny egress policy exists, external traffic is blocked unless explicitly allowed. Add an egress rule with ipBlock 0.0.0.0/0 for port 443.
2. Verify DNS resolution: kubectl exec <pod> -- nslookup google.com. If DNS fails, the problem may be DNS (see DNS scenario above), not actual routing. Fix DNS first.
3. Check NAT/masquerade rules: iptables -t nat -L POSTROUTING on the node. For pods to reach the internet, their pod IPs (non-routable externally) must be SNAT'd to the node IP. CNI plugins typically set this up, but it can break after upgrades.
4. Verify node can reach internet: SSH to the node and test curl -I https://google.com. If the node itself cannot reach the internet, the issue is infrastructure-level (VPC routing, NAT gateway, security groups) not Kubernetes.
5. Check cloud security groups / NACLs: In cloud environments, ensure the node security group allows outbound traffic on required ports (443, 80). Check VPC route tables point to an internet gateway or NAT gateway for the pod/node subnets.
