Kubernetes Workloads

Complete reference for all Kubernetes workload resources: Pods, Deployments, ReplicaSets, StatefulSets, DaemonSets, Jobs, and CronJobs. Production-ready YAML examples, interview Q&A, and troubleshooting scenarios.

Pods

A Pod is the smallest deployable unit in Kubernetes. It represents a single instance of a running process and can contain one or more tightly coupled containers that share network namespace, IPC, and optionally volumes.

Key Insight: You almost never create Pods directly in production. Instead, you use higher-level controllers (Deployments, StatefulSets, etc.) that manage Pod lifecycle. Understanding Pods is essential because every other workload resource ultimately creates and manages Pods.

1
Smallest Deployable Unit: A Pod is the atomic scheduling unit in Kubernetes. The scheduler places entire Pods on nodes -- you cannot split containers within a Pod across nodes. This is intentional: containers in a Pod are co-located and co-scheduled because they need to communicate via localhost.
2
Multi-Container Patterns: Pods support multiple containers for specific patterns: sidecar (log shipping, service mesh proxy), ambassador (proxy to external services), and adapter (transform output format). Each container has its own filesystem but shares the Pod's network namespace.
3
Shared Network Namespace: All containers in a Pod share the same IP address and port space. They communicate via localhost. This is implemented via the pause container which holds the network namespace alive even if application containers restart.
4
Init Containers: Run sequentially before app containers start. Used for setup tasks (wait for dependencies, populate volumes, configure permissions). If any init container fails, the Pod restarts. They run to completion before the next init container (or app container) starts.
5
Ephemeral by Design: Pods are mortal -- they are created, assigned a UID, scheduled to a node, and run until termination. They are never "moved" between nodes; if a node dies, identical replacement Pods are created elsewhere by the controlling resource.

Complete Pod YAML

pod-complete.yaml Production Reference 
apiVersion: v1
kind: Pod
metadata:
  name: app-pod
  namespace: production
  labels:
    app: myapp                    # Used by Services and Deployments for selection
    version: v1.2.0              # Track which version is running
    team: platform               # Ownership for cost allocation
  annotations:
    prometheus.io/scrape: "true" # Enable Prometheus metrics scraping
    prometheus.io/port: "9090"   # Port where metrics are exposed
spec:
  # --- Scheduling Controls ---
  nodeSelector:
    disktype: ssd                # Schedule only on nodes with SSD
  tolerations:
    - key: "dedicated"
      operator: "Equal"
      value: "high-memory"
      effect: "NoSchedule"       # Tolerate taint on high-memory nodes
  affinity:
    podAntiAffinity:
      preferredDuringSchedulingIgnoredDuringExecution:
        - weight: 100
          podAffinityTerm:
            labelSelector:
              matchLabels:
                app: myapp
            topologyKey: kubernetes.io/hostname  # Spread across nodes

  # --- Service Account ---
  serviceAccountName: myapp-sa   # Dedicated SA with least-privilege RBAC
  automountServiceAccountToken: false  # Disable if not calling K8s API

  # --- Security Context (Pod-level) ---
  securityContext:
    runAsNonRoot: true           # Container must run as non-root
    runAsUser: 1000              # UID for the process
    runAsGroup: 3000             # GID for the process
    fsGroup: 2000                # GID for volume mounts
    seccompProfile:
      type: RuntimeDefault       # Apply default seccomp profile

  # --- Init Containers (run sequentially before app containers) ---
  initContainers:
    - name: wait-for-db
      image: busybox:1.36
      command:
        - sh
        - -c
        - |
          until nc -z postgres-svc.production.svc.cluster.local 5432; do
            echo "Waiting for PostgreSQL..."
            sleep 2
          done
      resources:
        requests:
          cpu: 50m
          memory: 32Mi
        limits:
          cpu: 100m
          memory: 64Mi

    - name: init-config
      image: busybox:1.36
      command:
        - sh
        - -c
        - cp /config-template/app.conf /shared-config/app.conf
      volumeMounts:
        - name: config-template
          mountPath: /config-template
        - name: shared-config
          mountPath: /shared-config

  # --- Application Containers ---
  containers:
    - name: app
      image: registry.example.com/myapp:v1.2.0  # Always use specific tags, never :latest
      imagePullPolicy: IfNotPresent              # Always for :latest, IfNotPresent for tagged

      # --- Ports ---
      ports:
        - name: http
          containerPort: 8080
          protocol: TCP
        - name: metrics
          containerPort: 9090
          protocol: TCP

      # --- Environment Variables ---
      env:
        - name: APP_ENV
          value: "production"
        - name: DB_HOST
          valueFrom:
            configMapKeyRef:
              name: app-config
              key: database.host
        - name: DB_PASSWORD
          valueFrom:
            secretKeyRef:
              name: app-secrets
              key: database.password
        - name: POD_NAME                         # Downward API -- inject pod metadata
          valueFrom:
            fieldRef:
              fieldPath: metadata.name
        - name: POD_IP
          valueFrom:
            fieldRef:
              fieldPath: status.podIP
        - name: NODE_NAME
          valueFrom:
            fieldRef:
              fieldPath: spec.nodeName
        - name: MEMORY_LIMIT                     # Useful for JVM heap sizing
          valueFrom:
            resourceFieldRef:
              containerName: app
              resource: limits.memory

      # --- Resource Requests and Limits ---
      resources:
        requests:
          cpu: 250m              # 0.25 CPU cores (used for scheduling)
          memory: 512Mi          # Used for scheduling and OOM score
          ephemeral-storage: 1Gi
        limits:
          cpu: 1000m             # 1 CPU core (throttled, not killed)
          memory: 1Gi            # Hard limit (OOMKilled if exceeded)
          ephemeral-storage: 2Gi

      # --- Probes ---
      startupProbe:              # Protect slow-starting containers
        httpGet:
          path: /healthz/startup
          port: http
        failureThreshold: 30     # 30 * 10s = 5 minutes to start
        periodSeconds: 10

      livenessProbe:             # Restart container if unhealthy
        httpGet:
          path: /healthz/live
          port: http
        initialDelaySeconds: 0   # startupProbe handles initial delay
        periodSeconds: 15
        timeoutSeconds: 3
        failureThreshold: 3      # Kill after 3 consecutive failures

      readinessProbe:            # Remove from Service endpoints if not ready
        httpGet:
          path: /healthz/ready
          port: http
        initialDelaySeconds: 5
        periodSeconds: 10
        timeoutSeconds: 3
        successThreshold: 1
        failureThreshold: 3

      # --- Container Security Context ---
      securityContext:
        allowPrivilegeEscalation: false
        readOnlyRootFilesystem: true   # Prevent writes to container filesystem
        capabilities:
          drop:
            - ALL                       # Drop all Linux capabilities

      # --- Volume Mounts ---
      volumeMounts:
        - name: shared-config
          mountPath: /etc/app
          readOnly: true
        - name: tmp
          mountPath: /tmp               # Writable tmp since rootfs is read-only
        - name: data
          mountPath: /var/data

      # --- Lifecycle Hooks ---
      lifecycle:
        postStart:
          exec:
            command: ["/bin/sh", "-c", "echo 'Container started' >> /tmp/lifecycle.log"]
        preStop:
          exec:
            command: ["/bin/sh", "-c", "sleep 5 && kill -SIGTERM 1"]  # Graceful shutdown

    # --- Sidecar Container (log shipper) ---
    - name: log-shipper
      image: fluent/fluent-bit:2.2
      resources:
        requests:
          cpu: 50m
          memory: 64Mi
        limits:
          cpu: 100m
          memory: 128Mi
      volumeMounts:
        - name: shared-logs
          mountPath: /var/log/app
          readOnly: true

  # --- Volumes ---
  volumes:
    - name: config-template
      configMap:
        name: app-config-template
    - name: shared-config
      emptyDir: {}               # Shared between init container and app
    - name: tmp
      emptyDir:
        sizeLimit: 100Mi         # Limit ephemeral storage usage
    - name: data
      persistentVolumeClaim:
        claimName: app-data-pvc
    - name: shared-logs
      emptyDir: {}

  # --- Pod Termination ---
  terminationGracePeriodSeconds: 60  # Time to gracefully shutdown (default 30)
  restartPolicy: Always              # Always | OnFailure | Never

  # --- DNS Configuration ---
  dnsPolicy: ClusterFirst            # Use cluster DNS (CoreDNS)
  dnsConfig:
    options:
      - name: ndots
        value: "2"                   # Reduce unnecessary DNS lookups

  # --- Image Pull Secrets ---
  imagePullSecrets:
    - name: registry-credentials

  # --- Priority ---
  priorityClassName: high-priority   # Preemption priority for scheduling

Deployments

A Deployment provides declarative updates for Pods and ReplicaSets. It manages the rollout of new versions, supports rollback, and ensures zero-downtime updates through configurable rolling update strategies.

1
Manages ReplicaSets: A Deployment creates and manages ReplicaSets. Each time you update the pod template, the Deployment creates a NEW ReplicaSet and gradually scales it up while scaling the old one down. Old ReplicaSets are kept (by default 10) for rollback.
2
Rolling Update Strategy: The default strategy creates new pods before killing old ones, ensuring continuous availability. Controlled by two parameters: maxSurge (how many extra pods above desired count) and maxUnavailable (how many pods can be down during update).
3
maxSurge Explained: Maximum number of pods that can be created above the desired replica count. Set to 25% (default) means during a rollout of 4 replicas, up to 5 pods may exist simultaneously. Higher values = faster rollout but more resource usage.
4
maxUnavailable Explained: Maximum number of pods that can be unavailable during the update. Set to 25% (default) means at least 75% of desired pods are always running. Set to 0 for zero-downtime (requires maxSurge > 0). Lower values = safer but slower rollout.
5
Rollback Capability: Every Deployment change is recorded as a revision. Rollback with kubectl rollout undo deployment/name or to a specific revision with --to-revision=N. The Deployment simply scales the previous ReplicaSet back up.
6
Revision History: Controlled by revisionHistoryLimit (default 10). Keeps old ReplicaSets (scaled to 0) for rollback. Set higher for critical services that might need to rollback to older versions. Set to 0 if you never rollback (saves etcd storage).
7
progressDeadlineSeconds: Maximum time (default 600s) for a Deployment to make progress before it is considered failed. If no new pods become ready within this window, the rollout is marked as stalled (status condition Progressing=False). Does NOT trigger automatic rollback.

Rolling Update Process

Rolling Update: 4 Replicas, maxSurge=1, maxUnavailable=1
Initial State (v1 running):
┌──────────────────────────────────────────────────────┐
│  ReplicaSet-v1 (4/4 ready)    ReplicaSet-v2 (0/0)   │
│  ┌────┐ ┌────┐ ┌────┐ ┌────┐                        │
│  │ v1 │ │ v1 │ │ v1 │ │ v1 │                        │

Step 1 -- Scale up v2, scale down v1 (maxSurge=1, maxUnavailable=1):

│  ReplicaSet-v1 (3/3 ready)    ReplicaSet-v2 (1/1)   │
│  ┌────┐ ┌────┐ ┌────┐        ┌────┐                 │
│  │ v1 │ │ v1 │ │ v1 │        │ v2 │                 │
│  └────┘ └────┘ └────┘        └────┘                 │

└──────────────────────────────────────────────────────┘

Step 2 -- Continue rolling:
┌──────────────────────────────────────────────────────┐
│  ReplicaSet-v1 (2/2 ready)    ReplicaSet-v2 (2/2)   │
│  ┌────┐ ┌────┐               ┌────┐ ┌────┐          │
│  │ v1 │ │ v1 │               │ v2 │ │ v2 │          │
│  └────┘ └────┘               └────┘ └────┘          │
│  Total pods: 4    Available: 4                       │
└──────────────────────────────────────────────────────┘

Step 3 -- Continue rolling:
┌──────────────────────────────────────────────────────┐
│  ReplicaSet-v1 (1/1 ready)    ReplicaSet-v2 (3/3)   │
│  ┌────┐                      ┌────┐ ┌────┐ ┌────┐   │
│  │ v1 │                      │ v2 │ │ v2 │ │ v2 │   │

┌──────────────────────────────────────────────────────┐
│  ReplicaSet-v1 (0/0 kept)     ReplicaSet-v2 (4/4)   │
│                               ┌────┐ ┌────┐ ┌────┐ ┌────┐
│                               │ v2 │ │ v2 │ │ v2 │ │ v2 │
│                               └────┘ └────┘ └────┘ └────┘

Key: At EVERY step, available pods >= (desired - maxUnavailable) = 3
     And total pods <= (desired + maxSurge) = 5

Complete Deployment YAML

deployment-production.yaml Production Ready 
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-api
  namespace: production
  labels:
    app: web-api
    tier: backend
    environment: production
  annotations:
    kubernetes.io/change-cause: "Update to v2.1.0 - fix memory leak in connection pool"
spec:
  replicas: 4                          # Desired number of pod replicas
  revisionHistoryLimit: 5              # Keep 5 old ReplicaSets for rollback
  progressDeadlineSeconds: 300         # Fail rollout if no progress in 5 minutes

  # --- Update Strategy ---
  strategy:
    type: RollingUpdate                # RollingUpdate (default) or Recreate
    rollingUpdate:
      maxSurge: 1                      # Allow 1 extra pod during rollout
      maxUnavailable: 0                # Zero-downtime: never reduce below desired

  # --- Pod Selector (immutable after creation) ---
  selector:
    matchLabels:
      app: web-api                     # Must match template.metadata.labels

  # --- Pod Template ---
  template:
    metadata:
      labels:
        app: web-api                   # Must match selector.matchLabels
        version: v2.1.0
      annotations:
        prometheus.io/scrape: "true"
        prometheus.io/port: "9090"
    spec:
      serviceAccountName: web-api-sa
      terminationGracePeriodSeconds: 45

      # --- Topology Spread for HA ---
      topologySpreadConstraints:
        - maxSkew: 1
          topologyKey: topology.kubernetes.io/zone
          whenUnsatisfiable: DoNotSchedule
          labelSelector:
            matchLabels:
              app: web-api
        - maxSkew: 1
          topologyKey: kubernetes.io/hostname
          whenUnsatisfiable: ScheduleAnyway
          labelSelector:
            matchLabels:
              app: web-api

      # --- Security Context ---
      securityContext:
        runAsNonRoot: true
        runAsUser: 1000
        fsGroup: 2000
        seccompProfile:
          type: RuntimeDefault

      # --- Containers ---
      containers:
        - name: web-api
          image: registry.example.com/web-api:v2.1.0
          imagePullPolicy: IfNotPresent

          ports:
            - name: http
              containerPort: 8080
            - name: metrics
              containerPort: 9090

          env:
            - name: GOMAXPROCS
              valueFrom:
                resourceFieldRef:
                  resource: limits.cpu    # Match GOMAXPROCS to CPU limit
            - name: DATABASE_URL
              valueFrom:
                secretKeyRef:
                  name: web-api-secrets
                  key: database-url

          envFrom:
            - configMapRef:
                name: web-api-config     # Load all keys as env vars

          resources:
            requests:
              cpu: 500m
              memory: 512Mi
            limits:
              cpu: 2000m
              memory: 1Gi

          # --- Probes ---
          startupProbe:
            httpGet:
              path: /healthz
              port: http
            periodSeconds: 5
            failureThreshold: 30         # 150s startup window

          livenessProbe:
            httpGet:
              path: /healthz
              port: http
            periodSeconds: 15
            timeoutSeconds: 5
            failureThreshold: 3

          readinessProbe:
            httpGet:
              path: /readyz
              port: http
            periodSeconds: 5
            timeoutSeconds: 3
            failureThreshold: 3
            successThreshold: 1

          securityContext:
            allowPrivilegeEscalation: false
            readOnlyRootFilesystem: true
            capabilities:
              drop: ["ALL"]

          volumeMounts:
            - name: tmp
              mountPath: /tmp
            - name: tls-certs
              mountPath: /etc/tls
              readOnly: true

          lifecycle:
            preStop:
              exec:
                command: ["/bin/sh", "-c", "sleep 10"]  # Allow LB to drain connections

      volumes:
        - name: tmp
          emptyDir:
            sizeLimit: 100Mi
        - name: tls-certs
          secret:
            secretName: web-api-tls
            defaultMode: 0400            # Read-only for owner

      imagePullSecrets:
        - name: registry-credentials

ReplicaSets

A ReplicaSet ensures that a specified number of pod replicas are running at any given time. It is the successor to ReplicationControllers and is primarily used as a mechanism by Deployments to orchestrate pod creation, deletion, and updates.

Important: You rarely create ReplicaSets directly. Use Deployments instead, which manage ReplicaSets for you and provide rollout/rollback capabilities. Understanding ReplicaSets is valuable for debugging (e.g., inspecting why a rollout is stuck).

1
Ensures Replica Count: The ReplicaSet controller continuously watches the actual number of pods matching its label selector. If there are too few, it creates new pods. If there are too many (e.g., after scaling down), it deletes excess pods (selecting by creation timestamp -- youngest first).
2
Label Selectors: Uses matchLabels and/or matchExpressions to identify which pods it manages. A pod is "owned" by a ReplicaSet if its labels match the selector AND it has an ownerReference pointing to that RS. Orphaned matching pods are adopted.
3
Pod Template Hash: When created by a Deployment, each ReplicaSet gets a unique pod-template-hash label (hash of the pod template spec). This prevents different ReplicaSets from accidentally adopting each other's pods and enables precise rollback targeting.
replicaset.yaml Reference Only 
apiVersion: apps/v1
kind: ReplicaSet
metadata:
  name: web-api-6799fc88d8         # Typically auto-generated by Deployment
  namespace: production
  labels:
    app: web-api
    pod-template-hash: 6799fc88d8  # Hash of pod template spec
  ownerReferences:                 # Points to parent Deployment
    - apiVersion: apps/v1
      kind: Deployment
      name: web-api
      uid: a1b2c3d4-e5f6-7890-abcd-ef1234567890
spec:
  replicas: 4
  selector:
    matchLabels:
      app: web-api
      pod-template-hash: 6799fc88d8
    # matchExpressions example:
    # matchExpressions:
    #   - key: environment
    #     operator: In
    #     values: ["production", "staging"]
    #   - key: version
    #     operator: NotIn
    #     values: ["deprecated"]
  template:
    metadata:
      labels:
        app: web-api
        pod-template-hash: 6799fc88d8
    spec:
      containers:
        - name: web-api
          image: registry.example.com/web-api:v2.1.0
          ports:
            - containerPort: 8080
          resources:
            requests:
              cpu: 500m
              memory: 512Mi
            limits:
              cpu: 2000m
              memory: 1Gi

StatefulSets

StatefulSets manage stateful applications that require stable network identities, persistent storage, and ordered deployment/scaling. Each pod in a StatefulSet gets a persistent identifier that it maintains across rescheduling.

Use StatefulSets when you need: Stable unique network identifiers (pod-0, pod-1, ...), stable persistent storage (each pod gets its own PVC), ordered graceful deployment and scaling, ordered automated rolling updates. Common use cases: databases (PostgreSQL, MySQL), distributed systems (Kafka, ZooKeeper, etcd), and any app that requires stable identity.

1
Stable Network Identity: Pods are named predictably: {statefulset-name}-{ordinal} (e.g., postgres-0, postgres-1, postgres-2). Each pod gets a stable DNS record: {pod-name}.{headless-service}.{namespace}.svc.cluster.local. This identity persists across rescheduling.
2
Ordered Deployment: Pods are created sequentially (0, 1, 2, ...) and each must be Running and Ready before the next is created. This enables primary/replica patterns where the primary must initialize first. Scaling down happens in reverse order (highest ordinal first).
3
Headless Service Requirement: StatefulSets require a Headless Service (clusterIP: None) to provide network identity. Unlike regular Services that load-balance, a Headless Service creates individual DNS A records for each pod, enabling direct peer-to-peer communication.
4
volumeClaimTemplates: Each pod gets its own PersistentVolumeClaim created from the template. PVCs are named {template-name}-{statefulset-name}-{ordinal}. Critically, PVCs are NOT deleted when pods are deleted or the StatefulSet is scaled down -- data persists for scale-up.
5
Update Strategies: RollingUpdate (default) updates pods in reverse ordinal order. OnDelete only updates a pod when you manually delete it -- useful for databases where you want to control the update sequence. The partition parameter allows canary-style updates.
6
Pod Management Policy: OrderedReady (default) enforces sequential startup. Parallel launches all pods simultaneously -- use for stateful apps that do not require ordered startup (e.g., independent cache nodes).

Complete StatefulSet YAML

statefulset-postgres.yaml Production Ready 
# Headless Service (REQUIRED for StatefulSet DNS)
apiVersion: v1
kind: Service
metadata:
  name: postgres-headless
  namespace: production
  labels:
    app: postgres
spec:
  clusterIP: None                    # Headless -- no load balancing
  selector:
    app: postgres
  ports:
    - name: postgres
      port: 5432
      targetPort: 5432
---
# Client-facing Service (load-balanced, for read replicas)
apiVersion: v1
kind: Service
metadata:
  name: postgres-read
  namespace: production
spec:
  selector:
    app: postgres
    role: replica
  ports:
    - name: postgres
      port: 5432
      targetPort: 5432
---
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: postgres
  namespace: production
  labels:
    app: postgres
spec:
  serviceName: postgres-headless     # Must reference Headless Service
  replicas: 3                        # 1 primary + 2 replicas
  podManagementPolicy: OrderedReady  # Sequential startup (primary first)

  # --- Update Strategy ---
  updateStrategy:
    type: RollingUpdate
    rollingUpdate:
      partition: 0                   # Update all pods (set >0 for canary)

  # --- Persistent Volume Claim Retention (K8s 1.27+) ---
  persistentVolumeClaimRetentionPolicy:
    whenDeleted: Retain              # Keep PVCs when StatefulSet deleted
    whenScaled: Retain               # Keep PVCs when scaling down

  selector:
    matchLabels:
      app: postgres

  template:
    metadata:
      labels:
        app: postgres
      annotations:
        checksum/config: "abc123"    # Force rollout on config change
    spec:
      serviceAccountName: postgres-sa
      terminationGracePeriodSeconds: 120  # Allow time for graceful shutdown

      securityContext:
        runAsNonRoot: true
        runAsUser: 999               # postgres user
        fsGroup: 999

DaemonSets

A DaemonSet ensures that a copy of a pod runs on all (or a subset of) nodes in the cluster. As nodes are added, DaemonSet pods are automatically scheduled on them. As nodes are removed, those pods are garbage collected.

1
One Pod Per Node: The DaemonSet controller ensures exactly one pod per eligible node. Unlike Deployments where the scheduler places pods, the DaemonSet controller directly sets spec.nodeName (in older versions) or uses node affinity (current behavior) to guarantee placement.
2
Common Use Cases: Cluster storage daemons (GlusterFS, Ceph), log collectors (Fluentd, Filebeat), node monitoring agents (Prometheus node-exporter, Datadog), network plugins (Calico, Cilium), security agents (Falco, Sysdig), and node-level configuration (NTP, kernel tuning).
3
Node Scheduling: DaemonSets bypass the normal scheduler. They use nodeAffinity to target specific nodes. You can restrict to a subset of nodes using nodeSelector or affinity rules in the pod template.
4
Tolerations for Control Plane: By default, DaemonSet pods are NOT scheduled on control plane nodes (which have the node-role.kubernetes.io/control-plane:NoSchedule taint). Add a toleration if the DaemonSet must run everywhere (e.g., CNI plugin, monitoring).
5
Update Strategy: Supports RollingUpdate (default, maxUnavailable configurable) and OnDelete (manual pod deletion triggers update). maxSurge is supported since K8s 1.22 for DaemonSets, allowing zero-downtime updates.

Jobs and CronJobs

Jobs create one or more pods and ensure that a specified number successfully terminate. CronJobs create Jobs on a recurring schedule. Both are designed for run-to-completion workloads rather than long-running services.

Jobs

1
Run-to-Completion: Unlike Deployments which maintain pods indefinitely, Jobs track successful completions. Once the required number of completions is reached, the Job is done. Pods created by Jobs have restartPolicy: Never or OnFailure (never Always).
2
Parallelism: spec.parallelism controls how many pods run concurrently. spec.completions sets the total number of successful completions needed. Use parallelism=N with completions=M for embarrassingly parallel workloads (e.g., processing M items with N workers).
3
backoffLimit: Maximum number of retries before marking the Job as failed (default 6). Uses exponential backoff (10s, 20s, 40s, ..., capped at 6 minutes). Each pod failure counts toward this limit, so parallelism can exhaust the backoff limit faster.
4
activeDeadlineSeconds: Absolute time limit for the entire Job (all retries). If the Job runs longer than this, all pods are terminated and the Job is marked failed. Use to prevent runaway jobs consuming resources indefinitely.
5
ttlSecondsAfterFinished: Automatically clean up finished Jobs and their pods after the specified duration. Without this, completed Jobs and pods remain until manually deleted or garbage collected by a controller.

CronJobs

1
Schedule Syntax: Uses standard cron format: minute hour day-of-month month day-of-week. Examples: */5 * * * * (every 5 min), 0 2 * * * (daily at 2am), 0 0 1 * * (monthly), 0 */6 * * 1-5 (every 6h on weekdays).
2
Concurrency Policy: Allow (default) permits concurrent Jobs. Forbid skips new Job if previous is still running. Replace cancels the running Job and starts a new one. Use Forbid for database-dependent tasks that cannot safely overlap.
3
startingDeadlineSeconds: If the CronJob misses its scheduled time by more than this many seconds, it skips the run. Useful when a Job takes longer than the interval -- prevents queueing dozens of missed runs. If unset, all missed runs may execute simultaneously.
4
History Limits: successfulJobsHistoryLimit (default 3) and failedJobsHistoryLimit (default 1) control how many completed/failed Job objects are retained. Set higher for debugging, lower for cluster cleanliness.

Interview Questions and Answers

Q1: What are the differences between Deployment, StatefulSet, and DaemonSet?
They serve fundamentally different scheduling and identity guarantees:
  • Deployment: For stateless apps. Pods are interchangeable, no stable identity, no persistent storage guarantees. Supports rolling updates and rollback. Pods can land on any node and are named randomly (e.g., web-api-6799fc88d8-x4k2p).
  • StatefulSet: For stateful apps. Provides stable network identity (pod-0, pod-1), stable persistent storage (dedicated PVC per pod), ordered deployment/scaling, and ordered rolling updates. Requires a Headless Service.
  • DaemonSet: Ensures exactly one pod per node (or subset of nodes). Used for node-level infrastructure (monitoring, logging, networking). Does not use the scheduler -- directly targets nodes. Pods are tied to specific nodes.
Q2: Describe the complete Pod lifecycle from creation to termination.
The lifecycle spans creation, scheduling, initialization, running, and termination:
  1. Pending: Pod accepted by API server, stored in etcd. Scheduler evaluates placement. Image pulling may begin.
  2. Init Containers: Run sequentially. Each must exit 0 before the next starts. Failure restarts the pod.
  3. Running: All containers started. Startup probes gate liveness/readiness. Readiness probe success adds pod to Service endpoints.
  4. Termination: Pod receives SIGTERM, preStop hook executes, containers get terminationGracePeriodSeconds (default 30s) to shut down. SIGKILL sent if not terminated. Pod removed from endpoints first (before SIGTERM) to stop receiving traffic.

Pod phase reflects the aggregate: a pod is Running if at least one container is running, regardless of other container states.

Q3: How does a rolling update work internally? What happens if a new pod fails readiness?
The Deployment controller orchestrates ReplicaSet scaling based on maxSurge and maxUnavailable:

During a rolling update, the Deployment creates a new ReplicaSet and incrementally scales it up while scaling the old one down. At each step, it ensures: total pods <= desired + maxSurge AND unavailable pods <= maxUnavailable.

If a new pod fails readiness: The rollout pauses naturally. Since the pod never becomes Ready, it does not count toward "available" replicas. The Deployment cannot scale up the new RS further (violating maxUnavailable) or scale down the old RS (reducing available below threshold). If this persists beyond progressDeadlineSeconds, the Deployment condition is set to Progressing=False, but NO automatic rollback occurs -- you must manually rollback or fix the issue.

Q4: What are init containers and when would you use them?
Init containers run to completion before app containers start, providing setup guarantees:
  • Dependency waiting: Block until a database, API, or message queue is reachable (avoids CrashLoopBackOff from connection failures at startup).
  • Configuration generation: Populate shared volumes with configs, secrets from Vault, or dynamic data that the app container reads.
  • Security setup: Run privileged operations (sysctl, iptables) that the app container should not have permission to do.
  • Database schema migration: Apply migrations before the app starts (though a separate Job is often better for this).

Key properties: run sequentially, can have different images/security contexts from app containers, must exit 0 for the pod to proceed, and share volumes with app containers.

Q5: Explain Kubernetes QoS (Quality of Service) classes and how they affect pod eviction.
QoS classes determine eviction priority when a node is under resource pressure:
  • Guaranteed: Every container has requests == limits for both CPU and memory. Highest priority -- evicted last. OOM score is -997. Use for critical workloads.
  • Burstable: At least one container has a request or limit set, but requests != limits. Medium priority. OOM score calculated proportionally to memory usage vs request.
  • BestEffort: No resource requests or limits set at all. Lowest priority -- evicted first when the node runs low on resources. OOM score is 1000.

Eviction order: When the node hits memory pressure, kubelet evicts BestEffort pods first, then Burstable pods exceeding their requests, then Guaranteed pods (only if truly necessary). Within a class, pods using more memory relative to their request are evicted first.

Q6: What is the difference between liveness, readiness, and startup probes?
Each probe serves a distinct purpose in managing container health:
  • Startup Probe: Protects slow-starting containers. Disables liveness and readiness checks until it succeeds. If it never succeeds (failureThreshold * periodSeconds), the container is killed. Use for apps with long initialization (JVM warmup, data loading).
  • Liveness Probe: Detects containers that are running but broken (deadlocks, infinite loops). Failure triggers container restart. Does NOT remove from Service endpoints -- that is readiness's job. Be careful not to probe external dependencies here (database down should not kill your pod).
  • Readiness Probe: Determines if the container can serve traffic. Failure removes the pod from Service endpoints (stops receiving requests). Does NOT restart the container. Use for temporary conditions (loading cache, downstream dependency unavailable).

Common mistake: Setting liveness == readiness. If your app temporarily cannot serve traffic (database failover), readiness should fail (stop traffic) but liveness should pass (don't restart the container).

Q7: How does Kubernetes handle pod termination? What is the graceful shutdown sequence?
Termination follows a precise sequence to prevent dropped requests:
  1. Pod state set to Terminating. Pod is removed from Service endpoints (kube-proxy updates rules). New requests stop routing to this pod.
  2. preStop hook executes (if defined). Common pattern: sleep 5 to allow in-flight requests to complete and LB to propagate endpoint removal.
  3. SIGTERM sent to PID 1 in each container. Application should catch this and begin graceful shutdown (close connections, finish in-flight work).
  4. Wait up to terminationGracePeriodSeconds (default 30s). The timer starts from step 2 (includes preStop time).
  5. SIGKILL sent to any remaining processes. Container is forcefully terminated.
  6. Pod object deleted from API server.

Key insight: There is a race condition between endpoint removal (step 1) and SIGTERM (step 3). Some LBs may still send requests after SIGTERM. The preStop sleep ensures the app stays alive long enough for endpoints to propagate.

Q8: When would you use the Recreate strategy instead of RollingUpdate?
Recreate kills all existing pods before creating new ones -- accepting downtime for correctness:
  • Database schema incompatibility: New code requires schema changes that break old code. You cannot have v1 and v2 running simultaneously against the same database.
  • Shared volume with single-writer: If pods write to a ReadWriteOnce volume, only one pod can attach at a time. RollingUpdate would fail because the new pod cannot mount the volume while the old one holds it.
  • License constraints: Software licensed per-instance where running extra instances during rollout violates licensing.
  • Singleton requirement: Application that must not have concurrent instances (legacy apps with global locks, certain batch processors).

The Recreate strategy is simple: scale old RS to 0, wait for termination, then scale new RS up. Downtime = termination time + startup time.

Q9: How does a StatefulSet handle node failure differently from a Deployment?
StatefulSets are deliberately conservative about rescheduling to prevent data corruption:

When a node becomes unreachable, the node controller marks it NotReady after ~40s. For Deployments, the controller creates a replacement pod immediately on a healthy node (the old pod may still be running on the partitioned node -- this is acceptable for stateless apps).

For StatefulSets, Kubernetes does NOT automatically delete the pod or create a replacement. Reason: the pod may still be running and writing to its PVC on the unreachable node. Creating a second pod with the same PVC could cause data corruption (split-brain). An operator must either:

  • Confirm the node is truly dead, then force-delete the pod: kubectl delete pod postgres-0 --force --grace-period=0
  • Or fence the node (cordon + drain) to ensure the old pod is truly gone before rescheduling.
Q10: What is the difference between spec.containers and spec.initContainers resource requests for scheduling?
The scheduler uses the maximum of init container requests vs sum of app container requests:

Kubernetes computes the effective resource request for scheduling as:

effectiveRequest = max(max(initContainer[i].request), sum(container[j].request))

This is because init containers run sequentially (only one at a time) while app containers run in parallel. So the node must have enough resources to handle whichever is larger: the heaviest init container running alone, or all app containers running together.

Example: If init container requests 2Gi memory and the two app containers request 512Mi each (1Gi total), the pod's effective memory request is 2Gi (the init container dominates). If the app containers requested 1.5Gi each (3Gi total), the effective request would be 3Gi.

Troubleshooting Scenarios

⚠ Scenario: CrashLoopBackOff

Symptom: Pod repeatedly crashes and enters CrashLoopBackOff with exponentially increasing restart delays (10s, 20s, 40s, ... 5min cap).

Debug Steps:

1. Check container logs: kubectl logs <pod> -c <container> --previous -- the --previous flag shows logs from the last crashed instance. This is your primary diagnostic tool for application-level errors.
2. Describe the pod for events: kubectl describe pod <pod> -- look at the Events section for "Back-off restarting failed container", check Last State for exit codes. Exit code 137 = OOMKilled, 1 = application error, 127 = command not found, 126 = permission denied.
3. Check if it is OOMKilled: kubectl get pod <pod> -o jsonpath='{.status.containerStatuses[*].lastState.terminated.reason}' -- if OOMKilled, increase memory limits or fix memory leak. Also check kubectl top pod for actual memory usage.
4. Verify command and args: kubectl get pod <pod> -o yaml | grep -A10 'containers:' -- check that command and args are correct. Common mistake: shell syntax in command array (use ["/bin/sh", "-c", "my command"] for shell features).
5. Debug with ephemeral container or override: kubectl debug <pod> -it --image=busybox --target=<container> or create a temporary pod with the same image but command: ["sleep", "infinity"] to investigate the filesystem, environment variables, and network connectivity without the application crashing.
⚠ Scenario: ImagePullBackOff

Symptom: Pod stuck in ImagePullBackOff or ErrImagePull. Container never starts.

Causes and Fixes:

1. Image does not exist: Typo in image name/tag. Verify with kubectl describe pod <pod> -- Events will show the exact pull error. Test manually: docker pull <image> or crictl pull <image> from the node.
2. Authentication failure: Private registry requires imagePullSecrets. Verify secret exists: kubectl get secret <name> -o jsonpath='{.data.\.dockerconfigjson}' | base64 -d. Ensure the secret is in the same namespace as the pod.
3. Network issues: Node cannot reach the registry. Check from node: curl -v https://<registry>/v2/. Could be DNS, proxy, firewall, or VPC endpoint misconfiguration. Check /etc/resolv.conf on the node.
4. Rate limiting: Docker Hub enforces pull rate limits (100 pulls/6h for anonymous, 200 for authenticated). Error message will mention "429 Too Many Requests". Fix: use imagePullSecrets with Docker Hub credentials or mirror images to a private registry.
5. Platform mismatch: Image built for wrong architecture (e.g., amd64 image on arm64 node). Error: "no matching manifest for linux/arm64". Fix: use multi-arch images or add nodeSelector for the correct architecture.
⚠ Scenario: Pod Pending (Resource Issues)

Symptom: Pod remains in Pending state. kubectl describe pod shows FailedScheduling event with "Insufficient cpu" or "Insufficient memory".

Debug Steps:

1. Check the scheduling failure reason: kubectl describe pod <pod> | grep -A5 "Events" -- the message tells you exactly which resource is insufficient and on how many nodes. Example: "0/5 nodes are available: 5 Insufficient memory."
2. Inspect node allocatable vs allocated: kubectl describe node <node> | grep -A20 "Allocated resources" -- compare Requests column to Allocatable. If requests total exceeds allocatable, the node is full regardless of actual usage.
3. Check for resource over-requesting: kubectl top nodes vs kubectl describe nodes | grep -A5 "Allocated" -- if actual usage is low but requests are high, pods are over-requesting. Reduce requests to match actual needs (use VPA recommendations).
4. Check ResourceQuota: kubectl get resourcequota -n <namespace> -- namespace quotas can block pod creation even if nodes have capacity. Check if quota for requests.cpu or requests.memory is exhausted.
5. Resolution options: Scale the cluster (add nodes), reduce pod resource requests, evict lower-priority pods (use PriorityClasses), enable cluster autoscaler, or check if pods can be spread across namespaces with available quota.
⚠ Scenario: OOMKilled

Symptom: Container terminated with reason "OOMKilled" (exit code 137). Pod may enter CrashLoopBackOff if it happens repeatedly.

Diagnosis:

1. Confirm OOM: kubectl describe pod <pod> -- look for Last State: Terminated, Reason: OOMKilled, Exit Code: 137. This means the container exceeded its memory limit and the kernel's OOM killer terminated it.
2. Check memory limit vs actual usage: kubectl top pod <pod> --containers -- compare with the memory limit in the pod spec. If usage consistently approaches the limit, the limit is too low. If it spikes suddenly, suspect a memory leak.
3. Check node-level OOM events: kubectl get events --field-selector reason=OOMKilling -n <namespace> and on the node: dmesg | grep -i "oom\|killed" -- confirms which process was killed and its memory usage at death.
4. Profile memory usage: For JVM apps, check heap settings (-Xmx should be ~75% of container memory limit to leave room for off-heap). For Go, check goroutine leaks. For Node.js, check --max-old-space-size.
5. Resolution: Increase memory limit (if legitimate usage), fix memory leak (if a bug), or tune application memory settings. Note: setting requests == limits gives Guaranteed QoS class, making the pod less likely to be evicted but still killed if it exceeds its own limit.
⚠ Scenario: Deployment Rollout Stuck

Symptom: kubectl rollout status shows "Waiting for deployment to finish: X out of Y new replicas have been updated..." and never completes. Deployment condition shows Progressing=False.

Debug Steps:

1. Check new ReplicaSet's pods: kubectl get rs -l app=<name> to find the new RS, then kubectl get pods -l pod-template-hash=<hash> -- check if new pods are stuck in Pending, CrashLoopBackOff, or never become Ready.
2. Check pod readiness probe: kubectl describe pod <new-pod> -- if readiness probe is failing, the pod never becomes Ready, the rollout cannot proceed (new pod does not count as available), and old pods cannot be scaled down.
3. Check Deployment events: kubectl describe deployment <name> -- look for "has timed out progressing" condition. Check if progressDeadlineSeconds has been exceeded. Check for quota or admission webhook rejections.
4. Check resource availability: If maxSurge > 0 and the cluster is full, new pods cannot be scheduled. The rollout stalls waiting for capacity. Either free resources, scale the cluster, or adjust the strategy.
5. Resolution: Fix the underlying issue (image, config, resources), then let the rollout continue. If unrecoverable, rollback: kubectl rollout undo deployment/<name>. To investigate without pressure, pause the rollout: kubectl rollout pause deployment/<name>.

Common YAML Mistakes

These are the most frequent YAML errors that cause deployments to fail silently, workloads to misbehave, or pods to never start.

Selector Mismatch

Wrong: Labels do not match selector

spec:
  selector:
    matchLabels:
      app: web-api         # Selector expects this
  template:
    metadata:
      labels:
        app: web-server    # Template has different label!
        # Result: Deployment creates pods
        # but cannot find them. ReplicaSet
        # shows 0/3 ready forever.

Correct: Labels match selector exactly

spec:
  selector:
    matchLabels:
      app: web-api         # Selector expects this
  template:
    metadata:
      labels:
        app: web-api       # Template matches!
        version: v2.1.0    # Extra labels are fine
        # The selector labels must be a SUBSET
        # of the template labels.

Resource Requests/Limits Format

Wrong: Incorrect resource units

resources:
  requests:
    cpu: 250              # No unit! Means 250 CORES
    memory: 512           # No unit! Means 512 BYTES
  limits:
    cpu: 1000m
    memory: 1g            # Lowercase 'g' = NOT gibibytes
    # 1g = 10^9 bytes (gigabyte)
    # 1Gi = 2^30 bytes (gibibyte)
    # Off by ~7% -- causes unexpected OOM

Correct: Proper resource units

resources:
  requests:
    cpu: 250m             # 250 millicores = 0.25 CPU
    memory: 512Mi         # 512 mebibytes
  limits:
    cpu: 1000m            # or "1" for 1 full core
    memory: 1Gi           # 1 gibibyte (uppercase Gi)
    # Always use explicit units:
    # CPU: m (millicores)
    # Memory: Mi, Gi (binary units)

Probe Configuration

Wrong: Liveness probe checks dependencies

livenessProbe:
  httpGet:
    path: /health         # This endpoint checks DB
    port: 8080            # connectivity!
  # Problem: If the database goes down,
  # ALL pods get killed simultaneously.
  # Restart doesn't fix the DB.
  # You now have a cascading failure
  # with all pods in CrashLoopBackOff.

Correct: Liveness only checks the process

livenessProbe:
  httpGet:
    path: /healthz/live   # Only checks: is the
    port: 8080            # process alive and
                          # responsive?
readinessProbe:
  httpGet:
    path: /healthz/ready  # THIS one checks
    port: 8080            # dependencies (DB, cache)
  # If DB is down: readiness fails (stop
  # sending traffic) but liveness passes
  # (don't restart the pod).

Container Command Syntax

Wrong: Shell syntax in command array

containers:
  - name: app
    command:
      - "echo $HOME && ls -la"
    # Fails! No shell interprets this.
    # The string is passed as argv[0] to
    # exec() directly. $HOME is literal,
    # && is not a shell operator here.
    # Error: "exec: not found"

Correct: Use explicit shell invocation

containers:
  - name: app
    command: ["/bin/sh", "-c"]
    args:
      - |
        echo $HOME && ls -la
    # Shell features (pipes, variables,
    # &&, ||, redirects) require a shell.
    # Use command for the binary,
    # args for the shell script.
    # Or: command: ["/app/binary", "--flag"]

Security Context

Wrong: Running as root with full capabilities

containers:
  - name: app
    image: myapp:latest
    # No securityContext at all!
    # Defaults:
    #   runAsUser: 0 (root)
    #   allowPrivilegeEscalation: true
    #   capabilities: ALL inherited
    # Container escape → full node access
    # This fails PodSecurity "restricted"

Correct: Minimal privileges

containers:
  - name: app
    image: myapp:v1.2.0
    securityContext:
      runAsNonRoot: true
      runAsUser: 1000
      allowPrivilegeEscalation: false
      readOnlyRootFilesystem: true
      capabilities:
        drop: ["ALL"]
    # Passes PodSecurity "restricted"
    # Minimal attack surface
    # Use emptyDir for writable /tmp

ImagePullPolicy with Latest Tag

Wrong: Mutable tags with IfNotPresent

containers:
  - name: app
    image: myapp:latest
    imagePullPolicy: IfNotPresent
    # Problem: "latest" is a mutable tag.
    # Node A has cached v1 (tagged latest)
    # Node B pulls fresh v2 (also latest)
    # You now have DIFFERENT versions running
    # with no way to tell which is which.
    # Rollbacks are impossible.

Correct: Immutable tags or digest

containers:
  - name: app
    image: myapp:v1.2.0          # Immutable tag
    imagePullPolicy: IfNotPresent
    # Or use digest for absolute immutability:
    # image: myapp@sha256:abc123...
    # Every node guaranteed same binary.
    # Rollbacks target exact version.
    # Clear audit trail of what ran.
