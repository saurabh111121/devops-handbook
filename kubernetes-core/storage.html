Kubernetes Storage

Comprehensive guide to Kubernetes storage: volumes, PersistentVolumes, StorageClasses, ConfigMaps, Secrets, and CSI -- from fundamentals to production patterns.

Volume Types

Kubernetes volumes solve the ephemeral nature of container filesystems. When a container crashes and restarts, its filesystem is reset. Volumes provide persistent (or shared) storage that outlives individual container restarts.

emptyDir

1
Temporary Shared Storage: An emptyDir volume is created when a Pod is assigned to a node and exists as long as that Pod runs on that node. It starts empty. All containers in the pod can read and write the same files. Data is lost when the pod is deleted (but survives container restarts).
2
Storage Medium: By default, backed by the node's filesystem (disk). Setting medium: Memory uses a tmpfs (RAM-backed), which is faster but counts against the container's memory limit and is lost on node restart.
3
Size Limits: You can set sizeLimit to cap the storage usage. For memory-backed emptyDir, exceeding the limit causes pod eviction. For disk-backed, it triggers the node's eviction manager when disk pressure thresholds are crossed.
4
Use Cases: Scratch space for computation, sharing files between init containers and app containers, caching layers, temporary data processing, sidecar log collection, or as a shared workspace between multi-container pods.
5
Lifecycle: Created when pod is scheduled to a node. Persists across container restarts within the same pod. Destroyed permanently when the pod is removed from the node (deletion, eviction, node failure).

hostPath

1
Direct Node Filesystem Access: A hostPath volume mounts a file or directory from the host node's filesystem into the pod. The data persists beyond pod lifetime (it lives on the node), but the pod is tied to that specific node.
2
Security Risk: hostPath is inherently dangerous -- it grants pods access to the node filesystem. A compromised container could access sensitive host files, escalate privileges, or interfere with other pods. Restricted by Pod Security Standards (Restricted profile blocks it entirely).
3
Type Validation: The type field validates the path: DirectoryOrCreate (creates if missing), Directory (must exist), FileOrCreate, File, Socket, CharDevice, BlockDevice. Always specify a type for safety.
4
Legitimate Use Cases: Running cAdvisor (needs /sys), running a node-level log collector (needs /var/log), DaemonSets that need access to node configuration, or single-node development clusters where PVs are overkill.
5
Pod Scheduling Implication: Pods using hostPath data are effectively pinned to a specific node (the data is local). Use nodeSelector or nodeAffinity to ensure the pod always schedules to the correct node, or the data will appear empty on other nodes.

projected

1
Multiple Sources in One Mount: A projected volume maps several volume sources (Secrets, ConfigMaps, Downward API, ServiceAccountToken) into a single directory. This avoids cluttering the pod spec with many separate volumeMounts to the same path.
2
ServiceAccountToken Projection: The recommended way to get bound, audience-scoped, time-limited service account tokens. Automatically rotated by kubelet. Replaces the legacy auto-mounted token (which never expired).
3
Downward API: Exposes pod metadata (labels, annotations, resource limits, pod name, namespace, node name) as files inside the container without requiring API server access.
4
Use Case: Bundling a TLS cert (from Secret), an app config (from ConfigMap), and the pod's own identity information (from Downward API) into a single /etc/app-config/ mount point for clean application configuration.
5
File Permissions: Each projected source can specify a mode (file permissions). Useful for secrets that should only be readable by the application user (e.g., mode 0400 for private keys).

PersistentVolumes and PersistentVolumeClaims

PV/PVC decouple storage provisioning from consumption. Administrators create PersistentVolumes (the actual storage), and developers request storage via PersistentVolumeClaims without knowing infrastructure details.

1
PersistentVolume (PV): A cluster-wide resource representing a piece of storage provisioned by an administrator or dynamically by a StorageClass. It has a lifecycle independent of any pod. Think of it as the "physical disk" abstraction.
2
PersistentVolumeClaim (PVC): A namespaced resource representing a user's request for storage. It specifies size, access mode, and optionally a StorageClass. The control plane binds a PVC to a suitable PV that meets the requirements.
3
Access Modes: ReadWriteOnce (RWO) -- mounted read-write by one node. ReadOnlyMany (ROX) -- mounted read-only by many nodes. ReadWriteMany (RWX) -- mounted read-write by many nodes. ReadWriteOncePod (RWOP) -- mounted read-write by exactly one pod (K8s 1.27+).
4
Reclaim Policies: Retain -- PV is kept after PVC deletion (manual cleanup required; data preserved). Delete -- PV and underlying storage are deleted when PVC is deleted (cloud volumes destroyed). Recycle (deprecated) -- basic scrub (rm -rf) before reuse.
5
Binding Process: The PV controller watches for unbound PVCs and matches them to available PVs based on: requested size (PV must be >= PVC), access mode, StorageClass, label selectors, and volume mode. Once bound, the relationship is one-to-one and exclusive.
6
Volume Modes: Filesystem (default) -- volume is mounted as a directory with a filesystem. Block -- volume is presented as a raw block device without a filesystem. Block mode is used for databases that manage their own storage layout (e.g., some distributed databases).
7
Storage Capacity Tracking: Since Kubernetes 1.24 (GA), CSI drivers can report their available capacity. The scheduler uses this information to avoid scheduling pods on nodes where the required storage cannot be provisioned, preventing PVC-Pending situations.
persistentvolume-nfs.yaml PV - NFS 
# PersistentVolume backed by NFS storage
# Cluster-wide resource -- no namespace
# Suitable for ReadWriteMany access (shared file storage)
apiVersion: v1
kind: PersistentVolume
metadata:
  name: nfs-pv-data
  labels:
    type: nfs
    environment: production
    team: platform
spec:
  capacity:
    storage: 100Gi                # Advertised capacity
  volumeMode: Filesystem          # Mount as directory (default)
  accessModes:
    - ReadWriteMany               # NFS supports multi-node read-write
  persistentVolumeReclaimPolicy: Retain   # Keep data after PVC deletion
  storageClassName: nfs-slow      # Must match PVC's storageClassName
  mountOptions:
    - hard                        # NFS mount options
    - nfsvers=4.1
    - rsize=1048576
    - wsize=1048576
    - timeo=600
    - retrans=2
  nfs:
    server: 10.0.1.50            # NFS server IP
    path: /exports/data          # Export path on the server

StorageClasses

StorageClasses enable dynamic provisioning -- when a PVC references a StorageClass, Kubernetes automatically creates the PV and underlying storage. No manual PV creation needed.

1
Dynamic Provisioning: Instead of pre-creating PVs, administrators define StorageClasses with a provisioner and parameters. When a PVC references the class, the provisioner automatically creates the storage backend (EBS volume, GCP PD, Azure Disk, etc.) and a corresponding PV.
2
Provisioner: The component that creates the actual storage. Examples: ebs.csi.aws.com (AWS EBS), pd.csi.storage.gke.io (GCP PD), disk.csi.azure.com (Azure), kubernetes.io/aws-ebs (legacy in-tree). Each provisioner supports different parameters.
3
Parameters: Provisioner-specific key-value pairs that configure the storage (volume type, IOPS, encryption, filesystem type). These are opaque to Kubernetes -- they are passed directly to the provisioner.
4
Volume Binding Mode: Immediate (default) -- PV is provisioned as soon as PVC is created. WaitForFirstConsumer -- provisioning is delayed until a pod using the PVC is scheduled, ensuring the volume is created in the correct availability zone.
5
Default StorageClass: One StorageClass can be marked as default (annotation storageclass.kubernetes.io/is-default-class: "true"). PVCs that do not specify a storageClassName use the default class. If no default exists and no class is specified, the PVC remains Pending.
6
Allowed Topologies: Restrict where volumes can be provisioned (specific zones or regions). Useful in multi-zone clusters to control data placement for compliance or performance reasons.
storageclass-aws-gp3.yaml StorageClass - AWS gp3 
# StorageClass for AWS EBS gp3 volumes with encryption
# Enables dynamic provisioning -- PVCs referencing this class
# automatically get an EBS volume created
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: gp3-encrypted
  annotations:
    # Set as the default StorageClass for the cluster
    storageclass.kubernetes.io/is-default-class: "true"
# CSI driver for AWS EBS (replaces legacy kubernetes.io/aws-ebs)
provisioner: ebs.csi.aws.com
# Parameters passed directly to the EBS CSI driver
parameters:
  type: gp3                          # Volume type: gp3 (general purpose SSD)
  iops: "3000"                       # Baseline IOPS (gp3 supports up to 16,000)
  throughput: "125"                  # Throughput in MiB/s (gp3 supports up to 1,000)
  encrypted: "true"                  # Enable encryption at rest
  kmsKeyId: "arn:aws:kms:us-east-1:123456789:key/abc-def-123"  # Custom KMS key
  fsType: ext4                       # Filesystem type (ext4, xfs)
# Reclaim policy for dynamically provisioned PVs
reclaimPolicy: Delete                # Delete EBS volume when PVC is deleted
# Wait for pod scheduling before provisioning (zone-aware)
volumeBindingMode: WaitForFirstConsumer
# Allow volume expansion (requires controller support)
allowVolumeExpansion: true
# Restrict provisioning to specific availability zones
allowedTopologies:
  - matchLabelExpressions:
      - key: topology.kubernetes.io/zone
        values:
          - us-east-1a
          - us-east-1b
          - us-east-1c
# Mount options applied to all PVs created by this class
mountOptions:
  - discard                          # Enable TRIM/discard for SSD
  - noatime                          # Don't update access time (performance)

ConfigMaps

ConfigMaps decouple configuration from container images, allowing you to change application behavior without rebuilding images. They store non-confidential key-value pairs that can be consumed as environment variables, command-line arguments, or configuration files.

1
Non-Confidential Configuration: ConfigMaps are for non-sensitive data (feature flags, database hostnames, log levels, config files). They are stored unencrypted in etcd and visible to anyone with API access to the namespace. Use Secrets for passwords/keys.
2
Size Limit: Maximum 1 MiB per ConfigMap (enforced by etcd's per-object size limit). For larger configuration, use a volume mount from a PVC or package config into the container image.
3
Creation Methods: From literal values (--from-literal=key=value), from files (--from-file=config.yaml), from directories (--from-file=./config-dir/), or from env files (--from-env-file=app.env).
4
Consumption Methods: As environment variables (individual keys or all keys at once via envFrom), as command arguments (referencing env vars), or mounted as files in a volume (each key becomes a filename, value becomes file content).
5
Hot-Reload Behavior: Volume-mounted ConfigMaps are automatically updated by kubelet (sync period ~60s + cache propagation). The update is atomic via symlink swap. However, env vars injected from ConfigMaps are NOT updated -- they require a pod restart. Applications must watch for file changes to benefit from hot-reload.
6
Immutable ConfigMaps: Setting immutable: true prevents modifications after creation. Benefits: protects against accidental changes, significantly improves cluster performance (kubelet stops polling the API server for updates on immutable ConfigMaps, reducing API server load in large clusters).
7
subPath Limitation: When using subPath to mount a single key as a file (without overwriting the entire directory), automatic updates do NOT work. The mounted file is a snapshot at mount time. Use symbolic links or mount the entire volume to get updates.
configmap-creation.sh kubectl commands 
# Create ConfigMap from literal key-value pairs
kubectl create configmap app-config \
  --from-literal=DATABASE_HOST=postgres.database.svc.cluster.local \
  --from-literal=DATABASE_PORT=5432 \
  --from-literal=LOG_LEVEL=info \
  --from-literal=CACHE_TTL=300 \
  -n production

# Create ConfigMap from a file (key = filename, value = file contents)
kubectl create configmap nginx-config \
  --from-file=nginx.conf=/path/to/nginx.conf \
  --from-file=mime.types=/path/to/mime.types \
  -n production

# Create ConfigMap from an entire directory
kubectl create configmap app-configs \
  --from-file=/path/to/config-directory/ \
  -n production

# Create ConfigMap from .env file (each line becomes a key=value pair)
kubectl create configmap env-config \
  --from-env-file=app.env \
  -n production

Secrets

Secrets hold sensitive data (passwords, tokens, keys, certificates). They are similar to ConfigMaps but with additional protections: base64-encoded (NOT encrypted by default), RBAC-restricted, and optionally encrypted at rest in etcd.

Critical Security Note

Base64 is NOT encryption. By default, Secrets are stored in etcd as base64-encoded plaintext. Anyone with etcd access or API read permissions can decode them trivially. You MUST enable encryption at rest (EncryptionConfiguration) and restrict RBAC access to Secret resources for actual security.

Secret Types

1
Opaque (default): Arbitrary user-defined key-value pairs. Most common type. Values must be base64-encoded in YAML (or use stringData for plain text that gets auto-encoded). Used for database passwords, API keys, application secrets.
2
kubernetes.io/tls: Stores a TLS certificate (tls.crt) and private key (tls.key). Validated by Kubernetes to ensure both fields are present and the cert is PEM-encoded. Used by Ingress resources for TLS termination.
3
kubernetes.io/dockerconfigjson: Stores Docker registry credentials for pulling private images. Contains a .dockerconfigjson key with registry auth configuration. Referenced via pod's imagePullSecrets.
4
kubernetes.io/basic-auth: Stores username and password fields for basic authentication. Used by some tools and controllers that need credentials in a standardized format.
5
kubernetes.io/ssh-auth: Stores an SSH private key in the ssh-privatekey field. Used for git-based workflows (Flux, ArgoCD) that need to authenticate to private repositories.
6
kubernetes.io/service-account-token: Legacy auto-generated token for service accounts. Since Kubernetes 1.24, these are no longer auto-created. Use projected volumes with bound tokens instead (time-limited, audience-scoped).

Creating and Using Secrets

secret-opaque.yaml Opaque Secret 
# Opaque Secret for application credentials
# stringData accepts plain text (auto-encoded to base64 on creation)
# data accepts pre-encoded base64 values
apiVersion: v1
kind: Secret
metadata:
  name: app-credentials
  namespace: production
  labels:
    app: backend-api
type: Opaque
# stringData: plain text (easier to write, auto-encoded on apply)
stringData:
  DATABASE_PASSWORD: "s3cur3-p@ssw0rd-h3re"
  API_SECRET_KEY: "ak_live_xyzABC123defGHI456"
  JWT_SIGNING_KEY: "your-256-bit-secret-key-here"
# data: base64-encoded (required for binary data)
# echo -n 'value' | base64
data:
  REDIS_PASSWORD: "cjNkMXMtcEBzc3cwcmQ="    # base64 of "r3d1s-p@ssw0rd"
---
# Pod using the Secret
apiVersion: v1
kind: Pod
metadata:
  name: backend-api
  namespace: production
spec:

Encryption at Rest

1
EncryptionConfiguration: Enable via kube-apiserver flag --encryption-provider-config. Defines which resources to encrypt and which provider to use (aescbc, aesgcm, secretbox, or a KMS provider). Secrets are encrypted before being written to etcd.
2
KMS Provider (Recommended): Integrates with external Key Management Services (AWS KMS, GCP KMS, Azure Key Vault, HashiCorp Vault). The encryption key is managed externally, providing better key rotation and audit trails. Supports envelope encryption.
3
Key Rotation: When rotating encryption keys, add the new key as the first provider (used for writes) and keep the old key as a fallback (for reading existing data). Then run kubectl get secrets --all-namespaces -o json | kubectl replace -f - to re-encrypt all secrets with the new key.

External Secrets Operator

1
External Secret Management: The External Secrets Operator (ESO) synchronizes secrets from external secret stores (AWS Secrets Manager, HashiCorp Vault, Azure Key Vault, GCP Secret Manager) into Kubernetes Secrets. This keeps the source of truth outside the cluster.
2
ExternalSecret Resource: You define an ExternalSecret that specifies which secrets to fetch from which external store. The operator creates and maintains the corresponding Kubernetes Secret, refreshing it on a configurable interval.
3
Benefits: Secrets never stored in git. Centralized rotation (update in Vault, K8s Secret auto-updates). Audit trails via the external provider. Multi-cluster consistency (same source, multiple clusters). Separation of duties (security team manages secrets, dev team references them).

Container Storage Interface (CSI)

CSI is the standard interface between Kubernetes and storage systems. It replaced in-tree volume plugins, allowing storage vendors to develop and release drivers independently of the Kubernetes release cycle.

1
What is CSI: A specification defining a standard API for container orchestrators to interact with storage systems. Kubernetes communicates with CSI drivers via gRPC. Each driver implements three services: Identity, Controller (provisioning, attach/detach), and Node (mount/unmount on the node).
2
Why CSI: Before CSI, storage plugins were compiled into Kubernetes ("in-tree"). Adding or updating a storage backend required changing core Kubernetes code and waiting for a release. CSI decouples storage development from Kubernetes releases, enabling faster innovation and reducing core code bloat.
3
Architecture: A CSI driver consists of: (1) A Controller plugin (Deployment) that handles volume create/delete/attach/detach/snapshot. (2) A Node plugin (DaemonSet) that handles volume mount/unmount/format on each node. (3) Sidecar containers (external-provisioner, external-attacher, external-snapshotter) that bridge Kubernetes API objects to CSI gRPC calls.
4
Common CSI Drivers: AWS EBS CSI (ebs.csi.aws.com), AWS EFS CSI (efs.csi.aws.com), GCP PD CSI (pd.csi.storage.gke.io), Azure Disk CSI (disk.csi.azure.com), Ceph CSI (rbd.csi.ceph.com), Longhorn, OpenEBS, Portworx.
5
CSI Migration: Kubernetes is migrating all in-tree volume plugins (awsElasticBlockStore, gcePersistentDisk, azureDisk) to their CSI equivalents. Existing PVs using in-tree plugins continue to work via a translation layer that redirects calls to the CSI driver.

Ephemeral Volumes (CSI)

1
CSI Ephemeral Volumes: Inline volumes defined directly in the pod spec (no separate PVC). Created when the pod starts, destroyed when the pod terminates. Useful for injecting secrets from external stores, providing per-pod scratch storage with specific characteristics, or attaching node-local storage.
2
Generic Ephemeral Volumes: Like CSI ephemeral volumes but use standard PVC semantics under the hood. Kubernetes auto-creates a PVC (named <pod-name>-<volume-name>) that is bound to the pod's lifecycle. Supports all PVC features (StorageClass, access modes, snapshots) but with automatic cleanup.
3
Use Cases: HashiCorp Vault CSI (inject secrets as files without PVC management), AWS Secrets Manager CSI, local SSDs for batch jobs, temporary high-performance storage for ML training, and testing scenarios where cleanup must be automatic.

Interview Questions and Answers

Q1: What is the difference between a PersistentVolume and a PersistentVolumeClaim?
PV is the storage resource, PVC is the request for storage. A PersistentVolume (PV) is a cluster-wide resource representing actual provisioned storage (an EBS volume, NFS export, etc.) -- think of it as the "supply." A PersistentVolumeClaim (PVC) is a namespaced resource representing a developer's request for storage with specific characteristics (size, access mode) -- think of it as the "demand." The PV controller binds matching PVs to PVCs. This separation allows administrators to manage storage infrastructure independently from developers who consume it.
Q2: Are Kubernetes Secrets actually secure? How would you properly secure them?
Not by default -- base64 encoding is trivially reversible. To properly secure Secrets: (1) Enable encryption at rest via EncryptionConfiguration with a KMS provider (AWS KMS, Vault). (2) Restrict RBAC -- only pods/service accounts that need a Secret should have access. Avoid granting "get secrets" at the namespace level. (3) Use External Secrets Operator to sync from Vault/AWS Secrets Manager (secrets never in git). (4) Enable audit logging to track who accesses secrets. (5) Use short-lived tokens with projected volumes instead of static secrets. (6) Consider Sealed Secrets for GitOps (encrypted in git, decrypted in-cluster). (7) Limit etcd access to only the API server with mutual TLS.
Q3: Explain dynamic provisioning and why WaitForFirstConsumer volume binding mode is important.
Dynamic provisioning auto-creates storage; WaitForFirstConsumer ensures it is in the right zone. With dynamic provisioning, you only define a StorageClass and PVCs. The CSI driver automatically creates the underlying storage (e.g., AWS EBS volume) when a PVC is created. The problem with Immediate binding: the volume might be created in zone us-east-1a, but the pod gets scheduled to us-east-1b (where the volume is unreachable). WaitForFirstConsumer delays provisioning until a pod using the PVC is scheduled, so the volume is created in the same zone as the pod. This is critical for zone-restricted storage like EBS.
Q4: What are the PV access modes and which storage types support which modes?
RWO, ROX, RWX, and RWOP -- support depends on the underlying storage technology. ReadWriteOnce (RWO): mounted read-write by pods on a single node. Supported by: EBS, GCP PD, Azure Disk, iSCSI, most block storage. ReadOnlyMany (ROX): mounted read-only by pods on many nodes. Supported by: NFS, CephFS, AzureFile. ReadWriteMany (RWX): mounted read-write by pods on many nodes. Supported by: NFS, EFS, CephFS, GlusterFS, AzureFile. ReadWriteOncePod (RWOP): K8s 1.27+, exactly one pod can mount it. Provides stronger guarantees than RWO (which is per-node, not per-pod). Note: access mode is a constraint, not enforcement -- it determines scheduling eligibility.
Q5: What is the difference between ConfigMaps and Secrets? When would you use each?
ConfigMaps for non-sensitive config, Secrets for sensitive data -- with different handling by the system. Similarities: both store key-value pairs, both can be consumed as env vars or volume mounts, both are namespace-scoped. Differences: Secrets are base64-encoded (not plaintext), can be encrypted at rest, have size limits (1MB vs 1MB), get special RBAC treatment (can restrict access separately), are stored in tmpfs when mounted (never written to node disk), and the kubelet limits which pods can access which secrets. Use ConfigMaps for: application settings, feature flags, connection strings (non-secret), config files. Use Secrets for: passwords, API keys, TLS certificates, SSH keys, OAuth tokens.
Q6: How does a ConfigMap hot-reload work and what are its limitations?
Kubelet periodically syncs volume-mounted ConfigMaps via atomic symlink swaps, but with notable caveats. Process: Kubelet watches for ConfigMap changes (sync period ~60s). When updated, it writes new content to a timestamped directory, then atomically swaps the symlink (..data → new directory). Applications see the update within the kubelet sync period + cache propagation (total ~60-90s). Limitations: (1) Env vars from ConfigMaps are NEVER updated without pod restart. (2) subPath mounts are never updated (snapshot at mount time). (3) Applications must actively watch for file changes (inotify/fsnotify) to react. (4) The update is not truly instant -- there is a window where different pods see different versions. (5) Immutable ConfigMaps disable this entirely.
Q7: What happens to data when a PVC is deleted? How do reclaim policies work?
The reclaim policy determines whether data is kept, deleted, or recycled. When a PVC is deleted: (1) Retain: The PV moves to "Released" state. The underlying storage (EBS volume, NFS export) is preserved with all data intact. An admin must manually clean up the PV and storage (or rebind it). Used for critical data that cannot be lost. (2) Delete: The PV and its underlying storage are automatically destroyed. The EBS volume is terminated, the GCP PD is deleted. Used for ephemeral data where storage cost matters. (3) Recycle (deprecated): A basic rm -rf /volume/* is performed, then the PV becomes Available again. Replaced by dynamic provisioning. Best practice: Use Retain for production databases, Delete for development/testing, and VolumeSnapshots as backup before any deletion.
Q8: How would you handle storage for a StatefulSet that needs both high IOPS and data durability?
Use volumeClaimTemplates with a StorageClass configured for high-performance storage, combined with snapshots for durability. Setup: (1) Create a StorageClass with provisioner-specific params (gp3 with 16000 IOPS, io2 for consistent performance, or local NVMe with a topology constraint). (2) Define volumeClaimTemplates in the StatefulSet -- each replica gets its own PVC (database-0, database-1, etc.) that persists across pod rescheduling. (3) Set volumeBindingMode: WaitForFirstConsumer for zone-aware placement. (4) For durability: schedule periodic VolumeSnapshots via a CronJob or Velero, replicate at the application level (database replication), and use reclaimPolicy: Retain to prevent accidental deletion. (5) Monitor with CSI driver metrics for IOPS throttling and latency.

Troubleshooting Scenarios

⚠ Scenario: PVC Stuck in Pending State

Symptom: PersistentVolumeClaim remains in Pending state. Pods referencing it cannot start.

Root Causes and Diagnostics:

1. Check PVC events: kubectl describe pvc <pvc-name>. Look at Events for specific error messages like "no persistent volumes available" or "waiting for first consumer".
2. Verify StorageClass exists: kubectl get storageclass. If the PVC references a non-existent StorageClass, it will remain Pending forever. Check for typos in storageClassName.
3. Check provisioner is running: kubectl get pods -n kube-system | grep csi or kubectl get pods -n <csi-namespace>. If the CSI driver pods are crashed or missing, dynamic provisioning cannot work.
4. WaitForFirstConsumer: If using this binding mode, the PVC stays Pending until a pod is scheduled that references it. This is normal behavior -- create the pod and the PVC will bind. Check with kubectl get pvc -o yaml | grep volumeBindingMode.
5. Capacity and access mode mismatch: For static provisioning, verify a PV exists that satisfies the PVC's size requirement (PV >= PVC) AND access modes. Use kubectl get pv to list available PVs and compare.
6. Zone/topology constraints: EBS volumes are zone-specific. If no nodes are available in the zone where the PV exists, the PVC cannot bind. Check kubectl get pv -o yaml | grep -A5 nodeAffinity.
⚠ Scenario: Volume Mount Permission Denied

Symptom: Container starts but the application cannot read/write files in the mounted volume. Errors like "permission denied" or "read-only file system" in logs.

Root Causes and Diagnostics:

1. Check SecurityContext fsGroup: kubectl get pod -o yaml | grep -A5 securityContext. When a volume is mounted, the owning group is set to fsGroup. If your container runs as a non-root user (e.g., UID 1000), set fsGroup: 1000 in the pod's securityContext to ensure group write access.
2. Verify readOnly flag: Check if the volumeMount has readOnly: true set (or if the PVC is ROX). If the app needs to write, change to readOnly: false and ensure the access mode supports writing.
3. Init container for permissions: Use an init container running as root to fix permissions: initContainers: [{name: fix-perms, image: busybox, command: ["sh", "-c", "chown -R 1000:1000 /data"], volumeMounts: [...]}].
4. Check SELinux/AppArmor: On nodes with SELinux enabled, volume mounts may have incorrect labels. Use seLinuxOptions in securityContext or set the volume mount's seLinuxRelabel field to allow relabeling.
5. NFS root squash: For NFS volumes, the server may have root_squash enabled, mapping root (UID 0) to nobody. Ensure the container user's UID/GID matches the NFS export permissions, or configure no_root_squash on the NFS server (security trade-off).
⚠ Scenario: ConfigMap Changes Not Reflected in Pod

Symptom: You updated a ConfigMap but the application still uses the old values. No errors visible.

Root Causes and Diagnostics:

1. Check consumption method: kubectl get pod -o yaml | grep -A10 env. If ConfigMap is injected as environment variables (envFrom or valueFrom), updates NEVER propagate -- a pod restart is required. This is the most common misunderstanding.
2. Check for subPath mount: kubectl get pod -o yaml | grep subPath. Volumes mounted with subPath do NOT receive automatic updates. Remove subPath and mount the entire ConfigMap as a directory, or use a projected volume.
3. Wait for sync period: Volume-mounted ConfigMaps take 60-90 seconds to propagate (kubelet sync period + ConfigMap cache TTL). Check the timestamp of the mounted file: kubectl exec <pod> -- ls -la /path/to/config/..data (the symlink target shows the update time).
4. Application not watching for changes: Even after the file is updated, the application must detect and reload it. Many apps read config once at startup. Solutions: implement file watching (fsnotify), use a sidecar (configmap-reload by jimmidyson), or use rolling restart (kubectl rollout restart deployment).
5. Immutable ConfigMap: Check kubectl get cm <name> -o yaml | grep immutable. If set to true, updates are rejected entirely. You must create a new ConfigMap with a different name and update the pod spec to reference it.
⚠ Scenario: Secret Accidentally Committed to Git

Symptom: A Kubernetes Secret YAML (with base64-encoded credentials) was committed and pushed to a git repository. The credentials may be compromised.

Immediate Response and Remediation:

1. Rotate credentials immediately: Assume the secret is compromised. Change the password/key/token in the source system (database, API provider, cloud console). Do this BEFORE anything else -- time is critical.
2. Update the Kubernetes Secret: kubectl create secret generic <name> --from-literal=key=NEW_VALUE --dry-run=client -o yaml | kubectl apply -f -. Restart affected pods to pick up the new credentials.
3. Remove from git history: Simply deleting the file in a new commit is NOT enough -- the secret remains in git history. Use git filter-branch or BFG Repo-Cleaner to purge from all branches/history. Force-push the cleaned history.
4. Prevent recurrence: Implement pre-commit hooks (detect-secrets, gitleaks, trufflehog) that scan for high-entropy strings and known secret patterns. Add Secret YAML patterns to .gitignore. Use Sealed Secrets or External Secrets Operator so raw secrets never exist in git.
5. Audit access: Check git hosting platform (GitHub/GitLab) audit logs to see who accessed the repository after the commit. Review cloud audit logs (CloudTrail, etc.) for suspicious activity using the compromised credentials.
