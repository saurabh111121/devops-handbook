Kubernetes Observability

Health probes, logging architecture, monitoring with Prometheus, alerting, events, and distributed tracing

1. Health Probes

Probe Types Comparison

ProbePurposeFailure ActionWhen to Use
livenessProbe Is the container alive? Container is killed and restarted Detect deadlocks, unrecoverable states
readinessProbe Is the container ready to serve traffic? Pod removed from Service endpoints Warmup time, dependency checks, load shedding
startupProbe Has the container started successfully? Container is killed and restarted (after failureThreshold) Slow-starting applications (replaces liveness during startup)

Key Differences:

  • Liveness kills the container; readiness only removes it from the Service (no restart)
  • Startup probe disables liveness and readiness probes until it succeeds
  • A failing readiness probe means the pod stays running but receives no traffic
  • A failing liveness probe causes a restart (CrashLoopBackOff if repeated)

Probe Mechanisms

MechanismHow It WorksSuccess Criteria
httpGetHTTP GET to specified path and portStatus code 200-399
tcpSocketTCP connection to specified portConnection established
execExecutes a command inside the containerExit code 0
grpcgRPC health check on specified portSERVING status returned

HTTP Probe Example

probes-http.yaml All three probes 
apiVersion: v1
kind: Pod
metadata:
  name: web-app
spec:
  containers:
  - name: app
    image: myapp:v2
    ports:
    - containerPort: 8080
    # Startup probe - gives app up to 5min (30 * 10s) to start
    startupProbe:
      httpGet:
        path: /healthz/started
        port: 8080
      initialDelaySeconds: 5
      periodSeconds: 10
      failureThreshold: 30
      successThreshold: 1
    # Liveness probe - enabled after startup succeeds
    livenessProbe:
      httpGet:
        path: /healthz/live
        port: 8080
        httpHeaders:
        - name: X-Health-Check
          value: "liveness"
      initialDelaySeconds: 0
      periodSeconds: 10
      timeoutSeconds: 3
      failureThreshold: 3
      successThreshold: 1
    # Readiness probe - controls Service membership
    readinessProbe:
      httpGet:
        path: /healthz/ready
        port: 8080
      initialDelaySeconds: 0
      periodSeconds: 5
      timeoutSeconds: 2
      failureThreshold: 3
      successThreshold: 1

EFK vs Loki Stack

AspectEFK (Elasticsearch + Fluentd + Kibana)Loki (Promtail + Loki + Grafana)
IndexingFull-text index on all fieldsOnly indexes labels (metadata)
StorageHigh (indexes everything)Low (stores compressed chunks)
QueryPowerful full-text search (Lucene)LogQL (label-based, then grep)
CostExpensive (RAM-heavy)Cheap (runs on object storage)
Best ForComplex search, analytics, complianceK8s-native, cost-effective, Grafana users
CollectorFluentd / Fluent BitPromtail / Grafana Agent

Structured Logging Best Practices

  • Always use JSON format for application logs (machine-parseable)
  • Include: timestamp, level, message, request_id, user_id, service name
  • Use consistent field names across all services
  • Include correlation IDs (trace_id) for distributed tracing
  • Avoid logging sensitive data (PII, credentials)
  • Use log levels appropriately: DEBUG (dev only), INFO (normal ops), WARN (degraded), ERROR (failures)
  • Set resource limits on log collectors to prevent node pressure

PrometheusRule (Alert Rules)

prometheus-rules.yaml Alert rules 
apiVersion: monitoring.coreos.com/v1
kind: PrometheusRule
metadata:
  name: kubernetes-alerts
  namespace: monitoring
  labels:
    release: prometheus
spec:
  groups:
  - name: kubernetes.pod.rules
    rules:
    # Pod CrashLooping
    - alert: PodCrashLooping
      expr: |
        rate(kube_pod_container_status_restarts_total[15m]) * 60 * 5 > 0
      for: 15m
      labels:
        severity: warning
      annotations:
        summary: "Pod {{ $labels.namespace }}/{{ $labels.pod }} is crash looping"
        description: "Pod has restarted {{ $value | humanize }} times in last 15m"
        runbook_url: "https://runbooks.example.com/pod-crashloop"

    # Pod not ready
    - alert: PodNotReady
      expr: |
        kube_pod_status_ready{condition="true"} == 0
      for: 5m
      labels:
        severity: warning
      annotations:
        summary: "Pod {{ $labels.namespace }}/{{ $labels.pod }} not ready"

    # High memory usage
    - alert: ContainerMemoryHigh
      expr: |
        container_memory_working_set_bytes / container_spec_memory_limit_bytes > 0.9
      for: 5m
      labels:
        severity: warning
      annotations:
        summary: "Container {{ $labels.container }} in {{ $labels.pod }} using >90% memory"

  - name: kubernetes.node.rules
    rules:
    # Node not ready
    - alert: NodeNotReady
      expr: kube_node_status_condition{condition="Ready",status="true"} == 0
      for: 5m
      labels:
        severity: critical
      annotations:
        summary: "Node {{ $labels.node }} is NotReady"

    # Node disk pressure
    - alert: NodeDiskPressure
      expr: kube_node_status_condition{condition="DiskPressure",status="true"} == 1
      for: 2m
      labels:
        severity: critical
      annotations:
        summary: "Node {{ $labels.node }} has disk pressure"

  - name: slo.burn.rate
    rules:
    # Burn-rate alert (Multi-window, multi-burn-rate)
    - alert: ErrorBudgetBurnHigh
      expr: |
        (
          job:slo_errors_per_request:ratio_rate1h{job="web-api"} > (14.4 * 0.001)
          and
          job:slo_errors_per_request:ratio_rate5m{job="web-api"} > (14.4 * 0.001)
        )
      for: 2m
      labels:
        severity: critical
        slo: "availability"
      annotations:
        summary: "High error budget burn rate for {{ $labels.job }}"
        description: "Burning error budget 14.4x faster than sustainable. Will exhaust in ~1h."

    - alert: ErrorBudgetBurnSlow
      expr: |
        (
          job:slo_errors_per_request:ratio_rate6h{job="web-api"} > (1 * 0.001)
          and
          job:slo_errors_per_request:ratio_rate30m{job="web-api"} > (1 * 0.001)
        )
      for: 15m
      labels:
        severity: warning
        slo: "availability"
      annotations:
        summary: "Slow error budget burn for {{ $labels.job }}"
        description: "Burning error budget 1x faster than sustainable. Will exhaust in ~30 days."

Burn-Rate Alerts Explained

Multi-window, multi-burn-rate alerts balance detection speed with false-positive rate:

  • Fast burn (14.4x) - Consumes monthly budget in ~1 hour. Use short windows (1h + 5m). Page immediately.
  • Medium burn (6x) - Consumes budget in ~5 hours. Windows: 6h + 30m. Alert with urgency.
  • Slow burn (1x) - Consumes budget at sustainable pace over 30 days. Windows: 3d + 6h. Ticket/low priority.

Two windows per alert prevent alerting on very brief spikes (short window) while ensuring recent relevance (long window).

5. Kubernetes Events

Events are API objects that record state changes and important occurrences. They provide insight into what is happening inside the cluster without needing external monitoring.

Watching Events

kubectl get events --sort-by='.lastTimestamp' -n production
kubectl get events --field-selector type=Warning -A
kubectl get events --field-selector involvedObject.kind=Pod,involvedObject.name=my-pod
kubectl get events -w # Watch mode (stream new events)

Event Lifecycle

1
TTL - Events have a default TTL of 1 hour. After that, they are garbage collected. For persistence, export events to a logging system.
2
Deduplication - Repeated identical events are aggregated (count incremented, lastTimestamp updated) instead of creating new objects.
3
Types - Normal (informational) and Warning (something may be wrong).

6. Distributed Tracing

Distributed tracing follows a request as it traverses multiple services, providing visibility into latency, errors, and dependencies across the system.

Core Concepts

Trace

The entire journey of a request through the system. Contains multiple spans. Identified by a unique trace_id.

Span

A single operation within a trace (e.g., HTTP call, DB query). Has start time, duration, tags, and parent span reference.

7. Interview Q&A

Q: What is the difference between a liveness probe and a readiness probe?

Liveness probe: Determines if the container is alive. If it fails, kubelet kills and restarts the container. Use to detect deadlocks or unrecoverable states.

Readiness probe: Determines if the container can serve traffic. If it fails, the pod is removed from Service endpoints but NOT restarted. Use during startup, warmup, or when temporarily unable to serve (e.g., loading cache, waiting for dependency).

Common mistake: Using the same endpoint for both. Liveness should check "am I alive?" (simple), readiness should check "can I handle requests?" (may include dependency checks).

Q: How would you set up monitoring for a Kubernetes cluster from scratch?

Step-by-step approach:

  1. Deploy kube-prometheus-stack (Helm chart) - includes Prometheus, Alertmanager, Grafana, node-exporter, kube-state-metrics
  2. Configure ServiceMonitors for each application to define scrape targets
  3. Set up Grafana dashboards - import community dashboards for cluster, node, pod metrics
  4. Define PrometheusRules for alerts (pod crashes, node issues, SLO violations)
  5. Configure Alertmanager routing to Slack/PagerDuty based on severity
  6. Add long-term storage (Thanos/Cortex) if needed for retention beyond local TSDB
  7. Deploy logging (Loki + Promtail) and tracing (Tempo + OTel) for full observability

Q: Explain the USE and RED methods for monitoring.

USE Method (for infrastructure/resources - by Brendan Gregg):

  • Utilization - Percentage of time the resource is busy (CPU %, disk I/O %)
  • Saturation - Amount of queued work (CPU run queue, disk queue depth)
  • Errors - Count of error events (disk errors, network drops)

RED Method (for services/requests - by Tom Wilkie):

  • Rate - Requests per second
  • Errors - Error rate (failed requests / total requests)
  • Duration - Latency distribution (p50, p95, p99)

When to use: USE for nodes/infrastructure, RED for microservices/APIs. Together they provide full coverage.

Q: What is a burn-rate alert and why is it better than threshold-based alerting?

Burn-rate alerts measure how fast you are consuming your error budget relative to the budget period (usually 30 days).

Why better than thresholds:

  • Context-aware: A 1% error rate might be fine at low traffic but critical at high traffic. Burn rate considers the impact on SLO.
  • Reduced alert fatigue: Brief spikes that don't materially impact the SLO don't trigger alerts
  • Actionable: "You will exhaust your budget in 2 hours" is more actionable than "error rate is 2%"
  • Multi-window: Using both long and short windows prevents false positives while maintaining detection speed

Q: How do you handle alert fatigue in a Kubernetes environment?

Strategies to reduce alert fatigue:

  • SLO-based alerting: Only alert when error budget is being consumed too fast (burn-rate alerts)
  • Proper severity levels: Critical = page (customer impact), Warning = ticket (degraded), Info = dashboard only
  • Inhibition rules: Suppress downstream alerts when root cause is known (e.g., suppress pod alerts when node is down)
  • Grouping: Aggregate related alerts into a single notification
  • For/duration: Add for: 5m to prevent alerting on transient blips
  • Regular review: Delete alerts that are never actioned, tune noisy ones
  • Runbooks: Every alert should have a runbook_url with diagnosis steps

Q: What is OpenTelemetry and how does it relate to Jaeger/Zipkin?

OpenTelemetry (OTel) is a vendor-neutral observability framework that provides APIs, SDKs, and tools for generating and collecting telemetry data (traces, metrics, logs).

Relationship to Jaeger/Zipkin:

  • OTel is the instrumentation and collection layer (generates and ships data)
  • Jaeger/Zipkin/Tempo are backends (store and query traces)
  • OTel replaces vendor-specific SDKs (Jaeger client, Zipkin client) with one standard
  • OTel Collector can receive data in multiple formats and export to any backend

Think of OTel as the "universal adapter" - instrument once, export anywhere.

Q: How do Kubernetes events differ from logs and metrics?

Events: Kubernetes API objects recording cluster state changes. Short-lived (1h TTL), generated by controllers/kubelet. Great for understanding "what happened" (scheduling, scaling, probe failures).

Logs: Application output (stdout/stderr). Long-lived (stored in logging backend). Detailed application behavior and debugging info.

Metrics: Numeric time-series data. Aggregated, queryable, and alertable. Best for dashboards, trends, and automated alerting.

Together: Metrics tell you something is wrong, events tell you what Kubernetes did about it, logs tell you why the application failed.

Q: What is the startup probe and when should you use it instead of increasing liveness probe initialDelaySeconds?

Startup probe provides a dedicated mechanism for slow-starting containers. While it runs, liveness and readiness probes are disabled.

Why not just increase initialDelaySeconds on liveness?

  • A high initialDelaySeconds means if the app crashes after startup, detection is delayed by that same amount on restart
  • Startup probe allows: generous startup time (e.g., 5 minutes) + fast liveness detection after startup (e.g., 10 seconds)
  • Example: startupProbe.failureThreshold: 30 with periodSeconds: 10 = 5 min max startup, then livenessProbe.periodSeconds: 10 detects failures quickly

8. Real-World Scenarios

⚠ Scenario: Readiness Probe Failing - Pod Removed from Service

Symptom: Users report intermittent 503 errors. Some pods are Running but READY shows 0/1. Service has fewer endpoints than expected.

1. Check pod status: kubectl get pods -l app=web -o wide - look for pods with READY 0/1
2. Describe the pod: kubectl describe pod web-xyz - check Events for "Readiness probe failed"
3. Check endpoints: kubectl get endpoints web-service - verify which pods are in the endpoint list
4. Test the readiness endpoint manually: kubectl exec web-xyz -- curl -s localhost:8080/healthz/ready
5. Check application logs: kubectl logs web-xyz - look for dependency failures (DB, cache)

Common Causes: Database connection pool exhausted, downstream service unavailable, application overloaded (increase failureThreshold or adjust probe endpoint logic).

⚠ Scenario: Liveness Probe Restart Loop (CrashLoopBackOff)

Symptom: Pod restarts every 30 seconds. Events show "Liveness probe failed" followed by "Killing container." RESTARTS count climbing.

1. Check events: kubectl describe pod app-xyz - note the probe failure message
2. Check if app needs more startup time: compare initialDelaySeconds with actual startup duration
3. Add a startup probe if the app is slow to start (separates startup from ongoing liveness)
4. Verify the liveness endpoint is correct: wrong path or port causes perpetual failure
5. Check resource limits: OOM kills can look like liveness failures (container dies before probe runs)
6. Check if liveness probe does too much: liveness should NOT check external dependencies

Fix: Add startup probe, increase failureThreshold, ensure liveness is a simple "am I alive?" check (not a full health check including dependencies).

⚠ Scenario: Missing Metrics - Prometheus Not Scraping

Symptom: Grafana dashboard shows "No Data" for application metrics. Prometheus targets page shows the target as DOWN or missing.

1. Check Prometheus targets UI: http://prometheus:9090/targets - is the target listed? What error?
2. Verify ServiceMonitor labels match Prometheus operator's serviceMonitorSelector
3. Check that the Service has the correct labels matching the ServiceMonitor's selector
4. Verify the metrics port name in Service matches the ServiceMonitor's endpoints[].port
5. Test metrics endpoint directly: kubectl port-forward pod/app-xyz 8080 then curl localhost:8080/metrics
6. Check RBAC: Prometheus ServiceAccount needs permissions to discover services in target namespaces

Root Cause: Usually label mismatch between ServiceMonitor and Service, wrong port name, or namespace not included in namespaceSelector.

⚠ Scenario: Alert Fatigue - Too Many Non-Actionable Alerts

Symptom: On-call engineer receives 50+ alerts per shift. Most are ignored. Critical alerts get lost in the noise. Team morale dropping.

1. Audit all alerts: categorize as actionable vs informational over 2 weeks
2. Delete or downgrade alerts that never lead to action (move to dashboards)
3. Add for: durations to prevent alerting on transient spikes (5m minimum for warnings)
4. Implement inhibition rules: if node is down, suppress all pod alerts on that node
5. Switch to SLO-based (burn-rate) alerts instead of raw metric thresholds
6. Require every alert to have a runbook_url annotation before merging
7. Set up proper severity routing: only critical pages, warnings go to Slack/ticket

Goal: Every page should require human action within 5 minutes. If you can wait, it is not a page.
