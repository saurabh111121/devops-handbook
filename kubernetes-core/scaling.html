Kubernetes Scaling & Scheduling

HPA, VPA, Cluster Autoscaler, KEDA, node affinity, taints, tolerations, topology spread, resource management, and priority

1. Horizontal Pod Autoscaler (HPA)

How HPA Works

The HPA controller runs in a control loop (default 15s) that queries the metrics API, computes desired replicas using a formula, and scales the target workload.

1
Metrics Server - Collects resource metrics (CPU/memory) from kubelets. Required for resource-based autoscaling. Exposes metrics via metrics.k8s.io API.
2
Scaling Algorithm - desiredReplicas = ceil(currentReplicas * (currentMetricValue / desiredMetricValue))
3
Cooldown / Stabilization - Prevents flapping. Default: scale-up stabilization = 0s (immediate), scale-down stabilization = 300s (5 min). Configurable via behavior field.
4
Tolerance - Default tolerance is 10%. If the ratio is within 0.9-1.1, no scaling occurs.

HPA Decision Flow

HPA Control Loop
 Every 15 seconds (--horizontal-pod-autoscaler-sync-period)
 ============================================================

 +-------------------+     +-------------------+     +--------------------+
 |  Metrics Server   |     | Custom Metrics    |     | External Metrics   |
 |  (CPU / Memory)   |     | (Prometheus       |     | (Cloud provider    |
 |                   |     |  Adapter)         |     |  metrics)          |
 +---------+---------+     +---------+---------+     +----------+---------+
           |                         |                          |
           +-------------------------+--------------------------+
                                     |
                                     v
                    +----------------+------------------+
                    |        HPA Controller             |
                    |                                   |
                    |  1. Fetch current metric values   |
                    |  2. Calculate desired replicas    |
                    |  3. Apply stabilization window    |

                     +--------+--+   +------+------+
                     | Scale Up  |   | Scale Down  |

                    |    Update Deployment/ReplicaSet   |
                    |    .spec.replicas = desired       |
                    +----------------------------------+

Scaling Formula

Formula:

desiredReplicas = ceil[ currentReplicas * ( currentMetricValue / desiredMetricValue ) ]

Example: 3 pods, current CPU = 80%, target = 50%

desired = ceil(3 * (80/50)) = ceil(4.8) = 5 pods

Multiple metrics: HPA calculates desired replicas for each metric independently and takes the maximum.

Complete HPA YAML (CPU + Custom Metrics)

hpa-advanced.yaml autoscaling/v2 
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: web-app-hpa
  namespace: production
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: web-app
  minReplicas: 3
  maxReplicas: 50
  metrics:
  # CPU-based scaling
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 60
  # Memory-based scaling
  - type: Resource
    resource:
      name: memory
      target:
        type: Utilization
        averageUtilization: 75
  # Custom metric - requests per second
  - type: Pods
    pods:
      metric:
        name: http_requests_per_second
      target:
        type: AverageValue
        averageValue: "1000"
  # External metric - SQS queue depth
  - type: External
    external:
      metric:
        name: sqs_queue_length
        selector:
          matchLabels:
            queue: "orders"
      target:
        type: Value
        value: "100"
  # Scaling behavior configuration
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 0
      policies:
      - type: Percent
        value: 100
        periodSeconds: 15
      - type: Pods
        value: 4
        periodSeconds: 15
      selectPolicy: Max
    scaleDown:
      stabilizationWindowSeconds: 300
      policies:
      - type: Percent
        value: 10
        periodSeconds: 60
      selectPolicy: Min

2. Vertical Pod Autoscaler (VPA)

VPA Modes

ModeBehaviorUse Case
OffOnly provides recommendations, never applies themObserving right-sizing without disruption
InitialSets resources only at pod creation timeNew pods get right-sized; existing pods untouched
AutoEvicts and recreates pods to apply new resource valuesFull automatic right-sizing (causes restarts)

Important: VPA and HPA should NOT target the same metric (e.g., CPU). They can coexist if HPA uses custom metrics and VPA manages CPU/memory.

VPA YAML Example

vpa.yaml autoscaling.k8s.io/v1 
apiVersion: autoscaling.k8s.io/v1
kind: VerticalPodAutoscaler
metadata:
  name: web-app-vpa
  namespace: production
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: web-app
  updatePolicy:
    updateMode: "Auto"
  resourcePolicy:
    containerPolicies:
    - containerName: web-container
      minAllowed:
        cpu: 100m
        memory: 128Mi
      maxAllowed:
        cpu: 4
        memory: 8Gi
      controlledResources: ["cpu", "memory"]
      controlledValues: RequestsAndLimits
    - containerName: sidecar
      mode: "Off"  # Don't touch sidecar resources

When to Use VPA vs HPA

Use HPA When:

  • Workload is stateless and horizontally scalable
  • Traffic patterns vary predictably
  • You can add more replicas to handle load
  • Application handles load balancing well

Use VPA When:

  • Workload cannot scale horizontally (e.g., single-instance DB)
  • Resource requests are unknown or change over time
  • Pods are over/under-provisioned
  • You want to reduce waste without manual tuning

3. Cluster Autoscaler

The Cluster Autoscaler adjusts the number of nodes in a cluster when pods fail to schedule due to insufficient resources or when nodes are underutilized.

How It Works

1
Scale Up - Triggered when pods are in Pending state due to insufficient resources. CA simulates scheduling against node groups and adds a node from the best-fit group.
2
Scale Down - Triggered when a node's utilization is below threshold (default 50%) for 10 minutes. CA checks if all pods can be moved to other nodes before removing.
3
Safe to Remove Checks - Won't remove nodes with: pods with PDBs that would be violated, pods not managed by a controller, pods with local storage (unless configured), system pods.

Integration with Cloud ASGs

4. KEDA (Kubernetes Event-Driven Autoscaling)

KEDA extends HPA to support event-driven autoscaling from external sources (message queues, databases, custom endpoints). It can scale to/from zero.

Key Features

  • Scale to zero (and back from zero)
  • 60+ built-in scalers (Kafka, RabbitMQ, AWS SQS, Prometheus, Cron, etc.)
  • Works alongside native HPA
  • Supports ScaledObject and ScaledJob

Architecture

  • Agent - Activates/deactivates deployments (0 to 1 and 1 to 0)
  • Metrics Server - Exposes external metrics to HPA
  • Admission Webhooks - Validates ScaledObjects
keda-scaledobject.yaml KEDA 
apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: order-processor
  namespace: production
spec:
  scaleTargetRef:
    name: order-processor
  pollingInterval: 15
  cooldownPeriod: 300
  minReplicaCount: 0          # Scale to zero!
  maxReplicaCount: 100
  triggers:
  - type: aws-sqs-queue
    metadata:
      queueURL: https://sqs.us-east-1.amazonaws.com/123456789/orders
      queueLength: "5"        # Target messages per pod
      awsRegion: us-east-1
    authenticationRef:
      name: aws-credentials
  - type: prometheus
    metadata:
      serverAddress: http://prometheus:9090
      metricName: http_requests_total
      threshold: "100"
      query: sum(rate(http_requests_total{service="orders"}[2m]))

5. Node Affinity

Node affinity constrains which nodes a pod can be scheduled on based on node labels. It is more expressive than nodeSelector, supporting soft preferences and set-based operators.

Types Comparison

TypeBehaviorIf Not Satisfied
requiredDuringSchedulingIgnoredDuringExecution Hard requirement - pod MUST be placed on matching node Pod stays Pending
preferredDuringSchedulingIgnoredDuringExecution Soft preference - scheduler tries but doesn't guarantee Pod scheduled elsewhere

IgnoredDuringExecution means if a node's labels change after the pod is scheduled, the pod continues to run. A future RequiredDuringExecution will evict pods when labels no longer match.

YAML Examples

node-affinity-required.yaml Hard requirement 
apiVersion: v1
kind: Pod
metadata:
  name: gpu-workload
spec:
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
        - matchExpressions:
          - key: node.kubernetes.io/instance-type
            operator: In
            values:
            - p3.2xlarge
            - p3.8xlarge
          - key: topology.kubernetes.io/zone
            operator: In
            values:
            - us-east-1a
            - us-east-1b
  containers:
  - name: ml-training
    image: training:v1
    resources:
      limits:
        nvidia.com/gpu: 1
node-affinity-preferred.yaml Soft preference 
apiVersion: v1
kind: Pod
metadata:
  name: web-app
spec:
  affinity:
    nodeAffinity:
      preferredDuringSchedulingIgnoredDuringExecution:
      - weight: 80
        preference:
          matchExpressions:
          - key: node-type
            operator: In
            values:
            - compute-optimized
      - weight: 20
        preference:
          matchExpressions:
          - key: topology.kubernetes.io/zone
            operator: In
            values:
            - us-east-1a
  containers:
  - name: web
    image: nginx:1.25

6. Taints & Tolerations

Taints are applied to nodes to repel pods. Tolerations are applied to pods to allow scheduling on tainted nodes. Think of taints as "node rejects pods" and tolerations as "pod tolerates the rejection."

Taint Effects

EffectBehaviorAlready Running Pods
NoSchedule New pods without toleration won't be scheduled Not evicted
PreferNoSchedule Scheduler tries to avoid, but not guaranteed Not evicted
NoExecute New pods rejected AND existing pods evicted Evicted (unless tolerated with tolerationSeconds)

Applying Taints to Nodes

kubectl taint nodes node1 dedicated=gpu:NoSchedule
kubectl taint nodes node1 maintenance=true:NoExecute
kubectl taint nodes node1 dedicated=gpu:NoSchedule- # Remove taint

Pod Toleration YAML

tolerations.yaml Scheduling 
apiVersion: v1
kind: Pod
metadata:
  name: gpu-pod
spec:
  tolerations:
  # Exact match toleration
  - key: "dedicated"
    operator: "Equal"
    value: "gpu"
    effect: "NoSchedule"
  # Exists operator (matches any value for the key)
  - key: "node-role.kubernetes.io/control-plane"
    operator: "Exists"
    effect: "NoSchedule"
  # Tolerate NoExecute with eviction timeout
  - key: "maintenance"
    operator: "Equal"
    value: "true"
    effect: "NoExecute"
    tolerationSeconds: 3600   # Stay for 1 hour, then evict
  # Tolerate ALL taints (use carefully - for DaemonSets)
  # - operator: "Exists"
  containers:
  - name: gpu-workload
    image: cuda-app:v1
    resources:
      limits:
        nvidia.com/gpu: 1

7. Topology Spread Constraints

Topology Spread Constraints control how pods are distributed across topology domains (zones, nodes, racks) for high availability and balanced resource usage.

topology-spread.yaml HA Spread 
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-app
spec:
  replicas: 6
  selector:
    matchLabels:
      app: web
  template:
    metadata:
      labels:
        app: web
    spec:
      topologySpreadConstraints:
      # Spread across zones
      - maxSkew: 1
        topologyKey: topology.kubernetes.io/zone
        whenUnsatisfiable: DoNotSchedule
        labelSelector:
          matchLabels:
            app: web
      # Spread across nodes within each zone
      - maxSkew: 1
        topologyKey: kubernetes.io/hostname
        whenUnsatisfiable: ScheduleAnyway
        labelSelector:
          matchLabels:
            app: web
      containers:
      - name: web
        image: nginx:1.25
        resources:
          requests:
            cpu: 200m
            memory: 256Mi

Key Fields

  • maxSkew - Maximum difference in pod count between any two topology domains
  • topologyKey - Node label key that defines the topology domain
  • whenUnsatisfiable - DoNotSchedule (hard) or ScheduleAnyway (soft)
  • labelSelector - Which pods to consider for the spread calculation

8. Resource Requests & Limits

QoS Classes

QoS ClassConditionEviction PriorityBehavior
Guaranteed requests == limits (for all containers, both CPU and memory) Last to be evicted Most predictable performance
Burstable At least one container has a request or limit set, but doesn't meet Guaranteed criteria Evicted after BestEffort Can burst above requests up to limits
BestEffort No requests or limits set for any container First to be evicted Uses whatever is available

CPU Throttling vs OOM Kill

CPU (Compressible)

  • When a container exceeds its CPU limit, it is throttled
  • The process continues but runs slower
  • No kill or restart occurs
  • Measured in millicores: 1000m = 1 CPU core
  • CFS (Completely Fair Scheduler) enforces limits
  • CFS period = 100ms; limit of 200m means 20ms per 100ms period

Memory (Incompressible)

  • When a container exceeds its memory limit, it is OOM killed
  • The container is terminated immediately
  • Exit code 137 (128 + SIGKILL=9)
  • kubelet may restart it (based on restartPolicy)
  • Node-level OOM killer may also act under pressure
  • BestEffort pods killed first, then Burstable, then Guaranteed
resource-qos-guaranteed.yaml Guaranteed QoS 
apiVersion: v1
kind: Pod
metadata:
  name: guaranteed-pod
spec:
  containers:
  - name: app
    image: myapp:v1
    resources:
      requests:
        cpu: "500m"
        memory: "512Mi"
      limits:
        cpu: "500m"        # Same as request = Guaranteed
        memory: "512Mi"    # Same as request = Guaranteed
limitrange.yaml Namespace defaults 
apiVersion: v1
kind: LimitRange
metadata:
  name: default-limits
  namespace: production
spec:
  limits:
  - type: Container
    default:          # Default limits if not specified
      cpu: "500m"
      memory: "512Mi"
    defaultRequest:   # Default requests if not specified
      cpu: "100m"
      memory: "128Mi"
    max:              # Maximum allowed
      cpu: "4"
      memory: "8Gi"
    min:              # Minimum allowed
      cpu: "50m"
      memory: "64Mi"

9. Pod Priority & Preemption

Pod Priority allows marking some pods as more important. When the cluster is full, the scheduler can preempt (evict) lower-priority pods to make room for higher-priority ones.

priority-classes.yaml scheduling.k8s.io/v1 
# Critical system workloads
apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: system-critical
value: 1000000
globalDefault: false
preemptionPolicy: PreemptLowerPriority
description: "Critical system services that must always run"
---
# Standard production workloads
apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: production-high
value: 100000
globalDefault: false
preemptionPolicy: PreemptLowerPriority
description: "Production services with high priority"
---
# Default workloads
apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: default-priority
value: 1000
globalDefault: true
preemptionPolicy: PreemptLowerPriority
description: "Default priority for all workloads"
---
# Batch / preemptible jobs
apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: batch-low
value: 100
globalDefault: false
preemptionPolicy: Never    # Won't preempt others, but can be preempted
description: "Low-priority batch jobs that can be preempted"
---
# Using priority in a Pod
apiVersion: v1
kind: Pod
metadata:
  name: critical-service
spec:
  priorityClassName: system-critical
  containers:
  - name: app
    image: critical-app:v1
    resources:
      requests:
        cpu: "1"
        memory: "2Gi"

11. Interview Q&A

Q: What is the difference between HPA and VPA? When would you use each?

HPA scales the number of pod replicas based on metrics (CPU, memory, custom). Use for stateless workloads that handle load by distributing across replicas.

VPA adjusts the CPU/memory requests and limits of individual pods. Use for workloads that cannot scale horizontally (single-instance databases, monoliths) or to right-size resources.

Key rule: Never use both HPA and VPA on the same resource metric (e.g., both targeting CPU). They can coexist if HPA uses custom metrics and VPA handles CPU/memory.

Q: How does the HPA scaling formula work? What happens with multiple metrics?

Formula: desiredReplicas = ceil(currentReplicas * (currentMetric / targetMetric))

Multiple metrics: HPA calculates desired replicas for each metric independently and takes the maximum value. This ensures the workload satisfies all scaling criteria.

Tolerance: A 10% tolerance band prevents flapping. If the ratio is between 0.9 and 1.1, no scaling occurs.

Q: Explain the difference between requiredDuringScheduling and preferredDuringScheduling in node affinity.

required - Hard constraint. The scheduler MUST place the pod on a node matching the criteria. If no matching node exists, the pod remains Pending.

preferred - Soft constraint. The scheduler tries to place the pod on a matching node but will schedule elsewhere if necessary. Weights (1-100) determine preference strength.

IgnoredDuringExecution means if node labels change after scheduling, running pods are not evicted.

Q: What is the difference between taints/tolerations and node affinity?

Node Affinity - Pod-centric: "I want to run on these nodes." Attracts pods to specific nodes.

Taints/Tolerations - Node-centric: "I reject pods unless they tolerate me." Repels pods from nodes.

Together: Use taints to dedicate nodes (e.g., GPU nodes) and node affinity to direct specific workloads to those nodes. Taints prevent non-GPU pods from landing there; affinity ensures GPU pods prefer those nodes.

Q: What happens when a pod exceeds its CPU limit vs. its memory limit?

CPU (compressible): The pod is throttled. CFS limits its CPU time. The process slows down but is NOT killed. This manifests as increased latency.

Memory (incompressible): The pod is OOM killed (exit code 137). The kubelet restarts it according to restartPolicy. If node-level memory pressure occurs, kubelet evicts pods by QoS class: BestEffort first, then Burstable, then Guaranteed.

Q: How does Cluster Autoscaler decide to remove a node?

The CA considers a node for removal when its utilization drops below 50% for 10 minutes. Before removing, it verifies:

  • All pods can be rescheduled to other nodes
  • No pods with restrictive PDBs would be violated
  • No pods with local storage (unless --skip-nodes-with-local-storage=false)
  • No pods without a controller (bare pods)
  • No system-critical pods (kube-system with specific annotations)

It then cordons the node, drains it gracefully, and removes it from the cloud provider's node group.

Q: What is KEDA and how does it differ from standard HPA?

KEDA extends Kubernetes autoscaling to support event-driven sources. Key differences:

  • Scale to zero: KEDA can scale deployments to 0 replicas (HPA minimum is 1)
  • Event sources: 60+ built-in scalers (Kafka, SQS, RabbitMQ, Prometheus, etc.)
  • Architecture: KEDA manages 0-to-1 and 1-to-0 transitions; for 1-to-N it creates and manages an HPA
  • ScaledJobs: Can also scale Kubernetes Jobs based on queue depth

Q: Explain Pod Priority and Preemption. What is the preemptionPolicy field?

Priority: A numeric value assigned via PriorityClass. Higher value = more important. The scheduler queues pods by priority.

Preemption: When a high-priority pod cannot be scheduled, the scheduler identifies lower-priority pods to evict, ensuring PDBs are respected where possible.

preemptionPolicy:

  • PreemptLowerPriority (default) - Can evict lower-priority pods
  • Never - Will not preempt others; waits for resources. Still gets priority in queue.

Q: What are Topology Spread Constraints and when would you use them over pod anti-affinity?

Topology Spread Constraints ensure even distribution of pods across failure domains (zones, nodes). They control the maxSkew (maximum difference in pod count between domains).

vs Pod Anti-Affinity:

  • Anti-affinity: "Don't put two pods of this type on the same node" (binary)
  • Topology Spread: "Balance pods evenly across zones/nodes" (numeric skew)

Use topology spread when you need fine-grained distribution control (e.g., "no zone should have more than 1 extra pod compared to another").

12. Real-World Scenarios

⚠ Scenario: HPA Not Scaling Up

Symptom: HPA shows TARGETS: <unknown>/60% and replicas stay at minReplicas.

1. Check if metrics-server is running: kubectl get pods -n kube-system | grep metrics-server
2. Verify metrics are available: kubectl top pods (should return data)
3. Check HPA events: kubectl describe hpa web-app-hpa
4. Ensure pods have resource requests set (HPA needs requests for utilization calculation)
5. For custom metrics, verify the custom metrics adapter is working: kubectl get --raw /apis/custom.metrics.k8s.io/v1beta1

Root Cause: Usually missing metrics-server, pods without resource requests, or misconfigured custom metrics adapter.

⚠ Scenario: Pod Pending Due to Insufficient Resources

Symptom: Pod stuck in Pending state. Events show 0/5 nodes are available: 3 Insufficient cpu, 2 Insufficient memory.

1. Check node capacity: kubectl describe nodes | grep -A 5 "Allocated resources"
2. Check if Cluster Autoscaler is active: kubectl get pods -n kube-system | grep cluster-autoscaler
3. View CA logs: kubectl logs -n kube-system deployment/cluster-autoscaler
4. Verify node group hasn't hit max size in cloud provider ASG
5. Check for taints preventing scheduling: kubectl get nodes -o json | jq '.items[].spec.taints'

Resolution: Either reduce resource requests, add node capacity (increase ASG max), or fix scheduling constraints (remove taints, update affinity).

⚠ Scenario: CPU Throttling Causing Latency Spikes

Symptom: Application response times spike periodically. No OOM kills. Pods show high cpu_cfs_throttled_periods_total.

1. Check throttling metrics: kubectl top pods - compare usage vs requests/limits
2. Query Prometheus: rate(container_cpu_cfs_throttled_seconds_total[5m])
3. Review if CPU limits are too restrictive relative to burst needs
4. Consider removing CPU limits (set only requests) - Google's recommendation for non-adversarial clusters
5. Or increase CPU limits to accommodate burst patterns

Resolution: Either increase CPU limits, remove CPU limits entirely (Burstable QoS), or use VPA to right-size the requests/limits based on actual usage.

Debate: Some teams (including Google) recommend setting CPU requests but NOT limits, allowing burst. Others prefer Guaranteed QoS for predictability. Decide based on your cluster's contention levels.
