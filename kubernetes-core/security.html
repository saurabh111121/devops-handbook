Kubernetes Security

RBAC, Service Accounts, Pod Security Standards, SecurityContext, and advanced policy engines -- everything you need to secure a Kubernetes cluster from namespace to node level.

1. RBAC (Role-Based Access Control)

RBAC controls who can do what in a Kubernetes cluster. It uses four objects that work together to grant permissions:

The Four RBAC Objects

Object Scope Purpose
Role Namespace Defines permissions (verbs on resources) within a single namespace
ClusterRole Cluster-wide Defines permissions across the entire cluster or for non-namespaced resources
RoleBinding Namespace Binds a Role or ClusterRole to users/groups/SAs in a namespace
ClusterRoleBinding Cluster-wide Binds a ClusterRole to users/groups/SAs across the entire cluster

RBAC Relationship Diagram

│                        RBAC RELATIONSHIP MODEL                           │
│                                                                         │
│   WHO                      BINDING                    WHAT              │
│   (Subject)                (Glue)                     (Permissions)     │
│                                                                         │
│   ┌──────────┐        ┌──────────────┐          ┌──────────────┐      │
│   │  User    │───────▶│ RoleBinding  │─────────▶│    Role      │      │
│   │  Group   │        │              │          │ (namespace)  │      │
│   │  SA      │        │ (namespace)  │          └──────────────┘      │
│   └──────────┘        └──────────────┘                                 │
│                               │                                         │
│                               │ can also reference                       │

└─────────────────────────────────────────────────────────────────────────┘

Role YAML -- Read Pods in a Namespace

A Role grants permissions within a specific namespace only.

apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  namespace: development
  name: pod-reader
rules:
- apiGroups: [""]          # "" = core API group
  resources: ["pods"]
  verbs: ["get", "watch", "list"]
- apiGroups: [""]
  resources: ["pods/log"]
  verbs: ["get"]

ClusterRole YAML -- Read Nodes Cluster-Wide

A ClusterRole can grant access to cluster-scoped resources (nodes, PVs, namespaces) or be reused across namespaces via RoleBindings.

apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: node-reader
rules:
- apiGroups: [""]
  resources: ["nodes"]
  verbs: ["get", "watch", "list"]
- apiGroups: [""]
  resources: ["nodes/status"]
  verbs: ["get"]

RoleBinding YAML

apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: read-pods-binding
  namespace: development
subjects:
- kind: User
  name: jane@example.com
  apiGroup: rbac.authorization.k8s.io
- kind: ServiceAccount
  name: ci-bot
  namespace: development
roleRef:
  kind: Role
  name: pod-reader
  apiGroup: rbac.authorization.k8s.io

ClusterRoleBinding YAML

apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: read-nodes-global
subjects:
- kind: Group
  name: sre-team
  apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: ClusterRole
  name: node-reader
  apiGroup: rbac.authorization.k8s.io

kubectl auth can-i Commands

# Check if YOU can do something
kubectl auth can-i create deployments --namespace production

# Check if a specific user can do something
kubectl auth can-i get pods --namespace development --as jane@example.com

# Check if a service account can do something
kubectl auth can-i list secrets --namespace default \
  --as system:serviceaccount:default:my-sa

# List ALL permissions for current user
kubectl auth can-i --list

# List permissions in a specific namespace
kubectl auth can-i --list --namespace production

# Check cluster-scoped permissions
kubectl auth can-i get nodes

# Check subresource access
kubectl auth can-i get pods --subresource=log

2. Service Accounts

Service Accounts provide identities for processes running in pods. Every namespace has a default Service Account that is automatically mounted into pods unless configured otherwise.

Key Principles

  • Never use the default SA for applications -- it may accumulate permissions over time
  • Create a dedicated SA per application -- principle of least privilege
  • Set automountServiceAccountToken: false when the pod does not need API access
  • Use short-lived tokens (projected volumes) instead of long-lived Secret-based tokens

Complete Example: Create SA, Bind Role, Use in Pod

# 1. Create a dedicated Service Account
apiVersion: v1
kind: ServiceAccount
metadata:
  name: app-service-account
  namespace: production
  labels:
    app: my-app
automountServiceAccountToken: false   # Don't auto-mount; we'll be explicit
---
# 2. Create a Role with minimal permissions
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: app-role
  namespace: production
rules:
- apiGroups: [""]
  resources: ["configmaps"]
  verbs: ["get", "watch"]
  resourceNames: ["app-config"]       # Restrict to specific resource
- apiGroups: [""]
  resources: ["secrets"]
  verbs: ["get"]
  resourceNames: ["app-credentials"]
---
# 3. Bind the Role to the Service Account
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: app-role-binding
  namespace: production
subjects:
- kind: ServiceAccount
  name: app-service-account
  namespace: production
roleRef:
  kind: Role
  name: app-role
  apiGroup: rbac.authorization.k8s.io
---
# 4. Use the Service Account in a Pod
apiVersion: v1
kind: Pod
metadata:
  name: my-app
  namespace: production
spec:
  serviceAccountName: app-service-account
  automountServiceAccountToken: true    # Explicitly mount for this pod
  containers:
  - name: app
    image: my-app:1.0
    volumeMounts:
    - name: token
      mountPath: /var/run/secrets/tokens
      readOnly: true
  volumes:
  - name: token
    projected:
      sources:
      - serviceAccountToken:
          path: api-token
          expirationSeconds: 3600       # Short-lived (1 hour)
          audience: api

3. Pod Security Standards (PSS)

Pod Security Standards define three progressive levels of security restriction. They replace the deprecated PodSecurityPolicy (PSP) and are enforced via the built-in Pod Security Admission controller.

Security Levels

Level Description Use Case Key Restrictions
Privileged Unrestricted -- no policy applied System/infrastructure workloads (CNI, storage drivers) None -- allows everything
Baseline Minimally restrictive -- prevents known privilege escalations General application workloads No hostNetwork, hostPID, hostIPC; no privileged containers; limited volume types; no adding capabilities beyond defaults
Restricted Heavily restricted -- follows hardening best practices Security-sensitive workloads, multi-tenant clusters Must run as non-root; must drop ALL capabilities; read-only root FS recommended; seccomp profile required; no privilege escalation

Enforcement Modes

Mode Behavior Recommendation
enforce Rejects pods that violate the policy Use for production namespaces
warn Allows pod but shows warning to user Use during migration to stricter policies
audit Allows pod but logs violation in audit log Use for monitoring compliance

Namespace Labels YAML

# Apply Pod Security Standards via namespace labels
apiVersion: v1
kind: Namespace
metadata:
  name: production
  labels:
    # Enforce restricted level -- reject non-compliant pods
    pod-security.kubernetes.io/enforce: restricted
    pod-security.kubernetes.io/enforce-version: latest

    # Warn on restricted violations (shows in kubectl output)
    pod-security.kubernetes.io/warn: restricted
    pod-security.kubernetes.io/warn-version: latest

    # Audit restricted violations (logged to audit log)
    pod-security.kubernetes.io/audit: restricted
    pod-security.kubernetes.io/audit-version: latest
---
# Example: Development namespace with baseline enforcement + restricted warnings
apiVersion: v1
kind: Namespace
metadata:
  name: development
  labels:
    pod-security.kubernetes.io/enforce: baseline
    pod-security.kubernetes.io/enforce-version: latest
    pod-security.kubernetes.io/warn: restricted
    pod-security.kubernetes.io/warn-version: latest

4. SecurityContext

SecurityContext defines privilege and access control settings for a pod or container. Settings can be applied at two levels:

Level Field Scope
Pod-level spec.securityContext Applies to all containers in the pod (runAsUser, fsGroup, supplementalGroups, seccompProfile)
Container-level spec.containers[].securityContext Applies to a specific container; overrides pod-level where overlapping (capabilities, readOnlyRootFilesystem, allowPrivilegeEscalation)

Complete Hardened Pod YAML

This pod meets the Restricted Pod Security Standard and follows all hardening best practices.

apiVersion: v1
kind: Pod
metadata:
  name: hardened-app
  namespace: production
  labels:
    app: hardened-app
spec:
  # Pod-level security context
  securityContext:
    runAsNonRoot: true            # Refuse to start if image runs as root
    runAsUser: 1000               # Run as non-root UID
    runAsGroup: 3000              # Primary GID
    fsGroup: 2000                 # Volume ownership GID
    seccompProfile:
      type: RuntimeDefault        # Apply default seccomp profile

  serviceAccountName: app-sa
  automountServiceAccountToken: false

  containers:
  - name: app
    image: my-app:1.0
    ports:
    - containerPort: 8080
      protocol: TCP

    # Container-level security context
    securityContext:
      allowPrivilegeEscalation: false   # Block setuid/setgid binaries
      readOnlyRootFilesystem: true      # Immutable container filesystem
      capabilities:
        drop:
          - ALL                          # Drop ALL Linux capabilities
        # add: ["NET_BIND_SERVICE"]     # Only add if truly needed

    # Resource limits (also a security measure -- prevent resource abuse)
    resources:
      requests:
        memory: "64Mi"
        cpu: "100m"
      limits:
        memory: "128Mi"
        cpu: "500m"

    # Writable directories via emptyDir (since rootFS is read-only)
    volumeMounts:
    - name: tmp
      mountPath: /tmp
    - name: cache
      mountPath: /app/cache

  volumes:
  - name: tmp
    emptyDir:
      sizeLimit: 100Mi
  - name: cache
    emptyDir:
      sizeLimit: 200Mi

5. Advanced Security

OPA/Gatekeeper Overview

Open Policy Agent (OPA) Gatekeeper is a policy engine for Kubernetes that enforces custom policies via Admission Webhooks. Policies are written in Rego (a declarative language) and evaluated before resources are persisted.

Concept Description
ConstraintTemplate Defines reusable policy logic (Rego code)
Constraint Instantiates a template with specific parameters
Audit Periodically checks existing resources for compliance
Dry Run Evaluates policies without blocking requests

Kyverno Policy YAML -- Require Labels

Kyverno is a Kubernetes-native policy engine. Policies are written as Kubernetes resources (YAML) -- no new language to learn. It can validate, mutate, and generate resources.

apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: require-labels
  annotations:
    policies.kyverno.io/title: Require Labels
    policies.kyverno.io/description: >-
      All pods must have 'app' and 'team' labels for identification
      and cost attribution.
spec:
  validationFailureAction: Enforce   # Block non-compliant resources
  background: true                    # Also scan existing resources
  rules:
  - name: check-required-labels
    match:
      any:
      - resources:
          kinds:
          - Pod
    validate:
      message: "Labels 'app' and 'team' are required on all pods."
      pattern:
        metadata:
          labels:
            app: "?*"      # Must exist and be non-empty
            team: "?*"
  - name: check-deployment-labels
    match:
      any:
      - resources:
          kinds:
          - Deployment
          - StatefulSet
          - DaemonSet
    validate:
      message: "Labels 'app', 'team', and 'env' are required on workloads."
      pattern:
        metadata:
          labels:
            app: "?*"
            team: "?*"
            env: "?*"

Secrets Encryption at Rest

By default, Kubernetes Secrets are stored unencrypted in etcd. Anyone with etcd access can read all secrets. You MUST configure encryption at rest for production clusters.

# /etc/kubernetes/enc/encryption-config.yaml
# This file configures encryption of Secrets in etcd
apiVersion: apiserver.config.k8s.io/v1
kind: EncryptionConfiguration
resources:
  - resources:
    - secrets
    - configmaps    # Optionally encrypt ConfigMaps too
    providers:
    # Providers are tried in order; first one is used for WRITING
    # All listed providers are tried for READING (supports key rotation)

    - aescbc:
        keys:
        - name: key1
          secret: c2VjcmV0LWtleS1oZXJlLTMyLWJ5dGVz    # base64-encoded 32-byte key
    - identity: {}    # Fallback: read unencrypted data (for migration)

# To enable, add to kube-apiserver manifest:
# --encryption-provider-config=/etc/kubernetes/enc/encryption-config.yaml

# Verify encryption is working:
# kubectl create secret generic test-secret --from-literal=key=value
# ETCDCTL_API=3 etcdctl get /registry/secrets/default/test-secret | hexdump -C
# (Should show encrypted data, not plaintext)

6. Interview Q&A

Q1: Explain RBAC with a concrete example. How would you give a developer read-only access to pods in a specific namespace?

A: Create a Role with verbs [get, list, watch] on resources [pods, pods/log] in the target namespace. Then create a RoleBinding that binds this Role to the developer's User or Group. The developer can now view pods and their logs in that namespace only -- they cannot create, modify, or delete pods, and they have no access in other namespaces.

Q2: What is the difference between a Role and a ClusterRole?

A: A Role is namespace-scoped -- it defines permissions within exactly one namespace. A ClusterRole is cluster-scoped -- it can (1) grant access to cluster-scoped resources like nodes and PVs, (2) grant access to non-resource endpoints like /healthz, or (3) be reused across multiple namespaces via individual RoleBindings. A common pattern is to define a ClusterRole once (e.g., "pod-reader") and bind it in multiple namespaces via RoleBindings.

Q3: How would you grant a developer read-only access to all namespaces?

A: Two options: (1) Create a ClusterRole with read verbs and bind it with a ClusterRoleBinding -- this gives access across ALL namespaces including future ones. (2) Create a ClusterRole with read verbs and create individual RoleBindings in specific namespaces -- this gives access only to selected namespaces. Option 2 is more restrictive and follows least privilege.

Q4: What are Service Account best practices?

A: (1) Create a dedicated SA per application -- never use the default SA. (2) Set automountServiceAccountToken: false on both the SA and pod unless the pod actually needs API access. (3) Use projected volume tokens with short expiration (1 hour) instead of long-lived Secret-based tokens. (4) Restrict SA permissions with RBAC to the minimum needed. (5) Use resourceNames in Rules to restrict access to specific ConfigMaps/Secrets when possible.

Q5: Explain the three Pod Security Standard levels.

A: Privileged = unrestricted, for system workloads like CNI plugins. Baseline = prevents known privilege escalations (no hostNetwork, no privileged containers) while remaining compatible with most applications. Restricted = maximum hardening (must run as non-root, drop ALL capabilities, no privilege escalation, seccomp required). Start with Baseline for development, enforce Restricted for production.

Q6: How do you prevent privilege escalation in pods?

A: Multiple layers: (1) allowPrivilegeEscalation: false in securityContext blocks setuid binaries. (2) runAsNonRoot: true refuses to start if the image uses UID 0. (3) capabilities: drop: [ALL] removes all Linux capabilities. (4) readOnlyRootFilesystem: true prevents writing malicious binaries. (5) Enforce the Restricted Pod Security Standard at the namespace level. (6) Use Seccomp profiles to restrict system calls.

Q7: When would you use a Mutating vs Validating Admission Webhook?

A: Mutating webhooks modify the request before persistence -- use them to inject sidecar containers, set default resource limits, or add labels. Validating webhooks accept or reject requests without modification -- use them to enforce policies like "all pods must have resource limits" or "no containers can run as root". Mutating runs first so validators see the final mutated state.

Q8: How do you handle Secrets securely in Kubernetes?

A: (1) Enable encryption at rest in etcd (EncryptionConfiguration). (2) Use external secret stores (AWS Secrets Manager, HashiCorp Vault) with operators like External Secrets. (3) Restrict Secret access via RBAC -- most developers should not be able to read secrets directly. (4) Use stringData in manifests (auto-encodes to base64). (5) Never commit secrets to Git -- use sealed-secrets or external references. (6) Rotate secrets regularly and use short-lived credentials where possible.

7. Scenarios

Scenario 1: User Getting "Forbidden" Errors

Problem: A developer reports Error from server (Forbidden): pods is forbidden: User "dev-jane" cannot list resource "pods" in API group "" in the namespace "production"

# Step 1: Verify what permissions the user has
kubectl auth can-i --list --as dev-jane --namespace production

# Step 2: Check if any RoleBindings exist for this user
kubectl get rolebindings -n production -o yaml | grep -A 5 "dev-jane"

# Step 3: Check ClusterRoleBindings
kubectl get clusterrolebindings -o yaml | grep -A 5 "dev-jane"

# Step 4: Check if user is in a group that has access
kubectl get rolebindings -n production -o yaml | grep -B 5 -A 10 "kind: Group"

# Step 5: Fix -- Create appropriate RoleBinding
kubectl create rolebinding dev-jane-pods \
  --clusterrole=view \
  --user=dev-jane \
  --namespace=production

# Step 6: Verify fix
kubectl auth can-i list pods --as dev-jane --namespace production
# Output: yes

Scenario 2: Pod Running as Root -- Risk Assessment

Problem: Security scan reveals pods running as root (UID 0) in a multi-tenant cluster.

# Step 1: Find all pods running as root
kubectl get pods -A -o jsonpath='{range .items[*]}{.metadata.namespace}/{.metadata.name}: UID={.spec.securityContext.runAsUser}, containers={range .spec.containers[*]}{.name}(UID={.securityContext.runAsUser}) {end}{"\n"}{end}'

# Step 2: Check which pods have no securityContext at all (likely running as root)
kubectl get pods -A -o json | jq -r '
  .items[] |
  select(.spec.securityContext.runAsNonRoot != true) |
  "\(.metadata.namespace)/\(.metadata.name)"'

# Step 3: Enforce at namespace level
kubectl label namespace production \
  pod-security.kubernetes.io/enforce=restricted \
  pod-security.kubernetes.io/warn=restricted

# Step 4: Test with a dry-run
kubectl run test --image=nginx --dry-run=server -n production
# Should be rejected if image defaults to root

Scenario 3: Securing a Multi-Tenant Cluster

Problem: Multiple teams share one cluster. You need namespace-level isolation.

# 1. Create namespace with Pod Security Standards
apiVersion: v1
kind: Namespace
metadata:
  name: team-alpha
  labels:
    pod-security.kubernetes.io/enforce: restricted
    pod-security.kubernetes.io/warn: restricted
    team: alpha
---
# 2. NetworkPolicy -- deny all, then allow only within namespace
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: deny-all
  namespace: team-alpha
spec:
  podSelector: {}
  policyTypes:
  - Ingress
  - Egress
---
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-same-namespace
  namespace: team-alpha
spec:
  podSelector: {}
  policyTypes:
  - Ingress
  - Egress
  ingress:
  - from:
    - podSelector: {}
  egress:
  - to:
    - podSelector: {}
  - to:          # Allow DNS
    - namespaceSelector:
        matchLabels:
          kubernetes.io/metadata.name: kube-system
    ports:
    - protocol: UDP
      port: 53
---
# 3. ResourceQuota -- prevent resource hogging
apiVersion: v1
kind: ResourceQuota
metadata:
  name: team-alpha-quota
  namespace: team-alpha
spec:
  hard:
    requests.cpu: "10"
    requests.memory: 20Gi
    limits.cpu: "20"
    limits.memory: 40Gi
    pods: "50"
    services: "10"
---
# 4. LimitRange -- enforce per-pod limits
apiVersion: v1
kind: LimitRange
metadata:
  name: default-limits
  namespace: team-alpha
spec:
  limits:
  - default:
      cpu: "500m"
      memory: "256Mi"
    defaultRequest:
      cpu: "100m"
      memory: "128Mi"
    type: Container
