Kubernetes Architecture

Deep dive into the components, communication patterns, and design decisions that make Kubernetes work as a distributed container orchestration platform.

Cluster Architecture Overview

A Kubernetes cluster consists of a Control Plane (the brain) and one or more Worker Nodes (the muscle). The Control Plane makes global decisions about the cluster, while Worker Nodes run the actual application workloads.

Control Plane Components

The Control Plane is responsible for maintaining the desired state of the cluster. It makes scheduling decisions, detects and responds to cluster events, and provides the API that all other components interact with.

kube-apiserver

The API server is the front door to the Kubernetes control plane. Every interaction with the cluster -- whether from kubectl, other control plane components, or kubelets -- goes through the API server.

1
Central Communication Hub: All cluster components communicate exclusively through the API server. No component talks directly to another -- this ensures consistency and enables proper authentication/authorization for every operation.
2
RESTful API Interface: Exposes a RESTful HTTP API (typically on port 6443 with TLS) that handles CRUD operations for all Kubernetes objects. Supports JSON and Protocol Buffers serialization for efficient communication.
3
Authentication and Authorization: Implements a pluggable auth chain -- first authenticating the requester (via certificates, tokens, or OIDC), then authorizing the action (via RBAC, ABAC, or webhook), and finally running admission controllers.
4
Admission Controllers: Intercept requests after authentication/authorization but before persistence. They can mutate (MutatingAdmissionWebhook) or validate (ValidatingAdmissionWebhook) objects, enabling policy enforcement like resource quotas, pod security, and default values.
5
Watch Mechanism: Supports long-lived watch connections allowing components to subscribe to changes on specific resources. This event-driven model is what makes the entire controller pattern possible without polling.
6
Horizontal Scalability: Can be scaled horizontally by running multiple instances behind a load balancer. All instances are stateless -- they read/write state from etcd, enabling HA deployments with active-active configuration.
7
API Versioning and Aggregation: Supports multiple API versions (v1, apps/v1, etc.) simultaneously for backward compatibility. The aggregation layer allows extending the API with custom resources (CRDs) and API services without modifying core code.

etcd

etcd is the single source of truth for the entire cluster. If etcd is lost without backups, the cluster state is irrecoverable. Always maintain etcd backups!

1
Distributed Key-Value Store: A strongly consistent, distributed key-value store that holds all cluster state -- every pod, service, secret, configmap, and custom resource is stored here. Uses the Raft consensus algorithm for leader election and log replication.
2
Only Accessed by API Server: In a properly configured cluster, only the kube-apiserver communicates with etcd directly. This isolation simplifies security (only one component needs etcd credentials) and prevents inconsistencies from concurrent writes.
3
Raft Consensus: Requires a quorum (majority) of nodes to agree on writes. For a 3-node etcd cluster, it tolerates 1 failure; for 5 nodes, it tolerates 2. This is why etcd clusters always run an odd number of members.
4
Watch Support: Provides efficient watch notifications that the API server leverages to inform controllers of state changes. This event-driven architecture is fundamental to how Kubernetes achieves its declarative, self-healing behavior.
5
Data Storage Format: Stores data under the /registry prefix with paths like /registry/pods/namespace/name. Data is stored as Protocol Buffers by default (configurable to JSON). Encryption at rest can be enabled for sensitive data like Secrets.
6
Performance Considerations: Latency-sensitive -- requires fast disk I/O (SSD recommended). The recommended maximum database size is 8GB. Regular compaction and defragmentation are essential maintenance tasks to reclaim storage space.
7
Backup and Disaster Recovery: Supports snapshot-based backups via etcdctl snapshot save. Point-in-time recovery is possible by restoring from a snapshot. In production, automated periodic backups should be stored off-cluster.

kube-scheduler

1
Pod Placement Engine: Watches for newly created pods that have no node assigned (spec.nodeName is empty), then selects the optimal node for each pod. It does NOT actually place the pod -- it just sets the node assignment; the kubelet does the rest.
2
Two-Phase Scheduling: Uses a filter-then-score approach. First, it filters out nodes that cannot run the pod (insufficient resources, taints, affinity rules). Then, it scores remaining nodes using priority functions to find the best fit.
3
Filtering (Predicates): Checks hard constraints -- does the node have enough CPU/memory? Does it match required node selectors? Does it tolerate pod tolerations? Is there a matching PersistentVolume in the same zone? Any single filter failure eliminates the node.
4
Scoring (Priorities): Applies soft preferences to rank feasible nodes. Includes factors like spreading pods across failure domains, preferring nodes with images already cached, balancing resource utilization, and respecting pod topology spread constraints.
5
Scheduling Framework: Since Kubernetes 1.19, the scheduler uses a plugin-based framework with extension points (PreFilter, Filter, PostFilter, PreScore, Score, Reserve, Permit, PreBind, Bind, PostBind) enabling custom scheduling logic without forking the scheduler.
6
Preemption: If no node can accommodate a high-priority pod, the scheduler can preempt (evict) lower-priority pods to make room. Priority is determined by PriorityClass objects assigned to pods.
7
Leader Election: Only one scheduler instance is active at a time (leader). Additional instances run in standby mode. Leader election uses a Lease object in the kube-system namespace with a configurable renewal interval.

kube-controller-manager

1
Collection of Controllers: A single binary that runs multiple control loops (controllers). Each controller watches for specific resource types and works to reconcile actual state with desired state. Examples: ReplicaSet controller, Deployment controller, Node controller, Job controller.
2
Reconciliation Loop Pattern: Each controller follows the same pattern: observe current state (via API server watches), compare with desired state (from resource specs), take action to converge. This "level-triggered" approach means controllers self-heal even if events are missed.
3
Node Controller: Monitors node health via heartbeats (NodeStatus and Lease objects). If a node stops reporting, it marks the node as NotReady after 40s (default), then begins evicting pods after 5 minutes (pod-eviction-timeout).
4
ReplicaSet Controller: Ensures the correct number of pod replicas are running at all times. If pods crash or nodes fail, it creates replacements. If there are too many pods (e.g., after scale-down), it deletes extras.
5
Endpoints/EndpointSlice Controller: Watches Services and Pods to maintain the mapping between service selectors and pod IPs. Updates EndpointSlice objects that kube-proxy uses to configure network rules.
6
Service Account and Token Controllers: Automatically creates default ServiceAccount in new namespaces and provisions API access tokens for service accounts, enabling pod-to-API-server authentication.
7
Leader Election: Like the scheduler, only one instance is active. Uses a Lease-based leader election mechanism. Standby instances continuously attempt to acquire the lease, providing automatic failover if the leader crashes.

Worker Node Components

Worker nodes are the machines where application containers actually run. Each node runs a set of services that communicate with the control plane and manage the container lifecycle.

kubelet

1
Node Agent: The primary agent running on each node. It registers the node with the API server, reports node capacity/status, and ensures containers described in PodSpecs are running and healthy. It does NOT manage containers not created by Kubernetes.
2
Pod Lifecycle Management: Receives PodSpecs (primarily from the API server via watch) and drives the container runtime to create, start, stop, and delete containers. Handles init containers sequentially before starting main containers in parallel.
3
Health Checking: Executes liveness probes (restart container if failed), readiness probes (remove from service endpoints if failed), and startup probes (delay other probes until app is ready). Supports HTTP, TCP, gRPC, and exec probe types.
4
Resource Management: Enforces resource requests and limits. Works with the Linux cgroup subsystem to allocate CPU shares, set memory limits, and manage device plugins (GPUs, etc.). Reports allocatable resources to the scheduler.
5
Volume Management: Mounts volumes specified in pod specs. Coordinates with CSI drivers for cloud storage, handles volume attach/detach, and manages local storage like emptyDir and hostPath. Performs volume setup before containers start.
6
Node Status Reporting: Periodically reports node conditions (Ready, MemoryPressure, DiskPressure, PIDPressure), addresses, capacity, and allocatable resources to the API server via NodeStatus updates and Lease renewals (every 10s by default).
7
Static Pods: Can manage pods defined in local manifest files (typically /etc/kubernetes/manifests/) without API server involvement. This is how control plane components themselves are bootstrapped in kubeadm-based clusters -- a chicken-and-egg solution.

kube-proxy

1
Service Networking: Implements the Kubernetes Service abstraction by maintaining network rules on each node. These rules allow network communication to pods from inside or outside the cluster via virtual Service IPs (ClusterIPs).
2
iptables Mode (Default): Creates iptables rules that capture traffic destined for Service ClusterIPs and redirect to backend pod IPs. Random load balancing is applied across endpoints. Scales well to thousands of services but rule updates become expensive.
3
IPVS Mode: Uses Linux IPVS (IP Virtual Server) kernel module for load balancing. Supports multiple algorithms (round-robin, least connections, shortest expected delay). Better performance than iptables at scale (O(1) vs O(n) for connection routing).
4
nftables Mode (Newer): Available from Kubernetes 1.29+, uses nftables as the backend instead of iptables. Offers better performance and maintainability as the Linux networking ecosystem moves away from iptables.
5
EndpointSlice Watches: Watches the API server for changes to Service and EndpointSlice objects. When endpoints change (pod added/removed, readiness gate changes), kube-proxy updates the local networking rules accordingly on each node.
6
NodePort and LoadBalancer Support: Handles NodePort services by opening ports on all nodes and forwarding traffic to pods. For LoadBalancer services, it cooperates with cloud controllers to direct external traffic correctly.

Container Runtime

1
CRI (Container Runtime Interface): Kubernetes defines a gRPC-based interface (CRI) that any container runtime must implement. This decouples Kubernetes from specific runtimes. The kubelet communicates with the runtime over a Unix socket.
2
containerd: The most widely used CRI runtime. Originally extracted from Docker, it manages the complete container lifecycle (image pull, container create, start, stop, delete). Uses runc as the default OCI runtime for actually spawning containers.
3
CRI-O: A lightweight alternative built specifically for Kubernetes. Implements only what CRI requires -- nothing more. Popular in OpenShift/Red Hat ecosystems. Also uses runc under the hood.
4
Image Management: Handles pulling container images from registries, caching layers locally, and providing images to containers. Supports OCI image format and Docker image format for backward compatibility.
5
Sandbox (Pause Container): Each pod gets a "pause" container that holds the network namespace. Application containers join this namespace, which is why containers in the same pod share localhost. If the pause container dies, the entire pod is recreated.

Step-by-Step Breakdown

1
Client-Side Validation and Request: kubectl reads the YAML file, validates it against the OpenAPI schema (client-side), applies any strategic merge patches for existing resources, and sends an HTTP request (POST for create, PATCH for apply) to the API server at the appropriate endpoint (e.g., /apis/apps/v1/namespaces/default/deployments).
2
Authentication, Authorization, Admission: The API server authenticates the request (client certificate from kubeconfig), checks RBAC policies to verify the user can create/update Deployments in that namespace, then runs the admission controller chain (resource quotas, pod security admission, mutating/validating webhooks).
3
Persist to etcd: The validated and potentially mutated Deployment object is serialized (Protocol Buffers) and stored in etcd under /registry/deployments/namespace/name. etcd returns confirmation and the API server responds 201 Created (or 200 OK for updates) to kubectl.
4
Watch Notification: The API server sends a WATCH event (ADDED or MODIFIED) to all controllers watching Deployment resources. The Deployment controller in kube-controller-manager receives this notification through its informer cache.
5
Deployment Controller Creates ReplicaSet: The Deployment controller computes the desired ReplicaSet spec (applying its rolling update strategy), generates the RS name with a hash suffix (e.g., nginx-6799fc88d8), and sends a CREATE request to the API server for the new ReplicaSet.
6
ReplicaSet Stored: The new ReplicaSet object goes through the same API server pipeline (auth, admission) and is persisted to etcd. The API server sends WATCH events to ReplicaSet watchers.
7
ReplicaSet Controller Creates Pods: The ReplicaSet controller notices the RS has 0 ready replicas vs. desired count. It creates Pod objects (with ownerReferences pointing to the RS) by sending CREATE requests to the API server. Crucially, these pods have spec.nodeName empty.
8
Pods Stored (Unscheduled): Pod objects are persisted to etcd in a Pending state with no node assigned. The API server generates WATCH events, including to the scheduler which specifically watches for pods with empty spec.nodeName.
9
Scheduler Filtering and Scoring: The scheduler evaluates each unscheduled pod against available nodes. It filters out infeasible nodes (insufficient resources, taints not tolerated, affinity rules violated), then scores remaining nodes to find the best placement.
10
Scheduler Binds Pod to Node: The scheduler creates a Binding object (or PATCHes the pod's spec.nodeName) via the API server, assigning the pod to the chosen node. This is persisted to etcd and triggers another round of WATCH events.
11
Kubelet Picks Up the Pod: The kubelet on the target node, which watches for pods assigned to its node, receives the notification. It begins the pod admission process (checking resource availability, applying cgroups, preparing volumes).
12
Container Creation: The kubelet calls the container runtime via CRI: creates the pod sandbox (pause container + network namespace), calls CNI plugin to configure networking, pulls container images (if not cached), creates and starts containers. Once running, it reports pod status back to the API server.

High Availability (HA) Architecture

Production Kubernetes clusters require HA to survive component and node failures. The HA design ensures no single point of failure exists in the control plane.

Kubernetes HA Control Plane Architecture

Key HA Design Decisions:
────────────────────────
• API Server: Active-Active (all instances serve requests simultaneously)
• Scheduler: Active-Passive (leader election via Lease objects)
• Controller Manager: Active-Passive (leader election via Lease objects)
• etcd: Raft consensus (one leader, followers replicate)
• Load Balancer: Distributes API traffic; health-checks /healthz endpoint
• Minimum 3 control plane nodes to tolerate 1 failure (etcd quorum = 2/3)
• Leaders for scheduler/controller-manager can be on different nodes

Stacked vs External etcd Topology

Stacked (shown above): etcd runs on the same nodes as other control plane components. Simpler to set up, fewer machines needed, but a node failure loses both a control plane member and an etcd member simultaneously.

External etcd: etcd runs on a dedicated set of machines. More resilient (control plane and etcd failures are independent), but requires more infrastructure (6+ machines for HA instead of 3).

Interview Questions and Answers

Q1: What happens to running pods if the kube-apiserver goes down?
Running pods continue to run. The kubelet and container runtime on each node keep existing containers alive. However, no new pods can be scheduled, no scaling can occur, kubectl commands will fail, and controllers cannot reconcile state. The kubelet will continue to restart crashed containers based on the last-known PodSpec. If a node also fails during this window, its pods are lost since the controller-manager cannot reschedule them.
Q2: Why does etcd need an odd number of members?
Because of quorum math. Raft requires a strict majority (n/2 + 1) to commit writes. For 3 members, quorum = 2 (tolerates 1 failure). For 4 members, quorum = 3 (still only tolerates 1 failure). So 4 members gives no fault tolerance benefit over 3 but adds network complexity. The pattern is: 3 nodes = tolerate 1, 5 nodes = tolerate 2, 7 nodes = tolerate 3.
Q3: What is the difference between a controller and an operator?
An operator IS a controller, but with domain-specific knowledge. A controller is any control loop that watches resource state via the API server and reconciles toward desired state. An operator is a controller that encodes human operational knowledge for managing complex applications (databases, message queues) -- it knows how to deploy, scale, backup, upgrade, and heal the application. Operators typically use Custom Resource Definitions (CRDs) to extend the Kubernetes API.
Q4: How does the scheduler decide which node gets a pod?
Two-phase: Filter then Score. First, filtering eliminates nodes that cannot possibly run the pod (not enough CPU/memory, wrong node selector, unmatched taints, volume zone mismatch). Then, scoring ranks the remaining nodes using weighted priority functions: resource balance (LeastRequestedPriority), spreading across zones (InterPodAntiAffinity), image locality, and custom priorities. The highest-scoring node wins. If tied, random selection breaks the tie.
Q5: What is the role of the pause container in a pod?
It holds the pod's network namespace. The pause container (gcr.io/google_containers/pause) is the first container created in every pod. It acquires the network namespace (IP address, port space) and then simply sleeps forever. All other containers in the pod join this namespace via Linux namespace sharing. This ensures the network identity persists even if application containers crash and restart -- the pod IP stays stable.
Q6: How does Kubernetes achieve self-healing?
Through the declarative model and reconciliation loops. You declare desired state (3 replicas of nginx). Controllers continuously compare actual state vs desired state. If a pod crashes, the kubelet restarts it (restart policy). If a node dies, the node controller marks it NotReady, and the ReplicaSet controller creates replacement pods. Liveness probes detect stuck processes and trigger restarts. The level-triggered design means even if events are missed, the next reconciliation cycle will fix drift.
Q7: What is the difference between kube-proxy's iptables and IPVS modes?
Performance characteristics and load balancing algorithms differ significantly. iptables mode uses sequential rule matching (O(n) per connection for n services), supports only random probability-based load balancing, and becomes slow with 10,000+ services. IPVS uses hash tables (O(1) per connection), supports 8 load balancing algorithms (round-robin, least-connections, etc.), and handles large clusters efficiently. However, iptables is simpler to debug with standard Linux tools.
Q8: How do control plane components achieve high availability?
Different strategies per component. API servers run active-active behind a load balancer since they are stateless. The scheduler and controller-manager use leader election (via Lease objects in kube-system) -- only one instance actively reconciles at a time, with standbys ready to take over (typically within 15s lease duration). etcd uses Raft consensus for strong consistency with automatic leader election. All components are typically run as static pods managed by kubelet.
Q9: What are static pods and why are they important?
Static pods solve the bootstrap chicken-and-egg problem. They are pods managed directly by the kubelet daemon on a specific node, without the API server observing them. The kubelet watches a local directory (default: /etc/kubernetes/manifests/) and creates pods for any manifest files found there. This is how kubeadm bootstraps control plane components -- the kubelet starts apiserver, etcd, scheduler, and controller-manager as static pods before the API server even exists. Mirror pods are created in the API server for visibility.
Q10: If you have a 3-node etcd cluster and lose 2 nodes, what happens?
The cluster becomes read-only and effectively unavailable for writes. With only 1 of 3 members alive, quorum (2) cannot be reached. etcd will continue to serve stale reads (if configured with serializable reads) but will reject all write operations. The Kubernetes API server will return errors for any state-changing operations (creating pods, updating deployments). Recovery requires either restoring at least one more member or performing a disaster recovery procedure: restoring from a snapshot to a new single-node cluster using etcdctl snapshot restore --force-new-cluster.

Troubleshooting Scenarios

⚠ Scenario: Pods Stuck in Pending State

Symptom: Pods remain in Pending state indefinitely and never get scheduled to a node.

Root Causes and Diagnostics:

1. Check scheduler events: kubectl describe pod <pod> -- look at the Events section for FailedScheduling messages with reasons like "Insufficient cpu" or "no nodes match pod topology spread constraints".
2. Verify node resources: kubectl describe nodes | grep -A5 "Allocated resources" -- compare allocated vs allocatable. If nodes are full, scale the cluster or adjust resource requests.
3. Check taints and tolerations: kubectl get nodes -o json | jq '.items[].spec.taints' -- ensure your pods tolerate any node taints (especially the master/control-plane taint).
4. Verify node selectors/affinity: Ensure the pod's nodeSelector or nodeAffinity matches labels on at least one node. Check with kubectl get nodes --show-labels.
5. Check PVC binding: If the pod uses a PersistentVolumeClaim, ensure the PVC is Bound. Unbound PVCs will block scheduling. Check with kubectl get pvc.
⚠ Scenario: Node NotReady

Symptom: One or more nodes show NotReady status in kubectl get nodes.

Root Causes and Diagnostics:

1. Check kubelet status on the node: systemctl status kubelet and journalctl -u kubelet -f -- look for certificate expiry, API server connection failures, or OOM kills.
2. Check node conditions: kubectl describe node <node> -- look for MemoryPressure, DiskPressure, or PIDPressure conditions set to True, which indicate resource exhaustion.
3. Verify container runtime: crictl info or systemctl status containerd -- a crashed runtime will prevent the kubelet from managing containers and reporting healthy status.
4. Network connectivity: Ensure the node can reach the API server on port 6443. Check curl -k https://<api-server>:6443/healthz from the node. DNS resolution issues or firewall rules can break the connection.
5. Certificate validity: Check if the kubelet client certificate has expired: openssl x509 -in /var/lib/kubelet/pki/kubelet-client-current.pem -noout -dates. Expired certs prevent API server communication.
⚠ Scenario: etcd Cluster Degraded

Symptom: API server responds slowly or returns "etcdserver: request timed out" errors. Cluster operations become unreliable.

Root Causes and Diagnostics:

1. Check etcd member health: etcdctl endpoint health --cluster -- identifies which members are unhealthy and whether quorum is at risk.
2. Check disk I/O latency: etcdctl check perf -- etcd requires low-latency disk (p99 fsync < 10ms). Slow disks cause leader election churn and request timeouts. Move etcd to SSD/NVMe storage.
3. Monitor database size: etcdctl endpoint status --write-out=table -- if DB SIZE approaches the quota (default 2GB, max recommended 8GB), writes will fail. Run compaction and defragmentation.
4. Check network latency between members: High inter-node latency (>50ms) causes Raft heartbeat timeouts and frequent leader elections. Ensure etcd members are in the same failure domain or increase heartbeat-interval.
5. Review resource contention: etcd competes with other control plane components for CPU/memory. Monitor with etcdctl endpoint metrics -- look at leader changes, proposal failures, and snapshot duration.
⚠ Scenario: kubectl Commands Timeout or Return 403

Symptom: kubectl commands hang, timeout, or return "forbidden" errors even for admin users.

Root Causes and Diagnostics:

1. Verify connectivity: kubectl cluster-info -- if this times out, check network path to the API server. Verify kubeconfig has correct server URL and the load balancer (if any) is healthy.
2. Check certificate validity: kubectl config view --raw -o jsonpath='{.users[0].user.client-certificate-data}' | base64 -d | openssl x509 -noout -dates -- expired client certs cause TLS handshake failures.
3. Verify RBAC bindings: kubectl auth can-i --list -- if getting 403, check that ClusterRoleBindings/RoleBindings map the correct subject (user/group/SA) to the required role.
4. Check API server health: If you have node access, run curl -k https://localhost:6443/healthz directly on a control plane node. Check API server logs for webhook timeout issues -- a misconfigured admission webhook can block all requests.
5. Admission webhook issues: kubectl get validatingwebhookconfigurations,mutatingwebhookconfigurations -- a webhook with failurePolicy: Fail that is unreachable will reject all matching requests. Consider setting failurePolicy: Ignore or adding namespace exclusions.
⚠ Scenario: Pods CrashLoopBackOff After Node Restart

Symptom: After a node reboot, previously running pods enter CrashLoopBackOff. Container logs show connection errors or file permission issues.

Root Causes and Diagnostics:

1. Check container logs: kubectl logs <pod> --previous -- the --previous flag shows logs from the last crashed instance. Look for connection refused (dependency not ready) or permission denied errors.
2. Verify volume mounts: kubectl describe pod <pod> -- check for "Unable to attach/mount volumes" events. After a node restart, cloud volumes may need time to reattach. Local storage (emptyDir) is lost on node restart.
3. Check dependency ordering: If pods depend on other services (database, configmap from init), use init containers or readiness probes to handle startup ordering. Without these, fast-starting pods may crash connecting to slow-starting dependencies.
4. Examine resource limits: kubectl top pods and check for OOMKilled in kubectl describe pod -- after restart, all pods starting simultaneously may exceed node memory capacity, causing OOM kills and cascading restarts.
5. Review backoff timing: CrashLoopBackOff exponentially increases restart delay (10s, 20s, 40s, ... up to 5 min). Once the root cause is fixed, delete the pod to reset the backoff counter rather than waiting for the next restart attempt.
